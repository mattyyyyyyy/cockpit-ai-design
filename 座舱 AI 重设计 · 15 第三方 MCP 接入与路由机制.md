**§1 背景与目标**

**1.1 为什么做第三方 MCP 路由**

**核心问题**：同一种需求(点咖啡)有多个 MCP 服务商(瑞幸/星巴克/美团),谁来接?路由做不好,要么每次都问用户(烦),要么永远调同一个(用户没选择权)。

核心矛盾：

1. **多 MCP 冲突**：同能力多服务商,如何选择?
2. **动态 vs 静态**：用户偏好会变,路由策略不能写死
3. **开放 vs 安全**：越开放 MCP 生态越丰富,但安全风险越高

**目标**：

* **主目标**：MCP 路由准确率 >90%(用户对推荐的服务商满意)
* **安全目标**：所有第三方 MCP 需审核后上架,不可运行时动态加载
* **体验目标**：路由决策延迟 <500ms

**§2 用户价值**

* **不用每次选**：说"点咖啡",系统自动选你最常用的那个,但可改
* **有选择权**：不强制单一服务商,用户可设偏好或临时换
* **安全可信**：上架的 MCP 都经过审核,不会调用恶意服务

**反价值**

* 不做独家绑定——不因商业合作优先推某家
* 不做无审核开放——安全优先于生态丰富度
* 不做不可更改的默认——路由推荐可被用户覆盖

**§3 场景清单**

|  |  |  |
| --- | --- | --- |
| 场景 ID | 场景描述 | 优先级 |
| **S1-同能力多 MCP 路由** | "点咖啡"→ 自动选用户最常用的 | P0 |
| **S2-用户偏好覆盖** | 用户指定"用瑞幸" | P0 |
| **S3-MCP 降级** | 首选 MCP 不可用时自动换备选 | P1 |
| **S4-新 MCP 上架审核** | 第三方提交 MCP 需审核 | P1 |
| **S5-MCP 版本隔离** | 同一 MCP 多版本共存 | P2 |

**边界场景**

|  |  |
| --- | --- |
| 场景 | 处理策略 |
| **所有同类 MCP 不可用** | 提示"该服务暂不可用",不尝试其他类别 |
| **用户无历史偏好** | 首次随机+学习后续偏好 |
| **MCP 返回恶意内容** | 输出层过滤+立即下架该 MCP |

**非场景**

* 不做运行时动态加载 MCP
* 不做跨品类路由("点咖啡"不会路由到打车 MCP)
* 不做 MCP 间数据共享

**1. 核心问题**

**1.1 为什么需要 MCP 路由机制**

**问题场景**：

* 瑞幸 MCP 能点咖啡
* 星巴克 MCP 能点咖啡
* 美团外卖 MCP 也能点咖啡
* 用户说"我要一杯美式"，**走哪个？**

**不做路由的后果**：

* 方案 A：让用户选 → 用户烦（"我只是要咖啡，不想选渠道"）
* 方案 B：写死规则 → 不灵活（新 MCP 接入要改代码）
* 方案 C：随机选 → 体验差（用户偏好被忽略）

**做路由的价值**：

* AI 自己决定走哪个 MCP（根据用户偏好 + 性价比 + 上下文）
* 新 MCP 接入不影响已有流程
* 用户无感知，体验流畅

**1.2 设计目标**

|  |  |
| --- | --- |
| 目标 | 具体要求 |
| **可扩展** | 新 MCP 接入不需要改代码，不影响已有流程 |
| **智能** | 根据用户偏好 + 性价比 + 上下文自动择优 |
| **快** | 路由决策 < 200ms（不能让用户等） |
| **稳** | 某个 MCP 挂了自动切换备选，不影响用户 |
| **可控** | 人工审核新 MCP，防止低质量服务进来 |

**3. 第一层：意图识别（三级）**

**3.1 为什么要三级**

**问题**：大模型理解意图要 1-3 秒，车载场景等不了。

**解法**：分三级，80% 的常见意图走 L1（<50ms），只有长尾复杂意图才走 L3。

**3.2 三级定义**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 级别 | 方式 | 速度 | 覆盖场景 | 示例 |
| L1 | 关键词 + 规则匹配 | < 50ms | 80% 常见意图 | "咖啡" → 品类：咖啡 "外卖" → 品类：外卖 "打车" → 品类：出行 |
| L2 | 端侧轻量 NLU 模型 | < 200ms | 15% 模糊意图 | "我饿了" → 品类：外卖 "来点喝的" → 品类：咖啡/饮品 "找个地方吃饭" → 品类：本地生活 |
| L3 | 云端大模型理解 | 1-3s | 5% 复杂/长尾意图 | "帮我找个安静的地方喝杯东西" → 品类：咖啡 + 环境偏好：安静 |

**3.3 降级策略**

* L1 搞不定 → 升级到 L2
* L2 搞不定 → 升级到 L3
* L3 还搞不定 → 反问用户："你是想点咖啡、外卖还是其他？"

**4. 第二层：静态注册表**

**4.1 MCP 注册时必须声明**

每个 MCP 接入时必须提供：

|  |  |  |
| --- | --- | --- |
| 字段 | 说明 | 示例 |
| **能力描述** | 自然语言，MCP 自己写 | "提供瑞幸咖啡的在线点单服务，支持全国门店，5-10 分钟出杯" |
| **品类归属** | 大类（我们定义）+ 细分标签（MCP 自选） | 大类：咖啡 细分标签：["现磨咖啡", "连锁品牌", "快取"] |
| **API schema** | 入参/出参/调用方式 | order\_coffee(item, size, store\_id) |
| **服务范围** | 地域/时段/限制条件 | 地域：全国（除港澳台） 时段：06:00-23:00 限制：需要用户登录 |

**4.2 品类归属的冲突处理**

**问题**：瑞幸 MCP 标"咖啡"，美团 MCP 标"外卖 + 咖啡"，那"咖啡"这个品类下就有两个 MCP。

**解法**：标签不是路由终点，是候选池入口。

* 用户说"咖啡" → 查静态注册表 → 命中 3 个 MCP（瑞幸/星巴克/美团咖啡品类）
* 进入动态路由层 → 按优先级规则择优

**5. 第三层：动态路由（优先级规则）**

**5.1 五级优先级（从高到低）**

|  |  |  |
| --- | --- | --- |
| 优先级 | 规则 | 示例 |
| **P1** | 用户显式偏好 | 用户说"我要瑞幸" → 直接走瑞幸 MCP，跳过路由 |
| **P2** | 用户历史行为 | 过去 30 天点咖啡 10 次，8 次选瑞幸 → 权重 0.8 |
| **P3** | 性价比 + 评分 | 瑞幸 15 元/4.2 分，星巴克 28 元/4.5 分 → 计算性价比 |
| **P4** | 上下文适配 | 当前位置距离瑞幸 500m，距离星巴克 1.2km → 瑞幸更近 |
| **P5** | 实时可用性 | 瑞幸 MCP 响应正常，星巴克 MCP 超时 → 瑞幸可用 |

**5.2 性价比计算公式（老师拍板）**

**默认公式**（可按品类调整）：

|  |
| --- |
| Plain Text 性价比得分 = (评分 / 5.0) × 权重1 - (价格 / 参考价) × 权重2 + (时效 / 参考时效) × 权重3 |

**参数**：

* 权重1（评分权重）= 0.4
* 权重2（价格权重）= 0.4
* 权重3（时效权重）= 0.2
* 参考价 = 该品类的平均价格（咖啡参考价 20 元）
* 参考时效 = 该品类的平均时效（咖啡参考时效 10 分钟）

**评分下限**：< 4.0 分的 MCP 不进候选池。

**示例**：

* 瑞幸：15 元，4.2 分，5 分钟
* 性价比 = (4.2/5.0) × 0.4 - (15/20) × 0.4 + (5/10) × 0.2 = 0.336 - 0.3 + 0.1 = **0.136**
* 星巴克：28 元，4.5 分，8 分钟
* 性价比 = (4.5/5.0) × 0.4 - (28/20) × 0.4 + (8/10) × 0.2 = 0.36 - 0.56 + 0.16 = **-0.04**
* **推荐瑞幸**（性价比更高）

**5.3 品类差异化**

不同品类的性价比公式权重可以不同：

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 品类 | 评分权重 | 价格权重 | 时效权重 | 理由 |
| 咖啡 | 0.4 | 0.4 | 0.2 | 用户在意"快"（5 分钟 vs 15 分钟） |
| 外卖 | 0.3 | 0.5 | 0.2 | 用户更在意价格 |
| 打车 | 0.3 | 0.3 | 0.4 | 用户最在意"快"（3 分钟接驾 vs 10 分钟） |

**6. 新 MCP 接入流程（人工审核）**

|  |
| --- |
| Plain Text MCP 提供方提交接入申请  ↓ ① 自动校验  - API schema 是否合规  - 能否调通（ping 测试）  - 响应时间是否达标（< 3s）  ↓ ② 人工审核（产品 + 运营）  - 品类归属是否合理？  - 能力描述是否准确？  - 是否与已有 MCP 冲突？  - 冲突时：标记"共存"关系  - 服务质量是否达标？（评分 > 4.0，投诉率 < 5%）  ↓ ③ 灰度上线  - 先给 5% 用户（或内部测试用户）  - 观察 7 天：成功率、用户满意度、是否影响已有流程  - 成功率 < 90% → 下线整改  ↓ ④ 全量上线  - 进入正式候选池  - 初始优先级权重 = 0.1（最低，不抢走老用户）  - 随着用户选择逐步提升权重 |

**7. 版本隔离：新 MCP 不影响老流程**

**7.1 权重机制**

|  |
| --- |
| Plain Text 已有 MCP（瑞幸）：权重 0.8（用户历史选择积累） 新接入 MCP（库迪）：权重 0.1（初始值） |

* 新 MCP 不会抢走老用户的默认推荐
* 只有当用户主动选择新 MCP，或新 MCP 性价比明显更高时，才会被推荐
* 权重随用户行为逐步调整

**7.2 探索机制（防止新 MCP 永远出不了头）**

**问题**：新 MCP 初始权重 0.1，老 MCP 权重 0.8，新的永远排不上去。

**解法**：每 N 次推荐中，有 1 次随机推荐新 MCP（类似推荐系统的 explore/exploit）。

**参数**：

* 探索比例 = 10%（每 10 次推荐中，1 次探索新 MCP）
* 探索对象 = 权重 < 0.3 的 MCP
* 探索后：如果用户接受 → 权重 +0.05；如果用户拒绝 → 权重 -0.02

**8. 降级与兜底**

**8.1 单个 MCP 挂了**

**场景**：用户说"我要咖啡"，瑞幸 MCP 挂了。

**兜底**：

1. 自动切换到备选 MCP（星巴克）
2. 告诉用户："瑞幸暂时不可用，给你推荐星巴克"
3. 记录失败日志，触发告警

**8.2 多个 MCP 都挂了**

**场景**：用户说"我要咖啡"，瑞幸 + 星巴克 + 美团咖啡品类全挂。

**兜底**：

1. 告诉用户："咖啡服务暂时不可用，稍后再试"
2. 记录失败日志，触发高优先级告警
3. 不要硬推其他品类（比如"要不点个外卖？"）

**8.3 路由决策超时**

**场景**：动态路由层计算超过 200ms。

**兜底**：

1. 降级到默认规则：推荐评分最高的 MCP
2. 记录超时日志
3. 不影响用户体验（用户感知不到路由超时）

**9. 监控与优化**

**9.1 关键指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 定义 | 目标值 |
| **路由准确率** | AI 推荐的 MCP 被用户接受的比例 | ≥ 85% |
| **用户干预率** | 用户说"不要 A，给我 B"的比例 | ≤ 15% |
| **路由延迟** | 从意图识别到推荐结果的时间 | < 200ms（P95） |
| **MCP 可用性** | 各 MCP 的响应成功率 | ≥ 99% |
| **探索转化率** | 探索新 MCP 后用户接受的比例 | ≥ 30% |

**9.2 优化机制**

* **每周 review 路由准确率**：< 85% 时调整优先级规则权重
* **每月 review MCP 质量**：评分 < 4.0 或投诉率 > 5% 的 MCP 下线整改
* **A/B 测试**：新的优先级规则先在 10% 用户上测试，效果好再全量

**5. 功能点展开**

**5.1 R-F1 MCP 注册与能力声明**

**用户价值**

让新的第三方服务（咖啡/外卖/打车...）能以标准化方式接入系统，用户无感知地获得更多服务选择。

**场景脚本**

**S1：新 MCP 接入（运营视角）**

|  |
| --- |
| Plain Text 瑞幸提交接入申请  → 填写：能力描述 + 品类归属（咖啡）+ API schema + 服务范围  → 自动校验：API 能调通，响应 < 3s  → 人工审核：品类合理、能力准确、与星巴克 MCP 标记"共存"  → 灰度 7 天 → 全量上线 |

**S2：MCP 更新 API（不影响已有流程）**

|  |
| --- |
| Plain Text 瑞幸 MCP 升级 API v2  → 新版本注册，旧版本保留  → 新版本灰度验证  → 验证通过 → 切换到新版本  → 旧版本保留 30 天作为兜底 |

**成功指标**

* 新 MCP 从提交到上线 ≤ 14 天（含 7 天灰度）
* 注册信息完整率 100%（缺字段不允许提交）
* API 自动校验通过率 ≥ 90%（不通过的退回整改）

**放弃条件**

* 接入量 < 3 个 MCP / 季度 → 说明生态没起来，机制过重
* 人工审核成为瓶颈（积压 > 20 个待审）→ 需要半自动化

**失败路径**

* **F1**：MCP 声明的能力与实际不符（说能点咖啡，实际只能查菜单）→ 灰度期间发现 → 下线整改
* **F2**：品类归属争议（奶茶算"咖啡"还是"饮品"？）→ 人工审核时裁定，建立品类判例库
* **F3**：API schema 变更导致调用失败 → 版本管理兜底，自动回退旧版本

**技术钩子（给沈砚）**

1. 注册表数据结构设计（品类/标签/schema/版本/状态）
2. API 自动校验框架（ping + 功能测试 + 响应时间）
3. 版本管理策略（多版本共存 + 灰度切换 + 回退）
4. 注册表热更新（新 MCP 上线不需要重启系统）

**5.2 R-F2 意图识别三级路由（L1/L2/L3）**

**用户价值**

用户说任何话，系统都能快速（< 200ms）识别出要调用哪个品类的服务，不需要用户指定"我要用瑞幸 MCP"。

**场景脚本**

**S1：L1 命中（80% 场景）**

|  |
| --- |
| Plain Text 用户："来杯美式"  → L1 关键词匹配："美式" → 品类：咖啡  → 耗时 < 50ms  → 进入静态注册表查询 |

**S2：L1 未命中，L2 接管（15% 场景）**

|  |
| --- |
| Plain Text 用户："来点喝的"  → L1 关键词匹配：无精确命中  → L2 端侧 NLU："来点喝的" → 品类：咖啡/饮品  → 耗时 < 200ms  → 进入静态注册表查询 |

**S3：L1+L2 未命中，L3 接管（5% 场景）**

|  |
| --- |
| Plain Text 用户："帮我找个安静的地方坐坐，顺便来杯东西"  → L1：无命中  → L2：不确定（"安静的地方"+"杯东西"混合意图）  → L3 云端大模型：品类 = 咖啡 + 环境偏好 = 安静  → 耗时 1-3s（用户看到"思考中"动画） |

**S4：三级都搞不定**

|  |
| --- |
| Plain Text 用户："那个什么来着"  → L1/L2/L3 均无法确定品类  → 反问用户："你是想点咖啡、外卖还是其他？" |

**成功指标**

* L1 覆盖率 ≥ 80%（常见意图不需要走 L2/L3）
* L1+L2 覆盖率 ≥ 95%
* 整体识别准确率 ≥ 92%
* P95 延迟 < 200ms（L1+L2 场景）

**放弃条件**

* L1 覆盖率 < 60% → 关键词库太小，需要扩充或放弃 L1 直接走 L2
* 整体准确率 < 80% → 意图识别模型需要重训

**失败路径**

* **F1**：L1 误匹配（"冰"匹配到"冰淇淋"但用户要"冰咖啡"）→ 用户纠正后更新规则
* **F2**：L2 模型端侧推理超时（设备性能差）→ 降级到 L3 云端
* **F3**：L3 云端超时（网络差）→ 反问用户

**技术钩子（给沈砚）**

1. L1 关键词库维护机制（自动扩充 + 人工审核）
2. L2 端侧模型选型（轻量 NLU，< 10MB，推理 < 200ms）
3. L3 调用策略（超时时间 / 重试 / 降级）
4. 三级之间的升级判据（L1 置信度 < 0.7 → 升级 L2）
5. 与 03 NLU 的分工边界（03 做通用意图理解，R-F2 做品类路由）

**5.3 R-F3 动态路由决策（五级优先级）**

**用户价值**

当多个 MCP 都能满足需求时，AI 自动选出最适合用户的那个，不甩选择题给用户。

**场景脚本**

**S1：用户有明确偏好**

|  |
| --- |
| Plain Text 用户："我要瑞幸的美式"  → P1 命中：用户显式指定"瑞幸"  → 跳过路由，直接调用瑞幸 MCP |

**S2：用户无偏好，走历史行为**

|  |
| --- |
| Plain Text 用户："来杯美式"  → P1 未命中（没指定品牌）  → P2 命中：过去 30 天点咖啡 10 次，8 次选瑞幸  → 推荐瑞幸 |

**S3：新用户，走性价比**

|  |
| --- |
| Plain Text 用户（新用户）："来杯美式"  → P1 未命中  → P2 未命中（无历史）  → P3 命中：瑞幸 15 元/4.2 分 vs 星巴克 28 元/4.5 分  → 性价比计算 → 推荐瑞幸 |

**S4：用户纠正推荐**

|  |
| --- |
| Plain Text AI："给你推荐瑞幸的美式，15 元" 用户："不要，给我星巴克"  → 立刻切换到星巴克 MCP  → 更新用户偏好（星巴克权重 +0.1） |

**成功指标**

* 路由准确率（用户接受推荐）≥ 85%
* 用户干预率（"不要 A，给我 B"）≤ 15%
* 路由决策延迟 < 50ms（纯计算，不含 MCP 调用）

**放弃条件**

* 路由准确率 < 60% → 优先级规则失效，需要重新设计
* 用户干预率 > 30% → 用户不信任 AI 推荐

**失败路径**

* **F1**：历史行为过时（用户换了口味但系统还推旧的）→ 遗忘机制（30 天衰减）
* **F2**：性价比公式不适用某品类 → 按品类调整权重
* **F3**：多个 MCP 得分相同 → 按接入时间排序（先接入的优先）

**技术钩子（给沈砚）**

1. 优先级规则引擎（可配置、可热更新）
2. 用户行为数据存储（与 04 状态管理对接）
3. 性价比计算服务（实时获取价格/评分/距离/时效）
4. 权重衰减算法（30 天半衰期？指数衰减？）
5. A/B 测试框架（新规则灰度验证）

**5.4 R-F4 性价比计算与品类差异化**

**用户价值**

默认推荐性价比最高 + 评分最高的选项，不同品类（咖啡/外卖/打车）有不同的"性价比"定义。

**场景脚本**

**S1：咖啡品类（时效权重高）**

|  |
| --- |
| Plain Text 候选：  瑞幸：15 元，4.2 分，5 分钟出杯  星巴克：28 元，4.5 分，8 分钟出杯  便利店：10 元，3.8 分（< 4.0，不进候选池）  → 评分下限过滤：便利店淘汰  → 性价比计算（咖啡权重：评分 0.4 / 价格 0.4 / 时效 0.2）  → 瑞幸胜出 |

**S2：打车品类（时效权重最高）**

|  |
| --- |
| Plain Text 候选：  滴滴：预估 25 元，4.3 分，3 分钟接驾  高德：预估 22 元，4.1 分，8 分钟接驾  → 性价比计算（打车权重：评分 0.3 / 价格 0.3 / 时效 0.4）  → 滴滴胜出（时效优势大） |

**成功指标**

* 性价比推荐满意度 ≥ 80%（用户接受推荐且不纠正）
* 品类差异化覆盖率 = 100%（每个品类都有独立权重配置）

**放弃条件**

* 满意度 < 60% → 公式设计有问题，需要重新调研用户偏好
* 品类间权重差异无统计显著性 → 简化为统一公式

**失败路径**

* **F1**：价格数据不实时（MCP 返回的是昨天的价格）→ 缓存 TTL ≤ 5 分钟
* **F2**：评分数据来源不一致（美团评分 vs 大众点评评分）→ 归一化处理
* **F3**：时效预估不准（说 5 分钟实际 15 分钟）→ 用历史实际数据修正

**技术钩子（给沈砚）**

1. 实时数据获取（价格/评分/距离/时效的数据源和刷新频率）
2. 评分归一化（不同平台评分体系的统一）
3. 品类权重配置中心（可热更新，不需要重启）
4. 历史数据修正（预估 vs 实际的偏差学习）

**5.5 R-F5 新 MCP 接入审核流程**

**用户价值**

确保接入的 MCP 质量可靠，用户不会被推荐到"下单后没人做""评分造假""服务范围虚标"的垃圾服务。

**场景脚本**

**S1：正常接入流程**

|  |
| --- |
| Plain Text 库迪咖啡提交接入申请  → 自动校验：API 能调通 ✓，响应 1.2s ✓  → 人工审核：品类=咖啡 ✓，能力描述准确 ✓，与瑞幸/星巴克标记"共存" ✓  → 灰度 7 天：成功率 96% ✓，满意度 4.1 ✓  → 全量上线，初始权重 0.1 |

**S2：审核不通过**

|  |
| --- |
| Plain Text 某小众咖啡 MCP 提交接入  → 自动校验：API 响应 5s（> 3s 阈值）✗  → 退回整改："请优化 API 响应时间至 3s 以内" |

**S3：灰度期间发现问题**

|  |
| --- |
| Plain Text 某 MCP 灰度中  → 第 3 天：成功率降到 75%（< 90% 阈值）  → 自动下线 + 通知 MCP 提供方整改 |

**成功指标**

* 审核周期 ≤ 14 天（提交到上线）
* 灰度期间问题发现率 ≥ 95%（问题 MCP 不漏到全量）
* 全量上线后 30 天内投诉率 < 5%

**放弃条件**

* 审核积压 > 20 个 → 需要增加审核人力或半自动化
* 灰度期间问题发现率 < 70% → 灰度机制失效

**失败路径**

* **F1**：人工审核主观性强（A 审核员通过，B 审核员不通过）→ 建立审核标准 checklist
* **F2**：灰度 5% 用户量太小，统计不显著 → 延长灰度时间或扩大比例
* **F3**：MCP 提供方灰度期间表现好，全量后变差 → 持续监控 + 自动降级

**技术钩子（给沈砚）**

1. 自动校验流水线（CI/CD 式的 MCP 接入测试）
2. 灰度分流机制（按用户 ID hash 分流）
3. 灰度指标监控（成功率/延迟/满意度的实时看板）
4. 自动下线触发器（指标低于阈值自动执行）

**5.6 R-F6 版本隔离与探索机制**

**用户价值**

新 MCP 上线不会打扰老用户的习惯，同时新 MCP 有机会被用户发现和尝试。

**场景脚本**

**S1：新 MCP 不影响老用户**

|  |
| --- |
| Plain Text 用户一直点瑞幸（权重 0.8） 库迪新上线（权重 0.1）  → 用户说"来杯美式"  → 路由计算：瑞幸 0.8 >> 库迪 0.1  → 推荐瑞幸（老用户体验不变） |

**S2：探索机制让用户发现新 MCP**

|  |
| --- |
| Plain Text 用户第 10 次点咖啡（前 9 次都是瑞幸）  → 触发探索（10% 概率）  → 推荐库迪："发现一家新的咖啡，库迪美式 9.9 元，要试试吗？"  → 用户接受 → 库迪权重 +0.05  → 用户拒绝 → 库迪权重 -0.02，切回瑞幸 |

**S3：新 MCP 逐步上位**

|  |
| --- |
| Plain Text 库迪连续被用户选择 5 次  → 权重从 0.1 → 0.35  → 下次路由计算：瑞幸 0.8 vs 库迪 0.35  → 仍然推荐瑞幸（但差距在缩小）  → 如果库迪性价比明显更高，可能超过瑞幸 |

**成功指标**

* 老用户体验不变率 ≥ 99%（新 MCP 上线后老用户推荐不变）
* 探索转化率 ≥ 30%（探索推荐被用户接受的比例）
* 新 MCP 30 天内获得 ≥ 100 次曝光

**放弃条件**

* 探索转化率 < 10% → 探索机制打扰用户，需要降低探索频率
* 新 MCP 90 天内权重仍 < 0.2 → 说明用户不需要这个 MCP

**失败路径**

* **F1**：探索推荐让用户烦（"我就要瑞幸，别给我推别的"）→ 用户说"不要推荐新的"时关闭探索
* **F2**：权重只升不降（用户偶尔选了一次就一直推）→ 30 天衰减机制
* **F3**：多个新 MCP 同时上线，探索太频繁 → 每个品类每天最多 1 次探索

**技术钩子（给沈砚）**

1. 权重存储与更新（用户维度 × MCP 维度的权重矩阵）
2. 探索触发算法（固定概率 vs Thompson Sampling）
3. 衰减机制（时间衰减 + 行为衰减）
4. 探索频率控制（品类/用户/时间维度的限流）

**5.7 R-F7 降级与兜底**

**用户价值**

某个 MCP 挂了，用户不会感知到"服务不可用"，系统自动切换到备选，体验不中断。

**场景脚本**

**S1：单个 MCP 挂了，自动切换**

|  |
| --- |
| Plain Text 用户："来杯美式"  → 路由推荐瑞幸  → 调用瑞幸 MCP → 超时 3s  → 自动切换到星巴克 MCP  → AI："瑞幸暂时不可用，给你推荐星巴克的美式，28 元" |

**S2：多个 MCP 都挂了**

|  |
| --- |
| Plain Text 用户："来杯美式"  → 瑞幸 MCP 超时 ✗  → 星巴克 MCP 超时 ✗  → 美团咖啡品类 超时 ✗  → AI："咖啡服务暂时不可用，稍后再试"  → 不硬推其他品类 |

**S3：MCP 恢复后自动切回**

|  |
| --- |
| Plain Text 瑞幸 MCP 恢复  → 健康检查探针检测到恢复  → 下次用户点咖啡 → 重新走路由 → 瑞幸恢复候选 |

**S4：路由决策本身超时**

|  |
| --- |
| Plain Text 动态路由计算超过 200ms  → 降级到默认规则：推荐评分最高的可用 MCP  → 用户无感知 |

**成功指标**

* 降级成功率（切换到备选后完成服务）≥ 95%
* 用户感知中断率 < 1%（用户不知道发生了降级）
* MCP 恢复后自动切回时间 < 5 分钟

**放弃条件**

* 降级成功率 < 70% → 备选 MCP 也不靠谱，需要扩充备选池
* 降级频率 > 10% → 主 MCP 质量太差，需要下线整改

**失败路径**

* **F1**：降级后备选 MCP 也挂了（连环故障）→ 全挂路径：告知用户 + 高优先级告警
* **F2**：降级切换导致价格/时效变化大（瑞幸 15 元 → 星巴克 28 元）→ 必须告知用户价格变化
* **F3**：MCP 假恢复（探针通过但实际调用失败）→ 恢复后前 3 次调用仍走备选验证

**技术钩子（给沈砚）**

1. 熔断器（Circuit Breaker）实现（半开/全开/关闭状态机）
2. 健康检查探针（心跳 vs 主动探测 vs 被动观察）
3. 备选 MCP 表维护（每个品类至少 2 个备选）
4. 降级日志与告警（SRE 监控看板）
5. 恢复验证机制（探针通过 + 前 N 次调用成功才算真恢复）
