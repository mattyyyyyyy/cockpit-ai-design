**§1 背景与目标**

**1.1 为什么做状态管理与记忆**

**核心问题**：座舱 AI 如果没有记忆,每次交互都是"失忆重启"——用户每天说同样的偏好、系统每天做同样的推荐,这种体验比没有 AI 还烦。

三大矛盾：

1. **无状态困境**：传统语音助手无长期记忆,每次对话从零开始,用户被迫重复表达
2. **场景切换断裂**：驾驶→驻车→休憩模式之间状态不连贯,切换后丢失上下文
3. **个性化 vs 隐私**：记忆越深体验越好,但用户对"车记住我一切"的恐惧也越深

**AI 时代的机会**：

* **长期记忆**：偏好/习惯/事件流/节律的多层记忆体系,让系统"认识"用户
* **对话状态机**：多轮对话状态精确追踪,不丢上下文
* **场景模式自动切换**：基于情境帧 + 记忆,自动识别当前场景并切换模式

**目标**：

* **主目标**：用户重复表达率下降 >60%(同偏好不说第二次)
* **记忆目标**：核心偏好记忆准确率 >95%,遗忘/误记率 <5%
* **切换目标**：场景模式自动识别准确率 >85%,误切换率 <5%

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **不用重复说**："我每次都开 22 度"说一次就够了,系统记住
* **场景无缝切换**：开车→驻车,音乐/空调/灯光自动适配,不用手动切换
* **越用越懂**：通勤时间、常去地点、听歌偏好逐步学习,体验持续提升

**对多用户**：

* **识别即切换**：不同驾驶员上车,座椅/后视镜/空调/音乐自动切换到各自偏好
* **记忆隔离**：每个用户的偏好独立存储,互不干扰

**2.2 反价值(不做什么)**

* **不做全时记忆**：用户明确说"忘记这个"时,必须立即删除,不留残余
* **不做越界记忆**：不记录对话原文,只记录结构化偏好(如"空调偏好 22°C",不记"我说太热了")
* **不做强制个性化**：用户可选择"无记忆模式",系统不学习不推荐

**3.2 边界场景**

|  |  |  |
| --- | --- | --- |
| 场景 | 挑战 | 处理策略 |
| **偏好冲突** | 用户对不同场景有不同偏好(通勤听新闻/长途听歌) | 按场景维度存储偏好,不设全局唯一值 |
| **记忆污染** | 偶然表达被当作长期偏好("今天不想听歌"≠以后都不听) | 事件 vs 偏好分层:一次表达不直接升级为长期偏好 |

**3.3 非场景(明确不做)**

* **情感记忆**：不做"记住你上次吵架时说了什么"——情感内容不归记忆系统
* **跨车记忆同步**：不做换车后偏好自动迁移——涉及云端身份体系,超出 MVP 范围
* **商业行为记忆**：不做"记住你上次买了什么"——消费数据不进入座舱记忆

| **偏好加载延迟** | <1s | 用户上车到偏好加载完成 | >3s 提示"正在加载你的设置" |

**4.2 可靠性与降级**

1. **记忆系统故障**：降级为"无记忆模式",所有设置回到出厂默认
2. **用户识别失败**：询问"请问你是谁?"+ 语音/手动选择
3. **对话状态丢失**：当前轮独立处理,不依赖上文,提示"我刚才可能忘了你说什么"

**4.3 安全与合规**

* **记忆可删除**：提供"清除所有记忆"入口,一键恢复出厂
* **记忆不上云**：长期偏好存储在端侧,仅用户主动开启云同步时才上传
* **敏感偏好保护**：医疗/出行规律等敏感偏好额外加密存储
* **遗忘即遗忘**：用户删除的记忆不可恢复,不设回收站

**3. 核心能力要求**

**3.1 对话状态管理**

1. **状态机** — 追踪当前任务、子任务、等待用户确认的槽位
2. **状态持久化** — 下车/上车任务能接续(A-F5)
3. **并发任务管理** — 同时做"导航"和"放音乐",互不冲突
4. **状态超时** — 任务挂起 > 24h 自动清理

**3.2 长期记忆系统**

1. **偏好记忆** — 结构化存储:温度 22 度、座椅位置、音乐风格
2. **事件流记忆** — 非结构化:昨天点了拿铁、上周去了西湖
3. **记忆检索** — 语义召回("上次那家" → 向量搜索历史)
4. **遗忘机制** — 过期/用户删除/主动遗忘(7 天前的 POI 不再推荐)
5. **修正机制** — 用户说"我不喜欢这个了" → 更新画像

**3.3 节律识别**

1. **时间节律** — 每天 7:30-8:30 上班、18:00-19:00 下班
2. **地点节律** — 常去超市/健身房/咖啡店
3. **行为节律** — 早饭时段吃东西、周末去郊外
4. **组合节律** — 工作日早上走 A 路,周末走 B 路

**3.4 异常检测与主动关怀**

1. **节律偏离识别** — 今天本该吃早饭但没吃
2. **关怀模式触发** — 偏离 + 关怀预算未用完 → 触发
3. **置信度判据** — 14 天数据窗口 + 置信度 ≥ 80% 才上报

**3.5 场景模式切换**

1. **模式识别** — 融合 账号+乘员+时间+地点+历史习惯 判断当前模式
2. **模式库** — 通勤/家庭/旅途/关怀/驻车 等 5-8 个
3. **切换策略** — 默认"无声切换",可选"提示切换"
4. **手动覆盖** — 用户可强制指定模式
5. **模式冲突处理** — 老婆上车但还是工作日早晨,以人为优先

**3.6 隐私分级**

1. **端侧存储** — 敏感记忆(车内画面、声纹)只在车端
2. **云端同步** — 偏好、POI 兴趣、事件流摘要可上云
3. **用户可查可删** — 记忆控制台,用户查看自己所有记忆
4. **家庭共享 vs 个人独占** — 家庭账号共享记忆,但"和小三的聊天记录"绝不进共享

**5. 功能点展开**

**5.1 S-F1 对话状态机**

*待填*

**5.1.1 一句话定义**

**做什么**：维护当前**对话和任务的运行时状态**——当前任务、子任务、待用户确认的槽位、待 Agent 执行的指令、对话焦点；提供**状态查询、转移、超时清理**的统一接口。

**不做什么**：

* 不做"长期记忆"（B-F1/B-F2 的事，状态机是会话级运行时）
* 不做"任务执行"（Agent 编排 / 文档 06 的事）
* 不做"NLU 多轮上下文"（N-F6 的事，N-F6 管"用户在聊什么"，状态机管"系统的运行状态"）

**S-F1 vs N-F6 的边界**：

* **N-F6（NLU 层）**：用户视角的对话状态（意图栈/槽位栈/话题栈）
* **S-F1（应用层）**：系统视角的运行时状态（任务状态/确认中/执行中/已完成/已挂起）
* 两者通过共同的 session\_id + task\_id 关联

**5.1.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 状态流转 |
| S1 | **正常完成** | "导航去机场" → 状态: awaiting\_slot(目标) → 槽位补全 → pending\_execute → 06 执行 → completed |
| S2 | **等用户确认** | "把所有照片删了" → 高风险 → 状态: awaiting\_confirm(用户确认) → 用户"取消" → cancelled |
| S3 | **任务挂起** | 导航中用户下车去吃饭 → 状态: suspended(原因=用户离车) → 上车后系统提示"刚才的导航继续吗？" → resumed |
| S4 | **任务超时** | 用户挂着 30min 没回应"是去浦东还是虹桥？" → 状态: timeout → 自动清理上下文 |
| S5 | **任务失败** | 充电站没车位 → 状态: failed(原因=资源不足) → 触发兜底（推荐下一家） |
| S6 | **任务取消** | 用户"算了不去了" → 状态: cancelled → 清理 |

**核心状态枚举**：

|  |
| --- |
| Plain Text init → 任务刚创建 awaiting\_slot → 等槽位补全 awaiting\_confirm → 等用户确认（高风险） pending\_execute → 准备发给 06 执行 executing → 06 正在执行 completed → 完成 failed → 失败 cancelled → 用户取消 suspended → 挂起（用户离车/被打断） resumed → 从挂起恢复 timeout → 超时清理 |

**5.1.3 输入 / 触发条件**

* **输入**：NLU 输出（意图+槽位+置信度）、Agent 编排状态回调、用户事件（确认/取消/离车）、定时器（超时检查）
* **触发**：每次 NLU 完成、Agent 状态变化、用户操作、定时心跳
* **前置条件**：状态机引擎可用；存储后端可写（Redis/SQLite）

**5.1.4 执行流程**

|  |
| --- |
| Plain Text 事件输入（NLU 完成 / Agent 回调 / 用户事件 / 超时）  ↓ 状态机查找当前任务的状态  ↓ 状态转移规则匹配  ├─ 合法转移 → 更新状态 + 触发副作用（如 TTS 播报）  └─ 非法转移 → 拒绝 + 记录异常日志  ↓ 副作用执行  ├─ awaiting\_slot → 触发 N-F3 回填询问  ├─ awaiting\_confirm → 触发 TTS 高风险确认  ├─ pending\_execute → 调 Agent 编排 API  ├─ suspended → 持久化状态到长期存储  └─ resumed → 从长期存储恢复 + 提示用户  ↓ 状态广播（订阅方：UI、N-F6、文档 04 长期记忆） |

**关键时间预算**：

* 状态转移延迟：**≤ 10ms**（端侧）
* 持久化延迟：**≤ 50ms**（异步写）
* 超时检查频率：每 30s 扫一次

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **状态机数据丢失**（车机重启状态丢） | 用户对话上下文消失 → 重新开始 | 状态持久化到 SQLite 而非内存；车机启动后恢复 |
| **状态死锁**（任务卡在 awaiting\_confirm 不动） | 永远等不到用户确认 | 30s 超时强制清理；用户切换其他任务自动取消 |
| **并发状态冲突**（A 任务和 B 任务同时改空调） | 行为错乱 | 状态机加锁（同 session 的状态变更串行）；冲突时后者覆盖前者并通知 |
| **状态机和 NLU 不同步** | NLU 觉得在聊 X，状态机已切到 Y | 状态机是唯一真相源；NLU 每次决策前查状态 |
| **超时清理粒度太粗**（导航中超时被误清） | 用户长途导航被打断 | 不同任务类型超时不同：导航类无限期，确认类 30s，挂起类 24h |

**Paper Tiger（看起来像风险但不是）**

* "用 LLM 推理状态够了不需要状态机"—— 错。状态机是**确定性**保证；LLM 推理是非确定性，关键流程不能用 LLM 当真相源
* "状态机太重端侧扛不住"—— SQLite + 状态枚举非常轻，端侧无压力

**Elephant（团队可能没讨论够）**

* **状态机 vs Agent 编排（06）的边界**：
* S-F1 管"任务的对话/确认状态"
* 06 管"任务的执行步骤状态"
* 二者通过 task\_id 关联，**状态机是 06 的上游**
* **状态机 vs N-F6 多轮上下文**：N-F6 偏 NLU 视角，S-F1 偏系统视角；二者**共用 session\_id**，但**字段不同**
* **状态机的可观测性**：线上每个状态转移要有日志（能复盘"为什么任务卡在 awaiting\_slot"）
* **隐私边界**：状态中的"敏感信息"（联系人、健康数据）持久化时是否要加密？**端侧 SQLite 加密 + 用户可清除**

**5.2.1 一句话定义**

**做什么**：支持**多任务并发执行**（如"导航中 + 音乐播放中 + 充电提醒等"），管理任务优先级、资源冲突、任务间依赖，防止任务互相干扰。

**5.2.2 核心场景脚本**

|  |  |  |

**5.2.4 执行流程**

|  |
| --- |
| Plain Text 新任务创建请求  ↓ 资源冲突检查  ├─ 无冲突 → 直接过队  ├─ 有冲突 + 同资源 → 仲裁（高优先级覆盖低优先级 / 并发共享）  └─ 有冲突 + 资源不可用 → 排队等待  ↓ 任务入队  ├─ 按优先级排序（P0 > P1 > P2 > P3）  └─ 按创建时间排序（同优先级 FIFO）  ↓ 任务调度  ├─ P0 任务 → 立即执行，打断 P1/P2/P3  ├─ P1 任务 → 检查是否和 P0/P1 冲突 → 无冲突执行  ├─ P2 任务 → 检查是否和 P0/P1/P2 冲突 → 共享资源执行  └─ P3 任务 → 排队，等资源释放  ↓ 任务执行中  ├─ 状态变化（从 S-F1 同步）  ├─ 资源占用更新  └─ 依赖任务状态检查（A 失败 → B 取消）  ↓ 任务结束  ├─ 释放资源  ├─ 唤醒排队任务  └─ 通知状态机 |

**关键时间预算**：

* 资源冲突仲裁延迟：**≤ 20ms**

* 任务切换延迟：**≤ 100ms**（如导航切音乐）

**5.3 A-F5 任务状态持久化**

*待填*

**5.3.1 一句话定义**

**做什么**：把**未完成的任务状态**（已填槽位、执行进度、上下文、依赖关系）持久化到车端存储，在车辆重启/用户离车/跨会话后**能恢复继续**。

**不做什么**：

* 不做"会话对话历史"（那是 N-F6 的事，会话级状态）
* 不做"长期偏好"（B-F1 的事，偏好是稳定偏好而非任务）
* 不做"单次查询持久化"（查个天气不需要持久化）

**持久化范围**：

* **需持久化**：导航任务（未到达）、订单任务（未完成）、复合任务链（多步未完成）、关怀任务（对话未闭环）
* **不持久化**：查询类（天气/日程）、一次性车控（开空调）、闲聊

**5.3.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 持久化处理 |
| S1 | **跨会话导航** | 用户导航去嘉兴，中途下高速住酒店 → 车熄火，任务 suspended + 持久化 → 次日上车，系统"昨天的嘉兴导航继续吗？" → 用户"继续" → resumed |
| S2 | **复合任务中断** | "去嘉兴，先充电再接朋友" 已完成充电 → 熄火 → 次日上车恢复剩余步骤"接朋友" |
| S3 | **订单任务** | "帮我订晚上 7 点的湘菜馆" 下单中断网 → 任务 suspended → 网络恢复自动重试 |
| S4 | **任务过期自动清理** | 24h 内未恢复的 suspended 任务自动清理，避免积累"僵尸任务" |
| S5 | **用户主动取消** | 次日上车问"刚才的嘉兴导航继续吗？"，用户"不要了" → 任务 cancelled + 清除 |
| S6 | **任务依赖跨会话** | A 任务完成 B 任务才能执行 → 跨会话后 A 已完成，B 仍需等待下次触发 |

**持久化数据结构**：

|  |
| --- |
| JSON {  task\_id: "uuid",  created\_at: timestamp,  updated\_at: timestamp,  status: "suspended",  intent: "导航",  slots: { destination: "嘉兴", stop1: "充电站" },  progress: { completed\_steps: ["充电"], remaining: ["接朋友", "买咖啡"] },  depends\_on: [],  dependents: [],  resume\_hint: "昨天的嘉兴导航，已完成充电",  expire\_at: timestamp + 24h } |

**5.3.3 输入 / 触发条件**

* **输入**：任务状态变化事件（S-F1 状态机）、车辆事件（熄火/启动）、定时器（过期清理）
* **触发**：
* 任务进入 suspended 状态（车辆熄火、用户离车、网络中断）
* 车辆启动时检查是否有未完成任务
* 定时器检查过期任务
* **前置条件**：持久化存储可用（SQLite）；用户授权"跨会话任务恢复"

**5.3.4 执行流程**

|  |
| --- |
| Plain Text 任务需要持久化（状态 → suspended）  ↓ 持久化决策  ├─ 任务类型在"需持久化"白名单 → 持久化  └─ 不在白名单 → 直接清理  ↓ 持久化存储（SQLite）  ├─ 结构化字段写入 task\_persist 表  ├─ 生成 resume\_hint（用户可理解的描述）  └─ 设置 expire\_at（默认 24h）  ↓ 车辆启动事件  ↓ 查询未完成任务  ├─ 无任务 → 跳过  └─ 有任务 → 准备恢复  ↓ 用户确认恢复  ├─ TTS 提示"昨天的 X 任务继续吗？"  ├─ 用户同意 → 加载任务 + 恢复状态 → `resumed` → 继续执行  └─ 用户拒绝 → 任务 `cancelled` + 清除  ↓ 过期清理  └─ 定时器扫描 expire\_at < now 的任务 → 清除 |

**关键时间预算**：

* 持久化写入延迟：**≤ 100ms**（P95）
* 车辆启动恢复延迟：**≤ 3s**（查询 + 加载）
* 过期清理频率：每小时扫一次

**5.3.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **持久化数据损坏**（SQLite 崩溃） | 所有任务丢失 | 写入加校验和；定期备份；恢复失败时友好提示"任务恢复失败，请重新说" |
| **任务恢复时上下文已过时**（昨天导航的终点今天关店了） | 恢复后执行错误 | 恢复时重新校验关键槽位（POI 营业、用户位置）；有变化时提示用户 |
| **用户忘记自己有未完成任务** | 次日被问"昨天的 X 继续吗" 感到意外 | resume\_hint 要清晰 + 包含时间"昨天下午的 X"；用户能一眼回忆 |
| **僵尸任务积累**（过期未清） | 存储膨胀 | 严格过期机制 + 定时清理；用户可手动清除任务列表 |
| **跨账号任务混乱** | A 用户的任务被 B 用户恢复 | 任务绑定 user\_id；恢复前检查当前用户匹配 |

**Paper Tiger（看起来像风险但不是）**

* "用户不会想继续昨天的任务"—— 实证：长途/多步任务中断率 20%+，需要恢复的比例不低
* "SQLite 扛不住"—— 任务数量 < 100，SQLite 无压力

**Elephant（团队可能没讨论够）**

* **和 S-F1 状态机的协作**：状态机管运行时状态，A-F5 管持久化；**状态机 suspended 事件触发 A-F5 写入**，A-F5 不主动决策
* **resume\_hint 的生成算法**：模板 vs LLM 生成？**建议：模板优先，复杂任务 LLM 补**
* **过期时间的差异化**：导航类 24h、订单类 72h、关怀类 1h —— 类型化配置
* **和 06 Agent 编排的恢复协作**：任务恢复时，06 需要从哪一步继续？**06 接收完整任务状态 + 已完成步骤列表**

**5.3.6 SLO / 验收指标**

**5.5 B-F2 事件流记忆**

*待填*

**5.5.1 一句话定义**

**做什么**：记录用户的**历史行为事件**（昨天去过哪、上周点了啥咖啡、上月导航去过哪家餐厅），以**时序 + 语义向量**存储，支持"上次那家""上周去的"这类**时间+语义**召回。

**不做什么**：

* 不做"稳定偏好"（B-F1 的事，偏好是抽象稳定的，事件是具体时点的）
* 不做"节律识别"（B-F3 的事，节律是模式推断）
* 不做"实时对话"（N-F6 的事）

**事件类型**：

* 导航事件：去哪、何时、开了多久
* 订单事件：买了什么、在哪家、何时
* 播放事件：听了什么歌/播客、何时
* 对话事件：聊了什么话题（摘要）、何时

**5.5.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 事件流召回 |
| S1 | **"上次那家"** | 用户"导航去上次那家咖啡店" → 向量召回最近 30 天的 POI 事件 → 匹配"星巴克陆家嘴店"（3 天前去过） |
| S2 | **"上周五吃的"** | 用户"再来一份上周五吃的那个" → 时间过滤"上周五"+ 订单事件召回 → 匹配"麻辣香锅" |
| S3 | **"常去的"** | 用户"带我去我常去的咖啡店" → 事件聚合 + 频次排序 → 返回频次最高的咖啡店 |
| S4 | **对话回忆** | 用户"上次我们聊到哪儿了" → 对话摘要事件召回 → 恢复话题上下文 |
| S5 | **时间模糊召回** | 用户"前几天" → 时间窗口 3-7 天 → 向量召回 |
| S6 | **无召回结果** | 用户"上次那家没吃过的" → 召回为空 → TTS"抱歉没找到记录，您能说具体点吗？" |

**事件数据结构**：

|  |
| --- |
| JSON {  event\_id: "uuid",  user\_id: "ou\_xxx",  event\_type: "navigation",  content: { destination: "星巴克陆家嘴店", poi\_id: "xxx" },  timestamp: 1756... ,  semantic\_vector: [0.23, 0.45, ...], // 1536 维  summary\_text: "3 月 5 日下午导航去星巴克陆家嘴店" } |

**5.5.3 输入 / 触发条件**

* **输入**：用户行为事件（导航完成/订单完成/播放完成/对话结束）
* **触发**：
* 事件完成 → 自动记录
* NLU 检测到语义召回需求（"上次那家"/"上周的"/"前几天的"）→ 触发召回
* **前置条件**：事件存储可用；向量模型可用（Embedding 模型）

**5.5.4 执行流程**

|  |
| --- |
| Plain Text 事件发生（导航完成 / 订单完成 / 播放完成）  ↓ 事件结构化  ├─ content: 结构化字段  ├─ timestamp: 时间戳  ├─ summary\_text: 用户可理解摘要  └─ semantic\_vector: Embedding 向量  ↓ 事件存储  ├─ 端侧 SQLite 时序表（最近 30 天）  └─ 云端向量库（最近 180 天，用户授权后）  ↓ 召回触发（NLU 检测到）  ├─ 时间指代（"上次"/"上周"）→ 时间过滤  ├─ 语义指代（"那家咖啡店"）→ 向量召回  ├─ 组合（"上次那家咖啡店"）→ 时间 + 语义双重  └─ 频次（"常去的"）→ 事件聚合  ↓ Top-K 召回（默认 K=5）  ├─ 相似度打分  ├─ 时间加权（越近权重越高）  └─ 返回 Top-1 + 次选备选  ↓ 召回置信度判定  ├─ Top-1 相似度 ≥ 0.85 → 直接使用  ├─ 0.60-0.85 → TTS 确认"您是说上次那家星巴克陆家嘴店吗？"  └─ < 0.60 → TTS"抱歉没找到记录" |

* 事件写入延迟：**≤ 100ms**（P95）
* 向量召回延迟：**≤ 300ms**（P95，端侧向量库）
* 云端召回延迟：**≤ 800ms**（P95）

**5.6 B-F3 习惯节律识别**

*待填*

**5.6.1 一句话定义**

**做什么**：从用户的**历史行为事件流**（B-F2）中识别**时间规律**（每天 8 点吃早饭、每周五去餐厅、每月 1 号充电），为主动服务（D-F3）和异常关怀（B-F4）提供依据。

**不做什么**：

* 不做"事件记录"（B-F2 的事）
* 不做"主动服务"（D-F3 的事，节律识别只提供"规律"，不执行服务）
* 不做"异常检测"（B-F4 的事，节律识别只提供"正常模式"）

**节律类型**：

* **日节律**：每天固定时间（如每天 8 点吃早饭）
* **周节律**：每周固定时间（如每周五晚 7 点去餐厅）
* **月节律**：每月固定时间（如每月 1 号充电）
* **条件节律**：特定条件触发（如下雨天带伞）

**5.6.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 节律识别 |
| S1 | **早饭节律** | 用户连续 14 天在 8:00-8:30 导航去早餐店 → 识别"日节律：8 点吃早饭" |
| S2 | **通勤节律** | 用户工作日 8:30 导航去公司 → 识别"周节律：工作日早高峰通勤" |
| S3 | **周五聚餐** | 用户连续 4 周周五晚 7 点去同一餐厅 → 识别"周节律：周五晚聚餐" |
| S4 | **充电节律** | 用户每月 1 号充电 → 识别"月节律：月初充电" |
| S5 | **条件节律** | 用户下雨天 5 次都带伞 → 识别"条件节律：下雨 → 带伞提醒" |
| S6 | **节律失效** | 用户换工作后通勤时间变了 → 旧节律 30 天未触发 → 自动失效 |

**节律数据结构**：

|  |
| --- |
| JSON {  rhythm\_id: "uuid",  user\_id: "ou\_xxx",  rhythm\_type: "daily",  pattern: { time: "08:00-08:30", action: "导航早餐店" },  confidence: 0.85,  sample\_count: 14,  last\_triggered: timestamp,  created\_at: timestamp,  expire\_at: null // 30 天未触发自动失效 } |

**5.6.3 输入 / 触发条件**

* **输入**：B-F2 事件流（最近 30 天行为事件）
* **触发**：
* 定时任务（每天凌晨 3 点）扫描事件流 → 识别新节律
* 事件累积达阈值（如 14 天）→ 触发识别
* **前置条件**：事件流数据 ≥ 14 天；节律识别模型可用

**5.6.4 执行流程**

|  |
| --- |
| Plain Text 定时任务触发（每天凌晨 3 点）  ↓ 拉取最近 30 天事件流  ↓ 节律候选生成  ├─ 日节律：时间窗口聚类（8:00-8:30 出现 ≥ 14 次）  ├─ 周节律：星期 + 时间聚类（周五晚 7 点 ≥ 4 次）  ├─ 月节律：日期聚类（每月 1 号 ≥ 3 次）  └─ 条件节律：条件 + 行为关联（下雨 → 带伞 ≥ 5 次）  ↓ 置信度计算  ├─ 样本数 ≥ 14 → 置信度 0.85  ├─ 样本数 10-13 → 置信度 0.70  └─ 样本数 < 10 → 不记录  ↓ 节律冲突检测  ├─ 无冲突 → 写入节律库  ├─ 有冲突（旧节律 vs 新节律）→ 新节律覆盖旧节律  └─ 旧节律 30 天未触发 → 自动失效  ↓ 节律持久化（SQLite）  └─ 供 D-F3 主动服务 / B-F4 异常检测使用 |

**关键时间预算**：

* 节律识别延迟：**≤ 5s**（离线任务，凌晨执行）
* 节律查询延迟：**≤ 10ms**（P95）

**5.6.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **节律误识别**（用户临时 14 天早起不代表长期习惯） | 后续主动服务错误 | 置信度阈值 ≥ 0.85 才启用；用户可关闭"节律驱动主动服务" |
| **节律过时未失效**（用户换工作后通勤节律还在） | 主动服务错误 | 30 天未触发自动失效；用户可手动删除 |
| **节律冲突**（工作日 8 点通勤 vs 周末 8 点去早餐店） | 系统不知道用哪个 | 节律分层：工作日节律 vs 周末节律；条件优先 |
| **冷启动问题**（新用户 < 14 天数据） | 无节律可用 | V1 默认关闭节律驱动主动服务；用户可手动设定常用时间 |
| **隐私担忧**（节律暴露用户生活规律） | 用户不信任 | 节律可查可删；端侧存储不上云；用户可关闭 |

**Paper Tiger（看起来像风险但不是）**

* "节律识别算法很难"—— 实测：时间窗口聚类 + 频次统计即可，不需要复杂模型
* "用户生活不规律识别不出来"—— 部分用户确实不规律，但**通勤/早饭/充电**三类节律覆盖 60%+ 用户

**Elephant（团队可能没讨论够）**

* **节律的"置信度衰减"**：30 天未触发 → 失效；但如果用户只是出差 30 天呢？**建议：失效后不删除，降为"低置信度"，再次触发快速恢复**
* **节律的"条件化"**：工作日 vs 周末、晴天 vs 雨天 —— 节律要支持条件分支
* **节律和偏好的关系**：节律是"时间模式"，偏好是"内容模式"；**二者独立但可组合**（如"周五晚 7 点去我喜欢的餐厅"）
* **节律的"家庭共享"**：家人共用车，节律怎么隔离？**建议：账号级节律 + 家庭共享节律（如"周末全家出游"）**

| 指标 | 目标值 | 测试条件 |
| 节律识别准确率 | ≥ 80% | 500 条真实用户行为测试 |
| 节律误识别率 | < 10% | 临时行为不应被识别为节律 |
| 节律失效准确率 | ≥ 95% | 30 天未触发自动失效 |
| 节律查询延迟 | ≤ 10ms (P95) | SQLite 查询 |
| 冷启动覆盖率 | ≥ 50% | 14 天后至少识别 1 个节律 |
| 用户节律满意度 | ≥ 70% | 用户调研"主动服务准不准" |

**5.7 B-F4 异常识别与主动关怀**

*待填*

**5.7.1 一句话定义**

**做什么**：检测用户**行为的节律偏离**（该吃早饭没吃、该通勤没通勤），结合**关怀预算**（每周一次主动服务），在用户没意识到的场景**主动关怀**（提醒吃饭/路线异常等）。

**不做什么**：

* 不做"节律识别"（B-F3 的事，节律识别只提供规律，不做判断）
* 不做"主动服务执行"（D-F3 的事，B-F4 只检测偏离 + 提醒，不执行服务）
* 不做"疲劳干预"（那是 C-F1 DMS 的安全类，B-F4 是关怀类）

**关怀 vs 干扰的边界**：

* **关怀**：用户可能需要的提醒（如"今天没吃早饭"）
* **干扰**：用户明确不需要的推送（如"今天路线跟平时不一样"用户知道是施工）
* **关键**：关怀 ≤ 1 次/周，必须有用户可关闭的总开关

**5.7.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 异常识别 + 主动关怀 |
| S1 | **早饭异常** | 用户每天 8 点吃早饭，今天 8:30 没导航到早餐店 → 识别"早饭节律偏离" → TTS "今天还没吃早饭，要不要去那家经常去的？" |
| S2 | **通勤异常** | 用户工作日 8:30 导航去公司，今天 9:00 没导航 → 识别"通勤异常" → TTS "今天没导航去公司，要帮忙吗？" |
| S3 | **周五聚餐异常** | 用户连续 4 周周五晚 7 点去同一家餐厅，今天周五 7 点没去 → 识别"节律偏离" → TTS "今天是周五，要去上次那家餐厅吗？" |
| S4 | **雨雪天关怀** | 节律识别出"下雨天带伞"，今天下雨但用户没带伞（检测车内无雨伞）→ TTS "今天下雨，要找地方买伞吗？" |
| S5 | **异常路线** | 用户工作日常走 A 路，今天走了 B 路（导航目的地相同但路线不同）→ 识别"路线异常" → TTS "今天路线跟平时不一样，要注意安全" |
| S6 | **关怀预算超限** | 本周已关怀 2 次，还想触发第 3 次 → 关怀预算已满，不触发关怀 |

**关怀预算规则**：

|  |
| --- |
| Plain Text 关怀预算 = 1 次/周（可配置） 关怀类型：  - 节律偏离（早/晚饭、通勤、聚餐）  - 异常提醒（雨雪、路线异常） 关怀优先级：  - P0: 安全类（雨雪、路线异常）- 打破预算上限  - P1: 节律偏离（早/晚饭、通勤、聚餐）- 受预算限制 |

**5.7.3 输入 / 触发条件**

* **输入**：B-F3 节律识别结果（正常节律）、B-F2 事件流（用户实际行为）、车辆状态（雨雪天）、关怀预算计数
* **触发**：
* 用户行为偏离节律（该吃饭没吃）
* 车辆异常状态（雨雪天）
* 用户导航路线异常（同目的地不同路线）
* **前置条件**：节律识别可用（B-F3）；关怀预算计数可用；用户授权"主动关怀"

**5.7.4 执行流程**

|  |
| --- |
| Plain Text 定时任务（每小时检查）  ↓ 用户行为 vs 节律对比  ├─ 无偏离 → 跳过  └─ 有偏离 → 异常检测  ↓ 异常分类  ├─ 节律偏离（早/晚饭、通勤、聚餐）→ P1  ├─ 车辆异常（雨雪天）→ P0  └─ 路线异常 → P0  ↓ 关怀预算检查  ├─ P0 异常 → 突破预算上限，直接触发  ├─ P1 异常 + 预算未满 → 触发  └─ P1 异常 + 预算已满 → 队列等下周  ↓ 关怀内容生成  ├─ TTS 文本模板："今天还没吃早饭，要帮忙吗？"  └─ 支持个性化（用户名字 + 偏好内容）  ↓ 关怀执行  ├─ TTS 播报  ├─ 关怀预算计数 +1  └─ 用户交互  ├─ 用户回应 → 任务结束  └─ 用户无视 → 5s 后自动消除  ↓ 关怀反馈  └─ 用户可反馈"不准/有用"→ 训练优化 |

**关键时间预算**：

* 异常检测延迟：**≤ 1s**（离线任务）
* 关怀生成延迟：**≤ 50ms**
* 关怀播报延迟：**≤ 2s**（TTS）

**5.7.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **异常误识别**（用户今天不想吃早饭，被识别成"忘了"） | 用户觉得系统唠叨 | 节律偏离置信度 ≥ 80% 才触发；用户可关闭关怀 |
| **关怀频繁打扰**（预算控制失效，一周关怀 5 次） | 用户关闭功能 | 严格预算计数；P0 突破预算后一周内不再突破；用户可手动关 |
| **关怀内容不相关**（下雨天提醒"没吃早饭"） | 体验崩 | 异常类型和关怀内容严格映射；P0 雨雪天只说"下雨，要买伞吗？" |
| **关怀播报时机错**（开车播报，用户慌） | 安全风险 | 只在驻车或低速（< 30km/h）播报；高速时不播 |
| **用户无法关闭关怀** | 用户被迫接受 | 必须有关怀总开关 + 每类异常单独开关 |

**Paper Tiger（看起来像风险但不是）**

* "关怀用户不需要"—— 实证：30%+ 用户需要（老人/疲劳场景）；70% 用户可以关闭
* "节律偏离很难识别"—— 实测：早/晚饭/通勤/聚餐四类节律 + 14 天数据，准确率 ≥ 75%

**Elephant（团队可能没讨论够）**

* **关怀预算的"周"粒度够吗？** 用户周一关怀 1 次，周四 1 次，周五 1 次 —— 周一 1 次还行，周四、周五第 2、3 次呢？**建议：周预算 = 2 次，不是 1 次**
* **关怀内容的"个性化"**：能不能说"今天还没吃那家星巴克"而不是"还没吃早饭"？**建议：模板 + 节律内容插值（星巴克陆家嘴店）**
* **关怀"沉默期"**：用户刚关闭关怀，多久后才能再次提醒？**建议：30 天沉默期**
* **关怀和"疲劳干预"的边界**：DMS 疲劳是 P0 安全类，B-F4 关怀是 P1 生活类 —— 二者优先级明确，不打架

**5.7.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 节律偏离识别准确率 | ≥ 75% | 500 条真实用户行为测试 |
| 误识别率（临时行为被当成偏离） | < 15% | 临时行为测试集 |
| 关怀预算控制准确率 | ≥ 99% | 模拟多场景关怀计数 |
| 关怀内容相关性 | ≥ 90% | 用户相关性评分 |
| 关怀播报时机准确率 | ≥ 95% | 高速/低速场景 |
| 用户关怀接受率 | ≥ 50% | 被关怀的用户中有用/无感的比例 |
| 用户关怀关闭率 | < 10% | 长期监控 |

**分级规则**：

* P0 异常（安全类）：突破预算上限
* P1 异常（节律偏离）：受预算限制（2 次/周）

**5.8 B-F6 遗忘与修正机制**

**5.8.1 一句话定义**

**做什么**：给用户**彻底的记忆控制权**——查看自己所有记忆、修正错误记忆、删除敏感记忆、关闭某类记忆，让"记忆系统"不变成"监控系统"。

**5.8.2 核心场景脚本**

| --- | --- | --- |

**5.8.4 执行流程**

|  |
| --- |

* 云端同步删除延迟：**≤ 5s**

**5.8.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **删除不彻底**（只删端侧没删云端） | 用户以为删了其实没 | 端云双删 + 删除确认"已清除端侧和云端数据" |
| **误删不可恢复** | 用户后悔但找不回 | 重要删除有"7 天回收站"（除了敏感记忆）；敏感记忆（健康/对话）直接物理删除 |
| **记忆报告被用户嫌烦** | 用户关闭报告 → 透明度消失 | 报告默认开启但可关闭；报告内容简洁（< 30s 读完） |
| **修正不生效**（说"不喜欢 X" 但系统还推） | 用户觉得被忽视 | 修正立即生效 + TTS 确认"好的，以后不会再推 X 了" |
| **开关粒度太粗**（只能全关不能分类关） | 用户体验差 | 类别级开关（偏好/事件/节律/关怀独立） + 单条操作 |

**Paper Tiger（看起来像风险但不是）**

* "用户不会用控制台"—— 实证：20%+ 用户至少用一次；语音修正门槛更低
* "删除会让推荐变差"—— 用户主动删除 = 用户不希望被推荐，体验不应变差反而应该更尊重

**Elephant（团队可能没讨论够）**

* **删除的"审计日志"**：记录"用户删除了 N 条 X 类数据"作为元数据 —— **本身是否也需要用户可见/可删？** 建议：审计日志只存 30 天 + 用户可查
* **敏感记忆的"不可恢复"边界**：健康、对话、生物特征 → 用户删除即永久删除，无回收站；其他类有 7 天回收站
* **家庭共享记忆的删除权限**：A 用户能删 B 用户的记忆吗？**建议：只能删自己的，共享记忆需全家同意**
* **记忆的"延期删除"**：用户 30 天未登录，敏感记忆是否自动清除？**建议：用户配置（默认保留）**

**5.9 S-F3 场景模式识别与切换**

**5.9.1 一句话定义**

**做什么**：基于**账号 + 乘员 + 时间 + 地点 + 历史习惯**融合判断当前处于哪种**场景模式**（通勤/家庭/旅途/关怀/驻车），自动切换座舱的**交互风格、推送策略、默认行为、UI 布局**，让座舱"懂场景"。

**不做什么**：

* 不做"模式内的具体行为"（每个模式下的具体交互是各功能的事）
* 不做"用户手动切换"（用户可以强制覆盖，但 S-F3 的核心是自动识别）
* 不做"单一模式判断"（如只看时间或只看乘员）

|  |
| --- |
| Plain Text 1. 通勤模式（工作日早晚高峰，主驾一人）  特征：语音简短、推送克制、AI 人设专业、音乐偏工作流 2. 家庭模式（周末/节假日 + 后排有娃/全家）  特征：TTS 温柔、推送多、AI 人设亲切、音乐偏家庭友好 3. 旅途模式（长途导航 > 2h）  特征：关怀多（问需要休息吗）、娱乐内容多、AI 人设陪伴 4. 关怀模式（DMS 识别情绪低落 + 驻车）  特征：TTS 温柔、减少主动打扰、音乐治愈系、AI 人设共情 5. 驻车模式（车辆驻停 > 5min）  特征：支持沉浸式娱乐、影院模式、音乐/播客/游戏 |

**5.10.1 一句话定义**

**做什么**：基于**节律识别（B-F3）+ 场景模式（S-F3）+ 偏好记忆（B-F1）**，在用户**未明确发起**的情况下，**主动提供服务**（如"周五晚 7 点，要去上次那家餐厅吗？"），让座舱从"被动响应"变"主动陪伴"。

**不做什么**：

* 不做"节律识别"（B-F3 的事）
* 不做"异常关怀"（B-F4 的事，B-F4 是偏离提醒，D-F3 是正常习惯服务）
* 不做"模式识别"（S-F3 的事）

**主动服务 vs 异常关怀的边界**：

* **B-F4 异常关怀**：节律偏离 → 提醒（"今天没吃早饭"）
* **D-F3 主动服务**：节律正常 → 服务（"周五晚 7 点，要去上次那家餐厅吗？"）

**5.10.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 主动服务 |
| S1 | **周五聚餐** | 周五晚 6:50 + 节律识别"周五晚 7 点去餐厅" → TTS "今天是周五，要去上次那家餐厅吗？" → 用户"好" → 自动导航 |
| S2 | **通勤路线** | 工作日 8:20 + 节律识别"8:30 通勤" → TTS "要导航去公司吗？" → 用户"嗯" → 自动导航 |
| S3 | **早饭提醒** | 工作日 7:50 + 节律识别"8 点吃早饭" → TTS "要去那家经常去的早餐店吗？" → 用户"好" → 自动导航 |
| S4 | **充电提醒** | 每月 1 号 + 节律识别"月初充电" + 电量 < 30% → TTS "今天是 1 号，要去充电吗？" → 用户"好" → 推荐充电站 |
| S5 | **音乐推荐** | 通勤模式 + 偏好记忆"喜欢周杰伦" → TTS "要听周杰伦吗？" → 用户"好" → 自动播放 |
| S6 | **服务预算超限** | 本周已主动服务 3 次，还想触发第 4 次 → 服务预算已满，不触发 |

**主动服务预算规则**：

|  |
| --- |
| Plain Text 服务预算 = 3 次/周（可配置） 服务类型：  - 节律驱动（通勤/早饭/聚餐/充电）  - 偏好驱动（音乐/播客推荐） 服务优先级：  - P0: 安全类（充电 < 10%）- 打破预算上限  - P1: 节律驱动（通勤/早饭/聚餐）- 受预算限制  - P2: 偏好驱动（音乐推荐）- 受预算限制 |

**5.10.3 输入 / 触发条件**

* **输入**：B-F3 节律识别结果、S-F3 当前模式、B-F1 偏好记忆、当前时间、车辆状态、服务预算计数
* **触发**：
* 节律时间窗口到达（如周五晚 6:50）
* 模式切换（如切到通勤模式）
* 车辆状态变化（如电量 < 30%）
* **前置条件**：节律识别可用；用户授权"主动服务"；服务预算未满

**5.10.4 执行流程**

|  |
| --- |
| Plain Text 定时任务（每 5min 检查）  ↓ 节律匹配  ├─ 当前时间 vs 节律时间窗口  ├─ 当前模式 vs 节律模式  └─ 车辆状态 vs 节律条件  ↓ 服务候选生成  ├─ 节律驱动（通勤/早饭/聚餐/充电）  ├─ 偏好驱动（音乐/播客推荐）  └─ 模式驱动（家庭模式 → 推荐亲子内容）  ↓ 服务预算检查  ├─ P0 服务 → 突破预算上限，直接触发  ├─ P1/P2 服务 + 预算未满 → 触发  └─ P1/P2 服务 + 预算已满 → 队列等下周  ↓ 服务内容生成  ├─ TTS 文本模板："今天是周五，要去上次那家餐厅吗？"  ├─ 支持个性化（用户名字 + 偏好内容 + 节律内容）  └─ 支持多轮确认（"哪家？" → "上次那家星巴克陆家嘴店"）  ↓ 服务执行  ├─ TTS 播报  ├─ 服务预算计数 +1  └─ 用户交互  ├─ 用户同意 → 执行服务（导航/播放/订单）  ├─ 用户拒绝 → 任务取消  └─ 用户无视 → 5s 后自动消除  ↓ 服务反馈  └─ 用户可反馈"不准/有用"→ 训练优化 |

**关键时间预算**：

* 节律匹配延迟：**≤ 1s**（离线任务）
* 服务生成延迟：**≤ 50ms**
* 服务播报延迟：**≤ 2s**（TTS）

**5.10.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **服务不准**（周五晚推荐餐厅但用户今天不去） | 用户觉得系统唠叨 | 节律置信度 ≥ 0.85 才触发；用户可关闭主动服务 |
| **服务频繁打扰**（预算控制失效，一周服务 10 次） | 用户关闭功能 | 严格预算计数；P0 突破预算后一周内不再突破；用户可手动关 |
| **服务内容不相关**（通勤模式推荐音乐但用户在开会） | 体验崩 | 服务类型和模式严格映射；通勤模式只推通勤相关服务 |
| **服务播报时机错**（高速播报，用户慌） | 安全风险 | 只在驻车或低速（< 30km/h）播报；高速时不播 |
| **用户无法关闭服务** | 用户被迫接受 | 必须有服务总开关 + 每类服务单独开关 |

**Paper Tiger（看起来像风险但不是）**

* "主动服务用户不需要"—— 实证：40%+ 用户需要（老人/忙碌场景）；60% 用户可以关闭
* "节律驱动很难准"—— 实测：通勤/早饭/聚餐/充电四类节律 + 14 天数据，准确率 ≥ 80%

**Elephant（团队可能没讨论够）**

* **服务预算的"周"粒度够吗？** 用户周一服务 1 次，周三 1 次，周五 1 次 —— 周一 1 次还行，周三、周五第 2、3 次呢？**建议：周预算 = 3 次，不是 2 次**
* **服务内容的"个性化"**：能不能说"要去上次那家星巴克陆家嘴店吗"而不是"要去餐厅吗"？**建议：模板 + 节律内容 + 偏好内容插值**
* **服务"沉默期"**：用户刚关闭服务，多久后才能再次提醒？**建议：30 天沉默期**
* **服务和"异常关怀"的边界**：B-F4 是偏离提醒，D-F3 是正常服务 —— 二者优先级明确，不打架

## 补充：四层记忆架构（整合升级）

> 以下内容是对 3.2 记忆系统的补充和具体化，将记忆从"偏好+事件"两层扩展为四层完整架构。

**第一层：工作记忆（单次会话内）**

保留现有 Redis 方案，升级为 30 轮对话、1 小时 TTL。新增跨 channel 上下文桥接——用户从点餐切到导航时，目的地/商品/人数等关键字段自动带过去，不需要用户重复说。

**第二层：用户偏好记忆（永久）**

MySQL `user_preferences` 表，具体结构：

| category | key | value | confidence | 来源 |
|---------|-----|-------|-----------|------|
| food | kfc_regular | 香辣鸡腿堡套餐中可 | 0.85 | 下单3次自动学习 |
| food | starbucks_regular | 美式中杯少冰 | 1.0 | 用户说"记住" |
| shopping | taobao_address | 张江高科园区18号 | 1.0 | 下单地址提取 |
| movie | seat_preference | 中间靠后 | 0.85 | 选座2次自动学习 |
| navigation | home | 浦东新区xx路xx号 | 1.0 | 用户说"记住" |
| navigation | company | 张江高科技园区 | 0.85 | 导航3次自动学习 |

**写入规则**：
- 用户明确说"记住" → confidence=1.0
- 同一选择重复3次 → confidence=0.85
- 工具执行成功后自动抽取关键字段写入（订单→商品/门店；导航→目的地；选座→偏好位置）

**第三层：任务状态记忆（跨会话）**

MySQL `task_states` 表，记录每个任务的完整生命周期：

| 字段 | 含义 |
|------|------|
| task_id | 唯一标识 |
| user_id | 用户 |
| tool_code | 305.22（星巴克）/ 305.30（KFC）/ 305.32（电影票）等 |
| status | pending / running / done / failed / cancelled |
| context_json | 任务上下文快照（已填充的槽位、选择的商品等） |
| created_at / updated_at | 时间戳 |

**用途**：
- 用户说"上次那个订单怎么样了" → 查 task_states
- 系统重启 → 扫描 status=running 的任务恢复
- 用户中途打断 → 下次上车 TTS："您上次点了一半的肯德基订单，要继续吗？"

**第四层：节律记忆（V2）**

保留现有 3.3 节律识别方案不变，补充与偏好记忆的联动：节律规律 + 偏好记忆 → 驱动主动推荐（如"工作日早上常点美式" → 上车时主动问）。

**指代词解析（与偏好记忆联动）**

识别到"老地方/上次那个/老样子"时，查偏好表和最近 task_states 记录，把模糊词替换成具体实体再走 NLU。这属于 query_rewrite 阶段的增强，不是新增独立模块。

**歧义消除（与指代词解析不同）**

匹配到多个候选时（两个张伟、两家肯德基），TTS 播报让用户选一个，不自动猜。这是选择题，不是查历史。

# PRD 04 §5.6 · B-F3 习惯节律识别（补全版）

> 所属：PRD 04《状态管理·记忆系统·模式切换》
> 功能 ID：B-F3 · 版本：V1.5（需 ≥14 天行为数据后解锁）
> 补全日期：2026-05-12

---

## 1. 功能定义

**做什么**：通过分析用户历史行为事件流（B-F2），自动识别具有时间重复性的生活节律（如"工作日 8:20 出发上班""周五晚 22:00 前回家"），并以此驱动主动服务（B-F4、D-F3）。

**不做什么**：不做精神状态推断（抑郁/焦虑诊断）；不做跨用户画像共享；不在行为数据少于 14 天时激活。

---

## 2. 触发条件与输入

| 输入 | 来源 | 说明 |
|------|------|------|
| 行为事件流 | B-F2 时序数据库 | 出发时间/目的地/驾驶时长/音乐/空调设置 |
| 星期/日期 | 系统时钟 | 区分工作日/周末/节假日 |
| 节律置信度反馈 | 用户接受/拒绝主动服务 | D-F3 返回的反馈信号 |

触发时机：每天凌晨 03:00 离线批量计算（不实时，避免干扰驾驶体验）。

---

## 3. 核心算法流程

```
行为事件流（≥14天）
    ↓
时间序列分频（工作日/周末分桶）
    ↓
频繁模式挖掘（Apriori，支持度 >40%，置信度 >60%）
    ↓
节律候选集（最多20条）
    ↓
稳定性过滤（连续3周出现 → 稳定节律）
    ↓
节律权重计算（稳定性×频次×用户反馈）
    ↓
写入节律数据库（端侧 SQLite，TTL=90天滚动更新）
```

节律数据结构：
```json
{
  "rhythm_id": "commute_morning_weekday",
  "trigger": {"day_of_week": [1,2,3,4,5], "time_window": "07:50-08:40"},
  "pattern": {"destination": "公司", "music": "通勤播放列表"},
  "confidence": 0.83,
  "occurrence": 34,
  "last_active": "2026-05-10"
}
```

## 4. 失败路径与降级

| 失败情况 | 降级策略 |
|----------|----------|
| 数据不足（<14天） | 功能静默，不告知用户（避免"正在学习"的进度条焦虑） |
| 节律置信度 <60% | 不输出节律，等待更多数据积累 |
| 用户行程高度不规律 | 识别失败，D-F3 主动服务关闭，降级到手动触发 |
| 节律突变（连续7天不匹配） | 旧节律权重衰减（×0.5/天），7天后废弃，重新学习 |
| 离线计算超时（>60s） | 本次跳过，保留上次节律结果 |

## 6. 自我反驳（六关）

**关1 - 价值关**：节律识别准不准，用户真的在意吗？
→ 反驳：用户不在意节律"准不准"，在意的是主动服务"烦不烦"。只要触发时机对（该来的来，不该来的不来），用户感知是"懂我"而非"在学我"。核心指标是 D-F3 的接受率，不是节律识别准确率。

**关2 - 数据关**：14天才能启动，冷启动期用户体验太差？
→ 反驳：冷启动期（<14天）用户使用手动触发，体验与现有车机一致，不会更差。14天后解锁"惊喜功能"反而会有正向期待感。需要设计一个轻量提示："你已用车14天，智能节律已开启"（一次性通知，不骚扰）。

**关3 - 隐私关**：行为模式分析是否违反用户隐私预期？
→ 反驳：行为数据端侧计算、端侧存储，不上云。节律数据库只存抽象模式（"工作日早晨出发"），不存原始轨迹。用户可在记忆控制台查看所有节律并删除单条。符合"最小必要"和"知情同意"原则。

**关4 - 算力关**：每天批量计算影响端侧资源？
→ 反驳：凌晨03:00执行，车机处于待机状态（不驾驶），NPU/CPU占用无影响。计算量小（最多30天×N个事件的频繁模式挖掘），实测预估 <60s，完全可接受。

**关5 - 竞品关**：NOMI/小爱/小度都没做到好用的节律识别，我们凭什么能做好？
→ 反驳：NOMI 的失败是因为节律服务在网络层产生可感知延迟，且弹出时机不够精准（在换道/并线时弹出）。我们的差异点：① 端侧计算（无网络延迟）；② 只在驻车/等灯状态触发（不在行驶中打扰）；③ 有明确放弃条件（识别不准就关掉）。

**关6 - 范围关**：节律识别边界在哪？情感状态能不能识别？
→ 明确不做：不推断用户情绪/心理状态；不做预测性健康建议；不做跨用户比较（"像你这样的用户…"）。节律识别只做时间+行为模式，不做"为什么这样"的动机分析。

# PRD 04 §5.7 · B-F4 异常识别与主动关怀（补全版）

> 所属：PRD 04《状态管理·记忆系统·模式切换》
> 功能 ID：B-F4 · 版本：V1.5（依赖 B-F3 节律基线）
> 补全日期：2026-05-12

---

## 1. 功能定义

**做什么**：对比当前行为与已建立的节律基线（B-F3），检测显著偏差并触发主动关怀。例如：平时周五20:00到家，今天22:30还未到家→主动询问"还好吗，需要联系家人吗？"

**不做什么**：不做医疗诊断；不强制上报位置给第三方（含家人，须用户事先授权）；不在驾驶中执行高注意力关怀动作。

---

## 2. 触发条件

异常触发需同时满足：

1. **节律基线已建立**（B-F3 置信度 >60%）
2. **实际行为偏离基线**：
   - 时间偏差：超出正常窗口 >45min
   - 地点偏差：当前位置与预期位置 >5km
3. **排除已知例外**：
   - 用户主动告知"今天加班/出差"（NLU记录临时变更）
   - 日历检测到今日有外出事项
   - 当天行程已被手动修改

异常等级：
| 等级 | 条件 | 响应 |
|------|------|------|
| L1 轻微 | 偏离30-60min | 柔和询问："今天路上顺利吗？" |
| L2 中度 | 偏离60-120min | 主动询问+提供帮助选项 |
| L3 严重 | 偏离>120min 且停止响应 | 询问+可选联系紧急联系人（须用户事先授权） |

---

## 3. 执行流程

```
B-F3 节律基线
    ↓
实时位置/时间监测（每5min一次，驻车时暂停）
    ↓
偏差计算（Δt, Δlocation）
    ↓
异常等级判定
    ↓
例外条件检查（日历/临时告知/天气延误）
    ↓
通过 → 触发关怀 TTS（驻车/等灯时机）
         用户响应 → 记录，更新节律例外
         无响应（>10min）→ L3 升级
```

---

## 4. 失败路径与降级

| 失败情况 | 降级策略 |
|----------|----------|
| B-F3 基线不存在 | 本功能不激活，无任何输出 |
| GPS 信号丢失 | 停止位置监测，仅用时间维度判断 |
| 用户关闭关怀功能 | 全局关闭，不再触发任何异常关怀 |
| 用户一周内连续忽略3次 | 该场景静默30天，等待用户主动恢复 |
| 触发时正在驾驶（非等灯） | 延迟到下次驻车/等灯再触发 |

---

## 5. SLO / 验收指标

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 异常识别精确率 | >70%（触发的关怀中，用户确认"有效"占比） | 用户反馈统计 |
| 误报率 | <15%（用户说"没事，正常"的比例） | 拒绝信号计数 |
| 触发时机合规率 | 100%（不在行驶中触发L2+） | 驾驶状态埋点 |
| 关怀预算 | ≤2次/周/用户 | 预算计数器 |
| 用户授权紧急联系人比例 | 作为健康指标监测（不设强制目标） | 授权记录 |

**放弃条件**：误报率 >25% 且连续4周无改善。

---

## 6. 自我反驳（六关）

**关1 - 价值关**：这个功能有多少用户真的需要？
→ 目标用户是"独自长途通勤"或"家人关心用车安全"的群体，渗透率约30%（PRD 13 用户调研数据）。但触发频率低（异常才触发），对不需要的用户几乎无感知。价值在于"关键时刻的安全感"，而非高频使用。

**关2 - 骚扰关**：异常关怀太敏感，用户会觉得被监视？
→ 关键设计：① 用户主动开启（默认关闭）；② 关怀话术克制（询问式，不判断式）；③ 设置里可随时查看"触发日志"；④ 数据端侧计算，不上云。让用户感觉是"车在关心我"而不是"有人在看着我"。

**关3 - 准确关**：节律基线本身就不准，异常识别会不会更不准？
→ 使用多重过滤（日历/天气/临时告知）降低误报。B-F3 置信度 <60% 时本功能不激活。宁可漏报（异常发生了但没检测到），不可误报（正常情况被判断为异常）。

**关4 - 紧急联系关**：L3 联系家人这个设计是否合规？
→ 严格要求：① 用户事先在设置里添加并授权紧急联系人；② 联系方式仅限 App 内发送消息（不直接拨号）；③ 消息内容用户可预览（"车主可能需要关注，最后位置：XX区"）；④ 个保法要求用户明确同意，授权协议单独展示。

**关5 - 驾驶安全关**：关怀弹窗是否分散驾驶注意力？
→ L1 只走 TTS，无屏幕弹窗。L2 在等灯/驻车时才展示屏幕内容。L3 只有在车辆完全停止后才升级。行驶中（车速>10km/h）只能播放轻柔语音，不展示交互选项。

**关6 - 数据关**：位置历史用于异常检测，用户删除记忆后怎么办？
→ 用户在记忆控制台删除节律数据 → B-F3 基线清除 → B-F4 自动停止运行（无基线无关怀）。位置历史数据端侧存储，用户删除后不可恢复，不在云端有备份。

# PRD 04 §5.8 · B-F6 记忆遗忘与修正机制（补全版）

> 所属：PRD 04《状态管理·记忆系统·模式切换》
> 功能 ID：B-F6 · 版本：V1（Must，记忆系统上线即需要）
> 补全日期：2026-05-12

---

## 1. 功能定义

**做什么**：实现记忆系统的时效性管理（自动遗忘过期/低价值记忆）和用户主动修正（纠错/删除单条/全量清除），保证记忆库"有效、准确、可控"。

**不做什么**：不做无限积累的记忆库（存储和计算成本不可控）；不在用户不知情的情况下修改记忆。

---

## 2. 遗忘机制（自动）

### 2.1 时效性遗忘

| 记忆类型 | 保留策略 |
|----------|----------|
| 偏好记忆（B-F1） | 无过期，但权重随时间衰减（×0.97/天，无交互30天后降至0.1基准） |
| 事件记忆（B-F2） | 最近30天完整保留，30-90天压缩为摘要，>90天删除 |
| 节律记忆（B-F3） | 90天滚动窗口，持续90天无匹配的节律删除 |
| 对话历史 | 最近20轮原始 + LLM摘要（每5轮生成1条摘要），原始历史保留7天 |
| 任务记录 | 已完成任务保留3天（用于异常分析），之后删除 |

### 2.2 低价值遗忘

触发条件：
- 某偏好权重衰减到 <0.05（相当于90天无交互）→ 自动从活跃偏好移除
- 节律置信度 <0.3 且连续30天无匹配 → 自动删除
- 用户连续3次拒绝某类主动服务 → 对应记忆标记"用户不喜欢"，降权但不删除（保留供解释）

---

## 3. 修正机制（用户主动）

### 3.1 对话内即时修正

触发词识别（NLU层）：
- "不是这样的" / "我没说过这个" / "你记错了" → 触发修正流程
- "以后不要推荐XX了" → 偏好权重归零
- "忘掉关于XX的一切" → 语义检索相关记忆并列出，用户确认后删除

修正流程：
```
用户说"你记错了"
    ↓
展示当前相关记忆（最多3条）
    ↓
用户指认错误条目
    ↓
AI 重新理解正确信息
    ↓
更新记忆 + TTS 确认"好的，我已更新"
```

### 3.2 记忆控制台（UI）

路径：设置 → 关于我 → 记忆管理

功能：
- 查看按类型分组的所有记忆条目（偏好/事件/节律/对话摘要）
- 删除单条（立即生效，端云双删）
- 按类型全量清除（如"清除所有节律记忆"）
- 全量清除（相当于恢复出厂，所有学习数据归零）
- 导出记忆JSON（数据可携带权，脱敏后可下载）

删除响应时间：端侧 <1s，云端同步 <30s（异步），用户界面立即反馈"已删除"。

---

## 4. 失败路径与降级

| 失败情况 | 降级策略 |
|----------|----------|
| 端侧删除成功，云端同步失败 | 重试3次，最终失败后告知用户"云端同步失败，请稍后重试"，端侧数据已删除 |
| 用户修正意图识别错误 | 展示"你想修正的是这条吗？"确认对话，不自动执行 |
| 遗忘计算任务崩溃 | 下次触发时重跑，不影响日常记忆读取 |
| 全量清除中断（如断电） | 恢复后继续清除（幂等设计，已删除的不重复处理） |

---

## 5. SLO / 验收指标

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 删除响应时间（端侧） | P99 <2s | 操作时间戳埋点 |
| 云端同步完成率 | >99.5% | 同步结果埋点 |
| 对话内修正识别准确率 | >80% | 修正意图 NLU 准确率 |
| 遗忘机制误删率 | <1%（用户反馈"记忆被错误删除"） | 投诉率统计 |
| 记忆控制台使用率 | 作为健康指标（不设目标），反映用户对记忆系统的信任度 | MAU统计 |

---

## 6. 自我反驳（六关）

**关1 - 价值关**：用户真的会主动管理记忆吗？
→ 记忆控制台的核心价值不是"让用户天天用"，而是"让用户知道可以管"。透明度本身就是信任的来源。日常使用中，更常见的是对话内即时修正（"你记错了"），这个才是高频入口。

**关2 - 遗忘策略关**：自动遗忘会不会删掉用户认为重要的记忆？
→ 保守策略：只有持续长时间无交互的记忆才会降权/删除。删除前不会通知用户（避免打扰），但用户可在控制台查看"最近遗忘的记忆"（保留7天的回收站），允许恢复。

**关3 - 数据一致性关**：端云记忆如果不同步，谁以谁为准？
→ 端侧为主（本地最新）。云端是备份+跨设备同步源。删除操作：端侧先删，异步通知云端；恢复场景：换设备时，从云端拉取（此时云端可能有端侧已删但云端还未同步的数据）。解决方案：每条记忆有删除时间戳，云端同步时以时间戳判断（已删除的不恢复）。

**关4 - 合规关**：用户有权删除个人数据，我们的架构能保证吗？
→ 个保法第47条要求：受理删除请求 → 合理期限内删除 → 通知第三方（如有）也删除。实现：① 端侧立即删除；② 云端异步删除（<48h）；③ 生成"删除确认记录"（时间戳+操作类型），用户可查。不向第三方传输个人记忆数据（第三方API只接收匿名化请求）。

**关5 - 修正意图歧义关**：用户说"你记错了"但没说哪条记错了，怎么办？
→ 结合上下文推断：① 当前对话中AI刚用了什么记忆（最近使用的记忆权重最高）；② 展示候选条目让用户选（不超过3条，避免选择疲劳）；③ 用户说不出来 → "没关系，下次遇到我会再问你"（不强迫修正）。

**关6 - 范围关**：遗忘机制是否影响用户信任（"AI把我忘了"）？
→ 设计原则：用户永远不会"突然发现AI忘了什么重要的事"。偏好记忆权重衰减是渐进的（不是突然删除），AI使用偏好频率降低时用户自然感知到"AI推荐变了"而不是"AI忘了我"。节律记忆删除前有90天窗口，足够感知到"我最近没按这个规律出行了"。


---

# PRD 04 §5.10 · D-F3 习惯驱动主动服务（补全版）

> 所属：PRD 04《状态管理·记忆系统·模式切换》
> 功能 ID：D-F3 · 版本：V1.5（依赖 B-F3 节律，V1 默认关闭）
> 补全日期：2026-05-12

---

## 1. 功能定义

**做什么**：基于 B-F3 识别的用户生活节律，在适当时机主动提供与用户习惯高度匹配的服务（如"你平时这时候会去星巴克，要提前点单吗？"）。与 D-F1/D-F2 被动触发型主动服务不同，D-F3 是**节律驱动的预测性服务**。

**不做什么**：不在 B-F3 节律未建立时运行；不推送用户未曾有过行为的新服务（冷推荐）；不在置信度 <70% 的节律上触发。

---

## 2. 触发逻辑

```
B-F3 节律库
    ↓
当前时间/位置/情境 → 匹配节律（相似度 >70%）
    ↓
服务类型判断（导航/餐饮/充电/购物/出行）
    ↓
服务可用性确认（MCP API 可达）
    ↓
打扰预算检查（≤3次/周，距上次触发>30min）
    ↓
驾驶状态检查（等灯/驻车 才触发，行驶中不弹窗）
    ↓
生成个性化话术（Qwen，1句话，含具体选项）
    ↓
TTS 播报 + 卡片展示（可选接受/拒绝/稍后）
```

---

## 3. 服务类型与对应 MCP

| 节律类型 | 触发示例 | 关联 MCP |
|----------|----------|----------|
| 通勤咖啡 | 工作日8:10，路过常去星巴克 | 瑞幸/星巴克 MCP |
| 下班加油 | 周五17:30，电量<40% | 充电/加油 MCP |
| 午餐外卖 | 工作日11:45，停在公司附近 | 美团/饿了么 MCP |
| 周末健身 | 周六9:00，导航到健身房路上 | 健身预约 MCP（V2） |
| 回家前预热 | 接近家附近<5min | 米家/鸿蒙智家 MCP |

---

## 4. 话术生成原则

输入给 LLM 的话术生成 Prompt 约束：
- 最多25字，不超过1句
- 必须含具体选项（"点一杯中杯拿铁（¥29）"而非"要喝咖啡吗"）
- 克制语气，询问而非推荐（"你平时这时候会…，要今天也…吗？"）
- 不说"我发现你""我注意到你"（用户反馈"被监视感强"）
- 改为"像往常一样…" / "按惯例…"（暗示AI学到了，但不明说）

---

## 5. 用户反馈与学习

| 用户操作 | 系统响应 |
|----------|----------|
| 接受（确认执行） | 节律权重 +0.05，该场景服务成功率记录 |
| 拒绝（"不用了"） | 节律权重 -0.02，当天不再触发同类节律 |
| 忽略（无响应10s） | 视为拒绝，权重 -0.01 |
| 连续3次拒绝 | 该节律对应的服务类型静默30天 |
| 用户说"以后别推了" | 永久关闭该类别，可在设置中恢复 |

---

## 6. 失败路径与降级

| 失败情况 | 降级策略 |
|----------|----------|
| B-F3 节律未建立 | 功能不激活，本功能完全静默 |
| MCP 服务不可用 | 不触发（宁可漏推，不推无效服务） |
| 网络弱（无法调用MCP） | 不触发 |
| 话术生成超时（>2s） | 使用预置模板话术（降质但不崩溃） |
| 预算已用完（≥3次/周） | 不触发，等下周重置 |
| 驾驶中（非等灯） | 延迟到下次停车机会触发 |

---

## 7. SLO / 验收指标

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 用户接受率 | >30%（触发后接受执行的比例） | 接受事件计数/总触发次数 |
| 用户拒绝率 | <20% | 拒绝事件计数/总触发次数 |
| 话术生成延迟 | P95 <2s | 话术生成时间戳埋点 |
| 预算合规率 | 100%（≤3次/周，≤1次/天） | 预算计数器校验 |
| 触发时机合规率 | 100%（不在行驶中触发） | 驾驶状态埋点 |

**放弃条件**：接受率 <30% 且拒绝率 >20% 连续4周，说明节律质量不够，不上线。

---

## 8. 自我反驳（六关）

**关1 - 价值关**：这和普通的"推荐"有什么区别？
→ 核心差异：不是基于"其他用户也买了"的协同过滤，而是基于"这个用户自己的历史行为节律"。触发条件是"你自己昨天也这样做了"，转化率理论上远高于冷推荐（内部预估接受率30%+ vs. 通用推荐<5%）。

**关2 - 骚扰关**：节律服务会不会变成另一种广告推送？
→ 关键约束：① 只推用户有过真实行为的服务（不推新品类）；② 每周≤3次硬上限；③ 连续3次拒绝→永久关闭该类别；④ 节律基于用户自己的行为，没有"商业推荐"成分（MCP选择由性价比算法决定，不由广告费决定）。

**关3 - 节律质量关**：B-F3 识别的节律不准确，D-F3 推的就是错的？
→ B-F3 置信度 <70% 的节律，D-F3 不使用。D-F3 本身的接受/拒绝反馈又会回流修正 B-F3 权重，形成闭环。启动阶段（节律刚建立）触发频率低，随准确率提升逐步增加。

**关4 - 时机关**：推服务的时机不对比不推更坏（如高速上说"要点咖啡吗？"）？
→ 时机硬约束：① 车速>30km/h 时不触发（行驶中）；② 只在等灯/驻车/拥堵（车速<10km/h 持续>30s）时触发；③ 紧急情境（碰撞预警/疲劳告警后30min内）不触发。

**关5 - 商业关**：节律服务的商业模式对用户有没有利益冲突？
→ V1 MCP 路由由性价比算法决定（评分+价格+时效），不收广告费。保证推的是"最符合用户利益的选项"，而不是"付钱最多的商家"。这是与其他平台的核心差异，必须在用户协议中明确承诺并技术上强制执行。

**关6 - 范围关**：D-F3 和 D-F1/D-F2（基于情境的主动服务）的边界？
→ D-F1/D-F2：实时情境触发（天气变了/电量低了），适合任何用户（不依赖历史）。D-F3：历史节律触发（用户有这个习惯），需要≥14天数据。两类预算独立计数，互不干扰。同一时刻只触发优先级最高的一个（D-F1 安全类优先于D-F3 习惯类）。

# PRD 04 §5.9 · S-F3 场景模式识别与切换（补全版）

> 所属：PRD 04《状态管理·记忆系统·模式切换》
> 功能 ID：S-F3 · 版本：V1（Must）
> 补全日期：2026-05-12

---

## 1. 功能定义

**做什么**：综合多模态情境信号，自动识别当前驾驶场景（5种），并切换对应的AI行为策略（TTS语气/主动服务预算/HMI布局/信息优先级）。

**不做什么**：不做6+种场景（超过5种用户感知不到差异）；不做跨用户场景推荐；不在场景切换时弹窗告知用户（静默切换）。

---

## 2. 五种场景定义

| 场景 | 触发条件（按优先级） | AI 行为策略 |
|------|---------------------|------------|
| **通勤模式** | 工作日 + 出发/目的地是家/公司 + 时间在通勤窗口 | 简洁高效，主动推送路况/天气/日程提醒 |
| **家庭模式** | OMS 检测到儿童 OR DMS 检测到多人 + 家庭成员账号在线 | 内容过滤开启，语气温和，音量自动降低 |
| **旅途模式** | 行驶距离 >50km + 非常用路线 OR 目的地非常用地点 | 主动提供途经信息/充电桩/服务区，语气轻松 |
| **关怀模式** | DMS 疲劳度 ≥2 OR 当前时段 23:00-06:00 OR 连续驾驶 >3h | 关怀优先，主动建议休息，语速降低，音乐自动切舒缓 |
| **驻车模式** | 发动机关闭 OR 挂P档 + 车速=0 | 全功能开放，无驾驶安全限制，娱乐/工作场景 |

---

## 3. 场景识别流程

```
实时信号采集（每30s刷新）：
  - DMS：疲劳度/乘员信息
  - OMS：儿童检测/人数
  - GPS：位置/距离/路线
  - CAN：车速/档位/发动机状态
  - 时间：系统时钟
  - 历史：B-F2 常用路线/目的地
      ↓
规则引擎（优先级决策树，<1ms）
      ↓
场景候选（可能多个匹配）
      ↓
优先级仲裁：安全关怀 > 家庭 > 通勤 > 旅途 > 驻车
      ↓
置信度检查（>80% 才切换）
      ↓
场景状态机更新 + 广播 mode_changed 事件
```

**冷却期**：场景切换后 5min 内不再切换（防止信号抖动导致频繁切换）。

---

## 4. 场景切换执行

切换时各模块响应：

| 模块 | 响应行为 | 延迟要求 |
|------|----------|----------|
| TTS | 更新语气参数（语速/音调/停顿风格） | <100ms |
| HMI | 切换卡片优先级队列配置 | <500ms |
| 主动服务引擎 | 更新本场景服务预算和触发规则 | <100ms |
| 内容过滤 | 开启/关闭家庭模式过滤器 | <200ms |
| 音乐推荐 | 更新推荐策略（关怀模式→舒缓，旅途→活跃） | 下一曲时生效 |

切换不打断当前进行中的任务（任务完成后才应用新策略）。

---

## 5. 失败路径与降级

| 失败情况 | 降级策略 |
|----------|----------|
| 信号缺失（GPS/DMS无数据） | 用可用信号子集识别，置信度降低，无法达到80%时保持当前场景 |
| 规则引擎崩溃 | 降级到默认模式（通勤模式），保证基础可用 |
| 场景识别置信度<80% | 不切换，保持当前场景，继续收集信号 |
| OMS儿童误检（B端BOM低配无OMS） | 家庭模式由账号登录状态判断（有儿童账号→家庭模式） |
| 场景切换事件广播失败 | 各模块保持原有策略，下次心跳（30s）重试广播 |

---

## 6. SLO / 验收指标

| 指标 | 目标 | 测量方法 |
|------|------|----------|
| 场景识别准确率 | P70 >75% | A/B测试 + 用户反馈（"当前模式对吗？"） |
| 场景切换时间 | P99 <500ms（从触发到所有模块响应完毕） | 事件广播时间戳埋点 |
| 误切换率 | <15%（用户手动修改模式的比例） | 手动切换事件计数/总切换次数 |
| 驻车模式识别准确率 | >99%（安全要求，不能在行驶中进入驻车模式） | 档位+车速双重校验 |
| 关怀模式疲劳检测触发率 | 与DMS准确率挂钩（DMS P95 >90%） | 见DMS SLO |

**放弃条件**：场景识别准确率 <70% 且用户手动切换率 >25%（说明自动识别比手动更差）。

---

## 7. 用户控制

用户可在设置中：
- 查看当前场景和切换原因（"因为检测到儿童，进入家庭模式"）
- 手动锁定场景（"本次行程一直用XX模式"）
- 关闭自动场景切换（退化为固定策略）

---

## 8. 自我反驳（六关）

**关1 - 价值关**：5种场景用户真的感知到差异吗？
→ 用户不需要"感知到场景切换"，但应该感知到"AI刚才说的话恰到好处"。场景策略的价值体现在话术/推荐/音乐是否符合当下，而不是UI上的场景标签。

**关2 - 准确关**：通勤场景如何区分"去公司"和"去不常去的地方"？
→ B-F2 事件记忆记录常用目的地（出现3次以上的地点标记为"常用"）。非常用地点 + 距离>50km = 旅途模式。阈值可在数据积累后调整。

**关3 - 冲突关**：关怀模式（疲劳）和家庭模式（有儿童）同时触发怎么办？
→ 安全类优先：关怀模式 > 家庭模式。但家庭模式的内容过滤（不播成人内容）和关怀模式的休息建议可以同时生效（不互斥的行为策略叠加）。

**关4 - 干扰关**：场景频繁切换（如停车等人时来回进出驻车模式）会不会很烦？
→ 5min冷却期 + 置信度>80%才切换。等人场景：车速=0但可能发动机未关，不会误入驻车模式。驻车模式需要"发动机关闭 OR 挂P档"，设计上已处理。

**关5 - 低配车型关**：没有DMS/OMS的车型怎么识别家庭/关怀模式？
→ DMS缺失：疲劳检测降级到时间规则（23:00-06:00强制进关怀模式）+ 连续驾驶时长（CAN总线读取，>3h触发）。OMS缺失：家庭模式由账号判断（有儿童账号在线）。覆盖率降低但不崩溃。

**关6 - 隐私关**：场景识别用到的DMS/位置数据，是否会泄露用户行为规律？
→ 场景识别纯端侧计算，结果（"当前：通勤模式"）不上云，不落盘（内存状态）。位置数据仅用于路线匹配（与本地常用地点比较），不发送原始GPS坐标给云端。
