## §0 五大架构必答题

### A. 多 Agent 架构

| 问题 | 决策 |
|------|------|
| Agent 间通信协议 | **云端 gRPC / 端侧 in-process 调用**。不用消息队列（Broker 单点 + 延迟不可控）。Mother-bot → Sub-bot 走 Function Call 封装，返回值即工具结果。 |
| 主 Agent 模型 | **路由决策：Qwen2.5-7B（云端）/ 复杂任务规划：Qwen2.5-72B（云端）/ 端侧弱网降级：规则引擎（V1不跑端侧LLM）**。专项 Agent：车控/状态用规则引擎；导航/娱乐/信息用 Qwen2.5-7B。 |
| 状态共享分界 | **共享黑板（Shared Blackboard）**：Mother-bot 持全局上下文（Redis/SQLite），Sub-bot 只持本次工具调用临时状态，结果通过返回值传回。Sub-bot 崩溃不污染全局状态。 |
| Sub-bot 挂了怎么办 | 三级：① 超时3s→重试1次；② 重试失败→规则引擎降级；③ 规则也失败→TTS 告知用户。连续失败3次→熔断5min。 |

### B. 端云分工

| 能力 | V1 决策 |
|------|---------|
| KWS | 端侧，<10M 模型，不依赖网络 |
| ASR | 端侧 FunASR Paraformer-small（专为中文优化，CER ≈4-5%，~80MB ONNX INT8），云端 Paraformer-large 质量修正 |
| NLU | L1 关键词<5ms，L2 BERT-tiny端侧<100ms，L3 Qwen云端长尾 |
| TTS | 端侧 CosyVoice-lite（80M）首字出声，复杂情感走云端 |
| 大模型 | V1 纯云端，V1.5 评估端侧7B INT4 |
| 弱网降级 | L1全功能→L2核心功能（端侧NLU+规则）→L3安全兜底（规则引擎+离线地图） |

**端侧算力分配（骁龙8295，NPU 30 TOPS）：**
- KWS 0.5T / ASR 4T / NLU 2T / DMS 8T / OMS 4T / TTS 3T / 形象渲染 4T / 余量 4.5T

### C. 多音区与 BOM

- **麦克风：V1 选 6 麦**（主驾2+副驾2+后排2），BOM 增约 ¥60，方向准确率 >95%。前端 XMOS XU316 做 AEC/AGC/NS。
- **DMS：红外单目 OV9282（¥45）+ IR LED（¥8）**，不上双目（+¥60 但收益有限）。
- **OMS：广角 RGB 120° 共享1个（副驾+后排），¥35**。总增 BOM ¥88-120。
- **车外麦：V1 不加**。C-F3 紧急车辆识别用车内麦低频特征检测（准确率约85%），V2 再评估。

### D. 数据与合规

- **DMS/OMS**：只存结构化状态（`{fatigue_level, distraction, emotion, passenger_count}`），原始帧内存处理后即丢，绝不落盘/上云。
- **用户画像**：端侧 AES-256 加密本地存储 + 云端完整画像（分库分表）。上车触发增量 pull（<500ms，<50KB）。家庭多账号各自 namespace 隔离。
- **卖车清除**：App 触发工作流 → 云端删除 user_id 所有数据 → 车端加密密钥销毁（物理级不可恢复）→ 生成清除证明。

### E. MCP 路由

- **注册表**：Redis Hash（热）+ PostgreSQL（持久化）双写，端侧 SQLite 缓存 Top-20 MCP，<100ms 热更新。
- **性价比数据**：价格/评分 TTL=min(MCP声明freshness, 5min)，距离实时查询，时效用历史P50修正。
- **探索机制**：V1 ε-greedy（**5%固定探索**，车机场景比 App 更保守，10% 对习惯固定品牌的用户过于激进），V1.5 升 Thompson Sampling。每品类每用户每天最多1次探索。
- **审核工具链**：CI 4关（schema校验→功能测试→性能压测→隐私扫描）→人工审核（2工作日）→灰度5%（7天）→全量。

## 节点① 输入层（14条）

### 麦克风

**①-1 麦克风阵列方案**
决策：6 麦（主驾 A 柱×2 + 副驾 A 柱×2 + 顶篷后排×2），间距 12cm，朝向驾驶舱内侧倾斜15°。BOM 增约 ¥60/套，满足 >95% 方向估计准确率。V2 升 8 麦（支持3D声场）。

**①-2 车外麦克风**
V1 不加。用车内麦 500Hz 以下低频特征识别紧急车辆（准确率约85%，满足 C-F3 的 P95 要求）。V2 若做窗外噪音分析再补（IP67 防水，BOM +¥30）。

**①-3 采样率/位深**
16kHz / 16bit，AEC 回声消除 + AGC 自动增益 + NS 降噪全由 XMOS XU316 前端芯片硬件处理，CPU 零负担。

### 摄像头

**①-4 DMS 摄像头**
红外单目 1080p 30fps（豪威 OV9282，¥45/套）+ IR 补光 LED（¥8）。不上双目（成本增 ¥60，疲劳检测提升 <3%，不值得）。供应商：豪威/海康均可，V1 优先豪威。

**①-5 OMS 摄像头**
广角 RGB 120° 1080p（1个）覆盖副驾+后排。如后排有遮挡严重车型（SUV 第三排），可选2个独立摄像头（BOM +¥35），由整车厂按车型决定。

**①-6 车内手势摄像头**
复用 DMS 摄像头（主驾侧），不单独加。手势识别要求摄像头覆盖方向盘区域，OV9282 广角足够。

### 其他传感器

**①-7 方向盘握力传感器**
V1 不加（BOM +¥80，且与 DMS 视觉疲劳检测高度重叠）。DMS 多模态融合只用视觉+音频，握力作为 V2 选配。

**①-8 座椅压力传感器**
V1 不加独立传感器。OMS 广角摄像头做遗留物检测准确率已达 PRD 要求。车型有座椅传感器（部分豪华车型已有），可 CAN 总线接入作辅助信号，零 BOM 增量。

**①-9 振动传感器**
复用车身已有加速度计（CAN 总线读取），不加独立振动传感器。玻璃破碎声音特征（C-F3）用车内麦高频段识别，加速度计辅助置信度提升。

**①-10 温度传感器**
复用车内现有温度传感器（空调系统已标配），CAN 总线读取，用于 OMS 遗留物检测高温告警兜底。

**①-11 车速/方向盘转角/转向灯信号**
全部从 CAN 总线读取（ISO 15765-2），端侧 CAN 解析器模块处理，延迟 <10ms。用于手势识别过滤（转角 >30° 时屏蔽手势识别）。

### 触控/物理按键

**①-12 触摸屏多屏方案**
仪表+中控统一走 Android Automotive OS 触控协议（AOSP 标准）。HUD 无触控。副驾/后排屏通过内网总线与主系统通信（WebSocket）。

**①-13 方向盘按键**
KWS 唤醒兜底入口，映射到车控 Sub-bot 的 WAKE 事件，优先级最高（高于语音唤醒）。物理按键状态通过 CAN 总线同步。

**①-14 算力预算总览**
见 §0-B 端侧算力分配表。所有端侧模型总算力目标 <26 TOPS（NPU），余量 >4 TOPS 备用，防止多模型并发时超限。

## 节点② 感知层（27条）

### 语音感知

**②-1 端侧 KWS 模型**
自研轻量模型（基于 TDT/CRNN，参数 <3M），训练数据：内部采集唤醒词10万条+噪音增强。不用思必驰/讯飞商用 SDK（授权费高 + 黑盒无法迭代）。V1 唤醒词1个，V2 支持用户自定义。唤醒误触率目标 <1次/小时。

**②-2 多设备唤醒仲裁**
仲裁在端侧车机层。手机和车机同时响应时：① 车内有声音来自车机方向（麦阵列判断）→ 车机赢；② 蓝牙检测到手机在车内 → 车机赢；③ 距离最近设备优先。延迟 <50ms。

**②-3 Overlap buffer**
唤醒词检测确认瞬间，回溯缓冲区500ms的音频帧拼入 ASR 输入（环形缓冲区，16kHz×500ms=8000 采样点，内存 16KB）。丢包风险：环形缓冲区覆盖式写入，不会丢失；错位风险：用时间戳对齐保证。

**②-4 声源方向检测**
6 麦阵列，基于 GCC-PHAT 算法做 TDOA 时延估计，方向估计精度 ±15°（车内场景足够区分主驾/副驾/后排）。最少4麦可达95%准确率，6麦冗余更鲁棒。

**②-5 端侧声纹识别**
在 KWS 层做轻量声纹粗分类（主驾/非主驾 2 分类，不做精确身份识别）。模型 <5M，基于 d-vector。精确身份识别（账号绑定）由 DMS 人脸识别完成。

**②-6 端侧 ASR 引擎**
FunASR Paraformer-small（专为中文设计，参数 ~80M）ONNX INT8 量化，骁龙8295 NPU 推理，首字延迟 ≤150ms（P95）。中文 CER ≈4-5%（弱网独立运行时），远优于 Whisper-small 的 12-18%（多语言模型，中文权重不足）。流式 VAD 切句，每句结果增量返回。模型大小：量化后 ~50MB，常驻内存。

**②-7 端云协同 ASR**
正常网络：端侧 ASR 输出初始结果（低延迟），云端 Paraformer 异步修正（结果合并，最终采用云端）。弱网：端侧独立可用，CER ≈4-5%（Paraformer-small 中文专项优化）。端云结果合并策略：词级别置信度加权选择（云端权重0.8，端侧0.2），保证低延迟同时提升准确率。

**②-8 方言/口音支持**
V1 支持普通话+轻度口音（训练数据含8个地区口音）。七大方言区：粤语/川语/闽语/吴语 列入 V1.5 计划，V1 端侧 ASR 识别准确率对方言用户 CER 约15-20%，降级提示"没有听清楚，请说普通话"。

**②-9 拒识模型**
端侧轻量二分类（规则+BERT-tiny），区分「给AI的指令」vs「不需要识别的对话（电话/家庭闲聊）」。置信度 <0.5 → 拒识，不上报 ASR 结果，不记录原始音频。规则兜底：检测到通话状态（蓝牙HFP激活）→ 强制拒识。

**②-10 VAD（语音活动检测）**
Silero VAD（端侧，1.8M 参数），检测延迟 ≤30ms（P95）。全双工监听：VAD 常驻，KWS 仅在 VAD 触发后激活，降低 KWS 算力消耗约60%。算力代价：~0.3 TOPS。

**②-11 波束成形+多音区分离**
采用 MVDR（最小方差无失真响应）自适应波束成形，XMOS XU316 硬件实现，CPU零负担。多人并发 ASR：每个音区独立 ASR 实例（最多3路并行：主驾/副驾/后排），算力增加 ~8 TOPS，不超预算。不用 Cerence/Nuance 商用（授权费高）。

**②-12 唤醒仲裁**
主驾和副驾同时唤醒：① 主驾优先（安全驾驶优先级）；② 主驾未说话5s → 副驾接管。仲裁规则在母bot层（状态机），不在 KWS 层。仲裁延迟 <100ms。

### 视觉感知

**②-13 DMS 状态分类器**
4路分类（疲劳/分心/情绪/身份），单模型多任务输出（MobileNetV3-Small 改造，~2M 参数，NPU INT8）。端侧推理 <30ms/帧（30fps），分配算力 8 TOPS。量化方案：INT8 精度损失 <2%。

**②-14 OMS 状态分类器**
人数/年龄段/睡眠/情绪+遗留物检测，单模型（YOLOv8-nano 改造，~4M 参数）。多模态融合：视觉（主）+ 座椅压力（辅，如有）+ 温度传感器（异常兜底）。

**②-15 多模态融合（DMS）**
视觉（权重0.6）+ 音频/呼吸特征（权重0.3）+ 方向盘握力（权重0.1，如有）加权投票。规则优先于ML：血氧传感器（如有）触发直接报警，不经置信度模型。

**②-16 多模态融合（OMS）**
视觉（主）+ 座椅压力（辅）+ 温度传感器（兜底）三重保险。遗留物检测：视觉检测到后排有物体/婴儿 → 置信度 >0.8 时触发告警，同时检查车内温度（>35°C 升级P0）。

**②-17 手势识别模型**
骨骼点 + 时序分类（MediaPipe Hands + 轻量 LSTM，~3M）。端侧推理 <50ms/帧。V1 识别5种手势（音量+/-/接听/拒接/OK）。驾驶状态过滤：方向盘转角 >30° 或车速 >60km/h 时禁用。

**②-18 驾驶状态过滤**
CAN 总线读取车速/档位/方向盘转角，端侧状态机维护「驾驶中」标志。过滤规则：车速>60km/h → 禁用手势；转角>30° → 禁用手势（暗示正在转向）；驻车 → 全开放。

### 声音事件感知

**②-19 声音事件分类器**
多标签分类（MobileNet-Audio，~5M），识别：救护车/警车/消防车/鸣笛/玻璃破碎/儿童哭声/碰撞声共7类。端侧常驻，算力 ~1 TOPS。

**②-20 训练数据来源**
70% 公开数据集（ESC-50/FreeSound）+ 30% 自采集（国内各地警笛差异：上海/北京/广州各地警笛频率略有差异，需要覆盖）。V1 准确率目标：P95 >85%，V2 通过车队采集持续提升。

**②-21 声音+振动融合**
玻璃破碎：声音高频特征（5-8kHz 瞬态冲击）+ 车身加速度计（>2g 突变）双重确认，减少误报。单声音触发置信度 <0.7 时必须等加速度计确认。

### 外部情境感知

**②-22 GPS+地图融合**
V1 普通精度地图（高德）+ 车载 GPS（精度 ±5m）。不上高精地图（V1 不做自动驾驶辅助，BOM 增量不值得）。端侧离线地图包（高德离线 SDK），云端实时 POI 查询。

**②-23 天气 API 接入**
主：彩云天气 API（分钟级精度，覆盖全国）；备：和风天气。刷新频率：每15min 拉取一次，极端天气预警实时推送（Webhook）。端侧缓存最近1h数据，弱网可用。

**②-24 路况 API 接入**
高德实时路况（主）+ 百度路况（备），更新频率 5min。端侧缓存当前路段路况（10km范围），云端全量查询。

**②-25 车辆状态采集**
CAN 总线统一入口，端侧 CAN 解析器读取：电量/油量/胎压/续航/档位/车速/空调状态。解析延迟 <10ms，状态机持续维护最新值。

**②-26 时间感知**
本地系统时钟 + NTP 校准（每小时）。节假日日历：预置全年日历（OTA 年度更新），农历转换本地计算，不依赖网络。

**②-27 POI 数据库**
车外 POI 匹配用于视觉指代消歧（"去那个麦当劳"）。端侧轻量 POI 索引（1km范围内，SQLite）+ 云端全量查询。数据源：高德 POI API，本地缓存按区域切片（每城区约50MB），每周 OTA 增量更新。

## 节点③ 理解层（12条）

### NLU

**③-1 NLU 引擎选型**
两级架构：端侧轻量 NLU（BERT-tiny 蒸馏，~15M，覆盖Top-200高频意图）+ 云端大模型 NLU（Qwen2.5-7B，覆盖长尾复杂意图）。
分工边界：意图置信度 >0.85 且属于 Top-200 意图 → 端侧直接执行；其余 → 上云。
V1 端侧覆盖目标 ≥80% 请求量，云端处理 ≤20%。

**③-2 意图识别三级路由（L1/L2/L3）**
- L1 关键词规则（<5ms）：覆盖约100个高频固定短语（"导航回家""调低音量""下一首"），正则+关键词表，端侧，无模型推理。
- L2 端侧 BERT-tiny（<100ms）：覆盖 Top-200 意图，包括变体表达，置信度阈值0.7。
- L3 云端 Qwen2.5-7B（<1.5s）：复杂跨域意图、长句、指代消歧。
升级判据：L1 置信度 <0.9 → L2；L2 置信度 <0.7 → L3；L3 搞不定 → 反问用户。

**③-3 意图分类标准**
单轮意图：单一领域、单一动作、槽位完整（直接执行）。
多轮意图：跨轮次信息补全（"导航到…" "去哪？" "上次那家火锅"）。
并行意图：一句话含多个独立意图（"打开空调同时播放音乐"），由 Mother-bot 分解并行调度。

**③-4 槽位填充**
实体识别：端侧 NER 模型（基于 BERT-tiny，~10M），识别 POI/时间/人名/数字/车控参数等。
槽位缺失补全：L1 规则问询（固定问题）+ L3 LLM 上下文推断（复杂场景）。
补全策略：缺1个槽 → 直接问；缺2个+ → 优先问最关键槽，其余推断。

**③-5 多意图并行解析**
DAG 拆解：Mother-bot 用 LLM 将复杂意图分解为 DAG（有向无环图），并行调用多个 Sub-bot。
实现：Python asyncio 并发调用，超时保护3s（任何一个 Sub-bot 超时不阻塞其他）。
聚合：所有 Sub-bot 结果收集后统一播报（TTS串行播报各结果）。

### 上下文与多轮

**③-6 对话上下文存储**
最近5轮：结构化 JSON 存端侧 SQLite（{turn_id, intent, slots, result, timestamp}），单条 <1KB。
云端：最近20轮压缩后同步（用于跨设备连续对话），使用 LLM 生成摘要压缩（每5轮 → 1条摘要）。
上下文窗口：云端 LLM 调用时传入最近5轮原始 + 更早的摘要，控制 token 数 <2000。

**③-7 指代消歧融合**
三路信号：视线方向（DMS眼动，±20°精度）+ 手势指向（手势识别模型）+ 对话上下文（NLU历史槽位）。
融合：加权投票（上下文0.5，视线0.3，手势0.2），置信度 >0.7 自动消歧，<0.7 反问。
延迟：视线/手势实时采集，融合计算 <50ms。

**③-8 打断恢复上下文保留**
打断发生时：① 记录当前 TTS 播报位置（字符偏移量）和剩余文本；② 暂停播报，保存到端侧内存（TTL=5min）；③ 处理新意图。
恢复触发："刚才说到哪了" / 无新意图5s后自动询问是否继续。
数据结构：`{interrupted_text, offset, intent_context, timestamp}`。

**③-9 跨车型视线校准**
各车型屏幕位置（HUD/中控/仪表/后排）预存配置文件（JSON），OTA 下发。
视线校准：首次使用引导用户看5个校准点（<30s），存端侧。跨车型迁移：云端存用户校准配置，换车时 pull。

**③-10 错误样本回流**
用户纠正（"不是，我说的是…"）触发采集：原始 ASR 文本 + 错误 NLU 结果 + 正确意图（用户纠正内容）。
脱敏处理：PII 字段（人名/地址）哈希化，不保留原始文本。
上云：脱敏后异步上报，用于月度模型迭代。用户可关闭（设置页）。

### 情境推理

**③-11 情境融合规则**
信号优先级：DMS状态（安全类，最高）> 车辆状态（电量/速度）> 位置/时间 > 天气 > 用户历史偏好。
融合实现：端侧规则引擎（决策树，<1ms），不用 LLM（延迟不可接受）。
复杂情境（多信号组合）：端侧规则覆盖不了时，异步请求云端 LLM 推断（不阻塞主流程）。

**③-12 场景识别**
V1 五种场景：通勤/家庭/旅途/关怀/驻车。
判定规则（按优先级）：① DMS 检测到儿童 → 家庭模式；② 出发时间+目的地是公司/家 → 通勤模式；③ 行驶距离 >50km + 非常用路线 → 旅途模式；④ DMS 疲劳 + 夜间 → 关怀模式；⑤ 发动机关闭 → 驻车模式。
置信度 <0.8 保持当前模式（不切换），切换后5min冷却期。

## 节点④ 决策层（30条）

### 多 Agent 架构

**④-1 Agent 通信协议**
已在 §0-A 回答。补充：端侧 Agent 间 in-process 直调（零延迟）；云端 Agent 间 gRPC（内网 <5ms）；端云之间 WebSocket 长连接（保持心跳）。

**④-2 状态同步**
Mother-bot 维护全局任务状态（Redis，TTL=30min）。Sub-bot 只读全局状态，不直接写。写回机制：Sub-bot 执行完毕 → 返回结果 → Mother-bot 更新全局状态。冲突解决：乐观锁（version号），冲突时 Mother-bot 仲裁。

**④-3 失败兜底**
已在 §0-A 回答。补充降级路径细节：
- 车控 Sub-bot 挂了 → 端侧规则引擎直接执行（不经 LLM）
- 导航 Sub-bot 挂了 → 用缓存最近一次路线 + 告知用户"导航可能不是最优"
- 娱乐 Sub-bot 挂了 → 本地音乐库随机播放
- 通信 Sub-bot 挂了 → 提示"通信功能暂不可用，请手动操作"
- 信息 Sub-bot 挂了 → 端侧知识库兜底（覆盖常见查询）

**④-4 性能开销**
Mother-bot 路由决策（L1）：<5ms（规则查表）；L2：<100ms（端侧模型）；L3：<500ms（云端，含网络）。
多 Agent 并行调用额外开销：<50ms（asyncio调度开销）。
总端到端延迟预算：端侧路径 <800ms（ASR+NLU+执行+TTS首字），云端路径 <2.5s。

**④-5 LLM 模型选型**
Mother-bot 意图路由：**Qwen2.5-7B**（云端，分类任务，7B 准确率与 72B 相差 <3%，TTFT 更低）。复杂任务规划（DAG 拆解/多步推理）：**Qwen2.5-72B**（仅 L3 长尾路径触发）。端侧降级：规则引擎（V1 不跑端侧 LLM）。
专项 Sub-bot：车控/状态用规则引擎（无LLM）；其余用 Qwen2.5-7B。
备选：DeepSeek-V2.5（如果 Qwen 不稳定或合规有问题）。

**④-6 Agent 注册/发现**
静态注册（V1）：启动时加载配置文件（YAML），定义每个 Sub-bot 的能力、优先级、降级路径。不做动态注册（过于复杂，V1 5个 Sub-bot 固定不变）。
V2 支持动态注册（新增 Sub-bot 不需要重启）：基于服务注册中心（Consul/Nacos）。

### 任务规划

**④-7 任务图（DAG）表示**
用状态机 + DAG 混合：简单单步任务用状态机（CREATED→RUNNING→DONE），多步复杂任务用 DAG（每个节点是一个工具调用，边是依赖关系）。
存储：端侧 SQLite（任务持久化，防闪退丢失），云端 Redis（跨设备同步）。
中途修改支持：DAG 节点可标记为 SKIP/REPLACE，不重跑已完成节点。

**④-8 流式规划+执行**
LLM 流式输出（SSE）→ 边输出边解析工具调用 → 工具调用可并行启动（不等 LLM 输出完毕）。
实现：Python asyncio 生产者-消费者模型，LLM 流为生产者，工具调用为消费者。
协调：已启动的工具调用结果回注 LLM 上下文（function result），触发下一步规划。

**④-9 任务持久化**
端侧 SQLite 持久化所有任务状态（每次状态变更即写盘，WAL 模式保证不丢失）。
云端 Redis 同步（TTL=30min，用于跨设备恢复）。
任务粒度：每个 Sub-bot 调用记为一个任务节点，有独立 ID 和状态。

**④-10 任务中途修改**
用户说"取消"/"换成XX"/"先做XX再做YY"时：Mother-bot 接收新意图 → 标记当前运行节点状态为 CANCELLED → 插入新节点 → 重新规划 DAG。
已完成节点不重跑（幂等保证）。车控类操作：先撤销已执行指令（如已开窗 → 先关窗），再执行新指令。

### 工具调用

**④-11 工具协议**
V1 使用 OpenAI Function Calling 格式（JSON Schema 定义工具能力），兼容 MCP 协议（两者 schema 互通）。
车载特殊要求：① 超时硬限制（每个工具调用 ≤3s）；② 车控类工具调用必须端侧二次校验（不信任云端结果）；③ 工具调用结果必须带置信度字段。

**④-12 工具注册表**
按域分类，存 Redis（云端）+ SQLite（端侧缓存）：
- navigation: {高德导航, 百度导航}
- music: {QQ音乐, 网易云, 酷狗}
- vehicle: {空调控制, 车窗控制, 座椅控制}
- payment: {支付宝, 微信支付}
- food: {美团外卖, 饿了么}
- home: {米家, 鸿蒙智家}
工具 schema 含：capability, latency_sla, fallback, privacy_level 字段。

**④-13 工具调用安全沙箱**
第三方工具（外卖/支付/家居）运行在隔离 WebWorker / Docker 容器（云端）。
Prompt 注入防御：工具返回内容在注入 LLM 前经过扫描（规则+轻量分类模型），识别 "忽略之前指令" 类攻击，命中则丢弃工具结果并告警。
权限白名单：每个工具只能访问其声明的权限范围，超出权限的调用直接拒绝。

**④-14 工具调用超时熔断**
超时阈值：导航/音乐 2s，外卖/支付 3s，车控 1s（端侧规则引擎兜底）。
超时处理：① 超时→取消调用→走备选工具；② 备选也超时→告知用户"服务暂时不可用"。
熔断：同一工具连续失败3次 → 熔断5min，期间跳过直接走备选。

### MCP 路由（生活服务 Agent 内部）

**④-15 静态注册表存储**
见 §0-E 回答。

**④-16 动态路由计算位置**
在生活服务 Sub-bot 内部计算（不独立服务，减少网络跳数）。计算纯内存操作（Redis查权重→计算性价比→排序），延迟 <50ms。
仅在规则变更（新MCP接入/权重更新）时需要刷新缓存，不影响运行时性能。

**④-17 意图识别三级实现**
见 §0-E 和 ③-2。补充：L1 关键词库由产品运营维护（Excel → 自动构建 Trie 树），每周 OTA 更新。L2 端侧 NLU 模型每月随主模型一起 OTA。

**④-18 优先级规则引擎**
可配置规则引擎（Drools 轻量替代：自研 JSON 规则文件）。规则格式：`{condition: {intent, user_pref, context}, action: {mcp_id, priority}}`。热更新：规则文件云端下发，端侧 SQLite 存储，<100ms 生效，不需要重启。

**④-19 性价比计算**
实时从 MCP 拉取价格/评分，缓存 TTL=5min。距离实时计算（高德 API）。时效用历史 P50 修正。公式：Score = 评分×0.4 - 价格/参考价×0.4 + 时效/参考时效×0.2。评分 <4.0 的 MCP 不进候选池。

**④-20 品类权重配置**
配置中心（Nacos）热更新，格式 JSON，每品类独立权重配置。V1 初始权重：评分0.4/价格0.4/时效0.2（咖啡）；打车：评分0.3/价格0.2/时效0.5（时效最重要）。

**④-21 权重更新机制**
用户接受推荐 → 权重 +0.05；用户拒绝/手动选其他 → 权重 -0.02；30天无交互 → 权重指数衰减（×0.9/天直到0.1基准）。批量更新（T+1），不做实时更新（防止噪声干扰）。

**④-22 MCP 接入校验流水线**
见 §0-E 回答。

**④-23 熔断器（Circuit Breaker）**
三态状态机：CLOSED（正常）→ OPEN（熔断，连续失败N次）→ HALF_OPEN（试探恢复）。
参数：N=3次，熔断时长5min，HALF_OPEN 期间放行1次请求验证。
实现：端侧纯 Python 状态机，<100行代码，不引入重量级框架。

**④-24 健康检查探针**
主动探测：每60s 发送轻量 ping 请求（仅检查 API 可达性，不消耗配额）。被动观察：每次实际调用后更新成功率统计（滑动窗口5min）。两者结合：ping 通但成功率 <80% 也触发降级。

**④-25 探索机制**
见 §0-E 回答。

**④-26 A/B 测试框架**
与节点⑦数据闭环复用同一框架（Statsig 或自研灰度平台）。实验分组：按用户 ID hash（稳定分组，同一用户始终在同一组）。指标：路由准确率/用户干预率/转化率。显著性：p<0.05 才切换。

### 主动服务

**④-27 主动服务触发引擎**
端侧规则引擎（不用 LLM，延迟敏感）：情境信号（天气/位置/时间）+ 状态（DMS疲劳/电量）+ 历史节律（B-F3输出）→ 决策树匹配 → 触发主动服务。
云端 LLM 用于：个性化推荐话术生成（异步，不阻塞触发决策）。

**④-28 打扰阈值控制**
主动服务预算：P0安全类不受预算限制（疲劳/碰撞）；P1日常：3次/周，1次/天，单次触发间隔 >30min。
疲劳度计算：连续触发2次被用户忽略 → 该类主动服务静默24h。

**④-29 主动服务可解释性**
每次主动触发携带 reason 字段（"因为你今天快到公司了""因为电量低于20%"）。用户可查：记忆控制台展示最近10次主动服务记录+触发原因。

**④-30 拒绝学习**
用户说"不用了"/"知道了"/"别管我" → NLU 识别拒绝意图 → 该类主动服务权重 -0.1，静默30天。连续3次拒绝 → 该类永久关闭（用户可在设置中恢复）。

### 个性化

**④-31 用户画像存储**
端侧：加密 SQLite（AES-256），存本地画像（偏好/节律/历史），常驻内存 <50MB。云端：PostgreSQL 分库分表，按 user_id hash 分片。同步：增量同步（diff），每次上车触发，传输 <50KB。

**④-32 多用户身份识别融合**
DMS 人脸（权重0.6）+ 声纹粗分类（权重0.2）+ 蓝牙手机（权重0.2）加权投票。人脸识别置信度 <0.7 时：降级到蓝牙手机识别（谁的手机进了车）。都识别不了 → 默认主账号。切换延迟目标 <500ms。

**④-33 AI 人设跟随用户切换**
用户切换时：① 加载该用户画像（SQLite 查询，<50ms）；② 更新 LLM 系统 Prompt（人格/偏好/历史摘要注入）；③ TTS 音色切换；④ 广播人设切换事件（订阅者：娱乐Sub-bot/导航Sub-bot/HMI）。
长记忆：切换时拉取该用户最近20轮对话摘要注入 LLM 上下文。

**④-34 画像数据合规**
GDPR/个保法：用户删除请求 → 48h内完成端云双删 → 发送删除确认邮件/App通知。数据可携带：用户可申请导出画像 JSON（脱敏后）。迁移：账号注销时数据不迁移给新用户，直接删除。

## 节点⑤ 执行层（10条）

**⑤-1 车控指令协议**
CAN 总线（ISO 15765），端侧安全校验层（规则引擎）二次验证所有车控指令。AI 发出指令 → 安全层检查（车速/场景/权限）→ 合法才发 CAN 帧。高危操作（转向/制动）V1 绝对禁止，硬编码白名单。

**⑤-2 车控权限分层**
主驾：全权限（车控/空调/车窗/座椅）。副驾：非安全类（空调/娱乐/座椅）。后排：仅娱乐/温度。语音识别来源（DMS 判断说话人位置）决定权限层级，置信度 <0.8 默认主驾权限。

**⑤-3 高危操作二次确认**
支付/位置共享/删除数据/发消息 → 语音二次确认（"确认支付¥XX吗？"）+ 屏幕展示金额。驾驶中：≥200元支付直接拒绝，提示驻车后操作。撤销窗口：所有确认操作10秒内可语音撤销。

**⑤-4 导航 API 接入**
主：高德导航 SDK（深度接入，支持驾车/步行/骑行）。备：百度地图 SDK。故障切换：高德超时2s → 自动切百度，用户无感知。跨端导航：车→手机通过账号云端中转（任务状态云端存储，手机 App 上车时拉取）。

**⑤-5 导航重规划**
流式重规划：用户途中修改目的地 → 后台异步重算 → 前端立即展示"正在重规划"动画 → 新路线返回后切换（<3s P95）。用户在重规划期间可继续按旧路线行驶，不干扰当前路段指引。

**⑤-6 音乐服务接入**
主：QQ音乐 SDK（深度授权，支持播放列表/情境推荐）。备：网易云。Spotify：V1 暂不接（国内用户量小，授权复杂）。账号授权：OAuth2，车机一次授权，账号状态云端缓存。跨端：手机正在播 → 上车主动提示"是否继续播放"，用户确认后接续（状态云端同步）。

**⑤-7 后排独立娱乐**
V1 不做独立后排屏（BOM 成本，见 §0-C）。V1 在单屏上通过分区模式支持：主驾区（导航/助手）+ 后排区（娱乐卡片）。儿童内容过滤：DMS/OMS 检测到儿童 → 内容推荐自动进入儿童模式（过滤成人内容），由内容平台 SDK 提供分级接口。

**⑤-8 MCP 协议实现**
遵循 OpenAI Function Calling schema，兼容 MCP 协议（JSON-RPC 2.0 over HTTP）。能力声明必填字段：{name, description, parameters, latency_sla, data_freshness, privacy_level}。版本管理：语义化版本（major.minor），major 变更需重新审核，minor 自动兼容。

**⑤-9 MCP 通信协议**
HTTP/2（云端，支持多路复用）。连接管理：每个 MCP 维护连接池（min=2, max=10），超时3s。长连接：不做 WebSocket（MCP 调用是请求-响应模式，不需要双向推送）。gRPC：V2 高频 MCP 可升级 gRPC（更低延迟）。

**⑤-10 第三方故障降级**
每个品类维护备选表（至少2个备选 MCP）。主 MCP 挂了 → 自动切备选 + 告知用户"已切换到XX"（不静默，避免价格/时效预期差）。所有 MCP 都挂 → 告知用户"服务暂不可用" + 记录高优告警。恢复验证：探针通过后前3次调用仍走备选验证，确认稳定后才切回主。

## 节点⑥ 输出层（14条）

**⑥-1 TTS 引擎选型**
端侧：CosyVoice-lite（~80M ONNX INT8），首字延迟 <300ms，2种基础音色，弱网兜底。云端：CosyVoice-300M，支持流式+情感，>5种音色。切换策略：网络正常→云端；弱网（RTT>500ms）→端侧；无网→端侧固定音频库。

**⑥-2 TTS 流式输出**
LLM 流式输出 token → 端侧 TTS 监听 → 检测到完整短语（标点或30字）→ 立即送端侧 TTS 合成第一句 → 开始播放同时云端 TTS 合成后续句子。首字延迟目标 <300ms（P95）。音频缓冲队列：200ms，防卡顿。

**⑥-3 多音色支持**
V1 预置5种音色（专业女声/温暖男声/轻松女声/儿童友好/默认中性），用户选择存端侧+云端双存。音色切换：<100ms（参数切换，不重新加载模型）。自定义音色：V1.5 上线（需声纹验证）。

**⑥-4 TTS 中断/打断**
用户说话 → VAD 检测 → 50ms 内发送 INTERRUPT 信号 → TTS 立即 fade-out（50ms 渐出，无爆音）→ ASR 接管。已生成但未播放的音频缓冲清空。打断后上下文保留（见 ③-8）。

**⑥-5 TTS 优先级**
抢占优先级：P0安全告警（碰撞/疲劳/电量临界）> P1系统提示 > P2主动服务关怀 > P3导航播报 > P4娱乐背景。P0 可直接打断任何正在播报的内容，fade-out 仅100ms。

**⑥-6 多音区 TTS 输出**
V1 单通道 TTS（主扬声器）。V2 支持定向声场：主驾导航播报只从主驾侧扬声器输出（需硬件支持独立分区音频），副驾娱乐走副驾侧。需要整车音频系统支持（部分车型已有分区功能）。

**⑥-7 情感 TTS**
情感参数（语速/音调/停顿）由场景上下文决定：关怀模式（语速×0.85，音调降5%）；紧迫模式（语速×1.1，音调升3%）；日常（默认参数）。参数注入 CosyVoice SSML 标签，不做模型重训。幅度克制（用户反馈"假"的比例目标 <10%）。

**⑥-8 多屏协同协议**
仪表+中控：Android Automotive OS 内部 IPC（Binder）。HUD：独立渲染进程，接收 HUD 事件总线消息（内部 UNIX Socket）。跨屏通信延迟 <50ms（P95）。

**⑥-9 HMI 渲染引擎**
中控：Android Automotive + React Native（业务层），复用前端工程师。仪表：Qt（确定性渲染，车规要求）。HUD：专用 OpenGL ES 渲染（轻量，固定布局）。虚拟形象：Live2D SDK（2D骨骼动画，算力 <4 TOPS）。

**⑥-10 驾驶模式 HMI 简化**
驾驶中：中控自动降级（卡片最多2个，无复杂动画，禁止视频）。HUD：只显示导航箭头/车速/告警。切换触发：车速>20km/h 持续5s → 驾驶模式；驻车挂P档 → 退出。切换动画 <500ms（资源预加载）。

**⑥-11 HUD 内容呈现**
内容优先级：碰撞告警 > 疲劳告警 > 导航转向 > 限速提醒 > 其他。单时刻 HUD 只显示最高优先级1条。队列：FIFO + 优先级覆盖（高优先级直接替换低优先级，不排队）。显示时长：告警10s，导航按实际需要，其他5s后自动消失。

**⑥-12 虚拟形象渲染**
V1 决策：Live2D 2D 动画（推翻 PRD 11 静态 icon 判断，采用 PRD 09 H-F6 动态形象方案）。6状态：休眠（呼吸动效）/唤醒（睁眼）/聆听（声波）/思考（眼神微转）/成功（点头）/失败（皱眉）。驾驶中简化：仅表情+光效，停止大幅度动画（安全优先）。Live2D SDK 算力 <4 TOPS（GPU分担），内存 <200MB。

**⑥-13 多屏触控冲突**
仲裁规则：① 主驾触控优先级最高（驾驶安全）；② 副驾触控在驻车模式下与主驾平等；③ 后排触控只限娱乐区域，不能操作前排。同时触控：先到先得（<10ms 内先到的请求获得焦点锁），另一个触控忽略或排队。

**⑥-14 方向盘震动+氛围灯**
震动：安全告警（强震，3次 ×200ms）；导航提示（轻震，1次 ×100ms）；确认反馈（微震，1次 ×50ms）。通过 CAN 总线控制方向盘震动马达。
氛围灯：欢迎（蓝色渐入）；告警（红色闪烁）；关怀（暖白色柔和）；休眠（熄灭）。通过车身 CAN 总线控制 LED 驱动器。

## 节点⑦ 基础设施层（28条）

### 大模型车云协同

**⑦-1 端云分工默认路线**
端侧：KWS + 流式ASR + L1/L2 NLU + DMS/OMS + 端侧TTS + 规则引擎。云端：L3 NLU + 任务规划 + 工具调用 + 长期记忆检索 + 情感TTS。协同：ASR 端侧出首结果，云端异步修正；NLU 端侧出路由决策，云端做复杂推理。

**⑦-2 大模型选型**
主 LLM：Qwen2.5-72B（阿里云托管，国内合规，API 计费）。备 LLM：DeepSeek-V2.5（成本更低，用作降级或并发高峰分流）。不选 GPT/Claude：数据出境合规风险（个保法+汽车数据安全管理规定）。Embedding：bge-m3（向量召回，端侧 bge-small-zh 用于本地语义搜索）。

**⑦-3 端侧大模型**
V1：不上端侧7B+大模型（算力/功耗/内存不支持，V1 目标8295平台 NPU 30 TOPS 已分配满）。V1.5 评估：Qwen2.5-7B INT4（~4GB，骁龙8295 NPU 推理约4 TOPS）。弱网场景当前用规则引擎兜底，不是端侧大模型。

**⑦-4 端云协同决策**
路由决策因子：① 网络 RTT（>500ms → 端侧）；② 意图复杂度（L1/L2可端侧处理 → 端侧）；③ 隐私敏感性（车内画面/声纹 → 强制端侧）；④ 算力占用（NPU 占用 >80% → 排队或云端）。决策引擎：端侧规则引擎（<1ms），不用 ML 模型（太慢）。

**⑦-5 端云协同延迟**
正常网络：端侧首结果 <500ms，云端修正结果 <1.5s（P95）。网络抖动（RTT 200-500ms）：端侧结果直接用，不等云端。完全断网：端侧独立运行，无需任何云端调用。
兜底：云端调用超时3s → 端侧结果作为最终结果，不再等待。

**⑦-6 多模型 fallback**
主 LLM 挂了：① 自动切 DeepSeek 备用（<500ms 感知到故障）；② DeepSeek 也挂 → 端侧规则引擎（降级告知用户）。实现：熔断器（Hystrix 模式，Python tenacity 库）。调用方无感知：LLM 客户端封装 fallback 逻辑，对上层 Sub-bot 透明。

### 状态管理

**⑦-7 状态管理架构**
四层状态：① 用户状态（画像/偏好/节律，端侧SQLite+云端PG）；② 车辆状态（实时CAN数据，端侧内存，TTL=0即时）；③ 对话状态（最近5轮，端侧SQLite，TTL=30min）；④ 任务状态（DAG任务图，端侧SQLite+云端Redis，TTL=30min）。端侧优先（性能），云端同步（持久化/跨设备）。

**⑦-8 长记忆存储**
结构化偏好：SQLite（关系型，用于精确查询）。事件流：SQLite 时序表（按时间索引）。语义记忆：**Annoy**（端侧，树形索引，100万×384维向量常驻 <150MB，FAISS HNSW 在车机端建图时内存峰值可达 1.5GB+ 不可接受）+ 云端 **Milvus**（HNSW，跨设备，更大规模）。Embedding 模型：bge-small-zh（端侧，~90MB，推理 <30ms）。召回延迟：Annoy <80ms（vs FAISS HNSW <50ms，轻微劣势，内存安全性优先）。

**⑦-9 个性化推荐**
数据采集：行为埋点（最小必要，15-20个事件类型）+ 用户反馈（显式点赞/踩）。学习：离线批量训练（月度），不做实时在线学习（V1）。推荐引擎：协同过滤+规则混合（V1），纯ML推荐（V2）。端云分工：召回端侧（快），排序云端（准）。

**⑦-10 冷启动**
新用户默认配置：专业型人格/普通话/中等语速/无节律服务（开车30天后解锁）。加速冷启动：① 引导用户主动设置常用地点（家/公司）；② 上车前10次交互优先收集偏好。节律学习：需要14天数据，V1 默认关闭，30天后提示开启。

**⑦-11 模式切换**
见 ③-12。补充：模式切换事件通过内部事件总线广播（Publish-Subscribe），所有 Sub-bot 订阅 `mode_changed` 事件，收到后更新本地行为策略。事件总线：端侧用 Python asyncio EventBus（in-process），延迟 <1ms。

### 弱网降级

**⑦-12 网络状态检测**
信号源：系统网络 API（Android ConnectivityManager）+ 主动探测（每30s ping 云端健康检测接口）+ RTT 统计（每次云端调用记录）。融合：RTT<200ms=良好，200-500ms=一般，>500ms=弱网，无响应=断网。防抖：状态变化需持续30s才切换（防网络波动导致频繁切换）。

**⑦-13 弱网/无网降级链路**
每个能力的端侧版本见 §0-B。额外说明：LLM降级后，端侧规则引擎支持的场景列表需要与 PRD L1/L2 高频意图对齐，定期 review（每季度）确保覆盖率 ≥80%。

**⑦-14 离线时长上限**
连续离线 >24h：第三方服务（外卖/支付）功能不可用（提示用户）；连续离线 >72h：记忆同步暂停（端侧本地仍可记录，恢复网络后批量同步）；无上限：车控/导航/音乐（本地数据支撑）。

**⑦-15 网络恢复策略**
恢复检测：RTT<200ms 持续30s → 确认恢复。恢复动作：① 端侧积压的行为埋点批量上传（压缩后 <100KB）；② 状态同步（增量 diff）；③ 模型版本检查（有新版本则后台下载）。恢复提示：静默恢复，不打扰用户（除非有重要更新需告知）。

### 安全合规

**⑦-16 隐私分级**
必须本地：DMS/OMS 原始画面、声纹原始音频、车内对话录音。可上云（脱敏后）：ASR 文本（PII哈希）、结构化意图、行为埋点（去标识化）、地点（城市级别）。需用户主动同意：对话历史上云、位置精确同步、第三方服务数据授权。

**⑦-17 DMS/OMS 隐私边界**
原始画面：DSP→NPU 直通（不经过普通内存总线），处理完即丢，不落盘。结构化输出：`{fatigue:0-3, distraction:bool, emotion:enum}` 落端侧SQLite，聚合摘要上云（每10min一条）。审计：原始画面处理链路代码审计（每季度），确保无旁路录制。

**⑦-18 声音事件隐私**
声音事件分类后只上报事件类型（`{event_type: "siren", confidence: 0.92, timestamp}`），不上报原始音频。分类器输出即刻在端侧生成结构化事件，原始音频内存中处理完即释放。

**⑦-19 拒识日志隐私**
被拒识的对话（电话/家庭闲聊）：不记录文字内容，只记录「拒识事件」元数据（`{reason: "call_detected", duration_ms: 3200}`）。元数据保留30天用于准确率统计，之后自动删除。

**⑦-20 AI 决策可解释性**
用户问"为什么"时：① 查任务日志（trace_id对应的决策链）；② LLM 生成用户友好解释（"因为你今天快到公司了，我提前规划了路线"）。技术实现：每个决策节点记录 reason 字段，串联后输入 LLM 做自然语言解释。

**⑦-21 功能安全（ISO 26262）**
AI与车规的对齐方案：AI层（非确定性）→ 安全层（确定性规则引擎）→ CAN总线（确定性）。安全层：V1 ASIL-B 目标（不做舱驾融合则 ASIL-A 够），独立于 AI 的硬件冗余路径（紧急情况下物理按键直接触发车控，不经AI）。AI 建议 ≠ AI 执行：所有高风险操作（速度>30km/h时）只建议，不自动执行。

**⑦-22 儿童识别法律合规**
识别阈值：OMS 儿童检测置信度 >0.85 才触发儿童相关逻辑（高阈值降低误报率）。误报处理：误判为儿童 → 进入儿童模式 → 用户可手动关闭（不强制）。责任声明：用户协议明确"儿童检测为辅助功能，不替代人工监护"，不承担遗留物事故法律责任（仅辅助提醒）。

**⑦-23 数据合规（GDPR/个保法）**
用户权利实现：查阅权（记忆控制台）、删除权（端云双删，48h内）、可携带权（JSON导出）、撤回同意权（随时关闭任何数据收集）。同意记录：版本化JSON存储（`{consent_version, timestamp, user_id_hash}`），不可篡改（追加日志）。海外版：V1 不出海，V2 按目标市场本地化合规。

### 数据闭环

**⑦-24 数据采集协议**
埋点分类：① 行为事件（语音指令/车控/导航/音乐，T+1上报）；② 性能事件（延迟/错误率，实时上报）；③ 用户反馈（显式点赞/踩，实时上报）。采集最小必要原则：不采集原始语音/精确位置/通话内容。端侧缓存 + WiFi批量上传（降低流量成本）。

**⑦-25 数据脱敏**
PII 脱敏：人名/手机/地址 → SHA-256哈希（不可逆）。位置脱敏：精确GPS → 城市+区级别。语音脱敏：不上传原始音频，只上传ASR文本（再次哈希PII字段）。脱敏在端侧执行，脱敏后再上云。CI 自动检查：代码提交时扫描是否有未脱敏的PII字段。

**⑦-26 数据标注流水线**
自动标注（80%）：ASR输出 + NLU置信度 + 执行结果 → 自动生成弱监督标签。人工标注（20%）：失败样本 + 低置信度样本 → 外包标注（脱敏后）→ QA审核（10%抽检）。训练集生成：标注通过 → 自动加入训练集 → 触发增量训练。

**⑦-27 模型迭代闭环**
失败样本 → 端侧采集 → 脱敏上云 → 自动标注 → 人工review（低置信度）→ 进入训练集 → 月度离线训练 → 内部评测（badcase修复率/CER/用户满意度）→ 灰度5%（7天）→ 20%（7天）→ 100%。关键 badcase 修复目标 <7天（紧急通道：跳过灰度直接hotfix）。

**⑦-28 指标监控看板**
实时监控（Datadog/自研）：ASR CER / NLU准确率 / 任务成功率 / 端到端延迟P50/P90/P99 / 云端费用 / 端侧CPU/NPU占用。告警阈值：任务成功率 <85% → P1告警；端到端延迟P99 >3s → P2告警；云端费用超预算20% → P1告警。每周数据 review（产品+算法+工程三方）。
