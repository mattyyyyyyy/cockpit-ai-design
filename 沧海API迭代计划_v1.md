# 沧海API → 座舱AI重设计 · 迭代计划

> 日期：2026-05-13 · 作者：陆行（产品）× 沈砚（架构）
> 基于：沧海API 现有代码全量审查 × 16篇PRD + 架构文档 + P0/P1修正方案
> 本文件为迭代执行的顶层计划，各阶段具体技术方案另见对应文档。

---

## 现状判断

**基础可用，架构需重构。**

现有代码可跑通基础语音交互链路（ASR → NLU → 车控/工具 → TTS），但与理想设计存在三个根本差距：

1. **决策层**：现有是"NLU 决定 function_code → Agent 执行"的线性路由，缺 Mother-bot 自主编排 + DAG 任务规划
2. **记忆系统**：几乎为零——车不记得用户习惯，每次当陌生人
3. **多模态/输出层**：只有语音通道，缺视觉/手势/指代消歧/虚拟形象/卡片化 HMI

## M0 · 安全堵漏（2 周）

> 目标：修复 4 个线上安全漏洞，不修不能上线。

### M0-1：支付身份验证缺失

**问题**：`unionpay.py` 短信验证整段注释，支付链路跳过身份验证，驾驶中任何人可发起支付。

**怎么做**：
- 驾驶中支付改用声纹验证 + 语音确认（"确认支付 38 元"）
- 按驾驶状态分级：行驶中仅允许预授权品类（已绑定充电桩/停车/重复外卖），中额需声纹，大额(>500元)拒绝
- 驻车模式恢复全支付方式（扫码/声纹/指纹）

**做成效果**：支付有身份校验、驾驶中大额支付被拦截、预授权品类免二次确认

**关联**：P0-1 修正方案、PRD 07 §5.4

### M0-2：不可逆操作无法撤销

**问题**：下单/发消息/拨电话执行后无回退路径，用户误操作无法挽回。

**怎么做**：
- 所有有副作用的操作执行后插 10 秒撤销窗口
- TTS："订单已提交，10 秒内可以说取消撤销"，5 秒时提醒一次
- NLU 新增"撤销/取消"意图组，撤销意图优先级高于新意图
- 操作按可逆性分级：即时撤销（车控）→ 有窗口期撤销（下单）→ 部分可撤销（已支付）→ 不可撤销（消息已发）

**做成效果**：误操作 10 秒内可撤、不同操作有对应回退策略、撤销失败有兜底（推送 App 客服入口）

**关联**：P0-2 修正方案、PRD 06 §5.2

### M0-3：车机重启后任务悬空

**问题**：任务状态机缺 STALE 状态，重启后进行中的任务无法安全恢复。

**怎么做**：
- 状态机新增 STALE（状态未知）和 UNKNOWN（工具无法确认）两个状态
- 系统启动时自动扫描 RUNNING 任务 → 标记 STALE → 查询工具服务确认结果
- 车控类任务直接读 CAN 总线确认，第三方订单调平台 API 确认
- UNKNOWN 状态下次开车时 TTS 询问用户确认

**做成效果**：重启不丢任务、自动恢复确认、用户只需对"系统也拿不准"的操作做判断

**关联**：P0-3 修正方案、沈砚架构回答 ④-7

### M0-4：工具调用无幂等保护

**问题**：网络重试可能重复下单/扣款，工具层没有去重机制。

**怎么做**：
- 每次工具调用携带 idempotency key（格式：`{trace_id}-{node_id}-{attempt}`）
- MCP 服务端维护幂等缓存（Redis，TTL 24h），命中则返回原结果不重新执行
- V1 白名单 MCP 必须支持幂等（接入审核必过项）
- 不支持幂等的 MCP 只允许只读查询，不允许有副作用操作

**做成效果**：重复请求不重复扣款、MCP 接入有安全底线、重启恢复可安全重试

**关联**：P0-4 修正方案、沈砚架构回答 ④-9/⑤-8

## M1 · 架构重建（6 周）

> 目标：换决策引擎，这是最关键里程碑——架构不换，后面的能力都挂不上去。

### M1-1：Mother-bot 编排器 + DAG 任务规划

**问题**：复杂任务（"导航到公司顺便叫个咖啡"）无法拆步骤、无法并行、出错无法回滚。现有 NLU 输出 function_code → 直接分发给 Agent，是线性路由。

**怎么做**：
- 保留 AgentScope 作为 Sub-bot 运行时（现有 5 个 Expert Agent + 工具注册已跑通，不重写）
- 新建 Mother-bot 编排器（独立模块）：接收 NLU 意图 → 生成 DAG 任务图 → 调度 AgentScope Agent 执行
- DAG 支持：并行节点、串行依赖、条件分支、失败重规划
- 共享黑板：Sub-bot 之间通过黑板通信，不直接调用
- 简单任务（车控/单步查询）保留现有 function_code 路由，不经过 DAG——双轨并行

**做成效果**：一句话多意图自动拆解并行执行、某步失败能重新规划不影响其他步骤、复杂任务端到端可追溯、简单任务零改动

**关联**：PRD 05 §5（Agent 编排）、沈砚架构回答 ④-1~④-30

### M1-2：Harness 安全线束

**问题**：AI 没有安全兜底——车控指令没有端侧二次校验、失败不会自动隔离、降级策略无优先级、多个安全规则同时触发时谁赢不知道。

**怎么做**：
- 实现 7 条线束（优先级从高到低）：
  1. 安全红线 — 最高，不可被任何其他线束覆盖
  2. 失败隔离 — 防止故障扩散（熔断）
  3. 降级路径 — 保证可用性
  4. 置信度阈值 — 决策质量控制
  5. 人类监督 — 用户确认
  6. 状态机 — 流程管理
  7. 可观测性 — 诊断，不影响执行
- 冲突仲裁表：内存查表 <1ms，按优先级执行
- 车控指令走安全层：AI 输出 → Protobuf VehicleCommandRequest 安全校验 → CAN 总线，不信任云端直连
- 置信度模型 OTA 后自动下发 temperature scaling 校准参数，防止阈值漂移

**做成效果**：AI 不可控的操作被拦截、故障不扩散、降级无感切换、安全规则冲突有明确仲裁

**关联**：P1-5/P1-6 修正方案、沈砚架构回答 ⑦-27、安全层接口文档

### M1-3：三级 NLU 路由

**问题**：每次请求并发调多个 NLU 再取最佳，简单指令也走云端，延迟高、成本高、弱网下全挂。

**怎么做**：
- L1 关键词匹配（<5ms，端侧）：覆盖 Top 50 高频指令
- L2 BERT-tiny 端侧模型（<100ms）：覆盖 Top 200 意图
- L3 云端大模型（<1.5s）：兜底，处理复杂/模糊意图
- L1/L2 命中即返回，不再调云端，预计拦截 60%+ 请求
- 弱网时自动降级到 L1/L2，核心功能仍可用
- 现有并发 NLU 逻辑保留为 L3 兜底

**做成效果**：简单指令 100ms 内响应、云端调用成本降 60%+、断网仍可控制车控/导航

**关联**：PRD 03 §5（NLU 与意图理解）、沈砚架构回答 ③-1~③-12

### M1-4：trace_id 跨端云传播

**问题**：端侧 trace_id 和云端 LLM request_id 是两个孤立系统，故障时无法串联全链路。

**怎么做**：
- 统一 trace_id 格式：`{device_id_hash}-{session_id}-{sequence}-{timestamp}`
- 所有跨系统调用透传 X-Trace-Id header
- 云端 LLM 接入要求：请求携带 trace_id，响应返回 llm_request_id 关联存储
- 端侧关键节点（KWS→ASR→NLU→工具→TTS→播报完成）全链路 span 级日志

**做成效果**：一次请求全链路可在单一视图查看、故障定位从小时级降到分钟级

**关联**：P1-8 修正方案、沈砚架构草稿 §5.1

## M2 · 体验可感知（8 周）

> 目标：用户能直接感受到"车变聪明了"。

### M2-1：记忆系统全栈

**问题**：车不记得用户习惯，每次都当陌生人，"越用越懂你"无法实现。

**怎么做**：
- B-F1 偏好记忆：SQLite 存结构化偏好键值，从对话和操作中自动提取（"我喜欢听爵士"→ 记住音乐偏好）
- B-F2 事件记忆：时序存储 + FAISS 向量检索，支持"上次去那个餐厅叫什么"的回忆查询
- B-F3 节律识别：Apriori 挖掘行为模式（每周一早 8 点导航到公司），14 天数据积累后解锁
- B-F6 遗忘修正：时间衰减自动遗忘低频偏好 + 用户可手动纠正（"你记错了"） + 记忆控制台可查看/删除

**做成效果**：用两周后车自动记住空调温度/音乐偏好/通勤路线、用户可查看和删除记忆、记错可纠正

**关联**：PRD 04 补全文档（B-F3/B-F4/B-F6/S-F3/D-F3）

### M2-2：主动服务引擎

**问题**：车不会主动推荐，永远等用户开口，和传统车机体验无异。

**怎么做**：
- D-F3 习惯驱动主动服务：匹配节律后主动推荐（"你平时这个时间会去充电站，要不要导航？"）
- 预算控制：3 次/周，不烦人；用户正反馈 +0.05 权重，负反馈 -0.02 权重
- 首次主动服务强制附带触发原因（"因为你每周一到周五这个时间都会路过这里"），让用户区分"学到的"和"广告"
- B-F4 异常关怀：偏离日常路线/超速/疲劳时轻提示，2 次/周预算，关怀类默认关闭需用户开启

**做成效果**：车会主动提醒、用户知道为什么被推荐、正负反馈影响后续推荐频率

**关联**：PRD 04 补全文档、P1-1/P1-2 修正方案

### M2-3：冷启动体验

**问题**：节律服务需 14 天积累，用户前两周感知 AI 和传统车机无异，缺乏留存钩子。

**怎么做**：
- 探索期（1-3 天）：首次使用后 TTS "我会记住你的习惯，越用越懂你"
- 积累期（4-13 天）：每记住一个偏好→轻提示"我记住了你喜欢听爵士乐"，每天最多 1 次
- 解锁期（第 14 天）：TTS + 卡片"你已用车 14 天，智能节律已开启"

**做成效果**：新用户第一天就感知到"AI 在学我"、14 天后有惊喜感、不用等两周才有价值

**关联**：P1-1 修正方案、PRD 04 §5.4/PRD 11 §6

### M2-4：降级时机感知 + Sub-bot 协议

**问题**：云端超时 3 秒切端侧，用户 3 秒什么都看不到然后突然切换；Sub-bot 执行失败后 Mother-bot 不知道怎么重新规划。

**怎么做**：
- 降级感知时间线：800ms 显示"正在思考"、2000ms 后台预热端侧、2500ms TTS"稍等一下"、3000ms 无缝切换
- Sub-bot 标准化响应：status + error_code + suggested_action + partial_result
- Mother-bot 按 error_code 处理：TIMEOUT 重试、DESTINATION_CLOSED 问用户、CONTENT_UNAVAILABLE 自动换备选
- 并发控制：同时最多 3 个 Sub-bot，按优先级排队（车控>导航>通信>娱乐>信息）

**做成效果**：等响应时用户知道系统在工作、Sub-bot 失败有标准化处理、不会因一个 Sub-bot 超时阻塞全部

**关联**：P1-7/P1-9/P1-10 修正方案

## M3 · 竞品可对标（8 周）

> 目标：多模态 + 虚拟形象 + HMI 重构，补齐与蔚来/理想/极氪的体验差距。

### M3-1：多模态输入

**问题**：只有语音，没有视线/手势/指代消歧，"把那个调小"听不懂，后排说话和主驾分不开。

**怎么做**：
- V1 双音区起步：主驾 + 副驾声源分离，后排 v2
- DMS 视线追踪：识别"那个"指代目标（屏幕区域块精度 ~5cm）
- 基础手势：6 种（接听/挂断/音量/切歌/确认/取消），驾驶时 HUD 提示
- 多模态置信度融合：语音 + 视线 + 上下文加权，低置信时追问
- C-F3 声音事件识别：哭声/警报声/撞击声 → 触发安全/关怀

**做成效果**：说"把那个调小"配合视线就能识别目标、副驾和主驾互不干扰、不方便说话时手势兜底、车内异常声音能触发响应

**关联**：PRD 01（多模态输入）、PRD 02（外部情境感知）

### M3-2：Live2D 虚拟形象

**问题**：没有虚拟形象，竞品 NOMI/Eva/理想同学都有，我们的 AI 是"看不见的语音助手"。

**怎么做**：
- Live2D 2D 骨骼动画：6 状态（休眠/唤醒/聆听/思考/成功/失败）
- 驾驶简化模式：车速 >20km/h 持续 5s → 形象缩小为角落光效+波形（NPU 从 4 TOPS 降至 0.5 TOPS，释放给 DMS）
- 平滑过渡：全形象→简化 800ms ease-out，简化→全形象 600ms ease-in，防视觉跳变
- V1 预置 5-10 套官方模板，不支持用户自上传
- 位置：中控顶部状态条，不占主屏

**做成效果**：AI 有表情反馈、驾驶中不占算力不挡视线、切换无跳变

**关联**：PRD 09 §5.6 H-F6、文档修订备忘录 #1、P1-4 修正方案

### M3-3：卡片化 HMI + 多屏协同

**问题**：信息展示全靠文字，导航/音乐/订单信息密度低，多屏没有分工。

**怎么做**：
- AI 输出结构化卡片：导航卡片/音乐卡片/订单卡片/天气卡片，替代纯文字
- 主动服务视觉区分：学习触发的卡片带"根据你的习惯"+ 脑波图标，系统推送带"当前状况"+ 信号图标
- 多屏分工：仪表（驾驶信息）、中控（交互主屏）、HUD（驾驶辅助）、后排（娱乐）
- 虚拟形象嘴型同步：TTS 音频时间戳与 Live2D 嘴型对齐，误差 <30ms

**做成效果**：信息一屏看完、用户一眼区分"AI 学到的"和"系统推送的"、多屏各司其职

**关联**：PRD 09（HMI 多屏协同）、PRD 08（TTS 与语音风格）

### M3-4：儿童模式 + 场景模式

**问题**：OMS 检测到儿童直接触发过滤，无年龄细分，无临时关闭出口；无场景自动切换。

**怎么做**：
- 儿童模式三层架构：年龄段配置（0-3/4-12/13-17）→ OMS 自动检测触发 → 家长临时控制出口（方向盘长按 3s 声纹验证 / 中控设置页）
- S-F3 场景模式：5 种（通勤/家庭/长途/关怀/驻车），规则引擎 + 5min 冷却 + 80% 置信度阈值
- 场景切换联动：家庭模式→后排切儿童内容+音量降，长途模式→每 2 小时疲劳提醒

**做成效果**：儿童内容按年龄分级、家长随时可临时关闭、上下车自动切场景

**关联**：P1-3 修正方案、PRD 04 补全 S-F3、PRD 13 安全合规

## 关键风险与依赖

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Mother-bot 编排器与 AgentScope 集成复杂度 | M1 延期 | 简单任务保留现有路由，双轨并行降低切换风险 |
| 端侧 NPU 算力分配未确认（8295 30 TOPS 怎么分） | M1/M3 受阻 | 需硬件团队输入，M1 期间完成算力预算确认 |
| MCP 供应商不支持幂等 | M0 幂等方案落地难 | V1 白名单强制要求，不支持的只允许只读查询 |
| 记忆系统冷启动 14 天，用户可能流失 | M2 留存风险 | M2-3 冷启动体验专门解决此问题 |
| DMS/OMS 硬件选型未定 | M3 多模态受阻 | M1 期间完成摄像头 BOM 确认 |
