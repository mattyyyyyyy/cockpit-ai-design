# 座舱 AI 业务侧迭代方案 v1

> 视角：用户体验与业务目标优先 → 落到技术实现 → 可度量效果
> 2026-05-13

---

## 阅读方式

每个业务问题按三段式展开：
- **用户/业务痛点**：真实场景，现在是什么体验
- **技术落地方法**：解决它的工程手段
- **可达成的效果**：用户感知 + 可量化指标

---

## 业务问题一：AI 经常没听懂，用户要重复说

### 用户/业务痛点

> "我说'去老地方'，AI 说'对不起我没理解'，我只好重新说'导航到上海张江高科技园区'"

用户在驾驶中不愿意、也不方便长句子描述需求。他们习惯用省略、代词、简称来说话。现有 NLU 对这类"口语化、上下文依赖"的表达识别率很低，用户体验是频繁的"听不懂"。

现有代码问题：
- 查询改写（query_rewrite）虽然存在，但是串行跑在 NLU 之前，增加了延迟；且改写错了会让 NLU 更难识别
- LLM-NLU 返回的意图没有置信度，系统不知道自己"猜得有多准"，容易乱执行

### 技术落地方法

**① 查询改写并行化（不增加延迟）**
- NLU 和改写同时发出，NLU 先回来就直接用
- 改写完成后只在 NLU 置信度低时作为参考，不强制替换

**② LLM-NLU 输出置信度**
- 意图识别结果从 `{"305.22": "星巴克"}` 改为 `{"305.22": {"keyword":"星巴克","confidence":0.91}}`
- 置信度 < 0.75：TTS 反问确认（"您是说去附近的星巴克吗？"）
- 置信度 < 0.5：直接问用户（"我没太听清，您方便再说一遍吗？"）

**③ 偏好记忆支撑模糊表达**
- "老地方"→查用户偏好表→最近3次去过的高频地点
- "老样子"→查订单历史→上次星巴克的订单

### 可达成效果

| 指标 | 现在 | 目标 |
|------|------|------|
| 意图识别成功率 | 未度量（估计75%左右） | ≥ 90% |
| 用户重复说话次数 | 高频 | 降低 60%+ |
| 响应延迟增量 | +300-800ms（改写串行） | ≤ +50ms（并行化后）|

**用户感知**：说话方式从"向机器说话"变成"像对人说话"。

## 业务问题二：AI 回复太慢，影响驾驶节奏

### 用户/业务痛点

> "我问了一句话，等了5秒没反应，我以为没唤醒，又说了一遍，结果两条一起执行了"

驾驶场景对延迟极度敏感。用户注意力在路上，等待2秒就会开始怀疑"是不是没听到"。超过3秒会主动重试，造成重复执行。

现有代码问题：
- 云端 LLM 超时没有渐进感知（0到3秒都是无反应）
- 没有视觉/听觉的"正在处理"反馈
- 多工具并发调用没有上限，高峰期 GPU 实例被打爆，延迟飙升

### 技术落地方法

**① 渐进式用户感知（不改变实际处理速度，改变用户感知）**
```
0-800ms：无动作（快速请求不需要提示）
800ms：Live2D 切"思考"状态 + 思考动效
2000ms：后台预热端侧备用方案
2500ms：TTS轻播"稍等一下"（用户不焦虑了）
3000ms：切备用方案输出结果
```

**② 工具并发上限（防打爆）**
- asyncio.Semaphore 限制同时最多 3 个 MCP 工具并行
- 工具调用超时按类型差异化：车控 2s，导航 5s，餐饮下单 15s

**③ 车控走快速路（绕开 LLM）**
- 纯车控意图（开车窗/调空调）识别到就直接执行，不进 LLM
- 这条路 P99 延迟 < 500ms（传统 NLU 直通车控层）

### 可达成效果

| 指标 | 现在 | 目标 |
|------|------|------|
| 首字延迟（P95） | 未度量（估计2-5s） | ≤ 1.5s |
| 用户重复触发率 | 有（无法量化） | 降低 80%+ |
| 高峰期延迟尖刺 | 存在 | GPU 实例稳定，P99 ≤ 5s |

**用户感知**：AI 有即时反应，等待有预期，不会觉得"卡死了"。

## 业务问题三：每次上车都得重新说，AI 没有记忆

### 用户/业务痛点

> "我已经说了三次我喜欢少冰美式，第四次它还问我要什么"

> "上次导航到公司花了40分钟，这次它还是给我推荐了那条堵的路"

车主每天用车，对一个"好助手"的核心期待就是"记住我"。现在的系统对话历史只有1小时，换个会话什么都忘了。

现有代码问题：
- `PersistentCanghaiMemory` 只保留 15 轮，TTL 1 小时
- 没有跨会话的用户偏好存储
- 没有基于历史行为的推断能力

### 技术落地方法

**① 用户偏好 MySQL 表（持久化，永不过期）**

```sql
CREATE TABLE user_preferences (
  user_id INT,
  category VARCHAR(50),  -- food/navigation/music/climate
  key_ VARCHAR(100),      -- starbucks_regular/home_address/music_style
  value TEXT,             -- '美式咖啡中杯少冰' / '张江高科技园区'
  confidence FLOAT,       -- 0.0-1.0，越高越确定
  updated_at TIMESTAMP
);
```

**来源**：
- 用户明确说"记住" → confidence=1.0
- 重复3次相同选择 → confidence=0.85
- 工具执行成功后自动抽取关键字段

**② 跨会话上下文桥接**
用户首次开口前，注入个性化 context：
```
"用户常点：星巴克美式少冰；常去：公司（张江）；音乐偏好：爵士"
```
让 LLM 带着用户背景做意图理解，"老样子"自然能识别。

**③ 冷启动过渡（前14天）**
- 第1-3天：记录观察，TTS 不主动说，但悄悄学
- 第4-13天：记住了一个偏好就轻提示一次（"我记住了您喜欢少冰"）
- 第14天：TTS 播报"您的智能助手已了解您的习惯"，解锁主动推荐

### 可达成效果

| 指标 | 现在 | 目标 |
|------|------|------|
| 偏好重复询问次数 | 每次会话都问 | 确认过的偏好零重问 |
| 跨会话上下文连贯 | 无 | 常用场景直接识别 |
| 14天后主动推荐准确率 | 无能力 | ≥ 70% 用户接受 |

**用户感知**：从"工具"变成"懂我的助手"。

## 业务问题四：高危操作没有保护，一句话可能出大事

### 用户/业务痛点

> "我在高速上，AI 因为听错了开始导航变道，差点出事"

> "我说'给张伟发消息说我快到了'，结果发到了另一个张伟"

> "我在路上支付了一笔很大的钱，事后想撤销发现来不及"

驾驶场景有特殊的安全要求：注意力分散、操作精度低、误操作后果严重。

现有代码问题：
- `VehicleStateChecker` 只检查车辆当前状态（空调是否开着），没有速度感知的操作限制
- 没有"执行前确认"机制（直接执行，用户没有取消机会）
- 支付通道虽然是银联，但没有驾驶中金额上限

### 技术落地方法

**① 操作风险分级 + 驾驶感知**

| 风险级别 | 操作举例 | 行驶中处理 | 驻车处理 |
|---------|---------|-----------|---------|
| 低风险 | 调音量、切歌 | 直接执行 | 直接执行 |
| 中风险 | 下单餐饮（<500元）、发消息 | TTS播报+10秒取消窗口 | 确认后执行 |
| 高风险 | 支付>500元、拨打电话 | 行驶中拒绝，提示驻车操作 | 二次确认 |
| 车控 | 车窗/空调/座椅 | 速度<120km/h才允许 | 全允许 |

**② 10秒撤销窗口（所有中风险以上操作）**
```
操作执行后 → TTS："已完成，10秒内说'取消'可撤销"
→ 5秒时再提醒一次
→ 10秒超时 → 锁定，不可撤销
```
撤销窗口只靠语音，不在屏幕上显示（不分散驾驶注意力）。

**③ 支付分级（对接现有银联通道）**
- 车速 > 10km/h：上限 500 元，仅允许预绑定平台（充电/常用外卖）
- 车速 ≤ 10km/h 或驻车：金额不限，银联全流程

**④ 联系人歧义消除**
发消息/打电话时，如果匹配到多个同名联系人，先 TTS 播报确认："您联系人里有两个张伟，是公司的张伟还是家里的？"

### 可达成效果

| 指标 | 现在 | 目标 |
|------|------|------|
| 行驶中误操作率 | 有，无量化 | 高危操作零执行 |
| 支付安全事故 | 无保护 | 驾驶中支付有上限+撤销 |
| 用户误触撤销率 | 无能力撤销 | 10s内撤销成功率 ≥ 95% |

**用户感知**：放心交给 AI，知道它不会乱来。

## 业务问题五：一个功能挂了，整个 AI 都用不了

### 用户/业务痛点

> "我说打开导航，AI 一直转圈，然后什么都不回了"

> "今天小红书服务好像出问题了，我说什么都没反应"

用户对 AI 的容忍度很低：一次失败体验会让用户对整个系统产生不信任，甚至不再使用。

现有代码问题：
- MCP 工具调用超时没有熔断，一个工具慢会拖慢整个请求
- 异常处理是 `except Exception: log("warning", ...)` 直接吞掉，用户层没有反馈
- 没有降级策略：云端挂了就没有任何回答

### 技术落地方法

**① 熔断器（Circuit Breaker）——按工具独立**
```
正常状态：调用工具
连续3次失败/超时 → 熔断（打开）
熔断期30秒：直接返回"该功能暂时不可用"
30秒后：半开，允许1次探测
探测成功 → 恢复正常；失败 → 继续熔断
```
一个工具挂了，不影响其他工具。

**② 三级降级策略**
```
云端 LLM 正常 → 全功能
云端超时3s   → 切端侧简化版（车控 + 简单问答）
端侧也挂    → 离线模式（仅车控，TTS 本地 TTS）
```

**③ 用户友好的失败话术**
不能再让用户看到"请求超时"，而是：
- 工具失败："星巴克点单暂时有点问题，要我帮您记下来稍后提醒吗？"
- LLM 超时："网络慢了一点，我用本地方案帮您处理"
- 完全离线："现在网络不好，我只能帮您控制车内设备，其他稍后再试"

**④ 操作幂等保护（防重复执行）**
- 每次工具调用附带唯一 `idempotency_key`
- 网络抖动导致重试时，相同 key 不重复执行下单/支付
- 重启后进行中的任务可查询状态，不盲目重跑

### 可达成效果

| 指标 | 现在 | 目标 |
|------|------|------|
| 单工具故障影响范围 | 可能导致全局无响应 | 故障隔离，其他功能正常 |
| 重复下单/支付风险 | 存在 | 零重复执行 |
| 失败用户感知 | 沉默/转圈 | 有明确话术，不焦虑 |
| 云端故障时可用能力 | 全部失效 | 车控+本地问答可用 |

**用户感知**：AI 出问题时不是"死机"，而是"变笨了一点但还能用"。

## 业务问题六：AI 只会被动响应，不会主动帮忙

### 用户/业务痛点

> "每次快没电了我都要自己查充电桩，AI 从来不提醒我"

> "我每周三晚上都要接孩子，AI 从来没问过'要导航到学校吗'"

> "开了3小时长途，AI 不知道我累了，也不知道提醒我休息"

这是现有系统和"理想助手"之间最大的差距。用户对好的 AI 助手的核心期待是"懂我，能主动"，而不只是"叫一声才动"。

### 技术落地方法（分三步走）

**第一步：规则驱动主动服务（立即可做，不需要 AI）**

纯规则，确定性高，用户接受度高：

| 触发条件 | 主动内容 | 用户可操作 |
|---------|---------|-----------|
| SOC < 20% + 附近3km有充电桩 | "电量偏低，附近有充电桩要导航吗？" | 是/不用 |
| 行驶 > 2h + 高速服务区附近 | "已连续行驶2小时，前方3km有服务区" | 好的/继续走 |
| 进入常用停车场附近 | "到了，要帮您找空位吗？" | 要/不用 |
| 雨天 + 车速>100 | "雨天注意安全，建议降速" | 仅提示，无操作 |

**第二步：偏好驱动推荐（P1完成后可做）**

基于记忆系统里的用户偏好，在合适时机推：

| 已知偏好 | 触发时机 | 推荐内容 |
|---------|---------|---------|
| 周三接孩子 | 周三16:30出发 | "要导航去学校吗？" |
| 开车喜欢听爵士 | 上车 | 自动播放上次的歌单 |
| 常去公司路上点咖啡 | 工作日早上出发 | "要帮您提前下单美式吗？" |

**第三步：节律学习（V2，6个月后）**

AI 自己从行为数据里学习规律，不需要用户设置，不需要规则写死。

**主动服务的克制原则**：
- 每次出行最多主动推送 2 次（不打扰）
- 行驶中只播 TTS，不弹卡片
- 用户拒绝同类推送2次后，该类停止30天

### 可达成效果

| 阶段 | 能力 | 用户感知 |
|------|------|---------|
| 第一步（规则） | 充电/休息/停车提醒 | "比较贴心" |
| 第二步（偏好） | 个性化习惯推荐 | "它了解我" |
| 第三步（学习） | 主动推断，无需设置 | "真的是助手" |

## 总结：从用户体验角度的迭代优先级

| 用户问题 | 解决后用户感知变化 | 技术难度 | 优先级 |
|---------|----------------|---------|--------|
| 听不懂 | 从"工具"→"能沟通的助手" | 中 | 🔴 P0 |
| 太慢 | 从"等待焦虑"→"有节奏感" | 低 | 🔴 P0 |
| 不记得我 | 从"陌生人"→"懂我" | 中 | 🟠 P1 |
| 没有安全保护 | 从"不敢用"→"放心用" | 中 | 🔴 P0 |
| 功能挂了全挂 | 从"死机感"→"降级可用" | 中 | 🔴 P0 |
| 只被动不主动 | 从"工具"→"助手" | 高 | 🟡 P2→V2 |

**核心主线**：先让用户"敢用"（安全+稳定）→ 再让用户"好用"（准确+快速）→ 最后让用户"离不开"（记忆+主动）。
