**§1 背景与目标**

**1.1 为什么做外部情境感知**

**核心问题**：座舱 AI 的"主动性"依赖它对车外世界的理解程度。当前大多数座舱系统对情境的理解是"零散的"——天气是天气、路况是路况、电量是电量,没有融合成一个**可决策的情境画像**。

三大矛盾：

1. **数据孤岛**：车辆 CAN 信号、天气 API、导航路况、POI 数据各自独立,座舱 AI 拿到的是碎片,不是画面
2. **被动服务**：系统只在用户主动触发时响应,缺乏"因为外面下暴雨所以提前调好除雾"的预判能力
3. **时空盲区**：系统不知道"现在是工作日早高峰通勤时间"+"车在公司附近"+"电量 15%"这些信息组合意味着什么

**AI 时代的机会**：

* **情境融合**：把车辆状态 + 天气 + 路况 + POI + 时间节律 + 地理围栏融合为统一情境帧,喂给决策层
* **预判服务**：基于情境帧预测用户下一步需求(通勤场景提前热车、雨天自动调空调模式)
* **时空建模**：学习用户的出行规律(工作日通勤/周末亲子/季节性长途),形成个性化情境理解

**目标**：

* **主目标**：为决策层提供实时、融合、可推理的情境帧(刷新频率 ≤30s),覆盖 90% 以上常见驾驶情境
* **准确目标**：情境分类准确率 >88%,时空预测命中率 >75%(Top 3 候选)
* **延迟目标**：情境帧生成延迟 <1s(端侧),情境变更通知延迟 <5s

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **少操心**：系统提前感知"快没电了 + 前方 5km 有充电站"→ 主动建议,不用自己时刻盯仪表盘
* **更安心**：暴雨天系统自动调除雾/降速提醒/建议路线,不需用户手动做一堆操作
* **少操作**：通勤场景识别后,上车自动导航公司/回家,不用每天重复操作

**对全车**：

* **环境适应**：外部温度骤变时,空调/座椅加热自动调整,乘员体感无感过渡
* **行程智能**：基于历史规律推荐加油站/充电站/餐厅,减少"到了才发现没电/没位"的尴尬

**2.2 反价值(不做什么)**

* **不做全知预测**：不预测"你今天心情不好所以推荐去海边"——情境感知是客观世界的建模,不是读心术
* **不做过度干预**：不强制改导航/锁车门——感知结果供决策层参考,最终执行需用户确认或符合安全策略
* **不做隐私侵入**：不在未授权情况下持续追踪用户位置轨迹——时空建模基于本地学习,原始轨迹不上云

**3.2 边界场景(需明确处理策略)**

|  |  |  |
| --- | --- | --- |
| 场景 | 挑战 | 处理策略 |
| **天气 API 延迟/失败** | 情境帧缺少天气维度 | 缓存最近一次有效数据 + 标记"过期",降级为纯车辆状态推理 |
| **POI 数据不准** | 充电桩显示空闲实际占用 | 执行前不承诺,仅推荐"附近有充电站,建议提前确认" |
| **多情境冲突** | 暴雨 + 低电 + 附近无充电站 | 按安全优先排序:先处理暴雨适配,再推荐最安全路线到最近充电站 |
| **时空建模冷启动** | 新用户无历史数据 | 前 7 天不推送个性化预测,仅做通用情境感知 |
| **隐私模式下的位置** | 用户关闭位置服务 | 仅保留车辆状态 + 手动输入目的地,时空建模暂停 |

**3.3 非场景(明确不做)**

* **社交情境推理**：不做"你的朋友也在附近所以推荐见面"——社交数据不进入情境感知层
* **驾驶风格评估**：不做"你开得太激进"的判断——这是智驾域的事,不是座舱情境感知
* **商业推荐**：不做"路过某商场所以推荐你去逛"——情境感知不掺杂商业利益

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **情境帧刷新周期** | ≤30s | 从数据源更新到情境帧输出 | >60s 标记为"低置信度" |
| **情境帧生成延迟** | <1s(端侧) | 从数据输入到情境帧完成 | >3s 降级为"上一帧 + 过期标记" |
| **情境分类准确率** | >88% | 与人工标注对比 | <75% 暂停主动推荐,仅被动查询 |
| **时空预测命中率** | >75%(Top 3) | 预测目的地在 Top 3 候选中 | <50% 降低推荐频率 |

**4.2 可靠性与降级**

**降级策略**：

1. **网络断开**：本地缓存天气/POI 数据(24h 有效),时空建模继续(端侧)
2. **GPS 信号丢失**(隧道/地库)：惯性导航 + 最近已知位置,情境帧标记"位置精度低"
3. **CAN 总线异常**：关键信号(电量/车速)缺失时,使用最后已知值 + 标记过期

**容错机制**：

* **数据源冲突**(天气 API A 说晴,B 说雨)：取权威度更高的源 + 置信度标记
* **情境帧异常值**：跳变检测(如温度 1 秒内从 25°C 变到 -5°C)→ 触发数据校验,不直接使用

**4.3 安全与合规**

**功能安全**：

* **低电量不误报**：电量预警必须基于 CAN 实时数据,不允许基于估算值触发紧急推荐
* **天气预警延迟**：极端天气预警(冰雹/台风)必须 <10s 送达,优先级高于常规情境帧

**隐私合规**：

* **时空建模本地化**：用户出行规律的模型训练在端侧完成,原始轨迹数据 7 天后自动删除
* **位置数据最小化**：仅上传围栏触发事件(如"到家"),不上传连续位置流
* **用户控制**：提供"关闭时空学习"开关 + "清除出行历史"入口

|  |  |  |  |  |

**3. 核心能力要求**

**3.1 车辆状态感知**

1. **BMS 监控** — 电量(%)/里程(km)/能耗(kWh/100km)实时采集
2. **胎压/保养** — 胎压异常、保养里程预警
3. **故障码** — CAN 总线故障码抓取,上报给风险预判
4. **状态缓存** — 车端缓存 30 分钟历史状态,云端断开时能用

**3.2 外部环境感知**

1. **天气 API** — 当前+未来 2 小时天气(雨/雪/风/温)
2. **路况 API** — 实时拥堵、限行、封路信息
3. **POI 兴趣点** — 充电桩/加油站/停车场/餐饮兴趣点(基于位置偏好)
4. **地理围栏** — 家/公司/常去区域(基于用户历史)

**3.3 时空维度感知**

1. **时间节律** — 工作日/周末/早晚高峰/节假日识别
2. **地理节律** — 路线偏好(走 A 路口 vs B 路口)
3. **组合节律** — "工作日早上 8 点走 A 路"这种联合模式

**3.4 聚合输出给决策层**

1. **情境向量** — 把所有感知维度编码成"当前情境"向量,喂给《05 主动服务决策》
2. **置信度机制** — 单一源数据异常时,其他源补强
3. **降级信号** — 弱网/云端挂时,输出"有限模式"信号给决策层

**5. 功能点展开**

**5.1 C-F4 外部环境感知**

*待填*

**5.1.1 一句话定义**

**做什么**：实时获取\*\*天气（雨雪雾霾）、路况（拥堵/事故/施工）、POI 兴趣点（充电桩/餐厅/景点）\*\*等外部环境信息，聚合后输出给决策层（05 主动服务决策），让 AI 能说"外面下雨了，要不要关天窗""前面堵，换条路吧"。

**不做什么**：

* 不做"主动决策"（05 的事，02 只负责感知 + 聚合）
* 不做"车内人员状态"（01 DMS/OMS 的事）
* 不做"车辆自身状态"（5.2 C-F5 的事）

**核心价值**：AI 不是"车内助手"，是"知道外面发生什么的助手"。

**5.1.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 外部感知 |
| S1 | **雨天关窗** | 天气 API 检测到"雨" + 车辆状态"天窗开" → 输出给 05 → TTS "外面下雨了，要不要关天窗？" |
| S2 | **路况改道** | 路况 API 检测到"前方 2km 拥堵" + 导航中 → 输出给 05 → TTS "前面堵，换条路吧" |
| S3 | **充电提醒** | 电量 < 20% + POI API 检测到"附近 500m 有充电桩" → 输出给 05 → TTS "电量低，附近有充电桩，要去吗？" |
| S4 | **雾霾提醒** | 天气 API 检测到"AQI > 150" + 车辆状态"外循环" → 输出给 05 → TTS "外面雾霾，切内循环吧" |
| S5 | **弱网降级** | 地库/隧道无网 → 使用 30 分钟缓存 + 本地规则 → 降级服务 |
| S6 | **API 失败** | 天气 API 挂 → 降级到备选 API（和风天气 → 中国天气网）→ 都挂 → 不提供天气服务 |

**外部环境数据源**：

|  |
| --- |
| Plain Text 1. 天气：和风天气 API / 中国天气网 API（备选） 2. 路况：高德路况 API / 百度路况 API（备选） 3. POI：高德 POI API / 百度地图 API（备选） 4. 空气质量：和风天气 AQI / 环保部 API（备选） |

**5.1.3 输入 / 触发条件**

* **输入**：GPS 位置、时间、车辆状态（天窗/空调/导航）
* **触发**：
* 定时轮询（天气 10 分钟 / 路况 5 分钟 / POI 按需）
* 位置变化（进入新区域 → 刷新 POI）
* 车辆状态变化（天窗开 → 检查天气）
* **前置条件**：网络可用；GPS 定位可用

**5.1.4 执行流程**

|  |
| --- |
| Plain Text 定时轮询 / 位置变化 / 状态变化  ↓ 外部 API 调用  ├─ 天气 API（当前位置 + 未来 2 小时）  ├─ 路况 API（当前路线 + 前方 5km）  └─ POI API（当前位置 + 半径 5km）  ↓ 数据聚合  ├─ 天气：温度 / 天气状况 / 降水概率 / AQI  ├─ 路况：拥堵等级 / 事故 / 施工  └─ POI：充电桩 / 餐厅 / 景点 / 停车场  ↓ 缓存更新（30 分钟有效期）  ├─ 端侧缓存（SQLite）  └─ 云端同步（可选）  ↓ 输出给决策层（05）  ├─ 情境向量：{ weather: "rain", traffic: "jam", poi: [...] }  └─ 05 根据情境 + 车辆状态 + 用户偏好 → 决策是否主动服务  ↓ 失败处理  ├─ API 超时（3s）→ 重试 1 次  ├─ 重试失败 → 降级备选 API  └─ 都失败 → 使用缓存 + 本地规则 |

**关键时间预算**：

* 天气 API 延迟：**≤ 1s**
* 路况 API 延迟：**≤ 2s**
* POI API 延迟：**≤ 1s**
* 数据聚合延迟：**≤ 100ms**

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **API 超时/失败** | 外部感知失效 | 重试 1 次 + 备选 API + 缓存降级 |
| **弱网场景**（地库/隧道） | 无法获取实时数据 | 使用 30 分钟缓存 + 本地规则 |
| **GPS 漂移** | 位置错误 → POI 错误 | GPS 置信度 < 0.8 不刷新 POI |
| **API 返回数据污染**（恶意内容） | 系统注入 | 返回数据扫描（见 06 E-F3 沙箱） |
| **高频请求成本** | API 费用爆炸 | 10 分钟轮询 + CDN 缓存 + 车队聚合 |

**Paper Tiger（看起来像风险但不是）**

* "外部 API 不稳定"—— 实证：**和风天气 / 高德路况 SLA ≥ 99.5%**，备选 API 后可用性 ≥ 99.9%
* "用户不需要外部感知"—— 实证：**"雨天关窗""路况改道"是高频需求**

**Elephant（团队可能没讨论够）**

* **API 成本账**：天气 / 路况 / POI API 都有调用成本，**每车每天成本多少？商务怎么谈？**
* **车队聚合**：同一区域多车能否共享一次 API 调用？**技术可行，但需要车队管理平台**
* **隐私边界**：GPS 位置上传到云端 → 隐私风险。**建议：位置脱敏（只传区域码，不传精确坐标）**
* **和 01 DMS/OMS 的协作**：外部感知 + 车内人员状态 → 决策。**02 输出情境向量，01 输出人员状态，05 聚合决策**

**5.2 C-F5 车辆状态监控**

*待填*

**5.2.1 一句话定义**

**做什么**：实时采集**车辆自身状态**（电量/里程/胎压/保养/故障码/门窗状态/空调状态），监控异常，聚合后输出给决策层（05）和风险预判（5.5 D-F2），让 AI 能说"电量快没了""胎压异常""保养该做了"。

**不做什么**：

* 不做"外部环境"（5.1 C-F4 的事）
* 不做"车内人员"（01 DMS/OMS 的事）
* 不做"车辆控制"（执行层的事，02 只感知不控制）

**核心价值**：车自己的状态不能只存在仪表盘上，要**被 AI 看到 + 用于决策**。

**5.2.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 车辆状态监控 |
| S1 | **电量预警** | BMS 检测电量 < 20% + 外部 POI 检测到充电桩 → 05 决策 → TTS "电量低，附近有充电桩，要去吗？" |
| S2 | **胎压异常** | 胎压传感器检测胎压 < 2.0 bar → 立即提醒 → TTS "左前轮胎压异常，建议停车检查" |
| S3 | **保养提醒** | 里程 > 10000km 未保养 → 提醒 → TTS "该做保养了，要帮你预约吗？" |
| S4 | **故障码预警** | CAN 总线检测到 P0420（催化剂效率低）→ 上报 → TTS "车有点问题，建议去 4S 店检查" |
| S5 | **门窗状态** | 用户下车后天窗未关 → 检测 → TTS "天窗没关，要帮你关吗？" |
| S6 | **能耗异常** | 能耗突然高出平均值 30% → 识别异常 → 日志记录（不打扰用户，供后续分析） |

**车辆状态数据源**：

|  |
| --- |
| Plain Text 1. BMS：电量 / 里程 / 能耗 / 充电状态 2. 胎压：4 个轮胎胎压 / 温度 3. 保养：里程 / 时间 / 保养项 4. 故障码：OBD-II / CAN 总线 5. 门窗：4 门 / 天窗 / 后备箱状态 6. 空调：温度 / 风速 / 内外循环 |

**5.2.3 输入 / 触发条件**

* **输入**：CAN 总线数据、BMS 数据、OBD 数据
* **触发**：
* 定时轮询（电量每 30s / 胎压每 5min / 故障码每 1min）
* 事件驱动（充电开始 / 门开关 / 异常码）
* 用户查询（"剩多少电"）
* **前置条件**：CAN 总线可读；各 ECU 正常

**5.2.4 执行流程**

|  |
| --- |
| Plain Text 定时轮询 / 事件驱动  ↓ CAN 总线数据采集  ├─ BMS（电量 / 里程 / 能耗）  ├─ 胎压（4 个轮胎）  ├─ OBD（故障码）  ├─ 门窗（门 / 天窗 / 后备箱）  └─ 空调（温度 / 风速 / 循环）  ↓ 数据校验  ├─ 范围校验（胎压 0.5-5.0 bar 合理）  ├─ 连续性校验（电量不应跳变）  └─ 异常数据 → 重新采集 + 告警  ↓ 状态聚合  ├─ 当前状态快照  ├─ 历史趋势（最近 30 分钟）  └─ 异常标记（电量低 / 胎压异常 / 故障码）  ↓ 输出给决策层（05）  ├─ 状态向量：{ battery: 0.18, tire\_pressure: [2.3, 2.1, 2.0, 2.2], faults: [] }  └─ 05 根据状态 + 外部情境 → 决策主动服务  ↓ 本地缓存（30 分钟）  └─ 云端断开时也能用 |

**关键时间预算**：

* CAN 数据采集延迟：**≤ 100ms**
* 数据校验延迟：**≤ 50ms**
* 状态聚合延迟：**≤ 100ms**
* 总延迟：**≤ 500ms**（P95）

**5.5.1 一句话定义**

| S4 | **高速前胎压** | 导航即将上高速 + 胎压 2.1 bar（略低但正常）+ 路况 API"高速畅通" → 预判提醒 TTS "要上高速了，轮胎压力略低，建议提前检查" |
| S5 | **雾霾前换循环** | 天气 API AQI 突增（80 → 150）+ 车辆状态"外循环" → 预判提醒 TTS "外面空气变差了，切内循环吧" |
| S6 | **预判错误** | 预判"2 小时后下雨"但 2 小时后没下雨 → 用户反馈"你预判错了" → 记录失败用于优化算法 |

**风险类型**：

|  |
| --- |
| Plain Text 1. 天气风险：雨/雪/雾霾/大风 2. 续航风险：长途 + 低电 + 气温 3. 胎压风险：高速前胎压略低 4. 充电风险：低电 + 雨天 + POI 5. 空气质量风险：AQI 突增 |

**5.5.3 输入 / 触发条件**

* **输入**：外部情境（5.1 输出）、车辆状态（5.2 输出）、节律（5.3 输出）
* **触发**：
* 定时检查（每 30 分钟）
* 状态变化（导航启动 / 电量突变 / 天气突变）
* 事件触发（进入新地理围栏）
* **前置条件**：外部感知 / 车辆状态 / 节律识别都可用

**5.5.4 执行流程**

|  |
| --- |
| Plain Text 多源数据聚合  ├─ 外部情境：weather / traffic / POI  ├─ 车辆状态：battery / tire / HVAC  └─ 节律：time\_rhythm / geo\_rhythm  ↓ 风险预判规则引擎（05 负责）  ├─ 规则 1：weather.rain\_prob > 0.7 & windows.open → 预判提醒  ├─ 规则 2：nav.distance > 200km & battery < 0.5 & temp < 0°C → 续航预警  ├─ 规则 3：battery < 0.25 & weather.rain\_soon → 充电预警  └─ 规则 4：nav.highway\_approach & tire\_pressure < 2.2 bar → 胎压预警  ↓ 预判置信度打分  ├─ 高（≥ 0.8）→ 直接提醒  ├─ 中（0.6-0.8）→ TTS 确认  └─ 低（< 0.6）→ 不打扰  ↓ 预判提醒执行  ├─ TTS 播报  ├─ UI 卡片显示（预判详情）  └─ 用户可关闭"预判提醒"类别  ↓ 预判反馈  └─ 用户反馈"准/不准"→ 训练优化规则引擎 |

**关键时间预算**：

* 数据聚合延迟：**≤ 200ms**
* 规则引擎延迟：**≤ 100ms**
* 置信度打分：**≤ 50ms**
* 预判提醒延迟：**≤ 1s**（P95）

**5.5.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **预判错误**（雨前提醒但没下雨） | 用户烦躁 | 预判置信度 < 0.6 不打扰；用户可关闭预判提醒类别 |
| **预判误伤**（正常情况被当成风险） | 用户体验差 | 规则引擎严格；多源数据交叉验证 |
| **预判延迟**（已经下雨才提醒） | 预判失效 | 提前 2 小时预判；实时监控天气突变 |
| **多源数据冲突**（外部说没雨但节律说下雨） | 预判不准 | 置信度加权；冲突时取保守策略（提醒） |
| **预判过于频繁**（每 30 分钟提醒一次） | 用户关闭 | 单个场景 2 小时内只提醒一次 |
| **预判算法不可解释**（用户问"为什么说会下雨"答不上） | 用户不信任 | 规则引擎可解释；UI 显示"理由：未来 2 小时天气 API 显示降水概率 80%" |

**Paper Tiger（看起来像风险但不是）**

* "预判很难准"—— 实测：**2 小时内天气预判准确率 ≥ 70%**，30 分钟内 ≥ 90%。70% 准率比 0% 体验好 10 倍
* "预判成本高"—— 多源数据 + 规则引擎，总成本可控

**Elephant（团队可能没讨论够）**

* **规则引擎 vs 机器学习**：预判用规则还是 ML？**建议：V1 用规则（可解释），V2 引入 ML**
* **预判的"提前量"**：多久提前算合适？**建议：天气 2h / 胎压 10km / 续航 100km**
* **预判的"可配置性"**：用户能调"预判阈值"吗？**建议：V1 不开放，V2 开放"预判敏感度"**
* **预判的"负反馈"**：预判错了，用户怎么反馈？**建议：UI 卡片"准/不准"按钮 + 记录**
* **和 05 决策层的协作**：02 提供多源数据，05 做规则预判 + 决策 → **边界清晰**

**6. 自我反驳与边界**

**6.1 外部感知"堆 API"是不是太重?**

**自反**:如果每车都直连天气/路况 API,成本和延迟都是问题。**对冲方案**:

* 高频请求(天气每 10 分钟)走 CDN 缓存
* 车队聚合请求(一车队发一次,车内共享)
* 降级时用"过去 30 分钟缓存+本地规则"

**6.2 时间节律学习是不是伪需求?**

**自反**:用户有规律是事实,但"AI 学规律"可能不如"用户自己设定"。**反驳自反驳**:用户懒得设定,这是 AI 价值。**关键指标**:学出来的规律,用户确认"对"的比例要 > 80%,否则就是瞎猜。

**6.3 地理围栏和 C5 记忆驱动会不会重复?**

**自反**:地理围栏是"在哪里",C5 是"在哪里干了什么"。**本质不同**,但数据源可以共享——C5 的 POI 兴趣点就是地理围栏的候选。**结论**:合一个"地点兴趣库",服务两个场景。
