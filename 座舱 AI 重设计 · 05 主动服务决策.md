**§1 背景与目标**

**1.1 为什么做主动服务决策**

**核心问题**：感知和记忆是"知道",主动服务是"行动"。当前座舱 AI 绝大多数是"你问我答",缺乏**在正确时机主动提供正确服务**的能力。

三大矛盾：

1. **时机难题**：什么时候该主动说话?说早了烦(话痨),说晚了没用(事后诸葛亮)
2. **边界难题**：主动到什么程度?建议 vs 执行 vs 强制,尺度极难把握
3. **信任难题**：用户对"车主动做事"有天然警惕——猜对了是贴心,猜错了是越权

**AI 时代的机会**：

* **规则 + 模型双驱动**：安全/确定场景用规则(低电量必推荐充电),模糊场景用模型(情绪低落推荐音乐)
* **灵敏度可调**：用户自控"AI 主动程度"(沉默/温和/积极),不同人不同阈值
* **反馈闭环**：用户拒绝/接受/忽略主动服务的行为反馈回决策模型,持续校准

**目标**：

* **主目标**：主动服务接受率 >60%(用户觉得"确实该提醒我"的比例)
* **烦扰目标**：用户主动关闭主动服务的比例 <10%
* **安全目标**：安全类主动提醒(疲劳/碰撞风险)漏报率 <1%

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **关键不漏**：疲劳驾驶/低电量/恶劣天气等关键提醒永不错过
* **日常省心**：通勤自动导航、雨天自动除雾、低温自动预热——不用想就能享受到
* **可预期**：主动服务有规律可循,不是"突然冒出来吓一跳"

**对乘员**：

* **关怀恰当**：儿童哭闹时自动降低音量、老人上车自动调整座椅高度——不声张但有温度

**2.2 反价值(不做什么)**

* **不做无阈值主动**：每一条主动服务都有明确的触发条件和静默条件
* **不做不可关闭的主动**：所有非安全类主动服务均可被用户关闭
* **不做猜测性执行**：主动服务以"建议"为主,执行需确认(安全类除外)

**3.3 非场景(明确不做)**

* **商业推送**：不做"路过奶茶店推荐你买一杯"——主动服务不含商业利益
* **社交推荐**：不做"你朋友也在附近"——社交信息不进入决策层
* **情绪操控**：不做"检测到你开心,推荐你买歌"——情绪数据不用于商业

**4.2 可靠性与降级**

1. **决策模型不可用**：降级为纯规则触发(安全类规则不依赖模型)
2. **DMS 故障**：疲劳提醒降级为"每 2 小时定时提醒",不依赖视觉检测
3. **用户反馈数据不足**：新用户前 7 天使用保守阈值,逐步放开

**4.3 安全与合规**

* **安全提醒不可关闭**：疲劳/碰撞/低电等安全类提醒不可被用户关闭
* **执行需确认**：除安全类提醒外,主动执行操作(改导航/调空调)需用户确认
* **反馈数据脱敏**：用户接受/拒绝的行为数据仅用于模型优化,不关联个人身份

**3. 核心能力要求**

**3.1 触发规则引擎**

1. **规则驱动 + LLM 判断混合** — 简单触发用规则(电量<10% → 提醒),复杂触发用 LLM 综合判断
2. **触发优先级** — 安全(疲劳/电量极低) > 便利(节律提醒) > 推荐(POI 兴趣点)
3. **触发冷却** — 同类触发 30 分钟内不重复
4. **触发预算** — 每日上限,按场景分配

**3.2 关怀模式**

1. **情绪触发** — DMS 情绪 + 声音场景 + 用户历史
2. **场景关怀** — 娃哭/疲劳/心情差/晚归/到家 unwind
3. **关怀动作集** — 音乐/空调/座椅/灯光/屏幕的协同调整
4. **关怀克制** — "不打扰 ≠ 不主动",做完就安静,不喋喋不休

**3.3 风险预判**

1. **多源融合** — 车辆状态 + 外部环境 + 用户状态联合
2. **预判时间窗** — 30 分钟~2 小时
3. **可信度** — 预判置信度 < 60% 不主动说,只在用户问到时回答
4. **备选方案** — 提醒同时给 1-2 个备选行动

**3.4 用户可调灵敏度**

1. **三档** — 话痨 / 平衡(默认) / 沉默
2. **预算分配** — 话痨 15 次/天、平衡 8 次/天、沉默 3 次/天
3. **沉默档保留** — 仅"安全类"主动(疲劳/电量/路况风险)
4. **一键切换** — UI 醒目,即时生效

**3.5 反馈闭环**

1. **反馈入口** — 任何主动触发后用户可"喜欢/不喜欢/可以"
2. **画像更新** — 反馈进画像,影响未来阈值
3. **暂停机制** — 连续 3 次"不喜欢"该类主动暂停 7 天
4. **冷启动** — 默认按"平衡档"用,前 7 天数据用于个性化校准

**5. 功能点详细展开**

**5.1 D-F1 关怀模式触发**

**5.1.1 用户价值**

**做什么**：在检测到用户处于**需要关怀的状态**（娃哭 / 疲劳 / 情绪低落 / 晚归 / 到家卸压）时，**自动协同多个执行器**（音乐 / 空调 / 座椅 / 灯光 / 屏幕）做出一组恰到好处的物理调整，**动作先到位再说话、做完就安静**，不喋喋不休地表达"我在关心你"。

**不做什么**：

* 不做"持续陪聊式"关怀（NOMI 被骂的反例——调整完之后还要追问"感觉好些了吗？"）
* 不做"硬塞式"关怀（用户明确拒绝过的关怀模式，至少 7 天内不再触发同类）
* 不做诊断类表述（不说"你看起来抑郁了，要不要去医院"——越界、不准、吓人）
* 不做没有物理动作的"纯口头关怀"（只说"辛苦了"却没做任何事，是塑料关怀）

**核心价值主张**：

* **物理动作优先于语言表达**——座椅通风开了、空调调好了、音乐切到放松歌单了，这些是真关怀；只说"我懂你"是塑料关怀
* **做完就安静**——不追问"感觉好些了吗"、不持续陪聊、不表达"我在关心你"，给用户空间
* **可拒绝且记住拒绝**——用户说"不喜欢"后，7 天内同类关怀不再触发；拒绝 ≥ 3 次的关怀场景进入长期冷却（30 天）

**5.1.3 成功指标**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方式 | 备注 |
| 关怀触发准确率 | ≥ 70% | 用户"喜欢/可以"反馈 ÷ 总触发次数 | V1 目标；V2 提升到 80% |

| 关怀触发延迟（信号到动作） | ≤ 1.5s | 从信号检测到首个动作可见的时间 | 超过 2s 用户感知"反应慢" |
| TTS 发声延迟 | ≤ 1s | 从决策到 TTS 第一个字 | 流式 TTS 要求 |
| 关怀场景覆盖率 | ≥ 60% | 有关怀需求的场景中被触发的比例 | V1 预置 6 个场景包，覆盖高频场景 |

**5.1.4 放弃条件**

|  |  |  |
| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 关怀触发准确率持续低于 50% | 连续 2 周 | 关闭该场景包，回退到纯规则触发（如疲劳 L2 只做物理调整不说话） |
| 用户主动关闭关怀模式比例 > 30% | 上线后 1 个月 | 关怀模式默认关闭，改为 opt-in |
| 关怀动作执行成功率 < 80% | 连续 1 周 | 暂停关怀模式，排查执行器协同问题 |
| 用户投诉"关怀过度"/"监视感" | 单周 > 10 例 | 立刻收紧触发阈值，灵敏度默认档从"平衡"降为"沉默" |
| LLM 判断延迟 > 2s | 单日 > 20% 请求 | 关闭 LLM 慢路径，全部降级为规则快路径 |

**5.1.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 关怀触发但动作全部失败 | 执行器（音乐/空调/座椅）全部不可用 | TTS 说了"给你调好了"但什么都没变 → 塑料关怀 | 关键动作失败 → 回退整个场景包，不触发 TTS；记录失败原因到日志 |
| 关怀触发时机不对 | LLM 判断错误（如误判"心情差"） | 用户觉得"AI 瞎猜" | 用户点"不喜欢" → 该场景包 7 天冷却；置信度阈值提升 0.1 |
| 关怀动作与用户意图冲突 | 用户正在手动调音乐，AI 同时切歌 | 用户操作被打断 → 烦躁 | 检测到用户手动操作 → 关怀动作延迟 5s；若用户操作持续 > 10s 则取消关怀 |
| 副驾睡着但 TTS 发声 | OMS 睡眠态检测失败 | 吵醒副驾 → 投诉 | TTS 前强制查询 OMS 睡眠态；OMS 不可用 → 降级为"不发声"模式 |
| 关怀频次过高 | 预算池未生效 / 冷却时间过短 | 用户觉得"话痨" | 预算池强制限流；关怀类触发后锁定 15-30 分钟 |
| 隐私数据泄露感 | 用户发现 AI 知道"昨晚没开冰箱" | 用户觉得"被监视" | 隐私数据源（日历/消息/智能家居）独立开关；默认关闭，用户主动开启 |
| 关怀模式在海外市场引发文化冲突 | 欧美用户不习惯"主动关怀" | 投诉"过度干预" | 海外版默认"沉默档"；关怀模式 opt-in |

**5.1.6 技术钩子（给沈砚）**

1. **关怀触发判断器架构**：规则快路径（娃哭 L2+ / 疲劳 L2+ / 到家）≤ 100ms；LLM 慢路径（情绪低落 / 节律异常）≤ 800ms。混合架构怎么做？LLM 调用频率控制（成本）？
2. **关怀场景包定义格式**：6 个场景包（娃哭 / 晚归疲劳 / 心情差 / 到家 unwind / 长途疲劳 / 节律异常）预置在哪？YAML / JSON / DSL？V2 能否 OTA 增加/修改场景包？
3. **多执行器协同调用**：一个关怀场景包对应 3-5 个动作（音乐 / 空调 / 座椅 / 灯光 / HUD / TTS），并发执行 / 失败降级的事务模型是什么？关键动作失败如何识别并触发场景包回退？
4. **OMS 睡眠态强制检查接口**：触发 TTS 前必须查询"乘员是否睡着"信号（来自《01》C-F2 OMS），这个查询的延迟预算是多少？是实时查询还是订阅式？
5. **关怀锁定期实现**：同一场景包 15-30 分钟内不重触发，这个计时器是端侧持久化还是云端？断电重启后还保留吗（影响用户体验一致性）？
6. **隐私开关颗粒度**：用户能否单独关闭"从日历推断情绪"但保留"从 DMS 推断疲劳"？需要关怀模式的数据源依赖图。
7. **反馈数据回流**：用户"不喜欢"某场景包后，阈值调整是端侧立即生效还是等云端画像更新？时延预算？
8. **关怀 TTS 音量与方向控制**：定向播报（只让主驾听到）的技术方案——多音区扬声器？波束成形？需要硬件能力确认。
9. **预算池跨服务共享**：关怀类预算池（见 D-F5）需要跨进程/跨服务查询（音乐 agent / 空调 agent 都要查），用什么机制（Redis-like / 共享内存 / 总线消息）？

**5.2 D-F2 风险预判提醒**

**5.2.1 用户价值**

**做什么**：在用户**还没遇到麻烦之前**，基于多源数据（天气预报 / 续航计算 / 胎压监测 / 充电桩状态 / 限行政策 / 保养里程）做**2 小时内的风险预判**，提前告知并给出可操作建议（改路线 / 提前充电 / 预约保养），让用户有时间调整计划而不是临时抓瞎。

**不做什么**：

* 不做"已经发生"的告警（电量已低于 5% 才提醒 → 太晚了，那是 B-F4 偏离纠正的事）
* 不做"太远"的预判（3 天后可能下雨 → 用户记不住，噪音）
* 不做"低置信度"的猜测（置信度 < 0.6 的风险不提醒，宁可漏报不误报）
* 不做"无解"的风险提醒（告诉用户"明天限行"但用户没有备用车 → 徒增焦虑）

**核心价值主张**：

* **2 小时黄金窗口**——足够用户调整计划，不至于太远记不住
* **多源联合判断**——天气 API + 车辆传感器 + 地图数据 + 用户历史，交叉验证降低误报
* **可操作建议优先**——不只说"可能有问题"，还要说"怎么办"

| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 风险预判准确率持续低于 50% | 连续 2 周 | 关闭该类风险预判（如"天气 + 续航"），保留高准确率类别（如"胎压 + 长途"） |

| 多源数据融合延迟 > 1s | 单日 > 30% 请求 | 降级为单源预判（只用车辆传感器，不查天气/充电桩 API） |
| 用户投诉"过度提醒" | 单周 > 5 例 | 风险预判默认关闭，改为 opt-in |

**5.2.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 预判错误（天气预报不准） | 天气 API 数据过时或不准 | 用户按建议改路线，结果没下雨 → "AI 瞎猜" | 天气数据源多源验证（2 个 API 交叉）；误报后该类风险 24 小时冷却 |
| 预判太晚（已经在路上） | 触发条件判断延迟 / 用户突然改导航 | 提醒"前方暴雨"时用户已经开进雨区 → 无用 | 预判触发条件前置到"设置导航"时刻，不等出发 |
| 建议不可操作（充电桩都满） | 推荐的备选充电站也满了 | 用户点了建议，到了还是要排队 → 失望 | 充电桩推荐时查询实时占用率 + 预留 10 分钟到达时间的占用率预测 |
| 多源数据冲突（续航模型 vs 天气） | 续航模型说"够"，天气折损说"不够" | 用户不知道信哪个 | 置信度加权：续航模型 0.7 × 天气折损 0.6 = 综合置信度 0.42 < 0.6 → 不提醒 |
| 风险预判频次过高 | 预算池未生效 / 多个风险同时触发 | 用户觉得"AI 焦虑" | 风险类预算池独立（见 D-F5），单日最多 3 次风险提醒 |
| 用户已知风险但 AI 还提醒 | 用户刚查过天气 / 刚看过胎压，AI 不知道 | 用户觉得"AI 不懂我" | 用户主动查询行为（打开天气 app / 查看胎压）记入短期记忆，30 分钟内不提醒同类风险 |

**5.2.6 技术钩子（给沈砚）**

1. **多源数据融合架构**：天气 API（2 个源交叉验证）+ 充电桩实时占用 API + 车辆传感器（胎压/续航/里程）+ 地图 API（限行/路况）+ 用户画像（历史偏好/拒绝记录）。这 5 类数据源的查询并发 / 超时 / 降级策略是什么？单个源超时是否阻塞整个预判？
2. **置信度计算模型**：多源数据的置信度如何加权？例如"续航模型 0.7 × 天气折损 0.6 = 综合 0.42"，这个乘法规则是写死还是可训练？V2 能否用 ML 学习最优权重？
3. **风险预判触发时机**：是"设置导航"时触发、"出发前 30 分钟"触发、还是"实时监控"触发？不同触发时机对算力 / 电量 / 用户体验的影响？
4. **风险类别优先级**：安全类（胎压/续航极低）> 便利类（充电桩排队）> 提示类（保养里程）。多个风险同时触发时如何排序？是否合并提醒（"有 2 个风险，先说最重要的"）？
5. **风险预判的撤销机制**：用户点"不需要"后，该风险是立刻冷却 7 天、还是只冷却"当前这一次"？冷却粒度（按风险类别 / 按具体场景）？
6. **充电桩占用率预测**：当前占用率 + 到达时间 → 预测到达时占用率。这个预测模型是规则（历史统计）还是 ML？数据从哪来（众包 / 充电桩运营商 API）？
7. **天气数据的时空精度**：天气预报精确到"2 小时后 + 目的地 10km 范围"，这个精度能保证吗？天气 API 的 SLA 是什么？
8. **风险预判的 A/B 测试框架**：置信度阈值（0.6 / 0.7 / 0.8）、提前量（1h / 2h / 3h）、建议类型（改路线 / 充电 / 预约）都需要 A/B 测试优化。测试框架怎么搭？
9. **风险预判与安全告警的边界**：胎压 2.1 bar 是"风险预判"，1.8 bar 是"安全告警"（P0）。这个阈值在哪定义？两套系统如何避免重复提醒？

**5.3 D-F3 习惯驱动主动服务**

**5.3.1 用户价值**

**做什么**：基于用户的**长期出行习惯**（早晚通勤时间 / 常去地点 / 偏好歌单 / 习惯空调温度），在合适的时机**主动提供服务**，让车"懂用户的节奏"——早上上车自动放新闻 + 切到通勤模式、晚上回家路上自动切到放松歌单 + 空调调到家里温度。

**不做什么**：

* 不做**画像未稳定的主动服务**（数据不足 7 天的用户不主动，避免"瞎猜"）
* 不做**与当前情境冲突的主动服务**（用户今天破例去机场，不能还按通勤模式服务）
* 不做**侵入式个性化**（不主动提"上次你和老婆吵架后听了这首歌"——记得，但不说出来）
* 不做**强制覆盖**（用户主动改了音乐 / 空调 / 路线 → 立刻让位，不撤回）

**核心价值主张**：

* **节奏感优于功能感**——好的主动服务让用户感觉"自然"，不是"惊艳"
* **画像稳定后才主动**——冷启动期（前 7 天）只观察不主动；7 天后逐步开启，30 天画像稳定
* **当前情境优先于历史习惯**——情境（导航目的地 / 时间 / 同行人）和习惯冲突时，情境赢

**5.3.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 习惯识别 | 主动服务动作 | 关键约束 |
| S1 | **早高峰通勤**：工作日 8:00 用户上车，过去 30 天有 22 天此时段从家到公司 | 通勤习惯置信度 0.92 | 自动切"通勤模式"（HUD 显示路况 + 音乐切早间新闻 / 用户惯听播客 + 空调按用户习惯温度 22°C）+ 不发声 | 周末 / 节假日不触发；用户已设置导航到非公司目的地则不触发 |
| S2 | **晚归回家**：工作日 19:00-21:00 用户从公司方向上车，目的地未设置 | 通勤回家习惯置信度 0.88 | 自动设导航到家（仅建议，需用户确认）+ 切"放松模式"（音乐切晚间放松歌单 + 空调降到 20°C / 用户习惯回家温度 + 氛围灯切暖橙）+ TTS（一句）："导航回家，路上有点堵" | 用户上次明确设置非家目的地则不主动设导航 |
| S3 | **周末家庭出行**：周六 10:00 全家上车（OMS 识别 4 人 + 后排有儿童座椅），过去 4 周有 3 个周末此场景 | 家庭出行习惯置信度 0.75 | 自动切"家庭模式"（音乐切儿童歌单 + 后排空调单独控温 + 后排屏幕显示动画）+ 不发声 | 需要 OMS 识别成功；若识别只 2 人则降级为"普通出行模式" |
| S4 | **健身房常客**：每周二/四 19:30 用户去健身房（过去 8 周一致） | 习惯置信度 0.85 | 周二/四 19:00 推送通知："今天去健身房吗？路上有点堵，建议提前 10 分钟出发" + 一键导航 | 仅推送，不强制设导航；推送后用户没点击 → 30 分钟内不再推送 |
| S5 | **临时破例不打扰**：周三早上 8:00 用户上车，但导航设置为机场（不是公司） | 当前情境（机场）覆盖通勤习惯 | 不切通勤模式；按"机场出行"场景服务（HUD 显示航班动态 / 提醒登机时间） | 当前情境优先于历史习惯，无例外 |
| S6 | **新用户冷启动**：用户购车第 3 天 | 数据不足，画像未稳定 | 不做主动服务；只在用户问"我的习惯"时回答"还在了解你，再用 4 天就能更懂你" | 冷启动期 7 天；7-30 天逐步开启（先开通勤，30 天后开家庭模式） |
| S7 | **画像反馈机制**：用户连续 3 天上车后手动关掉自动播放的音乐 | 隐式"不喜欢"信号 | 自动播放习惯进入"待验证"状态，下次不主动播；7 天后用问询替代："要不要放音乐？" | 隐式反馈占画像权重 0.3，显式反馈（点喜欢/不喜欢）占 0.7 |

**5.3.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 误判通勤（用户今天请假） | 时间匹配但目的地变了 | 自动切通勤模式但用户去了医院 → 烦躁 | 检测到导航目的地与历史习惯不符 → 不切模式；若用户没设导航则等 5 分钟再判断 |
| 习惯过时（用户换了工作） | 画像没及时更新 | 还按旧公司路线推荐 | 检测到连续 5 天去新地点 → 画像更新；旧习惯进入"待验证"状态 30 天后清除 |
| 多用户混淆（夫妻共用一辆车） | 没识别当前驾驶员 | 老公上车放老婆喜欢的歌 → 尴尬 | 必须依赖 DMS 人脸识别；识别失败 → 不主动服务（只切默认模式） |
| 周末误判工作日 | 节假日 / 调休没识别 | 周末早上 8 点切通勤模式 | 接入节假日 API；用户手动取消后该日 24 小时内不再触发 |
| 习惯学习过快 | 1-2 次行为就当作习惯 | 用户偶尔去一次的地方变成"常去" | 习惯识别需要 ≥ 5 次/月 + 持续 ≥ 3 周才入画像 |
| 习惯学习过慢 | 数据足够但未识别 | 用户连续去 2 个月健身房，AI 还没"学会" | 画像更新频率：日终批量 + 关键事件实时；ML 模型每周重训 |
| 隐私泄露感 | 主动说出"上次你来过这" | 用户觉得"AI 监视我" | 习惯只用于"主动服务"不用于"对话"；用户问起再说，不主动暴露 |

**5.3.6 技术钩子（给沈砚）**

1. **画像数据源融合**：通勤习惯（位置 + 时间）/ 偏好（音乐 / 温度 / 路线）/ 家庭出行（OMS 识别）/ 兴趣（POI 频次）。这些数据在《04》记忆层管理还是单独的画像服务？画像更新频率（实时 / 日终 / 周更）？
2. **习惯置信度模型**：通勤 22/30 = 0.73，但要在不同时段、不同路线下加权。规则（频次 + 持续时长 + 一致性）还是 ML（贝叶斯 / 神经网络）？冷启动期到稳定期的过渡曲线？
3. **当前情境 vs 历史习惯优先级**：情境检测（当前导航目的地、同行人、时间）赢的判定规则。是简单 if-else 还是策略引擎？多个情境信号冲突时（如目的地是公司但同行人是孩子）？
4. **多用户识别**：DMS 人脸识别的可靠性（白天 / 夜晚 / 戴墨镜 / 戴口罩）。识别失败时的降级策略（用蓝牙手机ID / 账号登录）？
5. **隐式反馈与显式反馈的权重**：隐式 0.3 + 显式 0.7 是写死还是可调？随时间衰减（旧反馈权重降低）？
6. **画像存储与隐私**：习惯数据存端侧 / 云端？跨设备同步策略？用户切换账号 / 卖车后的数据清理？
7. **习惯识别的 A/B 测试**：习惯学习速度（5 次/月 vs 7 次/月）、画像更新频率、置信度阈值都需要测试。测试样本量和周期？
8. **节假日与特殊日期**：接入节假日 API 是必要的，还有用户的个人日历（请假 / 出差）。这些数据如何接入而不侵犯隐私？
9. **画像可解释性**：用户问"为什么自动放这个歌"，能否给出可读的解释（"因为周一晚上你常听这个"）？画像 → 解释的映射机制？

**5.4 D-F4 用户可调灵敏度**

**5.4.1 用户价值**

**做什么**：给用户一个**全局开关**，用三档（话痨 / 平衡 / 沉默）一键调整 AI 主动服务的频次。不同用户对"被打扰"的容忍度差别极大——有人喜欢车主动说话（NOMI 粉丝），有人只想要安静（特斯拉粉丝）——一套默认配置不可能满足所有人。用户自己拍板。

**不做什么**：

* 不做**过于复杂的设置**（不要求用户配 20 个开关，只给三档 + 分类细调）
* 不做**强制推荐档位**（不根据"画像推测用户喜欢话痨"默认给话痨档）
* 不做**静默档完全沉默**（安全类主动必须保留，否则失去座舱 AI 的底线价值）
* 不做**一键切换需要确认**（用户要立刻安静，点一下就安静，不问"你确定吗"）

**核心价值主张**：

* **三档 + 分类细调**——90% 用户用三档即可，10% 细分用户可进高级设置单独关掉某类
* **静默档保留安全类**——沉默 ≠ 失声，关键时刻（疲劳 L2+ / 电量极低 / 路况风险）必须说话
* **即时生效无延迟**——切档后下一次触发立刻按新档执行，不等下次启动

**5.4.6 技术钩子（给沈砚）**

1. **档位配置存储与同步**：用户档位设置存端侧 / 云端？跨设备（手机 app / 车机）同步？离线时生效规则？
2. **档位与预算池的映射**：话痨 15 次/天、平衡 8 次/天、沉默 3 次/天——预算池如何按档位动态切换？D-F5 的预算池实现需要考虑档位参数。
3. **切档事件广播**：切档后所有 agent（音乐 / 空调 / 座椅 / 关怀判断器 / 风险预判器）都要立刻知道。用什么机制（总线 / 共享配置 / 轮询）？广播延迟预算多少？
4. **安全类白名单**：疲劳 L2+ / 电量 < 10% / 胎压 P0 / 爆胎 / 事故 / 碰撞预警——这些必须沉默档也触发。白名单在哪维护？如何防止误改？
5. **高级分类细调的配置结构**：10 几类主动服务各有独立开关，配置的存储结构（flat 还是树形）、UI 渲染与配置的解耦机制？
6. **临时档位优先级**：客人模式（当次行程）> 蓝牙电话沉默（电话期间）> 用户档位 > 默认档。这个优先级栈如何实现？
7. **档位推荐模型**：根据用户行为（撤销次数 / 主动关闭次数）推荐"可能更适合的档位"。用规则还是 ML？模型训练数据从哪来（众包 / 用户埋点）？
8. **档位 A/B 测试**：默认档是"平衡"还是"沉默"，不同地区（中国 vs 海外）默认档不同，都需要 A/B 验证。测试框架？
9. **方向盘快捷键绑定**：用户可选"方向盘某键 = 一键切沉默"。快捷键冲突管理（和语音唤醒 / 媒体控制）？

**5.5 D-F5 触发阈值与冷却**

**5.5.1 用户价值**

**做什么**：座舱 AI 的"话痨综合症"是量产车的头号反例（NOMI / 理想都踩过）。D-F5 是**后端治理机制**——通过**预算池 + 冷却 + 阈值**三把锁，强制约束"每天说多少次 / 同一类多久不能重触发 / 置信度多高才触发"，让"主动"不会失控。用户不直接看到这层，但感知到的"不话痨"就是它的功劳。

**不做什么**：

* 不做**无限制的主动服务**（一天 100 次主动服务必然惹人烦）
* 不做**按单个功能限流**（一个功能限 5 次，10 个功能各限 5 次 = 每天 50 次，依然话痨）
* 不做**忽略用户档位的全局限流**（沉默档和话痨档不能用同一套限流规则）
* 不做**硬性截断**（预算用完就什么都不做 = 关键时刻失声；安全类永远豁免）

**核心价值主张**：

* **三把锁联动**——预算池（总量控制）+ 冷却（同类重复抑制）+ 置信度阈值（质量门槛）
* **分类独立预算**——关怀 / 风险 / 习惯 / 推荐各有独立配额，不互相抢
* **安全类豁免**——P0 类风险不受预算 / 冷却限制，但有专门的"防抖"机制（同一告警 30s 内不重复）

**5.5.5 失败路径**

| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 预算池竞争条件（并发触发） | 多 agent 同时申请预算导致超额 | 某些场景下突然连续触发 3-4 个主动服务 | 预算池访问加分布式锁；批量申请原子化 |
| 冷却时钟错位（车辆断电后） | 计时器未持久化 | 本应冷却的场景包重启后立刻触发 | 冷却时间戳持久化到端侧存储；启动时校验 |
| 置信度阈值过于严格 | 阈值设得过高导致大量信号被拦截 | 用户感受不到主动服务 | 灰度期间观察拦截率，> 30% 则调低阈值 |
| 自适应阈值失灵 | 反馈数据没回流到阈值调整 | 用户拒绝多次但阈值没调 | 反馈数据实时入画像；阈值调整每日批量更新 |
| 安全类白名单遗漏 | 新增安全类场景未加入白名单 | 沉默档 / 预算满时该事件被拦截 | 白名单代码审计；新增安全类必须走白名单评审 |
| 防抖机制误伤 | 相似但不同的 P0 告警被去重 | 两个独立告警只响一个 | 防抖按"告警 ID + 来源"组合去重，不只按 ID |
| 预算池无法降级 | 关键服务因预算不足被拦截 | 关怀类预算用完，用户疲劳时没响应 | 关怀 / 风险 / 安全分池；关怀类独立预算 3 次/天保底 |

**5.5.6 技术钩子（给沈砚）**

1. **预算池实现**：关怀 / 风险 / 习惯 / 推荐 4 类独立预算池 + 全局总预算。存储（内存 / Redis / 端侧数据库）？跨进程访问机制？持久化策略（断电后是否保留）？
2. **冷却时钟实现**：每个场景包独立冷却计时器，最多 6 个关怀场景包 + 10+ 风险场景 + N 个习惯场景 = 20+ 计时器。存储与查询效率？断电持久化？
3. **置信度阈值配置**：每个场景包独立阈值，自适应调整。配置存储（YAML / 数据库）？热更新 vs 冷启动？自适应调整的回退机制（出错时回到默认值）？
4. **预算池 + 档位联动**：D-F4 切档时预算池上限立刻变化（话痨 15 / 平衡 8 / 沉默 3）。已消耗预算如何处理（清零 / 保留 / 按比例换算）？
5. **安全类白名单维护**：硬编码 vs 配置。维护流程（新增安全类需要走什么评审）？
6. **分布式锁 / 原子操作**：预算池并发访问（多个 agent 同时申请），用什么机制保证原子性？性能（并发量 / 延迟）？
7. **防抖与冷却的区分**：防抖（去重同一告警）vs 冷却（节奏控制不同时刻的相似事件）。实现上是否共用一套机制？
8. **预算池监控 / 告警**：预算池使用率、阈值拦截率、自适应调整频次——这些监控指标如何采集和可视化？
9. **A/B 测试支持**：预算上限（8 vs 10 vs 12）、冷却时长（15min vs 30min）、置信度阈值（0.7 vs 0.75 vs 0.8）都需要 A/B。框架如何支持按用户分组的配置覆盖？

**5.6 D-F6 用户反馈闭环**

**5.6.1 用户价值**

**做什么**：每次主动服务触发后，给用户**一个极轻量的反馈入口**（喜欢 / 可以 / 不喜欢，三选一），把反馈**即时回流到画像和阈值**，让 AI 越用越懂用户——这一次错了，下一次就会调整。**闭环的意义是让 AI 可教育**，而不是让用户一次次忍受同样的错误。

**不做什么**：

* 不做**弹窗式反馈**（不弹对话框问"这次关怀怎么样" → 打断体验）
* 不做**必填式反馈**（不阻塞用户后续操作）
* 不做**反馈后不行动**（反馈数据不影响任何阈值 = 形同虚设）
* 不做**过度采集**（不采集未涉及到的行为数据作反馈信号）

**核心价值主张**：

* **隐式 + 显式双通道**——显式（点击喜欢/不喜欢）+ 隐式（手动撤销 / 关闭 / 修改）
* **反馈即时生效**——"不喜欢"点击后该场景阈值立刻调整，不等批量处理
* **反馈可解释**——用户问"为什么你记得我不喜欢这个" → AI 能回答"上次这个时间段你点了不喜欢"

| # | 场景（颗粒度到具体人/时间/状态） | 反馈入口与行为 | 关键约束 |

| S8 | **画像可解释性**：用户问"你为什么不给我放音乐了" | 回答："上周三晚上 9 点你说过这时间别放音乐，我记住了，要不要现在恢复？" | 反馈记录可查询；回答时不暴露过多细节（不说"你连续点了 3 次不喜欢"） |
| S9 | **反馈冷却机制**：用户一天内对同一场景包反馈 5 次以上 | 判定为"异常反馈"（可能是测试 / 误操作），不调整阈值，人工介入核查 | 防止单次大量反馈污染画像 |

| 显式反馈入口不明显 | UI 设计问题 | 用户不知道在哪反馈 | 方向盘物理键 + 中控浮标 + 语音"喜欢"三通道；用户第一次使用时一次性教学 |

| 隐式信号误判 | 用户手动切歌不等于不喜欢（可能只是想换个心情） | 错误地把正常行为标为"不喜欢"，阈值飘移 | 隐式信号权重低（0.3）；需多次累积才调整；关键决策仅依赖显式反馈 |
| 反馈数据污染（他人使用） | 用户借车给朋友，朋友点了不喜欢 | 下次车主用时 AI 行为变了 | 反馈数据绑定当前驾驶员（DMS 识别）；识别不到驾驶员则不采集反馈 |
| 阈值调整过度振荡 | 一次喜欢 / 一次不喜欢来回调 | AI 行为不稳定 | 阈值调整步长小（±0.05）；触发调整需要累积信号（如连续 3 次同向反馈） |
| 反馈可解释性暴露隐私 | 回答"为什么不放音乐"时暴露详细历史 | 用户觉得"被监视" | 回答颗粒度：只说"上周你说过这时间别放" → 不说"你连续点了 3 次不喜欢" |
| 反馈数据跨账号泄露 | 家庭共享车 / 账号切换 | A 的反馈影响 B 的体验 | 反馈数据强账号隔离；切账号后加载对应画像 |

**5.6.6 技术钩子（给沈砚）**

1. **反馈入口的多通道统一**：物理按键 + UI 浮标 + 语音三路输入，如何统一到同一反馈事件？事件时序去重（用户连点两路）？
2. **反馈到阈值的即时调整**：端侧立即生效（不等云端），但云端画像需要最终一致。端云同步机制（push / pull / 双写）？冲突解决？
3. **隐式信号识别**：用户手动撤销（切歌 / 改空调 / 改路线）的识别。哪些动作算"反馈信号"、哪些是"正常操作"？判断逻辑是规则还是 ML？
4. **反馈误操作回滚**：3 秒内反向点击的回滚机制。存储上回滚 / 逻辑上覆盖？回滚超过 3 秒怎么处理？
5. **自然语言反馈的 NLU**：用户说"这别放了"要识别为反馈信号。和 03 NLU 模块的边界？是 03 增加新意图，还是 05 独立识别？
6. **反馈数据的账号隔离**：家庭共享车、朋友借车场景。DMS 识别失败时反馈数据存还是不存？
7. **反馈可解释性查询接口**：用户问"为什么" → AI 查询该场景包的反馈历史。查询接口的数据结构 / 隐私边界 / 颗粒度控制？
8. **异常反馈检测**：一天内同一场景包反馈 > 5 次，判定为异常。规则阈值 vs ML 异常检测？人工介入流程？
9. **反馈数据 A/B 测试支持**：反馈入口（物理键 vs 浮标）、窗口长度（5s vs 10s）、权重（0.3 vs 0.5）都需要 A/B。框架支持？
10. **反馈数据的生命周期**：超过 90 天的反馈是否过期？用户换车 / 卖车 / 注销账号时的数据清理？

**5.7 D-F7 虚拟形象人格层**

**5.7.1 用户价值**

**做什么**：给座舱 AI 一个**可感知的人格**——不是炫技的 3D 动画小人，而是**一致的性格、语气、回应风格**，让用户感觉"这个 AI 有脾气、有边界、有温度"，区别于其他品牌。V1 用**静态 icon + 人格化文案**，V1.5 升级到**动态情绪 IP**，V2 探索**品牌代言人级 IP**（如理想的"理想同学"、蔚来的"NOMI"）。

**不做什么**：

* 不做**过度拟人化**（不强迫用户把 AI 当成"朋友" / "管家" / "伴侣"，那是用户的选择不是品牌的强推）
* 不做**情感承诺**（AI 不说"我会一直陪着你"这种煽情 + 不兑现的话）
* 不做**人格分裂**（一会儿幽默一会儿严肃 → 用户对品牌无认知）
* 不做**大量动画占屏**（炫技但干扰驾驶）

**核心价值主张**：

* **人格 ≠ 可爱**——理想同学的设计好在"靠谱 + 专业 + 偶有幽默"，不是"装可爱"。我们学这个
* **三层结构**：音色（见《08》T-F1）+ 文案风格（D-F7 负责）+ 视觉形象（D-F7 负责）
* **V1 轻量起步**——静态 icon + 固定文案模板；用数据决定 V1.5 要不要动态 IP

**5.7.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 人格层表现 | 关键约束 |
| S1 | **日常问候 - 早晨上车**：用户上车，AI 识别为本人 | icon 显示"问好"（微笑）状态；TTS："早上好，今天有雨，给你打开雨刷" | 问候限早晨一次；问候语言简意赅，不寒暄 |
| S2 | **关怀场景 - 疲劳**：用户疲劳 L1 | icon 切"关心"表情（带一点担忧的眼神）；TTS："路上有点远，给你把歌单切到放松的了"（不说"辛苦了"塑料话） | 关怀不煽情；动作优先语言 |
| S3 | **严肃场景 - 安全告警**：胎压 P0 | icon 切"警示"（红色边框闪烁）；TTS 语气严肃："胎压异常，建议靠边停车检查" | 严肃场景无幽默 / 无装饰；动画简洁有力 |
| S4 | **幽默场景 - 用户开玩笑**：用户说"车，给我来杯咖啡" | icon 小机灵表情；TTS："这个真没法，不过给你导了最近的星巴克" | 幽默限用户主动开玩笑时回应；AI 不主动说段子 |
| S5 | **拒绝场景 - 模糊指令**：用户说"随便放点什么" | icon 略思考状；TTS："放你上周听得最多的 A 还是昨天那首 B" | 拒绝不生硬；给出具体选项让用户选 |
| S6 | **道歉场景 - 执行失败**：TTS 响应，但用户要的操作实际失败（如蓝牙断开没切歌） | icon 略歉意；TTS："刚才没切成，现在再试一下"（不说"对不起"过度道歉） | 道歉不重复（一次即可）；重点是下一步动作 |
| S7 | **沉默档场景**：用户切沉默档 | icon 小淡出动画（表示"收到，我会安静"）；无 TTS | 人格在沉默档仍"在"，只是不说话 |
| S8 | **车辆锁车后**：用户下车锁车 | icon 淡出；TTS（可选）："到家了，歌单存起来" | 离车的"再见"场景限 1 秒内；不煽情 |
| S9 | **V1.5 动态 IP 场景**：升级后 icon 变为简单 3D 形象（如车载"管家"），有呼吸动画 + 语音同步唇动 | icon 成为"可见的人物"；动画不超过 2 秒；不占用主屏 | 动态 IP 限定在某个区域（HUD / 中控小窗）；不全屏出现 |

**5.7.4 放弃条件**

|  |  |  |
| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 用户识别一致性 < 3.0 | 月度调研 | 人格设计失败，需重新定义风格（和品牌战略同步） |
| 人格层干扰驾驶投诉 > 5% | 单月 | 动画简化；icon 改为更低调的显示方式 |
| "塑料感"投诉 > 10% | 单月 | 文案风格全盘检讨（可能过度拟人化 / 装可爱） |
| V1.5 动态 IP 好评率 < 3.0 | 发布 3 个月后 | 回退到静态 icon；动态 IP 不再投入 |
| 情绪与场景匹配率 < 70% | 抽样 | icon 状态机有 bug；或者触发逻辑错误 |

**5.7.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 情绪表达过度 | icon 动画过多 / 过大 | 用户觉得"花哨" / "幼稚" | 动画默认最小化；只在关键场景（关怀 / 告警）显示 |
| 文案风格不一致 | 多团队写文案没对齐 | 有的文案幽默有的生硬，品牌无辨识 | 统一的文案风格指南（术语 / 语气 / 长度）+ 自动化检查工具 |
| 严肃场景出现表情 | icon 状态机错误 | 胎压告警时 icon 还在"微笑" → 轻佻 | icon 状态机优先级：告警 > 关怀 > 日常；状态冲突时以高优先级为准 |
| 情感承诺无法兑现 | 文案写了"我一直在" 但 AI 重启后不记得 | 用户觉得"AI 撒谎" | 禁止情感承诺类文案；风格指南硬约束 |
| 动画分心 | HUD 动画占屏 / 频繁闪烁 | 干扰驾驶 | HUD 区域动画严格限制（颜色 / 大小 / 频次）；驾驶状态动画简化 |
| 多用户混淆导致人格错乱 | 家庭共享车，AI 对不同用户用同一人格 | 对老人用年轻化幽默 → 不合适 | 人格本身是品牌统一的，但**文案口吻按用户年龄段微调**（不是改人格，是改细节） |
| 动态 IP 硬件性能问题 | V1.5 3D 形象占用 GPU 资源 | 中控屏卡顿 / 发热 | 动态 IP 限低功耗模式；硬件不支持则回退静态 icon |

**5.7.6 技术钩子（给沈砚）**

1. **人格层与音色层的配合**：D-F7 管文案 / 视觉，《08》T-F1 管音色。配合机制：同一"关怀场景"下，T-F1 音色（care 变体）+ D-F7 文案（关怀风格）+ icon（关心表情）如何协同触发？
2. **icon 状态机**：默认 / 问好 / 关心 / 警示 / 思考 / 歉意 / 小机灵 / 淡出——至少 8 个状态。状态转移规则 + 优先级（告警 > 关怀 > 日常）？状态机实现位置（端侧本地）？
3. **文案风格指南的自动化检查**：术语白名单 / 禁止词（如"辛苦了" / "我一直陪着你"）/ 长度限制 / 语气匹配。CI 集成？上线前自动卡点？
4. **动态 IP 的渲染性能**：V1.5 3D 形象不能占用过多 GPU。性能预算（占用率 / 内存 / 功耗）？硬件降级策略？
5. **多语言人格一致性**：V2 海外版中英文双语，人格风格如何跨语言保持一致？本地化怎么做（直译 vs 文化适配）？
6. **人格与用户画像的交互**：文案口吻按用户年龄段微调（如对 60+ 用户不用网络流行语）。数据如何从画像流到文案生成？
7. **icon 与 HUD 的空间冲突**：HUD 有限空间内 icon + 导航 + 仪表信息。优先级和空间分配？驾驶状态下 icon 是否显示？
8. **人格在分布式多屏的表现**：中控 / HUD / 后排屏的人格表达——全部一致（都显示 icon）还是分工（中控主 + HUD 淡化）？
9. **V1.5 升级的平滑过渡**：从静态 icon 到动态 IP 的升级，用户是否可选保留旧版？数据和配置的向前兼容？
10. **人格层埋点**：每次 icon 状态切换、TTS 文案使用哪个模板，都需要埋点用于后续优化。数据结构 / 采样率？

**6. 自我反驳与边界**

**6.1 "克制"和"智能"是矛盾吗?**

**自反**:用户希望"少打扰",但又抱怨"AI 不智能"。**反驳自反**:智能 ≠ 多说话。智能是"该说的时候说,不该说的时候沉默"。**判据**:每次主动触发,用户"可以/喜欢"反馈率 > 60% 就是智能;否则就是"瞎主动"。

**6.2 关怀预算太苛刻会不会让人觉得 AI 冷漠?**

**自反**:平衡档 8 次/天,听起来不算多,实际可能不够。**对冲方案**:(1)安全类不算预算 (2)预算用完不是"什么都不说",而是"被动响应保留" (3)V1 上线后用真实数据迭代

**6.3 关怀模式的"做完就安静"会不会让人觉得 AI 不关心?**

**自反**:对比 NOMI 的"持续陪聊",我们看起来"冷"。**反驳自反**:NOMI 话痨被关掉是反例,我们做"恰到好处的关怀+给用户空间"。**关键**:动作要先到位(音乐/空调/座椅都调好了)再沉默,不是"什么都没做就闭嘴"。

**6.4 LLM 判断 vs 规则,什么时候用哪个?**

**自反**:LLM 判断慢、贵、可解释差;规则快、便宜、可靠。**对冲方案**:(1)安全类 100% 规则(疲劳/电量极低) (2)便利类规则触发后 LLM 二次判断"现在说合不合适" (3)推荐类全 LLM 决策(因为推荐错代价低)
