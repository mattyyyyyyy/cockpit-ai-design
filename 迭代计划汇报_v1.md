## 执行摘要

现有代码是一套**可运行的云端 AI 后端**，核心链路完整（ASR→NLU→Agent→MCP→TTS），已接入银联支付和10余个 MCP 工具。但距离"生产级智能座舱助手"还有 6 个关键维度需要迭代，核心问题是：**Harness 缺位、记忆浅薄、可观测性弱、无主动服务能力**。

## 一、意图理解与 NLU 路由

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| 模型地址硬编码 | `route_agent.py` 里 `base_url` 直接写死 GPU 实例 IP | 🔴 高 |
| Prompt 散落在代码里 | `keepRouter`、`intentRouter` system_prompt 是 Python 字符串，无法热更新 | 🟠 中 |
| LLM-NLU 无置信度输出 | ReActAgent 返回结构化意图但没有 confidence score，无法做二次确认判断 | 🟠 中 |
| 查询改写串行增加延迟 | query_rewrite 先于 NLU 执行，每次多一次 LLM 调用，实测增加 300-800ms | 🟠 中 |
| 传统 NLU 黑盒调用 | nlu_service 调用外部 API，超时无降级到 LLM-NLU 的显式机制 | 🟡 低 |

### 迭代方案

**1.1 Prompt 配置化**
- 所有 system_prompt 迁移到 `configs/prompts/` 目录，YAML 格式
- 支持从数据库/Apollo 热加载，无需重启服务

**1.2 LLM-NLU 输出规范化**
```python
# 现在：只输出意图名
{"mcp_tools": {"305.22": "星巴克"}}

# 改成：输出意图 + 置信度
{"mcp_tools": {"305.22": {"keyword": "星巴克", "confidence": 0.92}}}
```
置信度低于 0.75 的意图触发二次确认流程（对接 Harness 置信度门）。

**1.3 查询改写并行化**
- NLU 和查询改写同时发出，取先到结果
- 若改写未完成而 NLU 已返回明确结果（code=200），直接用 NLU 结果，放弃改写
- 节省 200-500ms 平均延迟

**1.4 模型配置中心化**
- 所有模型 endpoint 迁移到 `core/config.py → settings.MODEL_CONFIGS`（已有框架，只需规范）
- 从 Apollo/环境变量读取，零代码改动切换模型

### 目标状态
- P95 NLU 延迟 ≤ 300ms（含改写）
- 意图识别准确率可量化（有 confidence score）
- Prompt 更新无需发版

## 二、Agent 编排与工具管理

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| Agent 结构扁平 | consultation_expert 统一处理所有 MCP，无专家分工，工具数增加后 context 膨胀 | 🔴 高 |
| 工具注册无 Schema | 305.x 编码散落在各处 if/elif，新增工具要改 nlu_orchestrator + mcp_handler + config | 🟠 中 |
| 无并发上限 | 多意图并发调用工具无 Semaphore，高峰期 GPU 实例可能被打爆 | 🟠 中 |
| 无幂等保护 | 工具调用失败重试可能导致重复下单/重复支付 | 🔴 高 |
| Sub-bot 失败无重规划 | 工具返回 error 时只有 log，Mother-bot 层无重规划逻辑 | 🟠 中 |

### 迭代方案

**2.1 工具注册表中心化**
```python
# 新增 core/tool_catalog.py
TOOL_CATALOG = {
    "305.22": {
        "name": "starbucks_order",
        "domain": "food",
        "agent": "lifestyle_expert",
        "idempotent": True,
        "side_effect": True,
        "timeout": 15.0
    },
    "305.4": {
        "name": "navigation",
        "domain": "navigation",
        "agent": "navigation_expert",
        "idempotent": True,
        "side_effect": False,
        "timeout": 5.0
    }
}
```
NLU路由、MCP Handler、AgentScope 全部从这里查，不再散落。

**2.2 分域 Sub-bot 架构（渐进式，不推倒重来）**
- 第一步（V1.1）：按 domain 分组工具，同 domain 工具进同一个 consultation_expert context
- 第二步（V1.2）：高频域（导航/娱乐/车控）独立 Sub-bot，有自己的 system_prompt 和 memory

**2.3 并发控制 + 幂等 Key**
```python
# asyncio.Semaphore 限制最多 3 个工具并发
TOOL_SEMAPHORE = asyncio.Semaphore(3)

# 幂等 Key 格式
idempotency_key = f"{request_id}-{tool_code}-{attempt}"
# 写入 Redis，TTL 24h，重试时相同 key 不重复执行
```

**2.4 工具失败重规划**
- 工具返回 error_code 后，MCP Handler 返回标准化的 ToolResult（含 is_retryable、suggested_action）
- AgentScope 收到后自动选备选工具或告知用户

### 目标状态
- 新增 MCP 工具只需在注册表加一行，不改其他代码
- 重复下单/重复支付风险降至零（幂等保护）
- 3 个工具并发上限，GPU 实例稳定

## 三、安全与风险控制（Harness）

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| 无统一 Harness 中间件 | VehicleStateChecker 是唯一安全检查，但不在请求主路径上形成完整链 | 🔴 高 |
| 无置信度门 | LLM-NLU 置信度低时仍直接执行，未触发二次确认 | 🔴 高 |
| 无驾驶中支付限制 | 银联通道已有，但无驾驶速度→支付金额联动的安全判断 | 🔴 高 |
| 无熔断器 | 某个 MCP 工具持续超时，会持续被调用拖慢整个请求链 | 🟠 中 |
| 无 STALE 状态恢复 | 车机重启后进行中的任务状态丢失，可能重复执行 | 🟠 中 |

### 迭代方案

**3.1 Harness 中间件链（最高优先级）**
在 `websocket_handler.py` 主请求入口实现 7 层检查：
```python
async def handle_request(self, query, context):
    async with HarnessChain(context) as h:
        await h.check_state_machine()        # ① 状态门
        intent = await h.run_nlu(query)
        await h.check_confidence(intent)     # ② 置信度门（新增）
        await h.check_safety_guardrail(intent) # ③ 安全红线（升级 VehicleStateChecker）
        await h.check_human_approval(intent) # ④ 高风险确认
        result = await h.execute_with_isolation(intent)  # ⑤ 失败隔离
        await h.record_trace(intent, result) # ⑥ 可观测性
    return result
```

**3.2 驾驶安全支付分级**（对接现有银联通道）
- 车速 > 10km/h：金额 > 500 元直接拒绝，50-500 元只允许预授权品类
- 车速 ≤ 10km/h 或驻车：全额放开
- 声纹验证作为替代扫码的认证方式（对接 ASR 侧已有的说话人识别）

**3.3 Circuit Breaker（熔断器）**
```python
class ToolCircuitBreaker:
    # 连续 3 次超时 → 熔断，30s 后进入半开
    # 半开状态允许 1 次探测，成功则恢复
    thresholds = {"consecutive_failures": 3, "reset_timeout": 30}
```
按 tool_code 独立熔断，一个工具挂了不影响其他工具。

**3.4 STALE 状态恢复**
- 系统启动时扫 Redis/MySQL 中 status=RUNNING 的任务
- 按 tool 类型差异化恢复：车控查 CAN 状态、订单查平台 API
- 无法确认的标 UNKNOWN，下次会话 TTS 告知用户

### 目标状态
- 驾驶中零扫码支付风险
- LLM 置信度 <0.75 的意图 100% 触发确认
- 单 MCP 工具故障不影响其他功能
- 重启后任务状态可追溯恢复

## 四、记忆与个性化

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| 记忆仅 15 轮对话 + 1 小时 TTL | 跨会话记忆清零，用户每次上车都要重新说偏好 | 🔴 高 |
| 无用户偏好存储 | 星巴克常点什么、喜欢什么风格音乐——系统不记得 | 🟠 中 |
| 无节律/场景识别 | 工作日早晨 vs 周末下午——完全无差异 | 🟠 中 |
| 对话历史按 channel 隔离太细 | 用户问完导航转问点餐，上下文断裂 | 🟡 低 |

### 迭代方案

**4.1 三层记忆架构**

```
第一层：短期记忆（现有）
  Redis，15轮对话，1小时 TTL，按 user_id+channel 存储
  → 保留，升级：扩展到 30 轮，跨 channel 的上下文摘要

第二层：用户偏好记忆（新增）
  MySQL user_preferences 表
  数据结构：{user_id, category, key, value, confidence, updated_at}
  例：{user_id=1, category="food", key="starbucks_regular", value="美式咖啡中杯少冰", confidence=0.9}
  来源：对话中显式确认 + 重复行为自动学习

第三层：节律记忆（V2，先设计后实现）
  按 (user_id, weekday, hour_bucket) 存储高频行为模式
  驱动主动服务推荐
```

**4.2 跨 Channel 上下文桥接**
- 当用户 channel 切换时（305.22 → 305.4），提取前一个 channel 的关键信息（目的地、时间、人数）注入新 channel context
- 避免用户重复说"去星巴克" → "导航到刚才那个星巴克"的断层

**4.3 偏好学习触发点**
在 starbucks.py、navigation_planning.py 等工具完成后，提取关键字段写入偏好表：
- 订单完成 → 记录商品、门店、时间
- 导航完成 → 记录目的地类型、出发时段
- 音乐播放 → 记录风格、歌手

### 目标状态
- 用户偏好跨会话持久化（"老样子"能识别）
- 同一次出行内，跨意图上下文连贯
- 14 天后节律服务可用（V2 目标）

## 五、可观测性与稳定性

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| trace_id 不完整 | request_id 有，但 trace_id 在 mcp_handler 里是 Optional，云端 LLM 调用无法关联 | 🔴 高 |
| 大量 `except Exception` 吞异常 | nlu_orchestrator、mcp_handler 里大量 try/except 只打 warning，排查困难 | 🟠 中 |
| 无 SLO 监控 | 不知道 ASR/NLU/工具调用的 P95 延迟和成功率 | 🟠 中 |
| 无结构化日志 | log("info", "xxx") 是非结构化字符串，无法聚合分析 | 🟡 低 |

### 迭代方案

**5.1 统一 trace_id 规范**
```python
# 端侧生成，全链路透传
trace_id = f"{device_hash_8}-{session_id_8}-{seq_4}-{ts_8}"

# 所有跨服务调用必须携带
headers = {
    "X-Trace-Id": trace_id,
    "X-Request-Id": request_id
}
# LLM API 调用、MCP 工具调用、TTS 调用——全部透传
```

**5.2 结构化 span 日志**
```python
# 替换现有 log("info", "xxx")
tracer.span({
    "trace_id": trace_id,
    "span": "nlu_complete",
    "latency_ms": 148,
    "result": "vehicle_control",
    "confidence": 0.92,
    "level": "INFO"
})
```
按 trace_id 聚合，一次请求全链路可视化。

**5.3 SLO 埋点**
在 5 个关键节点埋点（ASR完成、NLU完成、工具返回、TTS开始、TTS完成），计算：
- 首字延迟（ASR完成→TTS第一帧）目标 ≤ 1.5s
- 全链路延迟 P95 ≤ 3s
- 工具调用成功率 ≥ 98%

**5.4 异常处理规范化**
- 明确区分 `ToolTimeout`、`ToolError`、`LLMError`、`SafetyReject` 等异常类型
- 只有真正预期的异常才 catch，其余上报告警
- 每周自动生成"沉默失败"报告（被吞掉的异常统计）

### 目标状态
- 任何故障 5 分钟内定位到根因
- SLO 仪表盘实时可见
- 全链路 trace 串联，端侧 → 云端 → 工具服务

## 六、TTS 与音频体验

### 现有问题

| 问题 | 具体表现 | 风险等级 |
|------|---------|---------|
| 多引擎无优先级 + 无降级链 | edge-tts/MiniMax/智谱三套并存，没有明确的主备关系和自动切换 | 🟠 中 |
| TTS 文本过滤是基础 regex | 只过滤 Markdown 符号，LLM 输出的 JSON 片段/代码块未过滤 | 🟡 低 |
| 无音色人格一致性 | 换 TTS 引擎时音色突变，用户感知到"不是同一个人" | 🟡 低 |

### 迭代方案

**6.1 TTS 引擎优先级链**
```
MiniMax（主力，低延迟流式）
  ↓ 失败/超时 2s
智谱 TTS（备用一，质量稳定）
  ↓ 失败/超时 3s
edge-tts（兜底，永远可用，免费）
```
统一在 `tts_module.py` 里封装，调用方不感知引擎切换。

**6.2 TTS 文本预处理增强**
新增过滤规则：JSON 片段、URL、过长数字串（电话/订单号 → "已发送到您手机"），避免播出机器音。

**6.3 音色配置化**
每个引擎维护同一"AI 人格"的音色参数（语速/音调/情感），切换引擎时自动映射，保持听感一致。

### 目标状态
- TTS 可用性 99.9%（三层降级兜底）
- 用户不会听到 JSON/代码被朗读
- 换引擎时音色感知变化不超过 20%（主观评测）

## 七、主动服务能力（V2 规划，V1 先打地基）

### 现有问题
现有系统是**完全被动的**——用户说话才响应。设计文档里的主动服务（节律推荐/情境感知/关怀服务）在代码里完全没有。

### V1.5 打地基（先做，不上线）

**7.1 情境事件总线**
- 新增 `core/context_event_bus.py`
- 车机 WebSocket 上报结构化车辆状态（车速/SOC/位置/时间）
- 事件总线分发给各"情境监听器"

**7.2 规则驱动的主动服务（先上）**
不用 AI，纯规则：
- SOC < 20% + 附近有充电桩 → 推充电提醒
- 行驶 > 2h + 高速 → 推休息区提醒
- 到达常用地点附近 → 推导航完成提示

**7.3 偏好驱动推荐（中期）**
基于第四章的偏好记忆，每次到达常用时间段触发：
"您平时这个时候会点星巴克，要我帮您提前下单吗？"

### 目标状态（V2）
- 用户无需主动说，AI 在合适时机主动响应
- 主动服务触发准确率 ≥ 70%（用户接受/不拒绝）
- 不在行驶中弹出高打扰通知

## 最终目标架构（详见架构图）

完成全部迭代后，系统架构分为 6 层：

1. **车端感知层**：ASR（FunASR SenseVoice）+ VAD（Silero）+ KWS + 车辆状态上报
2. **Harness 线束层**：7 个中间件串联，每个请求必须通过
3. **智能决策层**：查询改写 → NLU双轨 → LLM-NLU → 分域 Agent 编排
4. **工具执行层**：统一注册表 + 幂等 + 熔断 + 并发控制
5. **记忆与个性化层**：短期（Redis）+ 用户偏好（MySQL）+ 节律（V2）
6. **输出与可观测层**：TTS 降级链 + 全链路 trace + SLO 监控

> 架构图见 `architecture-diagram.html`（已更新为目标态）
