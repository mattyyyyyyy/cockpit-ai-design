**§1 背景与目标**

**1.1 为什么做语音输出优化**

**核心问题**：TTS 是座舱 AI 和用户对话的"嘴"——嘴不好使,脑子再聪明也白搭。当前车载 TTS 两大痛点:机器人味重、响应慢。

三大矛盾：

1. **自然度 vs 延迟**：越自然的 TTS 模型越大,推理越慢;快速响应往往意味着机械音
2. **一致性 vs 个性化**：品牌需要一致的语音人格,用户想自定义声音
3. **端侧 vs 云端**：端侧延迟低但音色有限,云端音色丰富但依赖网络

**AI 时代的机会**：

* **端到端 TTS**：去掉传统三段式(文本→韵律→声码器),直接文本→语音,延迟降 50%+
* **流式输出**：第一个词出来就开始播,不等整句生成完,体感延迟 <300ms
* **情绪化 TTS**：根据语境调整语气(提醒时严肃、闲聊时轻松),不再一个腔调念到底

**目标**：

* **主目标**：首字延迟 <300ms,用户感知"即时响应"
* **自然度目标**：MOS(平均意见分)>4.0/5.0
* **个性化目标**：支持 >5 种预置音色 + 1 种自定义音色

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **即时响应**：说完话立刻听到回复,不再等 2~3 秒沉默
* **听得舒服**：自然语音不刺耳,长时间交互不疲劳
* **情绪匹配**：导航指令干脆利落,闲聊轻松活泼,紧急提醒严肃明确

**对多用户**：

* **可选音色**：每个人选自己喜欢的声音,不被迫听同一个"机器人"
* **方言适配**：粤语/四川话等方言用户可选方言 TTS,沟通更亲切

**2.2 反价值(不做什么)**

* **不做拟人欺骗**：不追求"让人分不清是不是真人"——AI 就是 AI,自然但不必冒充人
* **不做无限制自定义**：不做"克隆任何人声音"——有声纹滥用风险,自定义需本人授权
* **不做过度情绪化**：不追求"哭着告诉你天气"——情绪是微调,不是表演

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **首字延迟** | <300ms(端侧) | 从文本到第一个音节 | >500ms 降级为简短确认音 |
| **MOS 自然度** | >4.0/5.0 | 人工评测 | <3.5 标记需模型升级 |
| **流式延迟** | <200ms | 文本到达后首音输出 | >500ms 降级为非流式 |
| **音色切换延迟** | <2s | 切换音色到生效 | >5s 提示"正在切换" |

**4.2 可靠性与降级**

1. **云端 TTS 不可达**：端侧 TTS 兜底(2 种基础音色)
2. **流式中断**：降级为非流式,完整生成后一次性播放
3. **情绪模型异常**：降级为中性语气,不尝试情绪适配

**4.3 安全与合规**

* **声纹保护**：自定义音色需本人声纹验证,不可使用他人声音
* **情绪真实性**：情绪化 TTS 不得用于欺骗(如"假装很高兴"来掩盖系统错误)
* **儿童保护**：面向儿童的 TTS 不使用成人化语气

**4.4 关键取舍**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 取舍点 | 选项 A | 选项 B | 决策 | 理由 |
| **TTS 架构** | 端到端(低延迟) | 三段式(可控) | **端到端为主,安全类用预制音** | 延迟优先,安全类场景不冒险 |

**3.3 流式 TTS**

1. **边想边说** — LLM 流式输出 → TTS 流式合成
2. **首字延迟** < 500ms
3. **打断响应** < 200ms

1. **关怀模式** — 慢语速(0.85x)、柔和音调

**3.5 多语种/方言**

1. **V1 主打普通话** — 标准普通话

**5.1 T-F1 音色与人格化**

**5.1.1 一句话定义**

**做什么**：定义**品牌主音色**（有辨识度，不抄 NOMI，但能让用户听了说"这是我们车"）+ **情境音色切换**（关怀/日常/紧迫/儿童 4 种变体），让同一个 AI 在不同场景听起来合适。

**不做什么**：

* 不做"人格对话风格"（11 个性化与人格层的事，T-F1 只管音色不管说话内容）
* 不做"TTS 合成技术"（技术选型见沈砚）
* 不做"语音内容生成"（LLM 生成，TTS 只负责念）

**核心价值**：音色是品牌的声音资产，不是"随便找个女声"。

**5.1.3 输入 / 触发条件**

* **输入**：待合成文本、当前场景上下文（04 S-F3 模式 / 01 DMS / D-F2 风险类型）、用户音色偏好
* **触发**：每次 TTS 合成请求
* **前置条件**：音色模型已加载；场景上下文可用

**5.1.4 执行流程**

|  |
| --- |
| Plain Text TTS 合成请求  ↓ 场景上下文读取  ├─ 04 S-F3 模式（通勤/家庭/关怀/驻车）  ├─ 01 DMS 状态（疲劳/正常/激动）  └─ 02 D-F2 风险类型（安全紧迫）  ↓ 音色变体决策  ├─ 安全紧迫 → urgent 变体（P0）  ├─ DMS 疲劳 / 关怀模式 → care 变体  ├─ 家庭模式 + 儿童在场 → kids 变体  └─ 默认 → daily 变体  ↓ 音色冲突仲裁  ├─ 优先级：safety > care > kids > daily  └─ 取最高优先级变体  ↓ TTS 合成  ├─ 传入音色 ID + 变体参数  └─ 流式输出（见 5.2 T-F2）  ↓ 播放 |

**关键时间预算**：

* 场景上下文读取延迟：**≤ 50ms**
* 音色决策延迟：**≤ 20ms**
* 总音色切换延迟：**≤ 100ms**（不阻塞 TTS 主链路）

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **音色切换过于频繁**（一句话内切换） | 听感混乱 | 单次 TTS 合成只用一个变体；下次请求才切换 |
| **变体参数失真**（关怀模式听起来"懒洋洋"） | 用户反感 | 变体参数严格调校；用户测试验证 |
| **紧迫音色太刺耳** | 用户反感 | 语速 ≤ 1.1x 上限；音调涨幅 ≤ 5% |
| **儿童音色误触发**（没小孩时用童声） | 违和 | 严格判定条件：家庭模式 + OMS 检测到儿童 |
| **音色资源加载失败** | TTS 失效 | 默认音色 fallback；异常时用系统 TTS |

**Paper Tiger（看起来像风险但不是）**

* "用户不需要多音色"—— 实证：**蔚来 NOMI 的专属音色是品牌护城河**，我们做但不抄
* "音色切换增加工程复杂度"—— 实测：参数化变体成本低，端侧可做

**Elephant（团队可能没讨论够）**

* **音色的"版权"**：自研合成音色有没有版权风险？**建议：用公开声音库训练，避免声优个人声音**
* **音色的"老化"**：3 年后音色听起来过时吗？**建议：V1 用经典风格，避免"网红风"**
* **音色 vs 人格层的边界**：音色是"声音本身"，人格层（11）是"说话风格/用词"。**T-F1 只管声学，11 管语义**
* **儿童音色的"儿童视角"**：儿童音色给大人听还是给小孩听？**建议：给小孩听的才用童声，给大人播报仍用主音色**
* **多音区多音色**：主驾听主音色，副驾听可选音色？**V1 不做，V2 考虑**

**5.2 T-F2 流式 TTS**

*待填*

**5.2.1 一句话定义**

**做什么**：**边想边说**—— LLM 流式输出文本 → TTS 流式合成 → 立即播放，而不是"等 LLM 说完再开始念"。首字延迟 < 500ms，打断响应 < 200ms。

**不做什么**：

* 不做"LLM 流式输出"（10 AI 大模型集成的事）
* 不做"ASR 流式"（01 多模态输入的事）
* 不做"音色切换"（5.1 T-F1 的事）

**核心价值**：用户感知延迟从"LLM 3s + TTS 1s = 4s"变成"LLM 流式 0.5s + TTS 并行 = 0.5s"。**首字延迟是语音交互的体验生死线**。

**5.2.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 流式 TTS 行为 |
| S1 | **短回答** | 用户"几点了？" → LLM 流式"现在是 10 点 15 分" → TTS 首字 400ms 出 → 流式播放 |
| S2 | **长回答** | 用户"讲个故事" → LLM 流式输出 → TTS 按句合成 → 边合成边播 → 总延迟感知 ≤ 500ms |
| S3 | **用户打断** | TTS 正在念"前面有个加油站"，用户插话"算了不要了" → 200ms 内停止 TTS + ASR 进入监听 |
| S4 | **流式规划播报** | 06 A-F4 流式规划 → LLM 流式输出"先充电"→ TTS 立即念 → 后续"再接朋友"→ TTS 接着念 |
| S5 | **LLM 流式中断** | LLM 网络中断，流式输出到一半 → TTS 念完已收到部分 + TTS "嗯……稍等一下"兜底 |
| S6 | **TTS 滞后 LLM** | LLM 输出快过 TTS 合成 → 缓冲队列缓存 → 按节奏播放（不会一下全念完） |

**流式架构**：

|  |
| --- |
| Plain Text LLM stream → token buffer → 文本切分（按标点/长度） → TTS stream chunks → 音频缓冲 → 扬声器  ↑  打断信号（200ms 内中断） |

**5.2.3 输入 / 触发条件**

* **输入**：LLM 流式文本输出、音色 / 变体参数（来自 5.1）
* **触发**：LLM 开始流式输出 token
* **前置条件**：TTS 支持流式合成；音频输出链路正常

**5.2.4 执行流程**

|  |
| --- |
| Plain Text LLM 开始流式输出（token by token）  ↓ 文本切分策略  ├─ 遇到句号 / 逗号 → 切分送 TTS  ├─ 连续 20 字无标点 → 按字切  └─ 首句切分：尽早（≤ 10 字）送出以降首字延迟  ↓ TTS 流式合成  ├─ 小片段音频（chunk）立即输出  ├─ 参数：音色 + 变体（来自 5.1）  └─ 合成并发：LLM 还在输出时 TTS 已开始合成  ↓ 音频缓冲队列  ├─ 缓冲 200ms 音频后开始播放（避免卡顿）  └─ LLM 输出快过 TTS → 队列缓存  ↓ 扬声器播放  ├─ 按节奏播放（不加速）  └─ 监听打断信号  ↓ 打断处理  ├─ ASR 检测用户说话 → 打断信号  ├─ 200ms 内停止合成 + 停止播放  └─ 未合成部分丢弃  ↓ 失败兜底  ├─ LLM 流式中断 → TTS 完成已收到部分 + 兜底话术  └─ TTS 合成失败 → 降级整句合成 + 提示延迟 |

**关键时间预算**：

* 首字延迟（LLM 首 token → 首音频）：**≤ 500ms**（P95）
* 打断响应：**≤ 200ms**
* 音频缓冲：**200ms**（防卡顿）
* TTS 合成速度：**> 350 TPS**（对标极氪）

**5.2.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **首字延迟 > 1s** | 用户感觉卡 | LLM 首句尽早切分 + TTS 预热 + 音色模型常驻 |
| **打断不及时**（用户插话 TTS 还在念） | 体验差 | 200ms 硬性目标；监听优先级高于合成 |
| **流式切分不自然**（半句被切开） | 听感差 | 按标点优先；无标点时按语义短语切（轻量 NLP） |
| **LLM 输出过慢**（token 间隔 > 300ms） | TTS 断续 | 缓冲 + 音频连接平滑处理；极端时切到整句等待模式 |
| **音频卡顿** | 体验崩 | 200ms 音频缓冲保底；网络抖动时优先保顺畅不保首字延迟 |
| **打断后状态错乱**（ASR/TTS 状态不一致） | 对话崩 | 严格状态机：tts\_playing → interrupt → tts\_stopped → asr\_listening |

**Paper Tiger（看起来像风险但不是）**

* "流式太复杂"—— 极氪 350 TPS 已证明可行
* "端侧跑不动"—— 端侧模型 + 云端模型联动，可实现

**Elephant（团队可能没讨论够）**

* **端侧 vs 云端 TTS**：端侧延迟低但质量低；云端质量高但网络敏感。**建议：混合——常用话术端侧 + 复杂长文本云端**
* **首字延迟和合成质量的 trade-off**：首字快但合成糙 vs 首字慢但合成精。**建议：首字优先体验**
* **切分算法的准确性**：半句切开是低级错误，**需要轻量的 NLP 句法感知**
* **多人打断的仲裁**：主驾说话 + 副驾说话谁打断？**04 S-F2 并发仲裁**
* **TTS 和音乐混音**：TTS 播报时音乐怎么处理？**建议：duck 到 30% 音量，TTS 完成恢复**

**5.3 T-F3 情绪化语音（4 种情境）**

*待填*

**5.3.1 一句话定义**

**做什么**：在 T-F1 音色变体基础上，**进一步细化 4 种情绪化语音**（关怀 / 紧迫 / 日常 / 儿童友好），不仅调语速音调，还做**情感表达**（如关怀模式有"叹息感"、紧迫模式有"清晰切分感"）。控制幅度适中，**克制不过度**。

**不做什么**：

* 不做"纯参数化变体"（T-F1 已做）
* 不做"人格对话风格"（11 人格层的事）
* 不做"情感识别"（01 DMS/C-F3 的事）

**核心价值**：情绪化是 TTS 的差异化，但**过度情绪化（NOMI 式话痨）是反例**。克制 > 炫技。

**5.3.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 情绪化表现 |
| S1 | **关怀模式**（晚归疲劳） | 语速 0.85x + 音调微降 + "嗯……" 柔和开头 —— TTS："到家了……今天辛苦了" |
| S2 | **紧迫模式**（安全风险） | 语速 1.1x + 音调微升 + 清晰切分 + 不拖音 —— TTS："前方！急刹！请注意！" |
| S3 | **日常模式** | 语速 1.0x + 中性音调 + 标准节奏 —— TTS："前方 500 米右转" |
| S4 | **儿童友好模式**（娃哭安抚） | 音调 1.15x + 活泼语调 + 韵律感 + 轻笑 —— TTS："小朋友，我们来唱首歌好不好呀？" |
| S5 | **情绪过度预警** | 关怀模式语速 0.7x（过慢）→ 用户反馈"听着懒"→ 记录 + 下次调整 |
| S6 | **情绪不匹配** | 本应紧迫却用了关怀（因为 DMS 判定疲劳）→ **安全优先** → 紧迫模式覆盖 |

**5.4.1 一句话定义**

| 风险 | 后果 | 缓解 |
| **英文发音错误**（"Apple" → "阿婆"） | 用户笑场 | 英文词典 + 用户反馈闭环 |
| **音色割裂**（中文女声 + 英文男声） | 体验差 | 统一音色模型支持多语种（VALL-E / CosyVoice） |
| **语种检测错误**（把"Apple"当中文） | 发音错误 | 语种检测准确率 ≥ 95%；常见品牌/地名白名单 |
| **韵律不连贯**（中文慢 + 英文快） | 听感差 | 统一韵律控制；语速保持一致 |
| **方言识别失败** | 用户说方言 TTS 听不懂 | V1 只做普通话 + 英文；方言 V2 |
| **多语种模型质量下降** | 中文质量降低 | 优先保中文质量；英文质量可接受即可 |

**Paper Tiger（看起来像风险但不是）**

* "多语种用户不需要"—— 实证：**中英混说是日常**（地名/品牌/专业词汇）
* "多语种模型难"—— VALL-E / CosyVoice 已支持，工程可行

**Elephant（团队可能没讨论够）**

* **方言的"优先级"**：上海/广东/四川方言都要支持吗？**建议：V1 不做方言，V2 按用户分布优先级做**
* **英文发音的"标准"**：美音 vs 英音？**建议：美音为主（国内用户习惯）**
* **多语种的"音色一致性"**：同一音色模型支持多语种 vs 不同语种不同音色？**建议：统一音色模型**
* **语种切换的"显式 vs 隐式"**：用户要不要说"切换到英文"？**建议：隐式自动切换**
* **多语种的"成本"**：多语种模型训练成本高。**V1 只做中英，其他语种 V2**

**6. 自我反驳与边界**

**6.1 定制音色要不要请明星/声优?**

**自反**:明星音色有传播效应,但有肖像权/授权成本/版本管理复杂性。**对冲方案**:V1 用自有合成音色(有辨识度即可),V1.5 考虑联名(类 NOMI 出道但不学 NOMI 那套拟人过度)。

**6.2 端到端 vs 三段式什么时候定?**

**自反**:这是沈砚领域决策,但产品视角有顾虑——端到端调试难、回滚难、失败不可解释。**建议 V1 三段式**,V1.5 再上。

**6.3 情绪化会不会让人觉得假?**

**自反**:过度情绪化(NOMI 的话太多)是反例。**对冲方案**:(1)情绪化幅度小(关怀模式只比日常慢 15%)(2)不是全程情绪化,只在场景触发时 (3)用户可关
