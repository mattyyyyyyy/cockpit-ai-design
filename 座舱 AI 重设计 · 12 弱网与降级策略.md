**§1 背景与目标**

**1.1 为什么做弱网与降级策略**

**核心问题**：车是移动物体,网络是变量。隧道、地库、山区、信号干扰——座舱 AI 如果依赖网络才能工作,它在最需要的时刻反而失灵。

核心矛盾：

1. **体验一致性 vs 网络不确定性**：用户不关心"你网络不好",他们要的是"随时能用"
2. **功能丰富度 vs 断网可用性**：功能越多越依赖云端,断网时能用的越少

**目标**：

* **主目标**：断网场景下核心功能(车控/导航/音乐)可用率 >90%
* **感知目标**：网络状态变化时用户无感知(不弹窗不闪烁),静默切换端云
* **恢复目标**：网络恢复后 5s 内自动回正,用户无需操作

**§2 用户价值**

* **随时可用**：进隧道不会突然变傻,出隧道自动恢复
* **无感切换**：用户不知道什么时候走的端侧什么时候走的云端——只觉得"一直能用"
* **明确降级提示**：功能确实不可用时(如语音订餐厅需网络),一句话告知"离线模式暂不支持,恢复网络后自动可用"

**反价值**

* 不做"网络差就什么都不做"——核心功能必须兜底
* 不做频繁弹窗提醒"网络已断开"——静默处理,只在用户触发需网络功能时告知

**3. 核心能力要求**

**3.1 网络状态感知**

1. **实时检测** — 4G/5G/WiFi/无网
2. **延迟估算** — 当前网络 RTT
3. **带宽估算** — 可用带宽(音频/视频/图片的加载能力)
4. **预测** — 地库/隧道/高速弱网段提前预判

**3.2 端侧兜底能力**

1. **离线 ASR/NLU** — 简单指令端侧能识别
2. **离线 TTS** — 基础播报端侧能合成
3. **离线地图** — 离线导航数据
4. **离线音乐** — 本地内容播放

**3.3 云端服务降级**

1. **LLM 调用降级** — 云端挂,退化到规则引擎
2. **记忆检索降级** — 云端挂,用端侧缓存
3. **第三方服务降级** — 高德挂,换百度;美团挂,换饿了么
4. **降级提示** — 用户明确知道"降级了"(不是静默)

**3.4 用户感知一致性**

1. **跨网络体验一致** — 有网/无网,核心功能(导航/车控)体验不塌
2. **降级不等于不用** — 降级后仍有基础能力
3. **状态透明** — 用户知道当前在"有限模式"

**6. 功能点展开**

**6.1 W-F1 网络状态感知**

**6.1.1 用户价值**

**做什么**：座舱 AI 必须**实时感知当前网络状态**——4G / 5G / WiFi / 无网、RTT / 带宽、是否即将进入弱网段（地库 / 隧道 / 高速）。感知是降级的前提——没感知的降级是"事后补救"，有感知的降级是"提前准备"。用户开车进地库前 10 秒 AI 就已经切到端侧兜底，用户感觉"无缝"。

**不做什么**：

* 不做"只看连接状态不看质量"（连着 4G 但 RTT 2 秒 = 实际弱网）
* 不做"频繁切换降级等级"（网络波动导致反复切 → 体验抖动）
* 不做"完全依赖外部 API 判断"（网络 API 本身就在弱网下不可用）
* 不做"静默降级"（用户不知道 AI 进了降级模式 → 体验预期错位）

**核心价值主张**：

* **多信号融合**：RTT + 带宽 + 信号强度 + 连接成功率
* **预判弱网**：地图数据标注弱网段（地库 / 隧道），提前切换
* **切换防抖**：短暂波动不触发切换（30s 持续差才切）

**6.1.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景 | 网络状态感知 | 触发动作 | 关键约束 |
| S1 | **进入地库**：地图标注"即将进入地库" | 预判弱网 | 预加载离线数据；预警用户 | 提前 10-30s 预判 |
| S2 | **RTT 突然上升**：5G 连接但 RTT 从 80ms → 800ms | 一般 → 弱网 | 逐步降级云端调用 | 持续 30s 才切换（防抖） |
| S3 | **完全无网**：隧道深处 | 无网 | 切离线模式 + 明确提示 | < 3s 切换 |
| S4 | **网络恢复**：出隧道 | 无网 → 良好 | 恢复云端能力 + 隐式提示 | 不打扰用户 |
| S5 | **带宽严重受限**：3G 连接但带宽 100kbps | 弱网 | 文字优先 + 禁用图片 / 视频加载 | 流量优化 |
| S6 | **WiFi 但网络慢**：连着 WiFi 但上网慢（公共 WiFi 被劫持） | 弱网 | 不盲信 WiFi；依然降级 | 连接状态 ≠ 网络质量 |
| S7 | **用户手动设置"省流量模式"** | 主动限流 | 全程走端侧 + 仅必要云端 | 用户偏好优先 |
| S8 | **弱网段预测不准** | 地图数据过时 | 实时 RTT 检测兜底 | 预测 + 实时双通道 |
| S9 | **多次切换抖动** | 网络反复变化 | 3 分钟内最多切换 3 次 | 限频保护体验 |
| S10 | **网络感知 API 失败** | 感知本身失败 | 假设"弱网"保守处理 | Fail-safe 原则 |

**6.1.3 成功指标**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方式 | 备注 |
| 网络状态感知准确率 | ≥ 95% | 感知状态与实际一致 | 核心指标 |
| 弱网段预判提前量 | ≥ 10s | 进入弱网前预判时长 | 预判有效性 |
| 切换延迟 | ≤ 3s | 从网络变化到模式切换 | 用户感知指标 |
| 切换抖动率 | ≤ 3 次 / 3 分钟 | 切换频次 | 防抖有效性 |
| 预判误报率 | ≤ 20% | 预判但未进入弱网 | 可接受范围 |
| 网络恢复识别延迟 | ≤ 5s | 网络恢复到模式切回 | 硬约束 |
| Fail-safe 生效率 | 100% | 感知失败时默认弱网处理 | 硬约束 |

**6.1.4 放弃条件**

|  |  |  |
| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 感知准确率 < 80% | 连续 2 周 | 感知算法重审 |
| 切换延迟 > 10s | 连续 1 周 | 切换机制优化 |
| 切换抖动过多 | 单周投诉 > 20 例 | 防抖时长延长 |
| 预判误报率 > 40% | 连续 1 个月 | 地图数据更新或算法改进 |
| Fail-safe 失效 | 任意一次 | P0 Bug |

**6.1.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 感知延迟大 | 信号采样周期长 | 已经弱网但还在云端模式 | 采样周期 ≤ 1s；多信号并发 |
| 误判弱网 | 临时抖动 | 体验降级但实际不需要 | 防抖 30s；30s 内恢复不切换 |
| 预判不准 | 地图数据过时 | 进地库没提前准备 | 实时 RTT + 预测双通道 |
| WiFi 质量误判 | 只看连接状态 | 连着 WiFi 但实际慢 | 多信号融合（RTT + 带宽 + 连接成功率） |
| 切换动画干扰 | 切换 UI 过重 | 用户看到"降级中"打扰 | 切换 UI 极简（顶部小图标） |
| 省流量模式用户体验差 | 限流过严 | 用户觉得"啥都不能用" | 省流量分级（轻度 / 中度 / 重度） |
| 跨运营商差异 | 不同运营商信号差异 | 部分用户感知不准 | 运营商级别适配 |

**6.1.6 技术钩子（给沈砚）**

1. **网络感知信号源**：RTT / 带宽 / 连接成功率 / 信号强度。采集周期 / 融合算法？
2. **弱网段地图数据**：地库 / 隧道 / 高速弱网段的数据来源？更新频率？
3. **切换防抖机制**：30s 持续差才切。状态机实现？
4. **预判算法**：地图数据 + 实时 RTT 混合。置信度计算？
5. **省流量模式分级**：轻度 / 中度 / 重度的功能配置。用户 UI？
6. **Fail-safe 机制**：感知失败时默认弱网。全链路兜底？
7. **切换 UI 设计**：状态栏图标 / 通知。最小干扰原则？
8. **运营商适配**：不同运营商信号差异。参数化配置？
9. **网络感知 API**：系统级 vs 应用级？权限？
10. **A/B 测试**：防抖时长、预判提前量等参数的 A/B。框架？

**6.2 W-F2 端侧兜底能力**

**6.2.1 用户价值**

**做什么**：无网 / 弱网时，**端侧能兜住一个可用的基础座舱**——离线 ASR / NLU 识别简单指令、离线 TTS 播报基础信息、离线地图导航、本地音乐播放。用户不会因为进了地库就"AI 全瘫了"，而是感觉"AI 少了一些高级能力，但够用"。

**不做什么**：

* 不做"端侧完全替代云端"（V1 不上 7B+ 端侧大模型）
* 不做"所有功能离线化"（成本 / 算力不支持）
* 不做"离线能力用完就丢"（还要定期更新离线数据）
* 不做"离线不告知用户"（用户该知道"当前是离线模式"）

**核心价值主张**：

* **基础能力兜底**：ASR / NLU / TTS / 导航 / 音乐离线可用
* **离线数据更新**：离线地图 / 本地音乐每月 OTA
* **状态透明**：UI 明确标注"离线模式"

**6.2.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景 | 离线兜底行为 | 关键约束 |
| S1 | **地库中车控**："调空调 22 度" | 端侧 ASR + 意图识别 + 执行 | 全链路端侧 < 500ms |
| S2 | **地库中导航**："导航回家" | 端侧地址查询（缓存）+ 离线地图路线规划 | 离线地图覆盖常用城市 |
| S3 | **地库中音乐**："放周杰伦" | 本地音乐库查询 + 播放 | 本地库需要用户预下载 |
| S4 | **地库中 TTS**：AI 回复"已调空调" | 离线 TTS | 基础音色；情感化音色需云端 |
| S5 | **地库中复杂指令**："订餐厅" | 端侧识别 → 告知"当前离线，无法完成，等网络恢复" | 明确告知；不假装能用 |
| S6 | **离线数据过时**：离线地图 3 个月未更新 | 提示用户"离线地图过时，建议更新" | OTA 提示 |
| S7 | **网络恢复**：出地库 | 自动恢复云端能力；未完成任务可继续 | 任务延续 |
| S8 | **离线功能有限**：无网时用户想看天气 | 告知"天气需要联网"；提供近期缓存 | 能给部分数据给部分 |
| S9 | **弱网部分可用**：弱网但 ASR 云端可用 | 关键功能走云端；次要功能走端侧 | 分级降级 |
| S10 | **长期离线**：用户长时间在无网环境 | 离线模式可用 8 小时以上 | 电量 / 存储限制 |

**6.2.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 离线 ASR 识别错 | 噪音 / 口音 / 模型小 | 指令执行错 | 置信度低时提示用户重说 |

| 离线 TTS 生硬 | 模型能力有限 | 机器感强 | 关键信息用固定音频预录 |
| 离线数据占存储大 | 数据量大 | 用户存储不足 | 分级下载（必需 / 推荐 / 可选） |
| 长期不更新离线数据 | 用户从不联网 | 越来越过时 | 每次联网强制检查更新 |
| 离线模式切换异常 | 状态机 bug | UI 显示有网但实际离线 | 状态机强约束；UI 与实际一致 |

**6.3 W-F3 云端服务降级**

**6.3.1 用户价值**

**做什么**：当**云端服务出问题**（主 LLM 挂了 / 高德地图挂了 / 美团外卖挂了），不让用户卡死——**自动降级到备用方案**：主 LLM 挂了用规则引擎、高德挂了换百度、美团挂了换饿了么。降级**对用户透明但有提示**——告诉用户"目前在用备用方案"，避免预期错位。

**不做什么**：

* 不做"云端挂了用户什么都做不了"
* 不做"静默切换不告知用户"（备用方案体验可能略差）
* 不做"无限重试"（卡死用户）
* 不做"降级体验比离线还差"（要么用备用要么走端侧，不是更糟）

**核心价值主张**：

* **多备用通道**：主 / 备 1 / 备 2 / 端侧兜底
* **自动切换**：检测到主服务异常立即切换
* **降级提示**：告知用户"用了备用方案"

**6.3.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景 | 主服务异常 | 降级行为 | 关键约束 |
| S1 | **主 LLM 超时**：用户复杂指令但主 LLM 卡 | 主 LLM 超时 | 切备 LLM；超时 ≤ 3s | 备 LLM 行为对齐 |
| S2 | **主 LLM 完全挂**：所有 LLM 都不可用 | 主 + 备 LLM 都挂 | 退化到规则引擎 | 规则引擎覆盖核心垂域 |
| S3 | **高德地图挂**：导航查询失败 | 高德 API 异常 | 切百度地图 | 用户告知"切到百度地图" |
| S4 | **美团挂**：点外卖失败 | 美团 API 异常 | 切饿了么 | 用户告知"切到饿了么" |
| S5 | **充电桩 API 挂**：特来电异常 | 特来电异常 | 切星星充电 / 小桔充电 | 多备份 |
| S6 | **TTS 云端挂**：复杂情感 TTS 失败 | 云端 TTS 异常 | 切端侧 TTS（基础音色） | 音色降级但可用 |
| S7 | **第三方 API 限流**：API 触发限流 | 限流 429 | 等待 + 队列；告知用户 | 不立即失败 |
| S8 | **降级后体验差**：备用方案明显不如主方案 | 用户体验下降 | 用户感知有提示 | 不掩盖真实情况 |
| S9 | **服务恢复**：主服务恢复 | 主服务可用 | 自动切回主服务 | 平滑切换 |
| S10 | **降级链路熔断**：连续多次失败 | 短期内快速失败 | 不再尝试主服务，直接走备用 | 熔断机制 |

**6.3.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |

**6.4 W-F4 用户感知一致性**

**6.4.1 用户价值**

**做什么**：不管用户处在什么网络状态（良好 / 一般 / 弱网 / 无网），座舱 AI 的**核心能力（车控 / 导航 / 音乐）体验不塌**，AI 的**人格 / 话术 / UI 风格一致**。降级不等于"换个 AI"——用户感觉仍是"同一个 AI"，只是某些高级能力暂时不可用。

**不做什么**：

* 不做"降级后人格突变"（在线时温暖型离线时冷漠型）
* 不做"降级后 UI 大换"（离线模式界面完全不同）
* 不做"降级影响安全类功能"（胎压告警 / 碰撞预警无论在线离线都必须工作）
* 不做"降级不可知"（用户不知道当前在什么模式）

**核心价值主张**：

* **核心能力跨网络一致**：车控 / 导航 / 音乐 / 安全告警
* **人格 / UI / 话术一致**：用户感知"同一个 AI"
* **状态明确标注**：UI 明确告诉用户当前模式

**7. 自我反驳与边界**

**7.1 端侧兜底能力会不会推高 BOM 成本?**

**自反**:端侧模型越多,芯片算力要求越高,成本越高。**对冲方案**:(1)端侧只做基础(ASR/NLU/TTS 基础版)(2)不依赖端侧 7B+ 大模型(3)V1 车型用主流芯片(8155/8295)够用

**7.2 降级提示会不会让用户焦虑?**

**自反**:用户看到"当前网络弱,功能受限"会觉得"这车这么菜"。**对冲方案**:(1)提示要柔和("当前网络较弱,部分功能暂时不可用,请稍后")(2)自动恢复后提示"网络已恢复"

**7.3 离线导航数据怎么更新?**

**自反**:离线地图要定期更新。**对冲方案**:(1)WiFi 下自动更新(2)用户手动触发更新(3)更新失败提醒
