**§1 背景与目标**

**1.1 为什么做数据闭环与持续优化**

**核心问题**：AI 系统不是"上线就完事"——用户在用,数据在产生,模型在老化。没有数据闭环,座舱 AI 三个月后就会"变笨"(新说法听不懂,新场景不会处理)。

核心矛盾：

1. **数据价值 vs 隐私风险**：数据越丰富模型越强,但用户对"车在偷听我"的恐惧越深
2. **快速迭代 vs 安全稳定**：模型更新可能引入新 bug,但不变新就会过时
3. **在线学习 vs 可控性**：实时学习响应快但不可控,离线训练可控但周期长

**目标**：

* **主目标**：月度模型迭代周期,关键 badcase 修复 <7 天
* **数据目标**：badcase 自动回流率 >80%,人工标注量 <10%(自动化标注)
* **隐私目标**：原始对话不上云,仅结构化意图+脱敏反馈上云

**§2 用户价值**

**2.1 核心价值主张**

* **越用越聪明**：被拒绝的主动服务会学习调整,识别错误的指令会自动优化
* **问题快速修复**：频繁出错的场景被自动捕获,下个版本修复
* **数据可控**：用户可查看"系统记住了什么"、可删除、可关闭学习

**2.2 反价值**

* **不做偷偷学习**：所有数据采集和学习行为必须用户可见可控
* **不做 A/B 实验**：不在用户不知情的情况下对部分用户使用不同模型
* **不做数据变现**：用户数据不用于广告、不用于第三方、不用于非产品优化目的

**3. 核心能力要求**

**3.1 用户行为埋点**

1. **埋点类型** — 语音/车控/导航/音乐/充电/会议等 15-20 个事件类型
2. **埋点维度** — 时间/地点/用户/车辆状态/场景模式
3. **埋点方式** — 端侧埋点+云端埋点混合
4. **脱敏** — 敏感字段(用户名/联系方式/车内画面)哈希化

**3.2 失败 case 回流**

1. **失败类型** — ASR 识别失败/NLU 理解失败/工具调用失败/任务失败
2. **失败上下文** — 用户原始语音/识别结果/NLU 结果/工具响应
3. **回流方式** — 自动上报(用户可关闭)
4. **人工 review** — 失败 case 样本人工 review

**3.3 主动服务反馈数据**

1. **反馈入口** — 每次 AI 主动触发后,用户"喜欢/不喜欢/可以"
2. **反馈维度** — 触发场景/触发时间/用户当前状态
3. **反馈影响** — 喜欢降低阈值/不喜欢提高阈值
4. **反馈可视化** — 记忆控制台,用户查看所有反馈

**3.4 模型迭代**

1. **离线训练** — 每月/每季度批量更新模型
2. **在线学习** — 小流量实时优化(慎用,防止模型漂移)
3. **A/B 测试** — 新模型灰度 5% → 20% → 100%
4. **效果评估** — CER/准确率/用户满意度

**3.5 隐私合规**

1. **日志加密** — 车端日志 AES-256 加密
2. **用户可查** — 记忆控制台查看所有埋点
3. **用户可删** — 一键清空埋点(车端+云端同步删除)
4. **数据最小化** — 只埋必要字段,不过度收集

**5. 功能点展开**

**5.1 D-F8 用户行为埋点**

**5.1.1 用户价值**

**做什么**：**最小必要原则**下记录用户在座舱的关键行为（语音 / 车控 / 导航 / 音乐 / 充电 / 会议等 15-20 个事件类型），让团队知道"用户真实怎么用产品"——哪些功能高频、哪些低频、哪个时段用啥、哪些场景出问题。埋点是"持续优化"的数据底座，没有埋点 = 盲人摸象。

**不做什么**：

* 不做"全量埋点"（像特斯拉那样激进收集 → 用户反感 + 合规风险）
* 不做"用户不知情的埋点"（违反合规 + 信任流失）
* 不做"埋点内容含敏感字段"（车内画面 / 声纹等绝不埋点）
* 不做"埋点数据永久保留"（定期清理 + 用户可删）

**核心价值主张**：

* **最小必要**：只埋产品优化必需的字段
* **用户可知**：隐私政策 + 记忆控制台透明可查
* **用户可删**：一键清空（端云同步）

**5.2 D-F9 失败 case 回流**

**5.2.1 用户价值**

**做什么**：当 AI 出错（ASR 识别失败 / NLU 理解错 / 工具调用失败 / 任务执行失败），**自动收集失败上下文**（用户原始语音 / 识别结果 / NLU 结果 / 工具响应）回流到训练 / 优化流程。**失败 case 是模型迭代的金矿**——没有失败就没有进步。但收集要合规、要用户可关闭。

**不做什么**：

* 不做"失败 case 默认全收集"（用户不知情 → 合规风险）
* 不做"失败 case 永久保留"（数据老化 / 存储压力）
* 不做"失败 case 关联用户"（隐私风险）
* 不做"只收集不分析"（数据进了仓库没人看 = 浪费）

**核心价值主张**：

* **自动收集 + 用户可关闭**：默认开启 + 设置中可关
* **匿名化处理**：数据不关联具体用户
* **优先级标注**：失败类型分级（高 / 中 / 低优先级回流）
* **闭环到模型迭代**：定期人工 review + 进入训练集

**5.3 D-F10 模型迭代（A/B 测试）**

**5.3.1 用户价值**

**做什么**：座舱 AI 模型上线后**持续迭代**——用埋点数据 + 失败 case 回流的结果训练新模型、通过 **A/B 测试**验证新模型效果（灰度 5% → 20% → 100%）、出问题可**一键回滚**。V1 以**离线训练**为主（每月/每季度批量更新），**V1.5 视用户量级决定是否上线学习**。特斯拉在线学习是激进路线，我们 V1 不跟。

**不做什么**：

* 不做"不测试直接上线"（模型变差也不知道）
* 不做"一次性全量推送"（出问题影响全网）
* 不做"V1 上线学习"（模型漂移风险）
* 不做"A/B 结果不透明"（团队决策盲人摸象）

**核心价值主张**：

* **离线为主，在线慎用**：V1 只做离线训练 + A/B 测试
* **严格灰度**：5% → 20% → 100%，每阶段观察 24-72h
* **可回滚**：出问题一键回滚到上版本
* **效果透明**：A/B 结果看板，核心指标明确

**5.3.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景 | 迭代行为 | 关键约束 |
| S1 | **每月模型更新**：ASR / NLU / 意图分类模型 | 离线训练 → 评估 → 灰度 5% | 月度常规节奏 |
| S2 | **灰度扩大**：5% 用户 24h 无异常 | 扩到 20% 观察 48h | 分阶段观察 |
| S3 | **全量推送**：20% 72h 无异常 | 扩到 100% | 一般 7 天完成全量 |
| S4 | **灰度期发现问题**：5% 用户 ASR 准确率下降 | 立即停止灰度 + 分析 + 回滚 | 异常检测自动化 |
| S5 | **紧急回滚**：全量后发现严重问题 | 1 小时内回滚所有用户 | 回滚流程 |
| S6 | **A/B 对比**：新 vs 旧模型 | 核心指标对比（准确率 / 延迟 / 用户反馈） | 指标看板 |
| S7 | **分层 A/B**：按车型 / 地区 / 用户画像分层 | 不同群体可能结果不同 | 分层决策 |
| S8 | **V1.5 试水在线学习**：小流量 | 监控模型漂移 + 快速回退 | 谨慎 |
| S9 | **模型依赖问题**：新模型需要新数据 schema | 数据 schema 版本化 | 兼容性 |
| S10 | **团队 review**：每次迭代前后评审 | 产品 + 算法 + QA + 合规 | 多角色把关 |

**6. 自我反驳与边界**

**6.1 特斯拉激进数据收集,我们要不要学?**

**自反**:学不好,用户反感(隐私争议)。**反驳自反驳**:V1 离线为主,不上线学习;埋点按"最小必要原则",不贪。**用户知情权 > 数据量**。

**6.2 主动服务反馈数据,用户真的会填吗?**

**自反**:用户懒得点"喜欢/不喜欢",数据量不够。**对冲方案**:(1)简化入口(一键反馈)(2)隐性收集(沉默≠同意,但"取消/跳过"算不喜欢)(3)V1 上线后用 A/B 测试看反馈率,如果 < 20% 说明入口设计有问题,要改

**6.3 失败 case 回流,用户会不会觉得被监控?**

**自反**:会。**对冲方案**:(1)明确告知"失败数据帮助我们优化"(2)用户可关闭(3)数据不关联具体用户(匿名化)
