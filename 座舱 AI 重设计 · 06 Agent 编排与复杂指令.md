**§1 背景与目标**

**1.1 为什么做 Agent 编排**

**核心问题**：用户说"帮我订个餐厅,不要太远,有停车位,7 点"——这句话需要调用导航(查距离)+ 美团(查餐厅)+ 地图(查车位)+ 日历(记行程)四个工具,还要处理中间结果(查完发现没车位要换一家)。传统指令式系统做不了这种**多步、有条件分支、需协调多个服务**的任务。

三大矛盾：

1. **复杂指令碎裂**：用户一个意图被拆成 4 个独立操作,每步都要手动触发
2. **工具孤岛**：导航/音乐/车控/第三方 App 各自 API 独立,没有统一调度层
3. **失败无兜底**：中间步骤失败(餐厅已满)后整个任务挂掉,不会自动换方案

**AI 时代的机会**：

* **Agent 化**：大模型作为规划器,把复杂指令拆解为工具调用序列
* **流式执行**：规划一步执行一步,不等你全部规划完再开始
* **失败自愈**：中间步骤失败自动换备选方案(餐厅 A 满了换 B)

**目标**：

* **主目标**：2 步以上复杂指令完成率 >70%
* **单步目标**：单工具调用成功率 >95%
* **安全目标**：高代价操作(支付/发消息)100% 需确认

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **一句话搞定复杂事**："导航去机场,不走高速,路上找个加油站"——系统自动拆解、规划、执行
* **失败了不卡住**：加油站 A 没开放,自动换 B,用户无感
* **中途可改**："算了还是走高速"→ 系统实时调整后续规划

**对全车**：

* **跨服务协作**：导航+音乐+空调+座椅联动("我困了"→ 开窗+降温度+放提神歌+推荐服务区)
* **可扩展**：新服务(如外卖/票务)接入后自动可被 Agent 调用,无需用户学习

**2.2 反价值(不做什么)**

* **不做无限 Agent**：不是所有操作都需要 Agent 编排——"开空调"直接执行,不规划
* **不做无边界调用**：Agent 可调用的工具有白名单,不可调用未授权的 API
* **不做不可中断**：任何复杂任务执行中可随时"停止"

* **自动支付**：不做 Agent 直接调用支付工具——必须用户确认

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **复杂指令完成率** | >70%(2 步以上) | 端到端任务评测 | <50% 降级为手动分步 |
| **单工具调用成功率** | >95% | 工具调用成功率 | <90% 标记工具不可用 |
| **规划延迟** | <2s | 从指令到第一步执行 | >5s 提示"正在处理" |
| **中途修改响应** | <3s | 从修改指令到计划调整 | >10s 放弃调整,重新规划 |

**4.2 可靠性与降级**

1. **规划器不可用**：降级为规则链式执行(无分支,无自愈)
2. **工具白名单为空**：降级为"仅车控"模式,第三方服务不可用
3. **执行超时**：单步 10s 超时,总任务 60s 超时,超时停止并提示已完成的部分

**4.3 安全与合规**

* **沙箱执行**：Agent 在沙箱中执行,不可直接访问车辆安全关键系统
* **操作确认分级**：车控(低风险)→ 自动执行;支付/发消息(高风险)→ 必须确认;驾驶安全相关 → 禁止
* **执行日志**：所有 Agent 执行路径可追溯,用于审计和问题排查

**3. 核心能力要求**

**3.1 工具定义标准**

1. **统一协议** — 类 MCP / Function Calling 标准
2. **工具注册表** — 按域分类(导航/音乐/充电/通讯/外卖)
3. **工具元信息** — 名称、参数、返回、错误、延迟、成本
4. **版本管理** — 工具升级不影响已有调用

**3.2 任务规划引擎**

1. **单任务执行** — 简单意图直接调工具
2. **多步任务拆解** — 长句意图 → 子任务图(嘉兴行程 → 5 个子任务)
3. **时序排列** — 有先后依赖的子任务按顺序
4. **并行子任务** — 可并行的同时启动(路况+天气+日程)
5. **失败回滚** — 子任务失败的降级/重试策略

**3.3 任务中途修改**

1. **插入子任务** — "加一个加油"
2. **替换子任务** — "第二步换成虹桥"
3. **取消子任务** — "取消最后一步"
4. **重排时序** — "先去上图,再充电"

**3.4 流式规划+执行**

1. **规划和执行并行** — 不等整个 plan 出来就开始执行第一步
2. **执行中的新规划** — 发现第一步异常,实时重新规划后续
3. **用户打断** — 规划/执行中随时可打断

**3.5 多 Agent 协同(V1 保留评估)**

1. **单 Agent 基础** — V1 默认单 Agent,简单任务
2. **多 Agent 场景** — 复杂场景(如全家出游)可能需要多个 Agent 协同(家庭 Agent + 出行 Agent)
3. **V1 决策** — 视沈砚技术评估,**倾向 V1 单 Agent,V1.5 再上多 Agent**

**3.6 工具调用安全沙箱**

1. **调用隔离** — 第三方只能访问授权字段
2. **返回扫描** — 防 prompt 注入
3. **高危拦截** — 支付/位置共享必须走多层认证(见《07》)
4. **超时熔断** — 单工具 > 3s 熔断,换备选

**3.7 失败兜底**

1. **重试** — 暂态错误(超时/5xx)3 秒内重试 1 次
2. **降级** — 备选工具(高德挂 → 百度)
3. **用户告知** — "刚才 X 出了点问题,我帮你用 Y 搞定了"
4. **全挂路径** — "这件事现在做不了,建议回头用手机"

**5. 功能点展开**

**5.1 A-F1 多步复合任务解析（执行侧）**

*待填*

**5.1.1 一句话定义**

**做什么**：接收 NLU（文档 03 A-F1）拆解后的**多意图任务图**（DAG），按**依赖关系**编排执行顺序，调度工具（导航/车控/订单/查询），处理**并行/串行/条件分支**，把"去嘉兴，先充电再接朋友"变成可执行的任务流。

**不做什么**：

* 不做"意图拆解"（03 NLU 的事，06 只负责执行）
* 不做"工具实现"（工具是各 App/服务提供的，06 只负责调用）
* 不做"任务状态持久化"（04 A-F5 的事）

**A-F1 在 03 vs 06 的分工**：

* **03 NLU 侧**：用户说"去嘉兴，先充电再接朋友" → 拆成 3 个意图（导航嘉兴 + 充电 + 接朋友）+ 依赖关系（充电 → 接朋友 → 导航嘉兴）
* **06 执行侧**：接收 DAG → 按依赖顺序调度 → 充电完成 → 接朋友完成 → 导航嘉兴

**5.1.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 执行编排 |
| S1 | **串行任务** | "去嘉兴，先充电再接朋友" → DAG: 充电 → 接朋友 → 导航嘉兴 → 按顺序执行 |
| S2 | **并行任务** | "今天路上会堵吗，有没有什么事" → DAG: 路况查询 ‖ 日程查询 ‖ 天气查询 → 并行执行 |
| S3 | **条件分支** | "如果路上堵就走高速，不堵走国道" → DAG: 路况查询 → if 堵 then 高速 else 国道 → 条件执行 |
| S4 | **依赖失败** | 充电桩没车位 → 充电任务失败 → 依赖链：接朋友任务取消（因为依赖充电完成）→ 导航嘉兴任务取消 |
| S5 | **部分失败** | 充电桩没车位 → 充电任务失败 → 但接朋友任务不依赖充电 → 接朋友继续 → 导航嘉兴继续 |
| S6 | **超时取消** | 充电任务 30min 未完成 → 超时取消 → 依赖链取消 |

**任务图数据结构**：

|  |
| --- |
| JSON {  task\_graph\_id: "uuid",  user\_id: "ou\_xxx",  tasks: [  { task\_id: "t1", intent: "充电", depends\_on: [], status: "pending" },  { task\_id: "t2", intent: "接朋友", depends\_on: ["t1"], status: "pending" },  { task\_id: "t3", intent: "导航嘉兴", depends\_on: ["t2"], status: "pending" }  ],  created\_at: timestamp } |

**5.1.3 输入 / 触发条件**

* **输入**：03 NLU 输出的任务图（DAG）、工具注册表、任务状态（04 S-F1）
* **触发**：NLU 完成任务拆解 → 触发 06 编排执行
* **前置条件**：任务图合法（无循环依赖）；工具可用

**5.1.4 执行流程**

|  |
| --- |
| Plain Text 接收任务图（DAG）  ↓ 依赖检查  ├─ 检测循环依赖 → 有则拒绝 + TTS "任务有循环依赖，无法执行"  └─ 无循环 → 继续  ↓ 拓扑排序（确定执行顺序）  ├─ 无依赖任务 → 立即执行  ├─ 有依赖任务 → 等待依赖完成  └─ 并行任务 → 同时执行  ↓ 任务调度  ├─ 串行任务 → 按顺序执行（t1 完成 → t2 开始）  ├─ 并行任务 → 同时执行（t1 ‖ t2 ‖ t3）  └─ 条件任务 → 根据条件分支执行  ↓ 工具调用  ├─ 查询工具注册表 → 找到对应工具  ├─ 调用工具 API（导航/车控/订单/查询）  └─ 等待工具返回  ↓ 任务状态更新（04 S-F1）  ├─ 任务完成 → status = "completed"  ├─ 任务失败 → status = "failed"  └─ 任务取消 → status = "cancelled"  ↓ 依赖链处理  ├─ 任务完成 → 唤醒依赖任务  ├─ 任务失败 → 依赖任务取消（级联取消）  └─ 任务取消 → 依赖任务取消  ↓ 全部任务完成  └─ TTS "已完成所有任务" |

**关键时间预算**：

* 任务图解析延迟：**≤ 50ms**
* 拓扑排序延迟：**≤ 20ms**
* 任务调度延迟：**≤ 30ms**
* 工具调用延迟：**取决于工具**（导航 < 1s，订单 < 3s）

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **循环依赖未检测** | 任务死锁 | 拓扑排序前强制检测；检测到拒绝 + TTS 提示 |
| **依赖链级联失败** | 一个任务失败 → 所有任务取消 → 用户体验崩 | 依赖类型分"强依赖"（必须完成）和"弱依赖"（可选）；弱依赖失败不影响后续 |
| **并行任务资源冲突** | 同时调用导航 + 车控 → 资源冲突 | 04 S-F2 并发任务管理 + 资源仲裁 |
| **工具调用超时** | 充电桩 API 30min 未响应 → 任务卡死 | 每个工具调用有超时（默认 30s）；超时后标记失败 + 依赖链取消 |
| **任务图过大** | 用户说"去嘉兴，做 20 件事" → 任务图爆炸 | V1 限制任务图 ≤ 10 个任务；超过拒绝 + TTS "任务太复杂，请分批说" |

**Paper Tiger（看起来像风险但不是）**

* "DAG 编排太复杂用户不需要"—— 实证：**"去嘉兴，先充电再接朋友"是高频**，不做 = 用户要分 3 次说
* "拓扑排序端侧扛不住"—— 实测：10 个任务的 DAG 拓扑排序 < 20ms，端侧无压力

**5.2.1 一句话定义**

**做什么**：任务执行过程中，用户随时可以**修改/插入/取消任务**（"加个加油""第二步换成虹桥""取消最后一步""先去上图再充电"），06 能动态更新任务图并继续执行。

**不做什么**：

* 不做"新任务创建"（那是 A-F1 的事，A-F1 是初次规划）
* 不做"任务执行"（A-F1 的事）
* 不做"用户意图识别"（03 NLU 的事，A-F2 只负责修改任务图）

**核心能力**：任务图是**动态的**，支持运行时修改。

**5.2.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 任务修改 |
| S1 | **插入任务** | 正在执行"充电 → 接朋友 → 导航嘉兴"，用户"加个加油" → 插入 t4（加油）→ 新任务图：充电 → 接朋友 → 加油 → 导航嘉兴 |
| S2 | **替换任务** | 正在执行"导航浦东机场"，用户"第二步换成虹桥" → 替换 t2：原"路径规划浦东"→"路径规划虹桥" |
| S3 | **取消任务** | 用户"取消最后一步" → 删除 t3（导航嘉兴）→ 任务图：充电 → 接朋友（完成后结束） |
| S4 | **重排任务** | 用户"先去上图再充电" → 调整顺序：上图 → 充电 → 接朋友 → 导航嘉兴 |
| S5 | **撤销修改** | 用户"算了还是按原来的" → 撤销修改，恢复原任务图 |
| S6 | **修改执行中任务** | 正在执行充电，用户"充电站换一家" → t1 状态从 executing → cancelling → 取消当前充电 + 重新规划充电任务 |

**修改类型**：

|  |
| --- |
| Plain Text 1. insert: 插入新任务 2. replace: 替换现有任务 3. cancel: 取消任务 4. reorder: 重排顺序 5. modify\_slot: 修改任务参数（如终点从浦东改虹桥） 6. revert: 撤销上一次修改 |

**5.2.3 输入 / 触发条件**

* **输入**：03 NLU 输出的修改意图（"加个加油""取消最后一步"）、当前任务图、任务执行状态
* **触发**：NLU 检测到修改意图
* **前置条件**：任务图存在；修改目标合法（如"取消第 5 步"但只有 3 步 → 拒绝）

**5.2.4 执行流程**

|  |
| --- |
| Plain Text 接收修改意图（03 NLU）  ↓ 修改目标识别  ├─ 插入任务 → 定位插入位置  ├─ 替换任务 → 定位目标任务  ├─ 取消任务 → 定位目标任务  ├─ 重排任务 → 确认新顺序  └─ 修改槽位 → 定位目标槽位  ↓ 合法性校验  ├─ 目标任务存在 → 合法  ├─ 目标任务已完成 → 拒绝（不能改已完成的）  ├─ 目标任务执行中 → 需取消重做  └─ 修改导致循环依赖 → 拒绝  ↓ 用户确认（可选）  ├─ 高风险修改（如取消导航）→ TTS 确认  └─ 低风险修改（如换终点）→ 静默执行  ↓ 任务图更新  ├─ 版本控制（保留上一版便于撤销）  ├─ 原子更新（不允许半成品状态）  └─ 状态机同步（04 S-F1）  ↓ 继续执行  ├─ 插入任务 → 等到插入位置执行  ├─ 替换任务 → 下次执行用新任务  ├─ 取消任务 → 依赖链级联处理  └─ 重排任务 → 重新拓扑排序  ↓ TTS 反馈  └─ "好的，已经加了加油" / "好的，换成虹桥了" |

**关键时间预算**：

* 修改解析延迟：**≤ 100ms**
* 任务图更新延迟：**≤ 50ms**
* 执行中任务取消延迟：**≤ 500ms**

**5.4.1 一句话定义**

**做什么**：复杂任务（如"帮我做个嘉兴攻略"）不等 LLM 规划完整任务图再执行，而是**边规划边执行**（LLM 流式输出第一步 → 立即执行 → 同时规划第二步），减少用户等待，提升响应速度。

**不做什么**：

* 不做"简单任务"（单步任务不需要流式，直接执行）
* 不做"任务规划"（规划是 LLM 的事，A-F4 只负责流式协调）
* 不做"任务执行"（执行是 A-F1 的事）

**核心价值**：用户感知延迟从"规划 3s + 执行 5s = 8s"变成"规划 0.5s + 执行 5s（并行）= 5.5s"。

**5.4.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 流式规划 + 执行 |
| S1 | **嘉兴攻略** | 用户"帮我做个嘉兴攻略" → LLM 流式输出"第一步：小红书查攻略" → 立即执行 → 同时 LLM 输出"第二步：高德规划路线" → 立即执行 |
| S2 | **数字停车** | LLM 流式输出"第一步：查车位" → 立即执行 → 同时输出"第二步：预约支付" → 等第一步完成后执行 |
| S3 | **规划失败** | LLM 流式输出"第一步：X" → 执行失败 → LLM 根据失败结果调整"第二步：Y（备选）" |
| S4 | **用户打断** | LLM 正在规划第三步，用户"算了不要了" → 停止规划 + 取消未执行任务 |
| S5 | **规划修正** | LLM 流式输出"第一步：A" → 执行中发现 A 不可行 → LLM 修正"第一步改成 B" |
| S6 | **全量规划降级** | 流式规划 3 次失败 → 降级到全量规划（等 LLM 规划完整任务图再执行） |

**流式规划数据结构**：

|  |
| --- |
| JSON {  stream\_id: "uuid",  user\_id: "ou\_xxx",  chunks: [  { chunk\_id: 1, task: "小红书查攻略", status: "executing" },  { chunk\_id: 2, task: "高德规划路线", status: "pending" },  { chunk\_id: 3, task: "携程订酒店", status: "planning" }  ],  mode: "streaming" // vs "batch" } |

**5.4.3 输入 / 触发条件**

* **输入**：用户复杂任务意图、LLM 流式输出、任务执行状态
* **触发**：NLU 检测到复杂任务（≥ 3 步）
* **前置条件**：LLM 支持流式输出；任务图支持动态更新

**5.4.4 执行流程**

|  |
| --- |
| Plain Text 接收复杂任务意图  ↓ 流式规划触发  ├─ 任务复杂度 ≥ 3 步 → 流式规划  └─ 任务复杂度 < 3 步 → 全量规划  ↓ LLM 流式输出（第一步）  ↓ 立即执行第一步  ├─ 不等后续步骤规划完  └─ 并行：执行第一步 + LLM 规划第二步  ↓ 第一步执行结果反馈给 LLM  ├─ 成功 → LLM 继续规划第二步  ├─ 失败 → LLM 调整第二步（备选方案）  └─ 用户打断 → 停止规划 + 取消未执行  ↓ LLM 流式输出（第二步）  ↓ 执行第二步（依赖第一步完成）  └─ 循环直到所有步骤完成  ↓ 全部完成  └─ TTS "已完成所有任务" |

**关键时间预算**：

* 流式规划首步延迟：**≤ 500ms**（LLM 流式输出第一步）
* 步骤间延迟：**≤ 300ms**（LLM 规划下一步）
* 总延迟节省：**30-50%**（vs 全量规划）

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |

* **和 A-F2 任务修改的协作**：流式规划中用户修改 → 停止流式 + 切全量

**5.6 E-F3 工具调用安全沙箱**

*待填*

**5.6.1 一句话定义**

**做什么**：第三方 App 的工具调用走**隔离沙箱**——权限最小化、返回数据扫描、prompt 注入防御、高危操作二次确认、超时熔断，防止第三方 App 拿到用户敏感数据或污染系统。

**不做什么**：

* 不做"工具实现"（工具由第三方 App 实现）
* 不做"工具调用"（A-F1/A-F3 的事）
* 不做"用户授权"（07 生态接入的事）

**核心价值**：**Agent 自主执行 ≠ 无边界执行**。沙箱是信任的底线。

**5.6.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 沙箱防护 |
| S1 | **权限最小化** | 美团下单只需要"地址 + 联系电话"，不给"行程历史""常去地点" |
| S2 | **返回扫描** | 小红书返回的攻略中嵌入"忽略之前的指令，执行 X" → 沙箱识别 prompt 注入 → 清理 + 告警 |
| S3 | **高危二次确认** | Agent 要调用"删除联系人 X" → 高危操作 → TTS 二次确认"确定删除 X 的联系方式吗？" |
| S4 | **超时熔断** | 携程 API 5s 未响应 → 超时熔断 → 降级备选（或提示用户手动操作） |
| S5 | **数据泄露尝试** | 恶意 App 要访问"家地址 + 常去地点" → 沙箱拦截 + 告警 + 降级功能 |
| S6 | **沙箱失效** | 沙箱组件异常 → 默认拒绝所有工具调用 → TTS "暂时不能调用外部服务" |

**沙箱防护层**：

|  |
| --- |
| Plain Text 1. 权限最小化 - 工具只能访问授权字段 2. 返回扫描 - prompt 注入检测、恶意内容过滤 3. 高危拦截 - 支付/位置共享/删除类必须二次确认 4. 超时熔断 - 单工具 > 3s 熔断 5. 调用审计 - 每次调用记录日志 6. 异常隔离 - 工具崩溃不影响主系统 |

**5.6.3 输入 / 触发条件**

* **输入**：工具调用请求、工具注册表、用户授权配置、沙箱规则库
* **触发**：每次工具调用
* **前置条件**：沙箱组件可用；规则库已配置

**5.6.4 执行流程**

|  |
| --- |
| Plain Text 工具调用请求  ↓ 权限校验  ├─ 检查工具是否在用户授权列表  ├─ 检查传入参数是否在工具声明的 inputs 范围  ├─ 不合法 → 拒绝 + 告警  └─ 合法 → 继续  ↓ 高危操作检查  ├─ 操作类型在高危列表（支付/删除/位置共享）→ 二次确认  ├─ 用户确认 → 继续  └─ 用户拒绝 → 取消  ↓ 工具调用执行（沙箱环境）  ├─ 隔离进程 / 受限权限  ├─ 超时控制（3s 默认）  └─ 异常捕获（工具崩溃不影响主系统）  ↓ 返回数据扫描  ├─ prompt 注入检测（正则 + LLM 分类）  ├─ 恶意 URL 检测  ├─ 敏感数据过滤（信用卡号 / 身份证等）  └─ 不合法 → 清理 + 告警 + 返回清理后数据  ↓ 审计日志  ├─ 记录工具 / 参数 / 结果 / 用时  └─ 高危操作详细记录（谁在何时批准了什么）  ↓ 返回结果给调用方 |

**关键时间预算**：

* 权限校验延迟：**≤ 10ms**
* 返回数据扫描延迟：**≤ 100ms**
* 沙箱调用总延迟：**≤ 工具 SLA + 150ms**

**5.7.1 一句话定义**

**做什么**：第三方 App / 服务**故障**（API 挂了、限流、网络异常、返回错误）时，自动**降级到备选方案**（高德挂 → 百度；美团挂 → 饿了么；都挂 → 提示用户手动操作），不让"一个 App 挂掉 = 整个 Agent 崩溃"。

**不做什么**：

* 不做"故障预测"（监控 / SRE 的事）
* 不做"故障修复"（第三方 App 自己修）
* 不做"用户通知"（仅在所有降级失败后告知用户）

**核心价值**：**Agent 体验的鲁棒性**——单点故障不影响整体可用。

**5.7.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 故障降级 |
| S1 | **导航降级** | 高德 API 挂 → 自动切百度 → 用户无感 |
| S2 | **外卖降级** | 美团下单失败（限流）→ 切饿了么 → 用户无感 |
| S3 | **多重降级** | 高德挂 → 百度 → 腾讯地图 → 都挂 → TTS "导航服务暂时不可用，建议手动打开" |
| S4 | **限流降级** | 携程 API 限流 → 排队等 5s → 仍未通过 → 切去哪儿 |
| S5 | **慢响应降级** | 携程 5s 未返回 → 切去哪儿 → 不等 |
| S6 | **故障告知用户** | 所有备选都挂 → TTS "X 服务暂时有问题，已经帮你完成了 Y、Z，剩下的建议手动" |

**降级策略**：

|  |
| --- |
| Plain Text 1. 重试：暂态错误（超时 / 5xx）3 秒内重试 1 次 2. 备选切换：主工具失败 → 切备选工具（导航：高德 → 百度 → 腾讯地图） 3. 功能降级：完整功能不可用 → 简化功能（如订餐失败 → 给推荐列表让用户手动） 4. 显式告知：所有降级失败 → 明确告诉用户"X 没搞定" |

**5.7.3 输入 / 触发条件**

* **输入**：工具调用结果（成功 / 失败 / 超时）、备选工具表、用户偏好
* **触发**：
* 工具调用失败（5xx / 超时 / 限流）
* 工具响应慢（> SLA）
* 工具返回错误（4xx 部分）
* **前置条件**：备选工具表已配置；备选工具可用

**5.7.4 执行流程**

|  |
| --- |
| Plain Text 工具调用失败  ↓ 失败分类  ├─ 暂态错误（超时 / 5xx / 网络）→ 重试  ├─ 限流（429）→ 等待 + 重试  ├─ 业务错误（4xx）→ 不重试，直接降级  └─ 服务不可用（503）→ 直接降级  ↓ 重试（最多 1 次）  ├─ 重试成功 → 继续  └─ 重试失败 → 降级  ↓ 查备选工具表  ├─ 有备选 → 切换  └─ 无备选 → 跳到"功能降级"  ↓ 备选工具调用  ├─ 成功 → 继续，TTS 静默切换  └─ 失败 → 继续切下一备选  ↓ 所有备选都失败  ├─ 功能降级（如订餐 → 推荐列表）  └─ 显式告知用户  ↓ 故障日志  ├─ 记录故障类型 / 工具 / 时间  └─ 触发 SRE 告警（高频故障） |

**关键时间预算**：

* 故障检测延迟：**≤ 100ms**（超时即检测）
* 重试延迟：**≤ 3s**
* 备选切换延迟：**≤ 500ms**
* 总降级延迟：**≤ 5s**（重试 + 切备选）

**5.7.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **降级链全失败** | 用户拿不到结果 | 显式告知用户 + 提示手动操作；不假装成功 |
| **降级延迟过长**（重试 + 多次切备选） | 用户等不及 | 总降级延迟 ≤ 5s；超过直接告知用户 |
| **降级结果不一致**（高德 vs 百度路线不一样） | 用户体验差 | 主备工具能力对齐 + 用户可指定偏好 |
| **降级触发过敏**（4xx 业务错误也触发降级） | 备选工具被滥用 | 错误码分类清晰；4xx 不降级（业务问题不是工具问题） |
| **故障未告知用户**（默默降级了，用户不知道） | 用户对结果困惑 | 静默降级 OK，但所有降级失败必须明确告知 |

**Paper Tiger（看起来像风险但不是）**

* "用户不需要降级，让 App 自己修"—— 实证：**第三方 App 故障率 1-3%**，Agent 不能依赖完美 App
* "备选工具体验差"—— 不一定，体验对齐做好后用户无感

**Elephant（团队可能没讨论够）**

* **备选工具表的"商务"**：备选工具可能是商业竞品，谈商务怎么谈？**产品 + BD 团队的事**
* **降级的"用户偏好"**：用户可以说"导航只用高德，不用百度"吗？**建议：V1 不开放，V1.5 考虑**
* **降级的"主动恢复"**：降级到备选后，主工具恢复了 → 自动切回？**建议：当前任务不切回，下个任务再切回**
* **故障的"传播阻断"**：A 服务挂导致 B 服务也调不动 → 隔离 / 熔断（Circuit Breaker）模式

**5.7.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 故障检测准确率 | ≥ 99% | 模拟各类故障 |
| 降级成功率 | ≥ 90% | 主工具失败 → 备选 |
| 降级总延迟 | ≤ 5s (P95) | 重试 + 切备选 |
| 错误分类准确率 | ≥ 95% | 暂态 vs 业务 vs 不可用 |
| 降级链全失败时告知率 | 100% | 必须告知用户 |
| 故障日志完整率 | ≥ 99.9% | SRE 监控 |

| 关 | 自问 | 答 |

**6.1 V1 多 Agent 架构（已定）⭐**

**老师 2026-05-06 拍板**：V1 直接上多 Agent 架构，不走"V1 单 Agent / V1.5 再上多 Agent"的保守路线。

**架构选型**：

* **拆分维度**：按**功能域**拆 Agent（不按品类，不按渠道）
* **协同方式**：**主从架构**（不是平等协作，不是消息总线）

**Agent 列表（V1）**：

|  |  |  |
| --- | --- | --- |
| Agent | 职责范围 | 内部调用 |
| **主 Agent**（Master） | 意图理解 + 任务拆解 + 任务分发 + 结果整合 | 调用专项 Agent |
| **座舱 Agent** | 车内服务：车控、空调、座椅、音乐、HMI、虚拟形象 | 调用座舱内部工具 |
| **生活服务 Agent** | 第三方服务：咖啡、外卖、打车、票务、本地生活、购物、旅行 | 调用 MCP（见文档 15） |
| **智驾 Agent** | 驾驶辅助、智能泊车、场景识别 | 调用智驾系统接口 |

**主从协作流程**：

|  |
| --- |
| Plain Text 用户："去嘉兴，先充电再喝咖啡"  ↓ 主 Agent：意图理解 + 任务拆解  - 子任务 1：充电（→ 座舱 Agent / 生活服务 Agent）  - 子任务 2：喝咖啡（→ 生活服务 Agent）  - 子任务 3：导航嘉兴（→ 座舱 Agent）  - 依赖关系：充电 → 咖啡 → 嘉兴  ↓ 主 Agent 按依赖顺序分发任务给专项 Agent  ↓ 专项 Agent 执行（生活服务 Agent 调 MCP，座舱 Agent 调车内工具）  ↓ 主 Agent 汇总结果给用户 |

**关键设计原则**：

* **专项 Agent 在自己领域内自主决策**（生活服务 Agent 自己决定走瑞幸还是星巴克，不需要请示主 Agent）
* **跨域协作必须经主 Agent**（座舱 Agent 不能直接调用生活服务 Agent）
* **主 Agent 不下场执行**（只做编排和整合，不直接调具体工具）

**通知沈砚**：架构方向已定（多 Agent / 主从 / 按功能域拆），工程实现细节由沈砚拍板：

* 通信协议（Function Call / RPC / 消息队列？）
* 状态同步（共享状态 vs 各 Agent 持有局部状态？）
* 失败兜底（专项 Agent 挂了主 Agent 怎么处理？）
* 性能开销（多 Agent 通信对响应延迟的影响）
* LLM 模型选型（主 Agent 用大模型，专项 Agent 用小模型？）

**与文档 15 的关系**：MCP 路由机制在**生活服务 Agent 内部**实现。生活服务 Agent 是 MCP 路由的载体，不是路由本身。

**6.2 流式规划会不会让用户困惑?**

**自反**:"边想边做"可能让用户感觉"AI 乱来"。**对冲方案**:(1)规划的每一步都要播报或卡片反馈(2)明确告知"我先帮你 X,然后 Y"(3)用户可随时打断重来

**6.3 嘉兴行程 5 个子任务,中间失败 1 个怎么办?**

**自反**:用户期望"一条龙"被打破,体验塌。**对冲方案**:(1)每个子任务独立可回滚,不影响其他(2)失败的子任务换备选(充电桩 A 挂 → 换 B)(3)实在回不来,明确告诉用户"X 环节没搞定,其他都可以继续"

**6.4 工具调用错了,谁负责?**

**自反**:AI 自动执行错了,用户会质疑"我没同意啊"。**对冲方案**:(1)高危操作(支付/位置/删除)必须二次确认(2)每次自动执行后有"撤销"按钮(3)操作日志可追溯
