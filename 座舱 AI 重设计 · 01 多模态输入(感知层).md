**§1 背景与目标**

**1.1 为什么做多模态输入**

**核心问题**：传统座舱交互以"触屏点按"为主,驾驶场景下存在三大矛盾:

1. **安全矛盾**：驾驶时视线离开路面操作屏幕,增加事故风险
2. **效率矛盾**：多层级菜单导航耗时长,高频操作(调空调/切歌/导航)平均需 3~5 次点击
3. **认知负担**：驾驶员需同时处理路况、车辆状态、乘员需求,屏幕交互占用过多注意力资源

**AI 时代的机会**：

* **语音**：从"命令式"(需记忆指令)→"对话式"(自然表达意图)
* **视觉**：DMS/OMS 从"监控"→"理解"(视线/手势/表情/唇动融合判断意图)
* **多模态融合**：语音 + 视线 + 手势组合消歧("把那个调小一点" + 视线锁定空调面板 → 降低温度)

**目标**：

* **主目标**：让 80% 的高频操作(车控/导航/娱乐/空调)可通过"说 + 看"完成,屏幕点按降为兜底手段
* **安全目标**：驾驶场景下交互时长 <3 秒,视线离开路面时间 <1.5 秒
* **体验目标**：多模态意图识别准确率 >92%,误唤醒率 <0.5 次/小时

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **更安全**：语音 + 视线交互让手不离方向盘、眼不离路面,降低分心驾驶风险
* **更高效**："打开空调 22 度"一句话完成,替代"点击空调图标 → 调节温度 → 确认"的 4 步操作
* **更自然**：不需记忆指令格式,像和副驾说话一样表达需求

**对乘员**：

* **独立控制**：副驾/后排通过多音区语音独立控制各自区域(空调/座椅/娱乐),不打扰驾驶员
* **情境感知**：系统通过 OMS 识别乘员状态(儿童哭闹/成人疲劳/宠物躁动),主动调整环境

**对全车**：

* **协同交互**：多人同时说话时,系统通过音区 + 视线 + 上下文判断"谁在对谁说、说给谁听"
* **无障碍友好**：老人/儿童/视障用户通过语音 + 简单手势完成操作,降低学习成本

**2.2 反价值(不做什么)**

* **不做炫技**：不追求"全语音控制"——复杂配置(如座椅记忆 8 向调节)仍保留屏幕精确操作
* **不做全知**：不强求系统"猜对一切"——识别不确定时主动询问,而非硬猜后出错
* **不做话痨**：不主动打断用户——除非安全相关(如碰撞预警),其他场景静默等待唤醒

**3.2 边界场景(需明确处理策略)**

|  |  |  |
| --- | --- | --- |
| 场景 | 挑战 | 处理策略 |
| **极端噪音**(高速开窗/暴雨) | ASR 准确率骤降 | 端侧 VAD 提前过滤 + 云端降噪模型 + 唇动识别兜底 |
| **多人同时说话** | 音区重叠 + 意图冲突 | 优先级:驾驶员 > 副驾 > 后排;冲突时语音提示"请一个一个说" |
| **儿童/宠物误触发** | 非目标用户唤醒系统 | 声纹 + 语义双重过滤:儿童咿呀学语/宠物叫声不触发唤醒 |
| **方言/口音** | ASR 泛化能力不足 | 端侧模型覆盖 Top 10 方言,长尾方言走云端 + 用户纠正后学习 |
| **隐私敏感场景**(商务通话/私密对话) | 用户不希望系统监听 | 物理按键"隐私模式":关闭麦克风阵列 + LED 指示灯熄灭 |

**3.3 非场景(明确不做)**

* **车外交互**：不做车外语音控制(如站在车外喊"打开后备箱")——安全风险 + 误触发高
* **全时监听对话内容**：不做"聊天内容理解"——只监听唤醒词,非唤醒状态下对话内容不上传
* **生物特征存储**：不在云端存储声纹/人脸原始数据——仅存特征向量,且用户可随时删除

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **唤醒响应延迟** | <500ms(端侧 KWS) | 从唤醒词结束到系统响应音 | >800ms 触发告警 |
| **ASR 识别延迟** | <1.5s(端侧) <3s(云端) | 从语音结束到文本返回 | 端侧 >2s / 云端 >5s 降级 |
| **多模态融合延迟** | <200ms | 语音 + 视线数据融合到意图输出 | >500ms 放弃融合,单模态兜底 |
| **误唤醒率** | <0.5 次/小时 | 非唤醒词触发 KWS 的频率 | >1 次/小时需调整唤醒阈值 |
| **ASR 准确率** | >95%(安静) >85%(噪音) | WER(词错误率) | <80% 提示用户"没听清,请再说一遍" |
| **多音区分离度** | >90% | 正确识别说话人位置的比例 | <75% 降级为全车单音区 |

**4.2 可靠性与降级**

**降级策略**：

1. **网络断开**：端侧模型兜底(覆盖 Top 50 高频指令)
2. **端侧算力不足**：关闭视觉融合,纯语音模式
3. **麦克风阵列故障**：单麦克风模式 + 提示用户"多音区功能暂不可用"
4. **DMS/OMS 摄像头遮挡**：纯语音模式,视线/手势/表情功能失效

**容错机制**：

* **识别不确定**(置信度 <0.7)：主动询问"你是要 A 还是 B?"
* **连续 3 次识别失败**：提示"抱歉没听懂,你可以点屏幕操作"
* **误操作撤销**：所有语音指令执行后 3 秒内可语音"撤销"

**4.3 安全与合规**

**功能安全**：

* **驾驶场景限制**：时速 >80km/h 时,禁用"复杂多轮对话"(如订酒店),仅保留车控/导航/娱乐简单指令
* **视线监控**：DMS 检测到驾驶员视线长时间(>3 秒)离开路面 → 暂停屏幕交互,语音提示"请注意路况"
* **紧急接管**：碰撞预警/车道偏离时,强制中断语音交互,优先播报安全提示

**隐私合规**：

* **数据最小化**：非唤醒状态下,麦克风数据仅在端侧 VAD 处理,不上传
* **用户控制**：提供"关闭语音助手"物理按键 + "删除历史录音"入口
* **透明度**：唤醒后 LED 指示灯常亮,用户明确知道"系统在听"

**4.4 关键取舍**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 取舍点 | 选项 A | 选项 B | 决策 | 理由 |
| **端云分工** | 全部云端(高准确率) | 全部端侧(低延迟) | **混合**:高频指令端侧,长尾指令云端 | 平衡延迟与准确率 |
| **唤醒词** | 固定唤醒词 | 自定义唤醒词 | **固定为主 + 自定义可选** | 固定词训练充分,自定义词误唤醒高 |
| **多音区数量** | 4 音区(主驾/副驾/后排左/后排右) | 2 音区(前排/后排) | **3 音区**(主驾/副驾/后排) | 成本与体验平衡点 |
| **视觉融合范围** | 全场景融合 | 仅驻车融合 | **驾驶限制融合**(仅视线,不做手势) | 驾驶安全优先 |
| **隐私模式** | 软件开关 | 物理按键 | **物理按键** | 用户信任度更高 |

**3. 核心能力要求**

**3.1 语音输入子系统**

1. **唤醒(KWS)** — 误唤醒率 ≤ 1 次/天/用户;漏唤醒率 ≤ 5%
2. **ASR** — 车内噪声环境 CER(字错率)≤ 5%;长句(>15 字)识别稳定
3. **拒识** — 基于意图置信度 + 上下文判断,避免话痨(蔚来反例)
4. **全双工** — 用户打断响应 ≤ 200ms,打断后 AI 立即停止当前说话
5. **多音区分离** — 主驾/副驾/后排独立音源分离,互不干扰
6. **流式处理** — ASR 边说边识别,不等整句(极氪路线)

**3.2 视觉输入子系统**

1. **DMS** — 疲劳识别(眨眼/哈欠/闭眼时长),置信度 ≥ 80% 才上报
2. **OMS** — 乘员数/位置/角色(儿童/老人),异常检测(哭声/呼救)
3. **视线追踪** — 用于"把那个调小"这种指代消歧
4. **手势识别** — 基础指令集(接听/挂断/音量调节)
5. **唇动识别** — 嘈杂环境 ASR 兜底
6. **表情识别** — 情绪输入(给关怀模式用)
7. **隐私边界** — 不做人脸识别也不存车内画面,只存结构化状态

**3.3 多模态融合**

1. **视觉+语音指代消歧** — 用户说"把这个调小",视线指向音乐卡片 → AI 理解指代目标
2. **置信度融合** — 单模态低置信时,其他模态补强
3. **降级机制** — 视觉挂了,退化到纯语音;语音挂了,退化到触控
4. **情境优先级** — 驾驶场景禁用手势/视线的需要"看屏幕"操作

**3.4 用户识别**

1. **融合判别** — DMS + OMS + 账号 + 蓝牙手机,置信度 ≥ 90% 才切换
2. **无感切换** — 切换时不打断当前任务,画像平滑过渡
3. **手动覆盖** — 用户可以一键"我是 X",强制切换
4. **家庭账号** — 家庭共享记忆 + 个人独占隐私(理想学起来)

**5.1 I-F1 唤醒 KWS**

**5.1.1 一句话定义**

**做什么**：在车内任何噪声环境下，用户说出唤醒词，系统在 300ms 内响应并进入"听命"状态。

**不做什么**：

* 不做开放唤醒（任何话都能唤醒，会误触）
* 不做跨账号共享唤醒（每个用户只能唤醒属于自己的人格）
* 不做无声唤醒（不通过 DMS 视线/手势直接激活 ASR，那是 I-F6 和 I-F7 的事）

**5.1.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 预期响应 |
| S1 | **晚上 7 点南北高架堵车**，司机单手握方向盘、车内放着娃要听的儿歌，说"小 X" | 500ms 内儿歌音量自动降到 30%，系统"嗯？"回应，进入听命状态 |
| S2 | **长途高速服务区刚出发**，副驾老婆刚醒没精神，小声说"小 X，开空调" | 即使声音弱（< 50 dB SPL），也能唤醒 + 识别出是副驾说的（多音区分离），执行"副驾侧空调打开" |
| S3 | **后排娃在吵闹**，噪声达 75 dB，司机喊"小 X！" | 后排噪声被分离（I-F5），前排唤醒词识别成功 |
| S4 | **电话会议中**，同事在说"嘿你那边有空唤醒一下刚才说的..." | **不唤醒**（拒识，I-F3 协同），会议正常继续 |
| S5 | **儿童模仿**：娃听爸爸说"小 X"学着说"小 X 放动画片" | 识别到是儿童声纹（I-F1 + DMS/OMS 辅助），走"儿童安全模式"，不是爸爸的完整画像 |

**5.1.3 输入 / 触发条件**

* **音频输入**：全车麦克风阵列，16kHz / 16bit，持续监听（低功耗 DSP 本地处理，不送云）
* **触发词**：默认一个主唤醒词（"小 X"），用户可从预设列表选 1-3 个备选（不开放自定义，避免生僻音识别困难）
* **前置条件**：车辆在 ACC 及以上电源状态；用户未手动关闭唤醒

**5.1.4 执行流程**

|  |
| --- |
| Plain Text 麦克风持续采样  ↓ 端侧 DSP 做 KWS 检测 (< 100ms)  ↓ 命中唤醒词？  ├─ 是 → 置信度判断（阈值分级见 §5.1.6）  │ ↓  │ 高置信 → 直接进入听命态 + 音频提示 + 抑制当前音频源  │ 中置信 → 进入"半听命态"，等下一句确认  │ 低置信 → 丢弃  │  └─ 否 → 继续监听 |

**关键时间预算**：唤醒词末尾 → 音频提示响起，**≤ 300ms**（端侧 KWS 100ms + 多音区分离 80ms + 音频处理抑制 120ms）

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **误唤醒**（车内电视里有人叫"小 X"） | 用户被动听到"嗯？"，信任度下降 | 引入声源方向检测：必须来自车内人声方向；引入上下文：媒体播放中提升阈值 |
| **漏唤醒**（吵闹环境、语速快、口音重） | 用户喊两三次才唤醒，崩溃 | 阈值分级（见 5.1.6）；用户自定义备选唤醒词；高噪声环境自动降低阈值 + 加"视线指向屏幕+说话"的多模态兜底（联动 I-F6） |
| **唤醒后的 200ms 空窗**（唤醒成功但 ASR 还没准备好） | 前半句被吃掉 | 唤醒触发时立刻启动 ASR 流式缓冲，保留唤醒前 500ms 音频给 ASR 回溯（overlap buffer） |

**Paper Tiger（看起来像风险但不是）**

* "唤醒词需要足够独特，用户记不住"—— 实证："小 X""同学""小爱"这类两到三字词，用户 1-2 天自然适应
* "儿童声纹和大人差异大，需单独模型"—— 只要触发层不分（先唤醒），后续个性化再分儿童模式（走 OMS + 声纹辅助），不是唤醒层的事

**Elephant（团队可能没讨论够）**

* **多设备竞争唤醒**：车主手机里也有同一 App 装着同一唤醒词，手机和车同时响应怎么办？结论：需要和手机端对齐唤醒仲裁机制（近场优先 / 上次活跃优先）—— 抛给沈砚，§7 加一条
* **唤醒词的情感延续**：用户喊"小 X"和冷冰冰"设备名"完全不同。唤醒词是"人格的第一声招呼"，这一声的 TTS 选型（轻快 vs 沉稳 vs 中性）是人格层的事，不是 KWS 层，但必须和人格层对齐（见文档 11）

**5.1.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 唤醒响应延迟 | ≤ 300ms (P95) | 室内 50 dB 背景噪声 |
| 漏唤醒率 | ≤ 5% | 日常行车 + 正常口音 + 中度噪声 |
| 误唤醒率 | ≤ 1 次 / 用户 / 天 | 连续 7 天实车日志 |
| 车内娃哭场景（B1） | 唤醒成功率 ≥ 90% | OMS 检测到后排哭声时 |
| 儿童唤醒识别 | 准确率 ≥ 85% | 3-10 岁儿童样本 |
| 声源方向判定 | 主驾/副驾/后排正确率 ≥ 95% | 多音区测试环境 |

**分级阈值（便利 vs 安全）**：

* 日常便利触发（个性化切换、娱乐控制）：高阈值 90%，避免打扰
* 安全相关触发（紧急呼救的潜在唤醒）：低阈值 70%，宁错不漏

**5.1.8 给沈砚的技术钩子**

1. **端侧 KWS 模型选型**：自研 vs 供应商（思必驰 / 云知声 / 科大讯飞）？推理资源预算？
2. **多设备唤醒仲裁**：车主手机（可能也装本 App）和车同时响应，谁赢？仲裁协议在哪层跑？（生态文档 07 的议题？）
3. **Overlap buffer 设计**：唤醒前 500ms 缓冲的音频怎么无缝拼给 ASR？有没有丢包/错位风险？
4. **声源方向检测精度**：麦克风阵列最少多少个麦才能做到 95% 方向准确？BOM 代价？
5. **端侧声纹识别**：是否在 KWS 层引入轻量声纹（主驾 / 副驾 / 后排 / 儿童粗分类）？这会影响 I-F1 的响应延迟吗？
6. **儿童安全模式的触发层**：KWS 侧只判断"像不像儿童声"，还是把这个判断下沉到 ASR + 后续 NLU？

**5.2 I-F2 ASR 流式识别（全双工核心：边听边识别 + 纠错 + 持续修改）**

**5.2.1 一句话定义**

**做什么**：用户说话的过程中，系统**边听边识别**，每 200-300ms 输出一次当前最优文字假设，直到用户停顿后给出最终结果。**与 I-F4 打断与恢复配合，构成座舱 AI 的"全双工体验"三支柱**：

* **支柱 1｜边听边识**（本功能）：用户说话时，系统不等说完就开始理解——屏幕实时显示灰字，延迟感减半
* **支柱 2｜纠错**（本功能 S2）：用户说到一半改主意（"导航到...呃不对"），系统检测回退信号，放弃未完成假设重新开始；或用户对已执行结果说"不对，我说的是 25 度"，系统理解为纠错而不是新指令
* **支柱 3｜持续修改**（本功能 S6 / I-F4）：对同一对象持续微调（"音量调大" → "再大一点" → "再小一点一点"），系统保持上下文不需要用户重复"音量"

**全双工 = 座舱语音体验的基础设施**——不是"多一个炫技功能"，而是让语音交互接近"跟真人对话"的门槛。做不好全双工，座舱 AI 体验会回到"对讲机"时代（说一句等一句）。

**不做什么**：

* 不做非流式 ASR（等用户说完再整段识别，延迟太大）
* 不在 ASR 层做意图理解（那是 I-F3 拒识和 NLU 的事）
* 不做跨语言实时翻译（商务场景需求，v2 考虑）
* 不做"每次修改都要重新唤醒"（违背全双工，用户烦）

**5.2.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 预期响应 |
| S1 | **晚上 7 点南北高架堵车**，司机说"小 X 帮我规划回家的路避开积水路段而且不要走北横" | 说到"规划回家"时屏幕已开始显示候选路径；说到"避开积水"时路径已自动过滤；说完 500ms 内最终路径出来 |
| S2 | **副驾老婆临时改口**："小 X 导航到...呃不对，先找一下最近的加油站" | 流式识别中途检测到"呃不对"回退信号，放弃"导航到..."的未完成假设，重新开始 |
| S3 | **后排娃插话**：司机说"明天早上七点提醒我..." 娃突然喊"我要吃冰淇淋！" | 主驾识别流不被后排声音打断（多音区分离 I-F5），继续输出司机的完整指令 |
| S4 | **口音场景**：四川司机说"给娘娘打个电话"（川普口音，"娘娘"实为"老婆"） | ASR 输出初稿"给娘娘打电话"，但通过用户通讯录上下文修正为"给老婆打电话"（需要 NLU 协同，本层只负责给出初稿 + 候选） |
| S5 | **中英混杂**："小 X 帮我找一下刚才发的那个 slides" | ASR 输出"刚才发的那个 slides"原文，不强制翻译；中英混合由 NLU 层处理 |

**5.2.3 输入 / 触发条件**

* **输入**：I-F1 唤醒后的音频流（含 overlap buffer 的前 500ms 回溯），16kHz 单声道主驾通道 + 副驾/后排辅助通道
* **触发**：唤醒成功后立即启动，静音检测（VAD）确认用户停顿 1.2s 后收尾
* **前置条件**：唤醒成功；网络可达时优先端云协同，网络差时端侧独立兜底

**5.2.4 执行流程**

|  |
| --- |
| Plain Text 唤醒成功 + overlap buffer 500ms  ↓ 端侧流式 ASR 启动 (20ms 窗口，滚动推理)  ↓ 每 200ms 输出一次 partial 结果 → 屏幕实时显示灰字  ↓ VAD 检测到用户停顿 > 1.2s？  ├─ 是 → 触发 final 结果 + 发给 NLU  │ 同时保留 Top-3 候选给上下文纠错  │  └─ 否 → 继续累加，每 200ms 更新 partial  网络可达时，云端大模型异步纠错 partial → 输出更优 final |

**关键时间预算**：

* partial 首出：唤醒后 ≤ 500ms
* partial 更新频率：200-300ms/次
* final 延迟：用户停顿末尾 → final 文本，≤ 400ms（端侧）；≤ 700ms（云端协同）

**5.2.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **弱网 / 无网场景**（地库/隧道/山路） | 云端 ASR 不可达，识别率骤降 | 端侧 ASR 必须独立可用，准确率下限 85%；云端只做"锦上添花"（上下文纠错、长句修正） |
| **口音/方言 ASR 翻车** | 用户"我说了好几遍车都听不懂" | 上线前至少覆盖七大方言区（四川/广东/东北/上海/河南/陕西/闽南）；线上"自学习"——用户纠正一次，个人画像记住 |
| **partial 结果跳字/闪烁**（前秒"导航到北京"，下秒变"导航到北海"） | 用户心里"这玩意儿不稳"，信任度跌 | 稳定性阈值：置信度 < 80% 的 partial 不对用户显示（仍在后台推理）；显示时用灰字表示"还在判断"，确认后转黑字 |
| **长句超过 15 秒** | 流式推理累计误差，final 质量下降 | 15s 软阈值触发"主动收尾"：TTS 提示"请稍等，我理解一下"，把已识别部分先给 NLU，后续当追问处理 |

**Paper Tiger（看起来像风险但不是）**

* "端云切换有感知延迟"—— 用户只关心 final 对不对，不关心背后是端还是云，后台降级无感
* "多音区音频混入会拉低主驾 ASR"—— 多音区分离（I-F5）解决，本层假设输入已干净

**Elephant（团队可能没讨论够）**

* **partial 对司机心智的影响**：屏幕上不断滚动的灰字会让司机分心。**是否让司机可关闭 partial 显示**（只靠 TTS 反馈）？交互文档 05 需对齐
* **隐私边界**：流式 ASR 的音频"边说边传"到云端，用户知情吗？车内"永远在听"的感知管理是法务/合规问题，抛给沈砚 + 法务，§7 加一条

| 指标 | 目标值 | 测试条件 |
| partial 首出延迟 | ≤ 500ms (P95) | 唤醒后用户开始说话 |
| final 延迟（端侧） | ≤ 400ms (P95) | 用户停顿末尾 → final 文本 |

| --- | --- | --- |
| 关 | 自问 | 答 |
| 场景真伪 | 流式真的比整句识别好吗？ | 好。"边说边看"减少焦虑，用户能提前判断"系统听对了没"；整句识别让用户说完盯着空屏幕 2 秒 |
| 失败路径 | 最糟糕失败？ | 弱网 + 口音重 + 长句三角暴击：用户说完 5 秒没反应，急了再说一遍，仍然没反应 → 砸手方向盘。**端侧独立兜底 + 15s 软收尾**必须做 |
| 替代方案 | 能不能只用云端 ASR？ | 不能。车载地库/隧道/山区网络抖动常态，端侧必须独立可用 |
| 代价账 | 端侧 ASR 模型占多少算力？ | 需沈砚给数字。产品底线：不能超过车载 SoC 20% 持续占用 |
| 竞品镜像 | 极氪端到端最激进，我们为什么不跟？ | 端到端 V1 风险太大：工程复杂度高、失败回滚难、训练数据缺口大。**三段式 + 云端大模型增强**——比华为"稳"更智能（大模型纠错上下文），比极氪"快"更稳（端侧独立兜底）。同时 overlap buffer + Top-3 候选为 v2 端到端留接口 |
| 撤退条件 | 什么指标不达预期就砍？ | 普通话 WER > 8% 或弱网可用率 < 85% 连续 2 个月，就要重构模型或换供应商 |

**5.2.8 给沈砚的技术钩子**

1. **端侧 ASR 模型选型和资源预算**：模型大小、内存占用、持续算力%？是否用量化版本？
2. **端云协同路由策略**：什么时候走端、什么时候走云、什么时候并行？切换延迟预算？
3. **上下文纠错机制**：云端大模型如何拿到用户通讯录/历史对话作为纠错输入？隐私保护怎么做？
4. **overlap buffer 无缝拼接**：I-F1 的 500ms buffer 喂给 ASR 时，如何避免帧边界错位导致首字丢？
5. **Top-3 候选输出**：ASR 返回 Top-3 候选（给 NLU 纠错用），代价是什么？
6. **隐私合规**：边说边传云，用户告知/开关/停止/删除机制怎么设计？法务必须介入

**5.3 I-F3 拒识（上下文+置信度）**

**5.3.1 一句话定义**

**做什么**：在唤醒成功后，判断用户接下来说的话是否真的是给系统的指令。**不是**指令（背景对话/电话内容/媒体声音/无意识自语）就静默拒识，不触发任何动作。

**不做什么**：

* 不在 KWS 层做（I-F1 只判断"听到唤醒词没"），拒识在唤醒后的语义层做
* 不靠纯关键词黑名单（那叫过滤，不叫拒识）
* 不做"自动播报式拒识反馈"（蔚来 NOMI 话痨教训：每次拒识都说"我不太明白"，反而让用户烦）

**5.3.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | 司机唤醒后接电话："喂老张，你那个文件我等下发你" | 系统判断"老张/文件"和车控/导航/娱乐意图都不沾边 → 静默拒识，不打断电话 |
| S2 | 唤醒后副驾对司机说："刚才那个出口你应该走右边的" | 识别"对人的话"（没有命令式动词指向系统），静默拒识 |
| S3 | 司机唤醒后说"小 X 你说今晚吃什么好" | **不拒识**——开放对话场景，人格层应接（走文档 11 NOMI 类闲聊路径） |
| S4 | 唤醒后司机叹气说"哎，这破天气" | 静默拒识（无指令，纯抒发）；**但记入情境**（可触发后续主动关怀，见 C-F4） |
| S5 | 唤醒后司机说"小 X，导航回家然后帮我把空调调到 22 度对了顺便放点轻音乐" | **不拒识**——复合指令，转给 NLU 拆解（I-F3 + NLU 协同） |
| S6 | 媒体声音误唤醒后：车载电台主持人说"那么我们现在来听一首歌" | 拒识（声源方向 = 车载扬声器而非车内人声 + 内容是被动接收非主动指令） |

**5.3.3 输入 / 触发条件**

* **输入**：I-F2 输出的 final 文本 + Top-3 候选 + 置信度；音频元数据（声源方向、音量、用户身份）；上下文（当前主屏状态、媒体播放状态、电话状态）
* **触发**：每次 ASR final 完成后立即触发，在送 NLU 前
* **前置条件**：I-F1 唤醒成功；I-F2 ASR final 已产生

**5.3.4 执行流程**

|  |
| --- |
| Plain Text ASR final 文本 + 上下文  ↓ 拒识判断模型（端侧 + 云端协同）  ├─ 第 1 关：声源 = 车内人声？（声源方向）  ├─ 第 2 关：有指令式语义？（有动词 + 客体）  ├─ 第 3 关：对话目标 = 系统？（不是对人说话）  ├─ 第 4 关：在车内能力范围内？（导航/车控/娱乐/通讯/服务）  ├─ 第 5 关：置信度综合判断  ↓  ├─ 拒识 → 静默不响应，记入"短期上下文窗口"（2 分钟内的话保留助下文判断）  ├─ 不确定 → 走"低成本确认"：TTS 简短"嗯？"/视觉提示待响应图标；用户继续说则进 NLU，无回应 5s 自动撤销  ├─ 不拒识 → 进 NLU 处理 |

**关键时间预算**：

* 拒识判断延迟：final 文本到达 → 决策完成，**≤ 100ms**
* 不能拖累 NLU 处理（NLU 自身预算约 300-500ms）

**5.3.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **过度拒识**（真指令被拒） | 用户："我说了没反应，这破车！" | "低成本确认"机制兜底——不确定时简短"嗯？"而非直接拒识；用户继续说就进 NLU，无回应 5s 自动撤销 |
| **过度不拒识**（车内闲聊被当指令） | 用户接电话说"导航回家"被真的导航了 | 多关串联（声源 + 指令式 + 对话目标 + 能力范围），不靠单一信号；电话进行中场景全局抑制 |
| **拒识反馈太重打扰用户**（蔚来 NOMI 模式） | 每次拒识都"嗯？我没听懂"，成了话痨 | **默认静默拒识**，不主动反馈；只在"低成本确认"分支才发声，且音量轻（TTS 30%） |
| **儿童/老人指令被错判**（童音 + 简短指令） | 娃说"小 X 我要听佩奇"被拒识 | 儿童语料覆盖 + 用户分级阈值——儿童身份（I-F1 声纹）下拒识阈值放宽，优先送 NLU |

**Paper Tiger（看起来像风险但不是）**

* "需要 100% 准确判断指令意图"—— 不需要。**95% 静默拒识 + 5% 低成本确认**就够了，确认机制兜住误判
* "不同口音拒识表现差异大"—— 拒识基于语义不基于发音，口音影响在 ASR 层（I-F2）解决

**Elephant（团队可能没讨论够）**

* **"开放对话"和"拒识"的边界**：S3 场景"小 X 你说今晚吃什么好"——闲聊式开放对话我们做还是不做？**做**则是 NOMI 模式（人格层接管），**不做**则用户感觉冷冰冰。**这是文档 11 人格层的核心议题**，本层只负责"识别为开放对话"分类，不决定 NOMI 怎么回
* **拒识日志的隐私边界**：拒识掉的话（电话内容/家庭对话）是否记日志？记多久？用户怎么删除？和 I-F2 隐私合规并案处理，抛给法务

**5.4 I-F4 全双工打断与恢复（与 I-F2 构成全双工体验两翼）**

**5.4.1 一句话定义**

**做什么**：系统在播报 / TTS 输出过程中，用户可以**随时说话打断**，系统立刻停止播报并进入听命状态，**无需再次唤醒**。**与 I-F2 一起构成"全双工体验"的两翼**：

* **I-F2 管"用户说"**：用户说话系统就在识别（不等说完）+ 用户改口能纠错 + 用户对同一对象可持续修改
* **I-F4 管"系统说"**：系统说话用户能打断 + 打断后不重唤醒 + 打断是否中断当前任务由意图智能判断

**没有 I-F4 的打断，全双工体验就断了一半**——系统自顾自把话说完再听，用户体验回到对讲机。这是座舱 AI 必须做对的基础体验，不是 nice-to-have。

**不做什么**：

* 不做"等系统说完再听"的半双工（上一代语音交互）
* 不做"打断后必须重新唤醒"（用户会崩溃）
* 不做"任何声音都打断"（后排娃哭 / 副驾说话不该打断主驾的播报）
* 不做"打断后强制清空上下文"（打断 ≠ 新对话，上下文应保留）

**5.4.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | 系统播报"前方 500 米右转进入..."司机打断："不对，走左边那条" | 播报立刻停止（< 200ms），进入听命态，识别"走左边那条" → 重新规划 |
| S2 | 系统播报长文本消息，司机说"停" | 播报立刻停止，不继续；保留"继续播报"入口（屏幕按钮 or 语音"继续"） |
| S3 | 系统播报天气，后排娃突然哭 | **不打断**——声源方向：后排非主驾，不触发打断；主驾播报继续 |
| S4 | 系统播报导航，副驾对司机说"你刚才那个路口应该..." | **不打断**——对话目标：副驾说话对象是司机不是系统 |
| S5 | 系统播报音乐推荐列表，司机说"第三个" | 播报停止，识别"第三个" → 播放列表第 3 首 |
| S6 | 系统播报复杂路况（长句），司机说"知道了" | 播报停止，系统记录"用户已确认"，不再重复 |

**5.4.3 输入 / 触发条件**

* **输入**：麦克风持续监听（全双工）；声源方向（主驾 vs 副驾 vs 后排）；当前播报状态（TTS 进行中 + 播报内容类型）
* **触发**：系统 TTS 播报进行中 + 检测到主驾说话（VAD + 声源方向）
* **前置条件**：系统处于"播报态"；多音区分离（I-F5）正常工作

**5.4.4 执行流程**

|  |
| --- |
| Plain Text 系统 TTS 播报中  ↓ 麦克风持续监听（全双工）  ↓ 检测到音频输入？  ├─ 是 → 声源方向判断  │ ├─ 主驾 → 打断判断  │ │ ├─ 有效打断（指令/停止词）→ 立刻停止 TTS + 进入听命态  │ │ ├─ 无效打断（咳嗽/叹气）→ 继续播报  │ │ └─ 不确定 → 降低 TTS 音量到 30%，等 1s 判断  │ ├─ 副驾/后排 → 不打断，继续播报  │ └─ 车外/媒体 → 不打断  │  └─ 否 → 继续播报 |

**关键时间预算**：

* 打断响应延迟：用户开始说话 → TTS 停止，**≤ 200ms**
* 打断判断延迟：音频输入 → 判断"是否有效打断"，**≤ 150ms**

**5.4.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **误打断**（咳嗽/叹气/后排说话被当打断） | 播报被频繁打断，用户："我又没说话你停什么？" | 声源方向 + VAD 能量阈值 + 打断词判断三重过滤；咳嗽/叹气（< 0.5s 短音）不触发打断 |
| **打断延迟 > 500ms** | 用户："我都说完了你还在说" | 端侧 VAD < 50ms 响应；TTS 停止指令优先级最高 |
| **打断后丢失上下文**（用户说"不对"系统不知道指什么） | 系统："您说什么不对？" 用户崩溃 | 打断时保留"被打断的播报内容"作为上下文送 NLU；例如播报"右转"被打断说"不对" → NLU 知道"不对"指"右转" |
| **副驾打断主驾播报**（多音区分离失败） | 副驾说话，主驾导航播报被停 | 多音区分离（I-F5）≥ 95% 准确；失败时宁可不打断（保守策略） |

**Paper Tiger（看起来像风险但不是）**

* "打断后用户忘了自己要说什么"—— 用户自己的问题，产品只需保证"打断响应快 + 进入听命态明确"
* "TTS 被打断会有爆音/卡顿"—— TTS 引擎工程问题，不是交互设计问题，抛给沈砚

**Elephant（团队可能没讨论够）**

* **"降低音量 vs 直接停止"策略**：不确定是否有效打断时，降低 TTS 音量到 30% 等 1s 判断，还是直接停？降音量更保守但可能让用户觉得"系统犹豫"，直接停更果断但误打断代价高。**默认降音量，用户可在设置里改成"直接停"**
* **打断后的"继续播报"入口**：屏幕按钮 or 语音"继续"？**两者都要**，语音"继续"优先级更高（驾驶场景手不一定能腾出来）

**5.4.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 打断响应延迟 | ≤ 200ms (P95) | 用户开始说话 → TTS 停止 |
| 打断判断延迟 | ≤ 150ms (P95) | 音频输入 → 判断有效打断 |
| 有效打断识别率 | ≥ 95% | 1000 条标注有效打断测试集 |
| 误打断率 | ≤ 3% | 咳嗽/叹气/后排说话被误判为打断 |
| 多音区打断准确率 | ≥ 95% | 主驾打断 vs 副驾/后排说话区分 |
| 打断后上下文保留率 | ≥ 90% | 打断后 NLU 正确理解"不对/停/继续"指代 |

**5.4.7 自我反驳（六关）**

|  |  |  |
| --- | --- | --- |
| 关 | 自问 | 答 |
| 场景真伪 | 全双工真的是刚需吗？ | 刚需。半双工让用户"等系统说完"是上一代体验，导航纠错/紧急打断场景完全不可接受 |
| 失败路径 | 最糟糕失败？ | 误打断 + 丢失上下文：咳嗽被当打断，播报停了，用户说"继续"系统不知道继续什么 → 砸方向盘。**上下文保留 + 误打断率 < 3% 是底线** |
| 替代方案 | 能不能只在"可打断场景"开全双工？ | 可以但"可打断场景"边界很难定。导航/音乐/天气几乎都该可打断。**全局开 + 打断判断过滤**更简洁 |
| 代价账 | 全双工持续监听的算力代价？ | VAD + 声源方向，端侧 < 3% 持续算力，可接受 |
| 竞品镜像 | 五家都支持全双工，我们凭什么做得更好？ | **差异化：多音区精准打断 + 上下文保留 + 用户可选策略**。华为/蔚来/理想还在"单音区 + 全局打断"层，我们做"多音区 + 智能过滤 + 上下文延续" |
| 撤退条件 | 什么指标不达预期就砍？ | 误打断率 > 10% 或打断响应延迟 > 500ms 连续 4 周，需重新设计打断判断策略 |

**5.4.8 给沈砚的技术钩子**

1. **TTS 停止指令优先级**：打断触发时，TTS 停止能否抢占其他任务？延迟预算？
2. **打断时的音频缓冲**：TTS 停止后，用户打断语音从哪一帧开始送 ASR？丢帧风险？
3. **上下文保留机制**：被打断的播报内容（文本 + 进度）怎么传给 NLU？数据结构？
4. **"降低音量 vs 直接停"策略切换**：用户设置改变时，TTS 引擎动态调整？
5. **多音区打断的声源判断**：I-F5 的声源方向数据在打断层怎么用？硬规则还是软特征？
6. **TTS 爆音/卡顿**：打断时 TTS 引擎的 fade-out 时间预算？< 50ms 能做到吗？

**5.5 I-F5 多音区分离**

**5.5.1 一句话定义**

**做什么**：在车内多人同时说话时，系统能准确识别"谁在说话"（主驾/副驾/后排），并把对应音频流分离出来，送给各自的 ASR + NLU 处理。

**不做什么**：

* 不做"只听主驾"的单音区（副驾/后排体验为零）
* 不做"全车混音"（所有人的话混一起识别，会互相干扰）
* 不做"后排细分到每个座位"（v1 后排当一个音区，v2 再细分）

**5.5.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | **A4 会议车载切换**：主驾说"小 X 导航回公司"，同时副驾说"小 X 把空调调到 24 度" | 两路音频并行识别 → 主驾导航 + 副驾空调同时生效，互不干扰 |
| S2 | **B1 娃哭夜路**：后排娃哭闹（75 dB），主驾说"小 X 导航最近的医院" | 后排哭声被分离（不送 ASR），主驾指令正常识别 → 导航医院 |
| S3 | **副驾接电话**：副驾打电话说"那个文件你发我一下"，主驾说"小 X 放点轻音乐" | 副驾电话内容被分离（不送 ASR），主驾指令正常识别 → 播放音乐 |
| S4 | **后排娃说话**：后排娃说"小 X 我要听佩奇"，主驾正在说"小 X 导航回家" | 主驾导航生效（优先级高），后排娃请求进"待确认队列"（主驾确认后再执行） |
| S5 | **副驾纠正主驾**：主驾说"小 X 导航到北京路"，副驾立刻说"不对，是北海路" | NLU 识别副驾是"纠正"（不是独立指令），合并处理 → 导航北海路 |

**5.5.3 输入 / 触发条件**

* **输入**：全车麦克风阵列（至少 4 麦：主驾 2 + 副驾 1 + 后排 1），16kHz 多通道音频
* **触发**：检测到多个音源同时说话（VAD 能量 + 声源方向）
* **前置条件**：麦阵列正常工作；波束成形算法已校准

**5.5.4 执行流程**

|  |
| --- |
| Plain Text 全车麦阵列采样  ↓ 波束成形 + 声源定位  ├─ 主驾方向 → 主驾音频流  ├─ 副驾方向 → 副驾音频流  └─ 后排方向 → 后排音频流  ↓ 各音频流独立送 ASR  ↓ ASR 输出 + 用户身份（主驾/副驾/后排）→ 送 NLU  ↓ NLU 判断：  ├─ 独立指令 → 并行执行（主驾导航 + 副驾空调）  ├─ 冲突指令 → 主驾优先，副驾/后排进"待确认队列"  └─ 纠正指令 → 合并处理（副驾纠正主驾） |

**关键时间预算**：

* 声源定位延迟：**≤ 50ms**
* 音频分离延迟：**≤ 80ms**（波束成形 + 降噪）
* 总延迟（分离 → ASR 输入）：**≤ 130ms**

**5.5.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **声源定位失败**（主驾/副驾混淆） | 主驾说"导航回家"被当成副驾说的 → 副驾画像的"家"地址 | 麦阵列 ≥ 4 麦；定位准确率 ≥ 95%；失败时用声纹辅助（I-F1 声纹识别） |
| **后排多人说话**（v1 后排当一个音区） | 后排两个娃同时说话，分不清 | v1 后排当一个音区不细分；v2 增加后排麦数量（2→4）细分到座位 |
| **噪声干扰**（车窗开/高速风噪） | 分离质量下降，ASR 识别率跌 | 自适应降噪 + 波束成形；高噪声（> 80 dB）提示用户"请稍后再说"或关窗 |
| **多人同时唤醒**（主驾和副驾同时说"小 X"） | 系统不知道响应谁 | 唤醒仲裁：主驾优先；副驾唤醒后 TTS 明确告知"副驾，我在听" |

**Paper Tiger（看起来像风险但不是）**

* "多音区会让系统响应变慢"—— 不会。分离和 ASR 都是并行的，不增加总延迟
* "副驾/后排用户不习惯多音区"—— 实证：理想/蔚来的多音区用户反馈正面，家庭场景刚需

**Elephant（团队可能没讨论够）**

* **主驾 vs 副驾的权限边界**：副驾能不能改主驾的导航？关主驾的音乐？**这是权限管理问题（文档 12 账号体系），不是多音区问题**。多音区只负责"识别谁说的"，不负责"谁能做什么"
* **后排儿童的"待确认队列"体验**：后排娃说"我要听佩奇"，系统说"主驾确认后播放"，娃等不及哭了怎么办？**需要和交互文档 05 对齐"儿童模式"的即时响应策略**

**5.6 C-F1 DMS 驾驶员状态感知**

**5.6.1 一句话定义**

**做什么**：通过驾驶舱摄像头持续监测驾驶员的**疲劳 / 分心 / 情绪 / 身份**四类状态，输出结构化信号给其他模块（安全干预、个性化切换、主动关怀、UI 调整）使用。

**不做什么**：

* 不做"疲劳 = 停车"的硬接管（这是智驾或 ADAS 的事，DMS 只提供信号）
* 不做人脸识别原始画面落盘（只提结构化状态，原始画面不持久化，隐私边界）
* 不做"给用户打分"类的 gamification（开累了再被系统评分，用户会反感）

**5.6.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | **A2 晚归疲劳**：晚上 10 点，司机连续 3 次打哈欠 + 眨眼频率下降 + 视线偏离前方 | DMS 输出"疲劳 L2"，触发主动关怀：座椅通风/冷风轻吹 + TTS"要不要休息一下？就近的服务区 5 公里" |
| S2 | **A1 早高峰通勤**：司机低头看中控屏超过 2 秒 | DMS 输出"分心 L1"，HUD 闪烁红点提示"请看路面"，不发声打扰 |
| S3 | **B3 全家上车**：爸爸进入驾驶座，DMS 识别身份 → 切换到爸爸画像（座椅/后视镜/音乐/导航偏好） | DMS 输出"身份切换：user\_A"，触发 B-F5 多用户画像切换（置信度 ≥ 90%） |
| S4 | **异常紧急**：司机突发身体不适，头部僵直前倾 + 双手脱离方向盘 + 眼睛闭合 > 3 秒 | DMS 输出"异常紧急 L3"，触发紧急处置：双闪 + 减速 + 呼叫 112（联动 ADAS + 紧急服务） |
| S5 | **长途疲劳 + 情绪低落**：副驾睡着，司机脸部表情疲惫 + 叹气 + 音乐节奏变慢 | DMS + 音频信号融合，输出"疲劳 L1 + 情绪低落"，主动推荐休息点 + TTS"还有 50 公里到服务区" |
| S6 | **儿童驾驶座**（异常）：车开始前 DMS 识别驾驶座是未成年儿童 | DMS 输出"异常身份：儿童"（置信度 ≥ 85%），车辆**禁止启动** + 中控弹窗警告 |

**5.6.3 输入 / 触发条件**

* **输入**：DMS 摄像头（红外 + RGB），30fps，分辨率 1080p+；车载音频（语音信号辅助情绪判断）；方向盘握力传感器（可选）
* **触发**：车辆在 ACC 及以上电源状态；DMS 摄像头正常工作（非遮挡）；用户未关闭 DMS（但安全类监测不可关闭）
* **前置条件**：摄像头校准完成；驾驶员在视野内

**5.6.4 执行流程**

|  |
| --- |
| Plain Text DMS 摄像头 + 音频 + 方向盘信号  ↓ 多模态特征提取（端侧）  ├─ 视觉：面部关键点 + 视线 + 眨眼 + 哈欠 + 头部姿态 + 表情  ├─ 音频：叹气 + 语音能量 + 情绪特征（辅助）  └─ 触觉：方向盘握力 + 脱离时长（辅助）  ↓ 状态分类器（端侧，4 路并行）  ├─ 疲劳分级：L0(清醒) / L1(轻微) / L2(中度) / L3(危险)  ├─ 分心分级：L0(专注) / L1(短暂偏离 < 2s) / L2(持续偏离 > 2s) / L3(长时分心 > 5s)  ├─ 情绪分级：L0(平静) / L1(愉悦) / L2(疲惫/低落) / L3(愤怒/焦虑)  └─ 身份：user\_A / user\_B / ... / unknown / child  ↓ 分级阈值过滤  ├─ 安全类（疲劳 L2+ / 分心 L2+ / 异常紧急 L3）→ 立即输出 + 强干预  ├─ 便利类（身份切换）→ 置信度 ≥ 90% 才输出  └─ 关怀类（情绪 / 疲劳 L1）→ 上下文感知，频次控制  ↓ 输出结构化信号 → 订阅方（安全 / 个性化 / 关怀 / UI） |

**关键时间预算**：

* 状态分类延迟：图像采集 → 分类输出，**≤ 200ms**（30fps 的 6 帧）
* 异常紧急识别延迟：**≤ 500ms**（L3 安全事件）
* 身份切换延迟：**≤ 1s**（含画像切换）

**5.6.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **疲劳误报**（用户只是揉眼睛） | 用户被频繁打扰"你疲劳了要不要休息"，崩溃关闭 DMS | **分级阈值 + 多信号融合**：安全类（疲劳 L2+）用 60% 阈值，便利类（疲劳 L1 推荐）用 80% 阈值；多信号（眨眼 + 哈欠 + 视线偏离）同时满足才触发 |
| **疲劳漏报**（用户已严重疲劳但没干预） | 用户出事故 | 安全类（疲劳 L3）采用**低阈值 + 多模态兜底**：视觉 + 音频 + 方向盘握力任一异常即触发；宁错不漏 |
| **身份识别错误**（爸爸上车识别成妈妈） | 画像切换错，座椅/音乐/导航全错 | 身份切换阈值 ≥ 90%；置信度不足时**不主动切换**，保持当前画像；用户可一键强制切换 |
| **儿童驾驶座未识别**（安全风险） | 儿童启动车辆 → 事故 | 儿童识别阈值 ≥ 85%；识别到儿童身份 → 车辆禁启动；法律/合规必须对齐 |
| **摄像头遮挡**（贴黑膜/灰尘/阳光直射） | DMS 功能失效 | 摄像头状态检测（信号丢失/遮挡/光照异常），主动提示用户清理；严重时**降级到纯音频 + 方向盘信号**的弱版 DMS |

**Paper Tiger（看起来像风险但不是）**

* "DMS 持续拍摄侵犯隐私"—— **原始画面不落盘，只输出结构化状态**；用户可查看/关闭 DMS（安全类除外）。合规边界清楚，不是真风险
* "太阳光照让疲劳识别不准"—— 红外摄像头 + 光照自适应算法，已是成熟工程问题

**Elephant（团队可能没讨论够）**

* **情绪识别的"可接受度"边界**：系统说"我看你心情不好，要不要听点欢快的音乐？"——听起来贴心，但可能让用户感觉"被监视 + 被评判"。**建议：情绪类关怀默认关闭，用户主动开启才上；上线时做 A/B 测试看打开率和关闭率**
* **安全类不可关闭的边界**：疲劳 L3 / 异常紧急 L3 / 儿童识别是否允许用户关闭？**法律合规必须介入**——某些地区（欧盟 GSR 法规）强制要求，国内暂无强制，但我们选择"默认开，不允许关"，对齐未来法规
* **多人驾驶员共享**（夫妻共用一辆车）：身份识别要不要主动询问"是 A 还是 B？"还是自动切？**建议：DMS 识别置信度 ≥ 95% 自动切，80-95% 询问，< 80% 保持当前画像**

**5.7 C-F2 OMS 乘员状态感知**

**5.7.1 一句话定义**

**做什么**：通过车内摄像头（副驾 + 后排）持续监测**副驾和后排乘员的状态**（儿童/成人、睡眠/清醒、情绪、遗留物检测），输出结构化信号给其他模块（儿童模式、空调调节、遗留物提醒、紧急呼叫）使用。

**不做什么**：

* 不做"监控乘员隐私"（原始画面不落盘，只输出结构化状态）
* 不做"乘员行为评分"（不是监控工具，是服务工具）
* 不做"后排细分到每个座位"（v1 后排当一个区域，v2 再细分）

**5.7.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | **B1 娃哭夜路**：后排儿童哭闹 + 挣扎（安全座椅解开风险） | OMS 输出"后排儿童异常"，中控弹窗提示 + TTS"后排宝宝可能不舒服" + 主动降低音乐音量 |
| S2 | **B2 副驾睡着**：副驾乘员闭眼 > 5 分钟 + 头部倾斜 | OMS 输出"副驾睡眠"，空调自动调低风速（避免吹醒）+ 音乐音量降低 20% |
| S3 | **B4 遗留物检测**：下车后，后排座椅检测到儿童或宠物未下车 | OMS 输出"后排遗留物：儿童/宠物"，车门锁定前弹窗警告 + 持续鸣笛 + 推送手机通知 |
| S4 | **B3 全家上车**：后排检测到 2 个儿童 + 1 个成人 | OMS 输出"后排 3 人：2 儿童 + 1 成人"，触发儿童模式：后排娱乐屏自动切换到儿童内容 + 空调后排独立调节 |
| S5 | **副驾情绪低落**：副驾表情疲惫 + 叹气 + 长时间沉默 | OMS + 音频信号融合，输出"副驾情绪低落"，主动推荐舒缓音乐（需副驾确认，不强制） |
| S6 | **后排冲突**：后排两个儿童打闹 + 哭声 | OMS 输出"后排冲突"，TTS 提示主驾"后排宝宝们可能需要关注" + 后排娱乐屏暂停（吸引注意力） |

**5.7.3 输入 / 触发条件**

* **输入**：OMS 摄像头（副驾 1 + 后排 1-2），30fps，分辨率 1080p+；车载音频（哭声/叹气检测）；座椅压力传感器（可选，辅助遗留物检测）
* **触发**：车辆在 ACC 及以上电源状态；OMS 摄像头正常工作；用户未关闭 OMS（但遗留物检测不可关闭）
* **前置条件**：摄像头校准完成；乘员在视野内

**5.7.4 执行流程**

|  |
| --- |
| Plain Text OMS 摄像头 + 音频 + 座椅压力  ↓ 多模态特征提取（端侧）  ├─ 视觉：人数 + 年龄段（儿童/成人）+ 睡眠状态 + 表情 + 姿态  ├─ 音频：哭声 + 叹气 + 打闹声（辅助）  └─ 触觉：座椅压力分布（辅助遗留物检测）  ↓ 状态分类器（端侧，3 路并行）  ├─ 乘员状态：人数 + 年龄段 + 睡眠/清醒 + 情绪  ├─ 异常检测：儿童哭闹 + 冲突 + 安全座椅解开  └─ 遗留物检测：下车后座椅仍有儿童/宠物/物品  ↓ 分级阈值过滤  ├─ 安全类（遗留物：儿童/宠物）→ 置信度 ≥ 80%，立即输出 + 强提醒  ├─ 便利类（睡眠 → 空调调节）→ 置信度 ≥ 85%  └─ 关怀类（情绪 → 音乐推荐）→ 置信度 ≥ 80% + 需用户确认  ↓ 输出结构化信号 → 订阅方（儿童模式 / 空调 / 娱乐 / 安全） |

**关键时间预算**：

* 状态分类延迟：图像采集 → 分类输出，**≤ 300ms**（30fps 的 9 帧）
* 遗留物检测延迟：**≤ 2s**（下车后持续检测）
* 儿童模式切换延迟：**≤ 1s**（检测到儿童 → 后排娱乐屏切换）

**5.7.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **遗留物漏报**（儿童被遗忘在车内） | 儿童中暑/窒息 → 品牌灾难 + 法律责任 | **多模态兜底**：视觉 + 座椅压力 + 车内温度传感器三重检测；遗留物检测**不可关闭**；置信度 ≥ 80% 即触发；宁错不漏 |
| **遗留物误报**（后排放了书包被当儿童） | 用户被频繁误报打扰 → 关闭 OMS | 误报 ≤ 1 次/周/用户；误报后用户可标记"这是物品不是人"，系统学习；连续误报 > 3 次自动降级到"仅座椅压力检测" |

| **副驾睡眠误判**（副驾只是闭目养神） | 空调/音乐被自动调整 → 副驾被打扰 | 睡眠判断需持续 > 5 分钟；调整幅度温和（音量 -20%，风速 -1 档）；副驾可一键恢复 |
| **摄像头遮挡**（后排挂衣服/贴膜） | OMS 功能失效 | 摄像头状态检测，主动提示用户清理；遗留物检测降级到**纯座椅压力 + 温度传感器** |

**Paper Tiger（看起来像风险但不是）**

* "OMS 持续拍摄侵犯乘员隐私"—— **原始画面不落盘，只输出结构化状态**；乘员可查看/关闭 OMS（遗留物检测除外）
* "后排儿童打闹被误判为冲突"—— 打闹和冲突的声学特征不同（打闹有笑声，冲突有哭声），音频 + 视觉融合可区分

**Elephant（团队可能没讨论够）**

* **遗留物检测的法律责任边界**：OMS 漏报导致儿童中暑，责任在车企还是用户？**法务必须介入**——欧美已有判例，国内法律模糊。建议：用户协议明确"OMS 是辅助工具，最终责任在监护人"，但技术上做到**宁错不漏**
* **儿童模式的内容审核**：后排娱乐屏自动切换到儿童内容，内容来源是谁？审核标准是什么？**需要和内容运营团队对齐**
* **副驾情绪识别的"可接受度"**：系统说"副驾心情不好"，主驾听到会尴尬？**建议：情绪类信号只给副驾自己看（副驾屏幕），不告诉主驾**

| Plain Text 麦克风阵列采样  ↓ VAD 区分语音/非语音  ├─ 语音 → 送 ASR（I-F2）  └─ 非语音 → 送声音事件分类器  ↓ 声音事件分类器（端侧，多标签分类）  ├─ 人类声：哭 / 笑 / 咳嗽 / 喊叫 / 叹气  ├─ 车辆声：喇叭 / 引擎异常 / 刹车异常 / 撞击  ├─ 外部声：救护车 / 消防车 / 警车警笛 / 喇叭  └─ 异常声：玻璃破碎 / 金属刮擦 / 爆胎声  ↓ 事件过滤  ├─ 安全类（紧急车辆 / 玻璃破碎 / 撞击）→ 立即触发 + 强提醒  ├─ 关怀类（哭声 / 咳嗽 / 喊叫）→ 置信度 ≥ 80% + OMS 联动确认  └─ 辅助类（后车鸣笛 / 笑声）→ 低优先级提示  ↓ 输出结构化事件 → 订阅方 |

**关键时间预算**：

* 事件识别延迟：**≤ 500ms**（P95）
* 紧急事件响应延迟：**≤ 300ms**（紧急车辆/玻璃破碎）

**5.9.1 一句话定义**

**做什么**：结合**视线方向 / 手势指向 / 上下文对话**，理解用户语言中的代词（"那个"、"这个"、"它"）指代的具体对象，打通"指什么 → 做什么"的认知闭环。

**不做什么**：

* 不做"无语音纯视觉指令"（必须以语音为主，视觉辅助）
* 不做"远距离指代"（只处理车内可见/可交互对象：屏幕/控件/车外 POI）
* 不做"多人并发指代"（v1 只处理主驾，副驾指代 v2 再做）

* **前置条件**：DMS/OMS 正常工作；对话上下文可用

|  |

**Tiger 级风险（真问题）**

| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **指代消歧错误**（"那个"理解错对象） | 用户说"那个别放了"被理解成"那条导航别导了" → 用户崩溃 | 置信度 < 80% 必须确认；错误后一键撤销；错误样本记录用于模型迭代 |

**Paper Tiger（看起来像风险但不是）**

* "视觉 + 语音融合太复杂用户用不起来"—— 实证：蔚来/极氪的指代消歧功能，小白用户只要说"那个"就会自然用，**不需要学习**。关键是不误识别，不是用户不会用
* "指代消歧会大幅增加 NLU 成本"—— 指代消歧是 NLU 的一个子模块，增加约 15% 计算量，不是独立成本

**Elephant（团队可能没讨论够）**

* **指代消歧的营销策略**：蔚来/极氪在营销上大讲"VLM 指代"，但实际用户认知度很低。**我们怎么让用户知道这功能？**——两种思路:①不营销，让用户自然发现（类似 iPhone Siri 的"抬起唤醒"）；②新手教程引导（前三次开车时弹窗"试试说'那个是什么'"）。**建议思路①**，不做营销，口碑自然扩散
* **指代消歧的"失败学习"机制**：用户每次纠正（"不对，我说的是 X"）都是宝贵数据，\*\*这些数据怎么采集、脱敏、训练？隐私边界？\*\*需要数据团队 + 法务介入
* **跨车型的视线校准**：HUD/中控/仪表/后排屏的位置在不同车型不同，视线命中哪块屏要和车型配置对齐。**平台化设计关键**

| S3 | **副驾睡觉主驾调音量**：副驾睡着，主驾不想说话吵醒，手势"音量+"（手掌向上抬） | 识别"音量+"手势 → 音乐音量 +10% + 无 TTS 提示（静默执行） |
| S4 | **导航确认**：导航弹窗"前方拥堵，是否重新规划"，主驾手势"确认"（点头 or 竖大拇指） | 识别"确认"手势 → 执行重规划 + HUD 提示"已重新规划" |
| S5 | **切歌**：主驾不喜欢当前歌曲，手势"下一首"（手掌向右挥） | 识别"下一首"手势 → 切下一首 + 中控屏显示新歌名 |
| S6 | **失败场景**：主驾在打方向盘，手部动作被误识别为手势 | 误触发 → 用户被打断。**需要"驾驶状态过滤"：转弯/变道时禁用手势识别** |

**5.10.3 输入 / 触发条件**

* **输入**：DMS/OMS 摄像头（主驾 + 副驾），30fps，分辨率 1080p+
* **触发**：车辆在 ACC 及以上电源状态；用户未关闭手势识别；**非驾驶关键动作时段**（转弯/变道时禁用）
* **前置条件**：摄像头正常工作；手部在视野内

**5.10.4 执行流程**

|  |
| --- |
| Plain Text DMS/OMS 摄像头采样  ↓ 手部检测 + 骨骼点提取  ├─ 主驾手部位置  └─ 副驾手部位置  ↓ 手势分类器（端侧，8 种基础手势）  ├─ 接听（手掌向上抓取）  ├─ 挂断（手掌向下推）  ├─ 音量+（手掌向上抬）  ├─ 音量-（手掌向下压）  ├─ 下一首（手掌向右挥）  ├─ 上一首（手掌向左挥）  ├─ 确认（竖大拇指 or 点头）  └─ 取消（摇头 or 手掌横切）  ↓ 驾驶状态过滤  ├─ 转弯/变道中 → 禁用手势识别  ├─ 方向盘转角 > 30° → 禁用  └─ 正常驾驶 → 允许  ↓ 置信度判定  ├─ ≥ 85% → 执行  └─ < 85% → 忽略（不误触发）  ↓ 执行 + 反馈（HUD 提示 or 静默执行） |

**关键时间预算**：

* 手势识别延迟：**≤ 300ms**（手势完成 → 分类输出）
* 执行反馈延迟：**≤ 100ms**（HUD 提示显示）

**5.10.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **驾驶动作误触发**（打方向盘被当手势） | 用户在转弯时音乐被切歌 → 分心 + 烦躁 | **驾驶状态过滤**：转弯/变道/方向盘转角 > 30° 时禁用手势识别；误触发率 < 1% |
| **手势识别延迟过高**（> 500ms） | 用户做完手势等半天没反应 → 放弃使用 | 端侧模型优化，延迟 ≤ 300ms；超时无响应不执行 |
| **副驾手势干扰主驾**（副驾伸手被误识别） | 副驾动作触发主驾功能 → 主驾被打断 | 手势识别分主驾/副驾两路，各自独立；副驾手势只控制副驾屏/音区 |

| **手势冲突**（用户自然动作和手势库重叠） | 用户挠头被当"取消"手势 → 误触发 | 手势库设计避开常见自然动作；置信度阈值 ≥ 85% |

**Paper Tiger（看起来像风险但不是）**

* "手势识别会让用户分心"—— 实证：极氪手势用户反馈"比语音更专注"（不需要组织语言）。关键是**不误触发**，不是手势本身分心
* "8 种手势用户记不住"—— 不需要记。高频手势（接听/音量/切歌）用户自然会用，低频手势（确认/取消）有 HUD 提示

**Elephant（团队可能没讨论够）**

* **手势识别的"学习成本"vs"自然发现"**：极氪在新手教程里教手势，但用户完成率 < 30%。**我们怎么让用户知道有手势？**——建议：①不做教程，HUD 在合适时机提示"试试手势接听"；②手势图标常驻 HUD 右下角（可关闭）
* **手势库的"文化差异"**：竖大拇指在中国是"赞"，在中东是侮辱。**出海版本手势库要本地化**
* **手势 vs 语音的"优先级"**：用户同时做手势 + 说话，听谁的？**建议：语音优先**（语音意图更明确），手势作为兜底

**5.11 C-F6 置信度融合与降级**

*待填*

**5.11.1 一句话定义**

**做什么**：当多个输入信号（语音/视觉/手势/上下文/DMS/OMS）同时到达或冲突时，按**置信度加权 + 场景优先级**融合成单一决策，并在信号质量不足时**自动降级到可用模态**。

**不做什么**：

* 不做"所有信号平权"（不同信号在不同场景下权重不同）
* 不做"用户手动选择模态"（系统自动融合，用户无感）
* 不做"信号冲突时弹窗让用户选"（系统内部仲裁，不打扰用户）

**5.11.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | **多信号一致**：用户看着导航说"换个路" + 手指指向导航 | 语音 + 视线 + 手势三路一致 → 置信度 95% → 直接执行重规划 |
| S2 | **信号冲突**：用户说"那个别放了"（语音），但视线看着导航（视觉），手指指向音乐卡片（手势） | 手势置信度最高（90%）+ 语音含糊（60%）→ 仲裁：停止音乐（手势优先）+ TTS 确认"已停止音乐" |
| S3 | **弱网降级**：用户说话，ASR 云端超时（> 3s），端侧 ASR 置信度 75% | 云端超时 → 降级到端侧 ASR → 执行（置信度 ≥ 70% 可用）+ 后台继续等云端结果用于学习 |
| S4 | **DMS 失效降级**：摄像头遮挡，视线信号不可用，用户说"那个调亮" | 视线信号缺失 → 降级到纯语音 + 上下文 → TTS 澄清"您是说屏幕亮度还是车灯？" |
| S5 | **噪音环境降级**：车窗开着，外部噪音 > 80dB，ASR 置信度 < 50% | 噪音过高 → 降级到手势 + 屏幕触控 → HUD 提示"环境嘈杂，试试手势或触屏" |
| S6 | **全模态失效**：摄像头 + 麦克风同时故障 | 所有输入失效 → 降级到纯触控 + 物理按键 → 中控弹窗"语音和手势暂不可用，请使用触屏" |

**5.11.3 输入 / 触发条件**

* **输入**：所有模态的置信度信号（I-F1~I-F7 + C-F1~C-F3）+ 环境状态（噪音/光线/网络）
* **触发**：任意模态输入到达；信号质量变化；环境状态变化
* **前置条件**：至少一个模态可用

**5.11.4 执行流程**

|  |
| --- |
| Plain Text 多模态信号并行到达  ├─ 语音（ASR 置信度）  ├─ 视觉（DMS 视线 + 手势置信度）  ├─ 上下文（对话历史匹配度）  └─ 环境（噪音/光线/网络质量）  ↓ 信号质量评估  ├─ 语音质量：ASR 置信度 + 噪音水平  ├─ 视觉质量：DMS/OMS 可用性 + 光线充足度  ├─ 手势质量：手势置信度 + 驾驶状态过滤  └─ 网络质量：延迟 + 丢包率  ↓ 场景优先级加权（动态权重表）  ├─ 安全类（疲劳/遗留物/紧急车辆）→ 视觉权重 0.7，语音 0.3  ├─ 指代类（"那个"）→ 视觉权重 0.5，语音 0.3，上下文 0.2  ├─ 快捷操作（接听/切歌）→ 手势权重 0.6，语音 0.4  └─ 复杂指令（导航/查询）→ 语音权重 0.7，视觉 0.2，上下文 0.1  ↓ 融合决策  ├─ Top-1 置信度 ≥ 80% → 直接执行  ├─ Top-1 置信度 60-80% → TTS 确认  ├─ Top-1 置信度 < 60% → TTS 澄清 or 降级  └─ 信号冲突（Top-1 和 Top-2 差距 < 10%）→ 按场景优先级仲裁  ↓ 降级策略（信号质量不足时）  ├─ 云端 ASR 超时 → 端侧 ASR  ├─ 视觉失效 → 纯语音 + 上下文  ├─ 语音失效 → 手势 + 触控  └─ 全模态失效 → 纯触控 + 物理按键 |

**关键时间预算**：

* 融合决策延迟：**≤ 100ms**（信号到达 → 融合输出）
* 降级切换延迟：**≤ 200ms**（检测到信号失效 → 降级模态激活）

**5.11.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **融合错误**（多信号冲突时仲裁错） | 用户说"那个别放了"想停音乐，被理解成停导航 → 用户崩溃 | 场景优先级表精细调优；冲突时 Top-1 和 Top-2 差距 < 10% 必须确认；错误样本回流训练 |
| **降级不及时**（云端超时 3s 才降级） | 用户等 3 秒没反应 → 放弃使用 | 云端超时阈值 ≤ 1.5s；端侧 ASR 预加载，降级无感 |
| **降级提示过度**（频繁弹"环境嘈杂"） | 用户被频繁打扰 → 关闭提示 | 降级提示最多 1 次/5 分钟；连续降级 > 3 次不再提示 |
| **权重表过时**（场景优先级不符合用户习惯） | 融合决策不符合用户预期 → 用户不信任 | 权重表可配置 + 用户反馈学习；A/B 测试优化 |
| **全模态失效未兜底**（摄像头 + 麦克风同时坏） | 用户完全无法交互 → 品牌灾难 | 物理按键 + 触控永远可用；全模态失效时中控弹窗明确告知 |

**Paper Tiger（看起来像风险但不是）**

* "多模态融合太复杂用户感知不到"—— 用户不需要感知融合过程，只需要感知"系统理解我了"。融合是内部机制，不是用户功能
* "降级会让用户觉得系统不稳定"—— 实证：华为弱网降级用户无感（端侧 ASR 足够好）。关键是降级无缝，不是不降级

**Elephant（团队可能没讨论够）**

* **场景优先级权重表的"可配置性"**：不同用户对模态的偏好不同（有人喜欢语音，有人喜欢手势）。**权重表要不要支持用户自定义？**——建议：V1 不支持，V2 根据用户行为自动学习调整
* **降级策略的"透明度"**：系统降级到端侧 ASR，要不要告诉用户"当前使用离线模式"？**建议：不告诉**，用户无需知道内部状态
* **多模态融合的"可解释性"**：系统为什么选了手势而不是语音？**调试时需要日志，但用户不需要看到**。产品上不做解释，工程上要有完整日志

**5.12 B-F5 多用户画像切换**

**5.12.1 一句话定义**

**做什么**：识别当前驾驶员/乘员身份，自动切换到对应用户的**画像配置**（座椅/后视镜/空调/音乐/导航偏好/AI 人设），让车成为"懂每个人"的座舱。

**不做什么**：

* 不做"主动询问身份"（DMS 自动识别，用户无感）
* 不做"画像深度个性化到极致"（V1 只覆盖核心 6-8 项配置，避免过度个性化）
* 不做"未识别用户的强制提示"（Guest 模式默认配置即可）

**5.12.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期响应 |
| S1 | **B3 全家上车 - 爸爸**：爸爸进入驾驶座，DMS 识别身份 user\_dad（置信度 92%） | 自动切换：座椅记忆位置 + 后视镜 + 爸爸常听的播客 + 导航历史 + AI 人设（直接型） |
| S2 | **妈妈接孩子**：妈妈进入驾驶座，DMS 识别 user\_mom | 自动切换：座椅 + 空调 22°C + 妈妈喜欢的轻音乐 + 学校接送路线 + AI 人设（温暖型） |
| S3 | **朋友借车**：朋友进入驾驶座，DMS 识别为 unknown（置信度 < 80%） | Guest 模式：默认配置（不切换历史画像）+ TTS"欢迎使用，您是新用户吗？要不要登录账号？" |
| S4 | **快速切换**：爸爸下车，妈妈立刻上车 | DMS 持续识别，无需手动切换；切换延迟 ≤ 1s（座椅/后视镜/音乐立即生效，导航历史 5s 内同步） |
| S5 | **共享车（夫妻共用）**：DMS 识别置信度 88%（80-95% 中间区） | 不主动切换，TTS 询问"您是 [user\_dad]，是吗？"，用户确认后切换 |
| S6 | **儿童误识别为成人**（边界 case）：青少年（高个）被识别为成人 user\_dad | 切换到爸爸画像 → 用户发现错误，一键"这不是我"按钮 → 系统记录纠正样本 + 切回 Guest |

**5.12.3 输入 / 触发条件**

* **输入**：DMS 身份识别结果（置信度 + user\_id）+ 账号体系（已注册用户列表）+ 历史使用习惯
* **触发**：用户进入驾驶座（座椅压力 + DMS 启动）；驾驶员变更（DMS 检测到不同人脸）
* **前置条件**：账号体系已建立；用户已完成首次画像注册（不少于 3 次驾驶）

**5.12.4 执行流程**

|  |
| --- |
| Plain Text 用户进入驾驶座  ↓ DMS 身份识别（人脸 + 声纹辅助）  ├─ 置信度 ≥ 95% → 自动切换  ├─ 置信度 80-95% → 询问确认  ├─ 置信度 < 80% → Guest 模式  └─ 儿童识别 → 禁止启动（C-F1 联动）  ↓ 画像加载（端侧缓存 + 云端同步）  ├─ 即时项（≤ 1s）：座椅 + 后视镜 + 空调 + 音乐  ├─ 快速项（≤ 5s）：导航历史 + 联系人 + 通讯录  └─ 后台项（≤ 30s）：AI 人设 + 长记忆 + 偏好学习  ↓ 切换确认（HUD/中控提示）  ├─ 静默切换（高置信度）：HUD 右下角小图标显示当前用户  └─ 询问切换（中置信度）：TTS"您是 [user\_dad]，是吗？"  ↓ 错误纠正机制  └─ 用户一键"这不是我" → 切回 Guest + 记录纠正样本 |

**关键时间预算**：

* 身份识别延迟：**≤ 500ms**（DMS 输出 user\_id）
* 即时项切换延迟：**≤ 1s**（座椅/后视镜/空调/音乐）
* 快速项切换延迟：**≤ 5s**（导航历史/联系人）

**5.12.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **身份识别错误**（爸爸被识别成妈妈） | 座椅/导航/音乐全错 + 隐私泄露（爸爸看到妈妈的导航历史） | 置信度 ≥ 95% 才自动切；80-95% 询问确认；错误后一键"这不是我"切回 Guest |
| **隐私泄露**（朋友借车看到车主历史） | 用户信任崩溃 | 朋友默认 Guest 模式；用户登录账号才显示个人数据；敏感数据（导航历史/通讯录）需二次确认 |
| **画像同步延迟**（导航历史 30s 才出来） | 用户上车想直接导航却找不到历史地址 | 即时项 ≤ 1s，快速项 ≤ 5s，后台项异步加载；上车前 5s 预加载（DMS 识别身份就开始） |
| **多用户冲突**（夫妻同时认证） | 切换混乱 | 只识别"驾驶座"的人；副驾/后排不影响主驾画像 |
| **新用户注册成本高**（前 3 次驾驶才完成画像） | 用户初期体验差，可能流失 | 默认配置足够好（不依赖个性化）；引导用户主动登录账号加速画像建立 |

**Paper Tiger（看起来像风险但不是）**

* "用户不喜欢被识别"—— 实证：理想/蔚来用户对自动画像切换满意度高（80%+）。**关键是切换无感 + 错误可纠正**，不是不识别
* "新用户没画像不能用"—— Guest 模式 + 默认配置足够好；用户感知不到差距

**Elephant（团队可能没讨论够）**

* **画像数据的"归属权"**：用户卖车时画像数据归谁？**法务必须介入** —— 建议：用户可一键"清除我的所有数据"；卖车前提示用户清理
* **画像的"过度个性化"边界**：是否要做到"系统比用户更懂自己"？**建议：克制**——只做用户明确表达过的偏好（不做猜测式个性化），避免"诡异谷"
* **共享车场景**（夫妻/父子共用）：身份切换频率高（每天 1-2 次）。**画像切换的 UX 不能成为用户负担** —— 静默切换 + HUD 小图标提示，不打扰

**5.12.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 身份识别准确率（多用户切换） | ≥ 90% | 多用户测试样本（含 2-4 用户家庭场景） |
| 身份识别延迟 | ≤ 500ms (P95) | DMS 输出 user\_id |
| 即时项切换延迟 | ≤ 1s (P95) | 座椅/后视镜/空调/音乐 |
| 快速项切换延迟 | ≤ 5s (P95) | 导航历史/联系人 |
| 错误纠正成功率 | ≥ 95% | 用户一键"这不是我" → 切回 Guest |
| 朋友借车隐私保护 | 100% | Guest 模式下不显示任何个人数据 |

**分级阈值**：

* 置信度 ≥ 95%：自动切换
* 80-95%：询问确认
* < 80%：Guest 模式

**6. 自我反驳与边界**

**6.1 "视觉+语音指代"是噱头还是刚需?**

**自反**:只有蔚来/极氪在做,用户认知度极低。如果 V1 花大力气做,用户可能根本不用。**反驳自反**:用户不用 ≠ 不需要,语言学研究表明"代词指代"占日常语言 20%+,只是用户没意识到车上可以这么用。**结论**:V1 做,但低调上线,不做营销,让自然发现的用户先用起来。

**6.2 DMS 置信度 ≥ 80% 才触发,会不会太保守?**

**自反**:保守一点不吃亏,误报一次"你疲劳了"用户就会烦。**但是** 漏报疲劳是安全问题——所以 DMS 在**安全类触发**(疲劳干预)用低阈值 60%,在**便利类触发**(个性化切换)用高阈值 90%。**分级阈值**才是对的。

**6.3 多音区是刚需吗?单音区能不能过?**

**自反**:单音区在"一人开车"场景足够,但 A4 会议车载切换和 B1 娃哭夜路两个场景**强依赖多音区**。单音区会让我们在"家庭"和"商务"两个人群体验打折。**结论**:多音区 V1 必做,双音区起步(主驾/副驾),后排靠单音区兜底。

**6.4 端到端 vs 三段式路线怎么选?**

**自反**:极氪端到端是激进路线,工程复杂度高、失败回滚难、数据需求大。华为三段式是稳路线,模块独立、易调试。**结论留给沈砚**:架构选型是他的决策,我的产品视角判断"V1 稳优先",沈砚你定。
