**§1 背景与目标**

**1.1 为什么做生态接入与跨端**

**核心问题**：座舱不是孤岛——用户的数字生活横跨手机/家居/车,但当前车机生态是"围墙花园",第三方接入浅、账号割裂、跨端任务不连贯。

三大矛盾：

1. **围墙花园**：车机只接了自家生态(如华为只接华为手机),用户被品牌绑架
2. **账号割裂**：手机上登录的 App 和车上是两套账号,数据不互通
3. **任务断裂**：手机上听了一半的播客,上车要重新找;车上设的导航,下车要重新搜

**AI 时代的机会**：

* **标准化接入**：类似 MCP 的标准工具协议,第三方按标准接入即可被 Agent 调用
* **账号融合**：手机扫码/蓝牙自动关联,第三方 App 免登录上车
* **任务延续**：跨端状态同步(手机播客→车上继续,车上导航→手机步行延续)

**目标**：

* **主目标**：核心服务(导航/音乐/支付/智能家居)深度接入率 100%,第三方接入 >30 个 App
* **跨端目标**：手机→车任务延续成功率 >85%
* **安全目标**：支付类操作 100% 需二次确认,第三方数据访问需用户授权

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **无缝上车**：手机上的导航/音乐/播客自动延续到车,不用重新操作
* **一句话触达服务**："订个外卖""查下快递"——不再被限制在预装的几个 App 里
* **车家联动**：快到家时空调自动开,出门时关灯锁门,不用掏手机

**对全车**：

* **账号自动识别**：不同用户的手机蓝牙/扫码上车,自动加载各自的 App 和偏好
* **安全支付**：车载场景下支付流程简化但不降安全,语音+确认双重保障

**2.2 反价值(不做什么)**

* **不做全 App 移植**：车机不是手机——短视频/社交/购物不做完整移植,只做语音可触达的核心功能
* **不做强制账号**：不要求用户必须注册车厂账号才能用第三方服务
* **不做数据变现**：用户第三方服务数据不用于广告推荐或商业变现

**3.2 边界场景**

|  |  |  |

| 场景 | 挑战 | 处理策略 |

**3.3 非场景(明确不做)**

* **完整 App 移植**：不做抖音/微信的完整车载版——只做语音可触达的核心功能
* **广告植入**：第三方服务接入不附带广告位——用户体验优先
* **社交分享**：不做"把你的行车轨迹分享给朋友"——位置数据不离开车

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **跨端延续延迟** | <5s | 手机操作到车机开始执行 | >15s 提示"正在同步" |
| **第三方调用成功率** | >90% | 工具调用成功/总调用 | <75% 标记服务降级 |
| **账号关联成功率** | >95% | 扫码/蓝牙到账号加载 | <85% 降级为手动登录 |

**4.2 可靠性与降级**

1. **第三方服务宕机**：提示"该服务暂不可用"+ 不影响其他服务
2. **账号同步失败**：降级为离线模式+手动登录
3. **车家互联断连**：本地缓存最后指令+重连后自动同步

**4.3 安全与合规**

* **支付沙箱**：支付操作在独立沙箱中执行,不可被其他 Agent 工具读取
* **数据最小化**：第三方服务只获取完成任务所需的最少数据
* **授权可撤**：用户可随时撤销任意第三方服务的车载访问权限

**4.4 关键取舍**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 取舍点 | 选项 A | 选项 B | 决策 | 理由 |
| **接入模式** | 深度定制(体验好) | 标准协议(扩展快) | **核心深度+长尾标准** | 核心服务深度定制保体验,长尾标准化保扩展 |
| **账号体系** | 车厂自有账号 | 直接用第三方账号 | **双轨:车厂账号+第三方直登** | 不强制用户注册,但自有账号解锁更多功能 |
| **支付方式** | 车载独立支付 | 复用手机支付 | **复用手机支付(扫码/蓝牙)** | 不让用户在车上输密码,复用手机已认证身份 |
| **生态开放度** | 审核制(严) | 开放注册(宽) | **审核制** | 车载场景安全性要求高于手机 |

**3. 核心能力要求**

**3.1 核心 App 深度接入(5-8 个)**

**V1 清单**:

1. **高德地图** — 导航/路况/POI
2. **酷狗/网易云/QQ 音乐** — 音乐播放(三选一或并存)
3. **特来电/星星充电/小桔充电** — 充电桩(接 3 家)
4. **滴滴出行** — 打车(主驾临时打车需求)
5. **美团外卖** — 路上点餐
6. **小红书** — 攻略内容
7. **ETCP 停车** — 停车场预约
8. **飞书日历/微信日历** — 日程

**深度要求**:

* 每个 App 支持 5-10 个核心指令("导航去最近不堵的火锅店"这种复杂指令)
* 结果展示在车机卡片,不跳转 App UI
* 语音 + 触控两种交互

**3.2 第三方 App 标准接入协议**

1. **协议选择** — 类 MCP(Model Context Protocol)或 Function Calling 标准
2. **开放接入** — 长尾 App(不在核心 5-8 里)可以自主接入
3. **审核机制** — 接入前有安全/性能/隐私审核
4. **沙箱执行** — 第三方 App 在隔离环境运行,不影响主系统

**3.3 支付多层安全**

1. **免密支付** — 小额(< 200 元)+ 白名单商家 + 声纹识别
2. **扫码支付** — 中额(200-500 元),用户手机扫车机码
3. **人脸+声纹双验证** — 大额(> 500 元)或非白名单
4. **撤销机制** — 所有支付 10 秒内可撤销
5. **流水可查** — 车机支付流水全部可追溯

**3.4 车家互联**

1. **场景联动** — 下班到家 → 家里空调/灯光预热
2. **设备控制** — 通过车机控制家里设备(Power Home 类)
3. **状态同步** — 家里设备状态同步到车机
4. **V1 边界** — V1 先做"下班到家预热"场景,其他 V1.5

**3.5 跨端任务延续**

1. **手机 → 车** — 手机上查了酒店,上车后主动提示
2. **车 → 手机** — 车上未完成任务,下车后推到手机
3. **车 → 家** — 到家继续听播客/看视频
4. **双向同步** — 状态跨端一致

**3.6 账号体系**

1. **统一账号** — 一个账号贯穿手机/车/家
2. **家庭账号** — 多人共享车,每人独立画像但共享部分偏好
3. **生物特征绑定** — 声纹/人脸与账号绑定,用于支付/隐私操作
4. **账号退出** — 退出时清理本车会话/临时缓存,保留上云记忆

**5.1 E-F1 核心 App 深度接入**

**5.1.1 用户价值**

**做什么**：把 V1 清单里的 8 个核心 App（高德 / 音乐三选一 / 三家充电桩 / 滴滴 / 美团外卖 / 小红书 / ETCP / 日历）**接到座舱原生能力层**，每个 App 至少 5-10 个**复杂指令**（不是简单的"打开 xx"）都能通过语音 + 触控完成，结果**呈现在车机卡片里**（不跳回 App 原生 UI 打断驾驶）。

**不做什么**：

* 不做"浅接"的 20+ App（每个只能"打开 / 关闭"——比不过小鹏的深度 + 比不过蔚来的品牌）
* 不做"把手机 App 原样搬上车"（UI 不适配驾驶场景 / 内容太密 / 操作路径深）
* 不做"非驾驶安全可用"的深度操作（小红书刷视频 / 美团浏览商品这种驾驶中危险，要在驻车才启用）
* 不做"只接 SDK 不做产品化"（上线了但没人会用 = 白做）

**核心价值主张**：

* **深度 > 数量**——每个 App 至少 5-10 个"真实使用场景下最频繁的复杂指令"原生化
* **卡片化呈现**——结果在车机卡片里显示，不跳转 App 原生 UI，减少驾驶分心
* **语音 + 触控双通道**——驾驶中用语音，驻车用触控，不强制单一交互

| 复杂指令成功率 | ≥ 85% | 用户 TOP 100 复杂指令的首次成功率 | 首次 = 第一次说就成功，不追问不重说 |
| 结果呈现时长（首条卡片） | ≤ 2s | 从指令识别到卡片显示 | 包含 API 调用 + 渲染 |
| 跨 App 组合成功率 | ≥ 70% | 两个 App 串联完成的比例 | V1 目标；V2 提升到 85% |
| 驾驶中卡片停留时长 | ≤ 5s（可视），≤ 2 个选项 | 代码审计 + QA 测试 | 硬约束，驾驶安全 |
| 用户"深度 App"满意度 | ≥ 4.2/5 | 月度调研 | 比竞品小鹏的"广度"营销更能让用户感知 |

**5.1.4 放弃条件**

|  |  |  |
| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 某个 App 复杂指令成功率持续 < 60% | 连续 2 周 | 该 App 降级为"浅接"（只支持打开 / 简单查询），不计入 V1 核心清单 |

| 跨 App 组合成功率 < 50% | V1 上线 3 个月 | 关闭跨 App 组合功能，让用户自己串联（降级体验但不出错） |

**5.1.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| API 超时 / 失败 | 网络波动 / 第三方服务宕机 | 用户说了指令但等半天没结果 | 2s 超时降级（规则）；超时后 TTS："网络有点慢，先给你看上次的结果" |
| 账号未绑定 | 用户没在车机登录对应 App | 指令触发但返回"请先登录" | 上车时自动识别已绑定账号；未绑定时给出"手机扫码登录"入口（30 秒内完成） |
| 结果与预期不符 | 语义理解偏差（"最近不堵" vs "距离最近"） | 用户说"最近的" AI 理解成"距离最近"但用户想"时间最近" | 卡片标注排序依据（"距离 / 时间 / 评分"）；用户可语音切换排序"按距离" |
| App 深度能力临时降级 | 第三方 API 改版 / 接口不稳 | 之前能用的指令突然不能用了 | 每个复杂指令有"浅接回退"版本（最起码能"打开 App"）；埋点监控成功率 |
| 驻车后仍限制 | 驾驶态识别错误 | 用户停车后点外卖仍被拒 | 驾驶态判断接 OBD（车速 0 + 挡位 P）；5s 内切换到驻车态 |
| 跨 App 组合中第 1 步失败 | 第 1 个 App 失败，第 2 个无法触发 | 用户等了很久什么都没发生 | 每一步失败都显式反馈（"没找到会议室，是否继续找餐厅？"），不静默失败 |
| 用户指令触发多个 App | 如"点餐 + 看小红书" 模糊归属 | AI 不知道调哪个 | 指令歧义时显式问"你是想点外卖还是看美食攻略"；不猜 |

**5.2 E-F2 第三方 App 标准接入协议**

**5.2.1 用户价值**

**做什么**：**不接的 App（长尾）也能用**。给长尾 App（不在核心 8 个里的）提供**一套标准接入协议**，第三方开发者按协议接入后就能被座舱 AI 调用。用户不需要知道这个 App 有没有被核心清单收录，只要装了就能用——核心 App 做深度，长尾 App 做广度，两条腿走路。

**不做什么**：

* 不做"闭门造车"的私有协议（开发者不愿接入 → 生态空转）
* 不做"零审核"的全开放接入（出事会影响整个座舱品牌）
* 不做"高成本"的接入（第三方开发成本 > 预期收益 → 没人愿意来）
* 不做"接入后不管"（接入后出问题不处理 → 一颗老鼠屎坏一锅汤）

**核心价值主张**：

* **类 MCP 标准**——向行业事实标准靠拢（MCP / Function Calling），降低开发者适配成本
* **四关审核**：功能 / 安全 / 性能 / 隐私——接入前过，接入后动态下架
* **沙箱执行**——第三方 App 跑在隔离环境，坏了只坏自己，不影响主系统

**5.2.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |

| S1 | **长尾 App 被调用**：用户说"用哈啰骑行看看附近有没有车"（哈啰不在核心 8 个） | 识别到指令意图 → 查"第三方接入库" → 哈啰已接入且通过审核 → 调用 | 调用时 UI 显示"由第三方 App 哈啰提供" |
| S2 | **未接入 App 的处理**：用户说"查一下拼多多的这单到哪了"（拼多多未接入） | AI 明确告知："这个 App 还没接入，我给你展示拼多多的打开入口" + 可触达方式 | 不假装能用但其实不能；明确告知未接入 |
| S3 | **接入前审核**：某充电桩厂商想接入（非核心 3 家之外的新厂） | 提交接入申请 → 功能评审（是否有价值）→ 安全审计（代码扫描 / 接口安全）→ 性能测试（响应时间 / 资源占用）→ 隐私审核（数据收集范围 / 合规）→ 四关通过上架 | 审核周期 ≤ 30 天；全程开发者可见进度 |

| S8 | **接入文档与 SDK**：开发者文档公开、SDK 可下载、Sandbox 测试环境可用 | 开放文档 + 示例代码 + 调试工具 + 测试沙箱 | 开发者体验 ≤ 7 天完成从了解到提交接入 |
| S9 | **付费 / 免费模式**：基础接入免费，但高级能力（如车控 / 支付）需商业合作 | 基础层（信息查询 / 数据读取）= 免费；高级层（车控 / 支付 / 数据写入）= 商业合作 | 付费层不在 V1 开放；V1 先打通免费生态 |

**5.3 E-F4 多层支付安全**

**5.3.1 用户价值**

**做什么**：座舱支付不是只有"输密码 / 扫码"两种。按**金额分级 + 场景白名单**做三层安全策略——**小额（< 200 元）免密 + 声纹识别 / 中额（200-500 元）手机扫码 / 大额（> 500 元）人脸+声纹双验证**——加上**所有支付 10 秒撤销**。目标是"**安全对用户无感**"：90% 场景下用户不会遇到额外确认步骤。

**不做什么**：

* 不做"一刀切输密码"（每次都输 → 驾驶中不安全 + 用户抱怨）
* 不做"完全免密"（无门槛支付 → 车载场景风险大，被盗刷不知道）
* 不做"支付失败就卡死"（支付通道故障要有备选路径）
* 不做"支付不可追溯"（出问题没法查 → 合规风险）

**核心价值主张**：

* **金额分级**——小 / 中 / 大三档不同策略
* **声纹活体**——小额场景的主要身份验证手段（"这是本人"）
* **10 秒撤销**——所有支付完成后 10 秒内可撤销，降低"误付"焦虑

**5.3.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 支付金额 / 类型 | 安全流程 | 关键约束 |
| S1 | **小额免密 - 充电**：司机充电 50 元 | 小额 + 白名单（特来电） | 声纹识别通过 → 免密支付 → 车机提示"已付 50 元，10 秒内可撤销" | 声纹识别耗时 ≤ 1s；白名单商家列表由平台维护 |
| S2 | **中额扫码 - 停车**：停车场 300 元 | 中额 | 车机生成二维码（含支付金额） → 用户手机扫码 → 手机支付完成 → 车机显示结果 | 车机二维码有效期 60s；扫码后在手机上最终确认 |
| S3 | **大额双验 - 加油**：加油 800 元 | 大额 | 人脸识别 + 声纹识别 → 双通过后支付 | 两个验证同时进行（并行，不是串行）；任一失败退回扫码支付 |
| S4 | **10 秒撤销**：小额充电 50 元已付，9 秒后用户说"取消刚才的付款" | 撤销窗口内 | TTS："已取消充电 50 元" + 实际发起退款 | 撤销窗口硬性 10 秒；超过 10 秒走正常售后流程（非 AI 秒退） |
| S5 | **声纹失败降级**：司机感冒声音沙哑，声纹识别失败 | 小额但声纹失败 | 自动降级到扫码支付；同时记录"声纹失败"事件 | 降级不提示用户声纹失败原因（避免暴露），只说"请扫码完成支付" |
| S6 | **非本人使用**：司机借车给朋友，朋友在车上想点外卖 | 任意金额 | 声纹不匹配 → 走扫码支付；账号不切换不允许免密 | 任何非本人支付强制走扫码 |
| S7 | **支付通道故障**：微信支付网关临时不可用 | 任意金额 | 降级到支付宝 / 云闪付；降级不给用户增加步骤 | 支付通道多备份；降级路径自动选择 |
| S8 | **隐式异常检测**：同一时段连续 3 次大额支付（可能被盗用） | 大额 + 异常模式 | 强制要求"人脸 + 声纹 + 手机验证码"三重验证 | 异常模式定义由风控团队维护（频次 / 地域 / 金额跳变等） |
| S9 | **撤销后支付记录保留**：10 秒内撤销的支付 | 撤销成功 | 流水里标记"已撤销"，不是删除记录 | 合规要求所有流水可追溯 |
| S10 | **驾驶中禁用某些支付**：驾驶中用户想大额转账 | 大额 + 驾驶中 | 禁用；TTS："驾驶中不建议做大额支付，到了再说" | 驾驶中禁用大额 / 风险操作；小额便利支付仍可用 |

|  |  |  |  |
| --- | --- | --- | --- |

**5.3.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 声纹识别误通过（非本人成功） | 模型误判 / 声纹相似（家人间） | 盗刷风险 | 声纹 + 金额阈值双重门槛；白名单商家 + 金额上限 |
| 声纹识别误拒绝（本人失败） | 感冒 / 环境噪音 | 用户无法支付 | 降级到扫码支付；多次失败后提示"今天嗓子不舒服吗，可以扫码" |
| 人脸识别光线差导致失败 | 夜晚 / 逆光 / 墨镜 | 大额支付卡住 | 车内补光灯（已有）；失败降级到扫码 |
| 车机生成二维码扫不上 | 屏幕反光 / 二维码模糊 | 中额支付卡住 | 提高二维码亮度对比度；备选"手机输入金额确认"路径 |
| 驾驶中支付被误触发 | 语音识别错误 / 误操作 | 司机分心 | 驾驶中支付前双重确认（先说"付 xx 元吗"再声纹）；大额直接禁用 |
| 支付撤销技术失败 | 后端退款接口失败 | 用户以为撤销了但钱没回 | 撤销失败立即告知 + 转人工客服；不静默失败 |
| 多账户混淆 | 家人共用账号 | 妻子支付但算到丈夫 | 账号 + 声纹双重绑定；每笔支付记录都绑定"由谁触发" |
| 第三方支付方要求签约合规 | 商家侧要求额外合规流程 | 上线延期 | 合规 / 风控团队提前介入；分阶段上线 |

**5.3.6 技术钩子（给沈砚）**

1. **声纹活体算法**：车载环境下（有 NVH 噪声 / 多人对话）的声纹识别准确率。用端侧模型还是云端？离线可用性？
2. **人脸活体算法**：白天 / 夜晚 / 墨镜 / 口罩等条件下的准确率。红外 + 可见光双模？车载算力支持？
3. **金额分级策略配置**：小额 / 中额 / 大额阈值（200 / 500 元）、白名单商家列表——这些配置存储位置、更新频率、在线下发机制？
4. **支付通道多备份**：微信 / 支付宝 / 云闪付 / 数字人民币——多个通道的接入 + 降级逻辑。用户账户绑定策略？
5. **10 秒撤销实现**：撤销窗口期内的支付状态（已支付？pending？）。用户申请撤销后的退款通道延迟？失败回退？
6. **风控系统接入**：异常模式检测（频次 / 金额跳变 / 地域异常）。实时风控 vs 事后审计？
7. **支付二维码的生成与安全**：车机 → 手机的扫码流程。二维码内容（含哪些字段、如何防伪）？过期时间？
8. **驾驶中支付禁用**：禁用清单（大额 / 非白名单 / 风险场景）。清单在哪定义？生效延迟？
9. **合规与审计**：所有支付流水的存储（保存多久？加密？本地化合规）？审计查询接口？
10. **账号与支付的绑定**：车机账号 / 车辆 VIN / 用户支付账号 的多维绑定关系。家庭共享车时不同用户的支付账号如何识别？

**5.4 E-F6 跨端任务延续**

**5.4.1 用户价值**

**做什么**：让**手机、车、家**三个场景之间**任务不中断**——手机上查了酒店上车后主动提示 / 车上听到一半的播客到家接着在音箱播 / 车上未完成的导航下车推到手机步行导航。用户不需要手动复制粘贴，也不需要记"我刚才在手机上做什么"，AI 自动感知上下文并无缝切换。

**不做什么**：

* 不做"强推式"同步（用户明明换设备就是想切换，你还追问"要不要继续"）
* 不做"实时推送"的高成本方案（iOS 后台受限严、电量敏感）
* 不做"不设置就不能用"（需要跨端同步功能必须先复杂配置 = 用不起来）
* 不做"跨品牌同步"的承诺（V1 只保障自己生态内跨端，苹果 / 米家 / 华为生态各自封闭）

**核心价值主张**：

* **上车主动拉取**——用户上车时 AI 主动查"手机上你最近在做什么"，不依赖实时推送
* **场景锚定**——同步场景聚焦"导航 / 音乐 / 视频 / 日程 / 购物清单"五类高频场景
* **用户可选触发**——"要不要继续看手机上那部剧"，用户说"要"才继续，不强推

**5.4.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 任务延续方向 | 交互形式 | 关键约束 |
| S1 | **手机 → 车 - 导航**：用户手机上查了"公司附近停车场"，10 分钟后上车 | 手机 → 车 | 上车 3 秒内 TTS："你在手机上查了公司附近停车场，要导航过去吗？" | 时间窗口 ≤ 30 分钟；超过则不提示（避免旧上下文） |
| S2 | **手机 → 车 - 音乐**：用户在手机上听播客，播到 20 分钟 | 手机 → 车 | 上车后车机卡片："继续听 xx 播客吗（已播 20 分钟）" | 只提示不自动播（避免打扰副驾 / 他人） |
| S3 | **车 → 手机 - 导航**：用户开到停车场停车，之后要步行 500 米去餐厅 | 车 → 手机 | 停车熄火后手机推送："这里距离目的地 500 米，要步行导航吗？" | 推送延迟 ≤ 30s；用户 5 分钟内没点则撤回推送 |
| S4 | **车 → 家 - 播客**：开车回家听播客到 25 分钟，到家下车 | 车 → 家 | 进家门 5 分钟内家里音箱询问："继续听 xx 播客吗？从 25 分钟开始" | 需要家里音箱账号打通；仅在家里设备有播放能力时触发 |
| S5 | **手机 → 车 - 日程**：手机上日历有"今天 15:00 会议"，当前 14:20 上车 | 手机 → 车 | 上车时 TTS："你 40 分钟后有会议在 A 栋，现在导航过去大约 25 分钟" + 卡片显示日程详情 | 会议时间距离当前 ≥ 10 分钟 & ≤ 4 小时才提示 |
| S6 | **手机 → 车 - 购物清单**：手机上记了"晚上买菜"，下班路过超市附近 | 手机 → 车 | 接近超市时 TTS："前面有超市，你今早记了要买菜，要不要顺路？" | 需要用户授权"从备忘录/便签推断" |
| S7 | **自然切换无打扰**：用户手机上在导航，上车后自动切到车机导航（不询问） | 手机 → 车 | 无声切换；车机自动接管导航任务 | 同一任务强切换；不同任务（手机听歌 vs 车机导航）并存，不互相打扰 |
| S8 | **跨端冲突解决**：用户在手机和车机同时听音乐 | 冲突 | 以最近的操作为准；先播放的方暂停 | 冲突检测 + 时间戳判断；不让用户手动解决 |
| S9 | **跨端数据不一致**：手机上新增日程但还没同步到车机 | 同步延迟 | 上车时主动拉取最新日程；显示"正在更新日程"微提示 | 拉取延迟 ≤ 3s；失败降级到上次缓存 |
| S10 | **用户拒绝跨端**：用户说"别跟我的手机同步" | 开关 | 设置里一键关闭跨端延续；关闭后所有五类场景都不同步 | 设置颗粒度：五类场景可独立开关 |

**5.4.3 成功指标**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方式 | 备注 |
| 跨端延续场景覆盖率 | 5 类 100%（导航 / 音乐 / 视频 / 日程 / 购物清单） | 功能清单核对 | V1 硬指标 |
| 上车 30 秒内跨端任务识别率 | ≥ 80% | 识别的跨端任务 ÷ 实际有的 | 手机 → 车同步时效性 |

| 跨端同步延迟（手机 → 车） | ≤ 3s | 拉取时的同步延迟 | 主动拉取而非实时推送 |
| 用户手动关闭跨端延续的比例 | ≤ 20% | 设置里关闭开关的用户 | 过高说明用户觉得烦 |
| 跨端冲突检测准确率 | ≥ 95% | 冲突被正确识别并解决 | 重要体验细节 |

|  |  |  |
| --- | --- | --- |
| 条件 | 阈值 | 后果 |
| 跨端任务识别率 < 50% | 连续 2 周 | 场景覆盖收窄（5 类 → 3 类最稳） |
| 用户采纳比例 < 30% | 连续 1 个月 | 跨端提示时机或内容设计失败，重新调研 |
| iOS 后台限制导致跨端功能不可用 | 上线后验证 | iOS 侧只做"上车拉取"不做"实时推送"；降级体验 |
| 用户关闭跨端延续比例 > 40% | 月度 | 跨端默认关闭，改为 opt-in |
| 跨端同步导致隐私泄露事件 | 任意一次 | P0 事件；全面审查数据流向 |

**5.4.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| iOS 后台限制 | iOS 不允许长连接 / 后台任务 | 手机 → 车同步延迟大或失败 | 上车主动拉取，不依赖 iOS 推送；Android 用推送，差异化设计 |
| 跨端提示太多 | 每次上车都弹一堆提示 | 用户烦 | 聚合为一条（"你有 3 个待办：..."）；优先级排序 |
| 用户手机没装座舱 App | 同步依赖 App | 功能不可用 | 降级到 iCloud / Google Calendar 公共接口；明确告知需要 App |
| 数据同步冲突 | 手机和车机同时改同一任务 | 状态不一致 | 以最后操作为准；冲突时保留历史版本供用户选择 |
| 隐私边界模糊 | 同步了用户不想同步的内容（如私人日历） | 用户觉得被侵犯 | 同步内容分类 + 独立开关（日历 / 消息 / 地图 / 备忘录） |
| 账号切换混乱 | 车机切账号后旧账号数据还在 | 跨用户数据泄露 | 账号切换时立刻清理跨端缓存；不保留旧账号任何状态 |
| 跨生态不兼容 | 用户的手机是小米（不是我们生态） | 跨端功能体验打折 | V1 仅保障自有账号生态；第三方账号（微信 / Apple ID）提供基础同步 |
| 用户觉得"AI 偷看手机" | 上车就说"你手机上刚看了 xx"暴露过多 | 监视感 | 提示语言保留适度（"你刚在手机上做了 xx"）；不暴露太多细节（如具体浏览记录） |

**5.4.6 技术钩子（给沈砚）**

1. **iOS 后台限制的技术方案**：iOS 不允许长连接。用什么机制（APNs 推送 / 远程拉取 / VoIP push）？合规风险？
2. **Android 跨端技术**：Android 更开放，实时推送可行。技术选型（WebSocket / MQTT / FCM）？电量敏感？
3. **上车拉取机制**：上车识别后（蓝牙 / NFC / 账号识别）主动拉取。拉取接口的延迟 SLO / 数据量控制？
4. **数据同步边界**：同步哪些数据、不同步哪些数据。配置在哪维护？如何让用户透明？
5. **冲突解决策略**：多设备同时修改同一任务（如日历事件）。冲突检测 + 合并策略？
6. **跨端账号打通**：手机 App / 车机 / 家里音箱的账号统一。SSO 机制？
7. **生态互通**：小米 / 苹果 / 华为等第三方生态的有限互通。接入他们的公共 API（iCloud / Google Calendar）？
8. **隐私数据流**：跨端同步的数据流动（手机 → 云 → 车 还是 手机 → 车 直连）？加密 / 合规？
9. **增量 vs 全量同步**：节省流量和电量，增量同步机制？
10. **用户开关的颗粒度**：跨端 5 大类场景独立开关 + 总开关。UI 设计 + 后端配置如何解耦？

**5.5 G-F1 车家互联**

**5.5.1 用户价值**

**做什么**：让车成为家的延伸——**下班离公司一定距离时，家里空调自动预热到合适温度 / 临近到家时家里灯光提前打开 / 到家熄火时家里背景音乐播起来**。这不是"车控制家"那种炫技，而是**让用户从车到家的过渡变得自然舒适**。V1 只做"下班到家预热"这一个场景，做深做对，再扩展。

**不做什么**：

* 不做"车控所有家电"（太广 / 厂商封闭 / 用户也不需要）
* 不做"家控车"（除非紧急情况，不让家里设备控车）
* 不做"非智能家庭场景"（没有智能家居的用户不强推）
* 不做"全平台支持"（V1 只做 1-2 个主流生态：米家 / 鸿蒙智家 / Apple HomeKit）

**核心价值主张**：

* **聚焦一个场景做透**——下班到家预热（学蔚来 Power Home + 理想家庭场景）
* **支持主流协议**——Matter（行业标准）+ 米家 / 鸿蒙智家 / HomeKit（用户基础）
* **用户可拒绝可调整**——所有联动场景都可关闭 / 微调（温度 / 触发时机 / 设备清单）

**5.5.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 触发条件 | 联动动作 | 关键约束 |
| S1 | **下班到家 - 空调预热**：晚上 19:00 用户从公司出发回家，距离家 5 公里 | 导航目的地 = 家 + 距离 ≤ 5 公里 + 室内温度低于 20°C | 家里中央空调启动 → 预热到用户偏好 22°C；客厅 / 卧室独立设备 | 仅夏冬两季触发；春秋温度适宜不预热 |
| S2 | **临近到家 - 灯光准备**：距离家 1 公里 | 距离 ≤ 1 公里 | 客厅主灯 + 走廊灯打开（暖色系，30% 亮度） | 仅夜间（日落后）触发 |
| S3 | **熄火到家 - 音乐继续**：到家停车熄火 | 进入家庭 WiFi 范围 + 用户在车上正在听音乐 | 家里音箱接续播放（从车上断点处继续） | 仅在用户此刻车上有音乐播放时触发；无音乐则不触发 |
| S4 | **进门 - 状态确认**：用户进入家门 | 蓝牙 / NFC / 智能门锁触发 | 车端取消"到家联动" 任务（避免离家半小时又返回造成误触发） | 进门信号需可靠；不可靠则用时间窗口（30 分钟） |
| S5 | **用户在路上改主意**：开了一半改导航不去家 | 导航目的地变更 | 立刻取消所有家庭联动任务；家里设备恢复原状 | 任务取消延迟 ≤ 10s |
| S6 | **家庭成员冲突**：妻子已在家开了空调 25°C，丈夫开车回家偏好 22°C | 检测到家里已有人 / 已有空调状态 | 不强行覆盖；TTS：" 家里已经在制冷，要保持现在温度还是按你习惯的 22°C？" | 多人冲突时不自动决策，由用户选 |
| S7 | **设备离线 - 优雅失败**：家里空调离线（断电 / 故障） | 联动指令发送但失败 | TTS（仅一句）："家里空调没响应，要不要打个电话给家人确认" | 失败不重试无限次；3 次失败后停止该次联动 |
| S8 | **用户调整阈值**：用户嫌"5 公里太早预热" | 设置入口 | 用户可调："3 公里 / 5 公里 / 8 公里" | 设置入口在"主动服务 - 车家互联" |
| S9 | **隐私敏感**：用户不想 AI 知道"什么时候到家" | 关闭开关 | 关闭后不触发任何车家联动 | 默认开启，用户可一键关闭 |
| S10 | **节能场景**：夏天用户希望"提前 5 分钟开空调即可" | 节能模式开关 | 开启后预热触发条件改为"距离 ≤ 1 公里"（≈ 5 分钟内到达） | 节能模式 = 牺牲一点舒适换电费 |

**5.5.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 设备离线 / 协议失败 | 家里设备宕机 / 网络波动 | 联动失败但用户不知道 | 失败后简短提示（不啰嗦）；用户可查"今天没联动到的设备" |
| 用户改导航后未取消联动 | 取消机制延迟 | 用户去了别的地方但家里空调白开 | 取消延迟 ≤ 10s 强约束；后台监控异常情况 |
| 多生态不互通 | 用户家是混合生态（米家 + 鸿蒙） | 部分设备能联动部分不能 | V1 优先支持单一主流生态；V2 才考虑多生态融合 |
| 误触发预热 | 经过家附近但不回家（去隔壁邻居） | 家里空调白开了 | 触发条件加"导航目的地 = 家"双重判断；不只是"距离" |
| 多人冲突场景处理差 | 家里已有人但 AI 不知道 | AI 强行覆盖造成冲突 | 设备状态查询前置；冲突时不自动决策 |
| 隐私担忧 | 用户觉得"AI 知道我什么时候到家" | 信任流失 | 隐私开关明显；车家互联关闭后不收集任何位置数据 |
| 联动延迟过大 | 云端服务慢 | 到家时家里还没预热完 | 触发条件提前（距离 5 公里 ≈ 7 分钟）；缓冲时间足够 |
| 设置入口找不到 | UI 设计问题 | 用户想调阈值找不到 | 一级菜单"主动服务 - 车家互联"明显入口 |
| 跨厂商协议变更 | 米家 API 改版 | 老的设备不工作 | 协议适配层；监控接口变化主动迁移 |
| 用户卖车后数据残留 | 账号 / 设备绑定不清理 | 新车主能控制旧车主家里设备 | 卖车 / 注销时强制清理所有绑定；二手车流转流程 |

**5.5.6 技术钩子（给沈砚）**

1. **协议选型**：Matter / 米家 / 鸿蒙智家 / HomeKit。每个的优劣 / 接入成本 / 用户覆盖率？V1 选 1-2 个？
2. **位置触发机制**：手机 / 车机的 GPS + 地理围栏。功耗影响？精度？多设备时哪个为准？
3. **设备状态查询**：联动前先查家里设备状态（避免冲突）。同步 vs 异步查询？延迟？
4. **指令分发**：车端 → 云端 → 家里设备。链路上每个节点的故障处理？
5. **取消任务的及时性**：用户改导航后取消 5s 延迟硬约束，怎么实现？
6. **多生态聚合**：V2 多生态融合时，统一的设备抽象层（Matter）+ 各生态适配层。架构设计？
7. **隐私数据流**：车 → 云 → 家的数据流向（位置 / 时间 / 偏好）。加密？数据最小化原则？
8. **设备绑定与解绑**：买车 / 卖车时家里设备的绑定迁移。流程？
9. **设置 UI 与配置存储**：阈值配置 / 场景开关 / 设备清单的 UI 与数据如何同步？
10. **跨家庭场景**：用户去亲戚家 / 短租场景的处理（这种不该触发联动）。如何识别"非自家"？

**5.6 G-F2 家庭账号与多用户**

**5.6.1 用户价值**

**做什么**：一辆车多人开（夫妻 / 父子 / 家庭），每个人都该有**独立的画像、独立的偏好、独立的支付 / 隐私边界**，但他们又**共享一辆车和部分家庭场景**（如 B5 关怀的"老婆在家等你"）。家庭账号要解决的是**身份识别 + 画像隔离 + 合理共享**三件事，让每个家庭成员都感觉"这辆车也是我的"。

**不做什么**：

* 不做"一人独占的账号"（家庭车 = 多用户是刚需）
* 不做"完全共享的账号"（偏好 / 支付全混在一起 = 尴尬）
* 不做"识别不准也强行切换"（识别错 → 体验比不切还差）
* 不做"复杂配置"（家里人多，让每个人单独配置 → 没人用）

**核心价值主张**：

* **三层账号模型**：主账号（车主）+ 家庭成员账号（共享）+ 临时账号（访客）
* **识别优先级**：DMS 人脸（首选）→ 蓝牙手机 → 账号主动登录 → 默认主账号
* **共享与隔离并存**：导航习惯 / 音乐偏好等私密画像独立；车家联动 / 车辆配置等共享

**5.6.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景（颗粒度到具体人/时间/状态） | 用户角色 / 识别方式 | 系统行为 | 关键约束 |
| S1 | **主账号（丈夫）上车**：DMS 人脸识别成功 | 主账号 | 自动加载丈夫画像（导航 / 音乐 / 座椅 / 空调偏好）；欢迎语"欢迎回来" | 识别准确率 ≥ 95% |
| S2 | **家庭成员（妻子）上车**：DMS 识别为妻子 | 家庭成员 | 加载妻子画像；欢迎语"欢迎回来"；支付限额遵循妻子设定（可能更高或更低） | 家庭成员数量 V1 ≤ 5 个 |
| S3 | **孩子（子账号）上车**：未成年，识别为孩子 | 子账号 | 加载孩子画像（儿歌 / 动画内容偏好）；禁用支付；禁用某些内容（暴力 / 成人） | 子账号有内容过滤 + 支付禁用 |
| S4 | **访客（临时账号）**：朋友借车 | 临时账号（需主账号授权） | 不加载任何画像；默认配置；访客行为不记录到任何账号画像 | 主账号可设置"临时访问时长"（1 天 / 1 周 / 永久） |
| S5 | **DMS 识别失败**：光线太差 / 戴墨镜 | 未识别 | 降级到蓝牙手机识别；仍识别不了则提示"请登录账号" | 降级链：DMS → 蓝牙 → 账号输入；不强制登录，仍可作为"访客模式"用 |
| S6 | **账号切换**：中途丈夫和妻子换位置开车（加油站换司机） | 新驾驶员识别 | 3 秒内识别新驾驶员，加载新画像；当前音乐 / 导航保留（不中断） | 切换时不中断任务流；只切换偏好层 |
| S7 | **共享配置 vs 独立画像**：车辆配置（如后视镜角度）保留上次主驾位置；导航习惯走画像隔离 | 两种维度分开 | 座椅 / 后视镜 / 驾驶模式 = 按识别驾驶员自动调整；不同驾驶员不同；但"车家联动"中的家庭场景（如老婆在家）不因驾驶员切换而变化 | 定义清晰哪些是驾驶员绑定 / 哪些是车辆绑定 |
| S8 | **家庭主账号的权限**：丈夫（主账号）想禁用女儿（子账号）的高速公路路权 | 主账号权限管理 | 设置 - 家庭 - 权限；可设置子账号"禁止高速"、"禁止夜间"等 | V1 基本权限（驾驶时段 / 最高速度 / 支付额度） |
| S9 | **家庭成员间共享场景**：B5 关怀"到家给老婆打电话"——丈夫和老婆都是家庭成员 | 共享场景 | 丈夫画像中的"到家关怀"触发"给老婆打电话"；老婆画像中的"到家关怀"触发"给老公打电话"；画像独立但场景共享 | 家庭关系维护在家庭账号结构里 |
| S10 | **用户卖车 / 注销**：车卖给二手车主 | 账号清理 | 主账号发起"卖车"流程 → 清理所有画像 / 支付绑定 / 车家联动 → 新车主激活 | 清理必须彻底；旧数据不能泄露给新车主 |

**5.6.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| DMS 识别错误（丈夫 vs 妻子相似） | 模型准确率不足 / 光线差 | 加载错画像，体验突兀 | 置信度 < 0.8 不切账号；人工确认"是 xx 吗"；用户可主动纠正 |
| 多人同时上车识别优先级 | DMS 只识别主驾但副驾才是车主 | 错误加载非主驾画像 | 以"主驾位置的人"为当前驾驶员；副驾不影响画像 |
| 账号数据跨成员泄露 | 隔离机制有漏洞 | 妻子看到丈夫的浏览历史 | 画像数据严格按账号 ID 隔离；访问前强制校验 |
| 临时账号权限过大 | 访客可做主账号才能做的操作 | 访客误操作影响主账号数据 | 临时账号权限白名单；默认最低权限 |
| 卖车数据清理不彻底 | 清理流程有遗漏 | 新车主能看到旧车主数据 | 清理流程三重校验 + 异地冷启动测试 |
| 子账号绕过限制 | 系统漏洞 / 用户技术破解 | 孩子看了不合适内容 | 内容过滤多层（账号级 + 内容级 + 时间级）；用户端日志可查 |
| 家庭成员信息不对称 | 妻子不知道丈夫也在用这辆车 | 家庭关系上的尴尬（如隐私泄露） | V1 不做过多社交功能；家庭成员仅共享必要场景 |
| 账号合并 / 分离困难 | 离婚 / 家庭结构变化 | 旧数据无法清理 | 主账号可发起"移除家庭成员"；分离后数据彻底隔离 |
| 访客借车频率高 | 经常换司机 | 每次都要主账号授权很烦 | 可设置"常用朋友"白名单，简化授权流程 |

**5.6.6 技术钩子（给沈砚）**

1. **DMS 人脸识别准确率**：不同光照 / 姿态 / 化妆 / 配饰下的准确率。模型大小 / 推理延迟 / 算力开销？
2. **识别优先级**：DMS → 蓝牙 → 账号主动 → 默认。每个环节的置信度阈值？降级链路？
3. **画像数据存储**：家庭成员的画像存储（端侧 / 云端）。多账号数据隔离的技术实现（数据库 schema / 加密）？
4. **账号切换延迟优化**：预加载策略（提前加载可能的账号画像）？缓存管理？
5. **子账号权限系统**：驾驶时段 / 最高速度 / 支付额度 / 内容过滤——这些权限的存储和生效机制？
6. **家庭关系模型**：家庭主账号 → 成员账号 → 子账号的树状结构。数据模型？
7. **共享 vs 独立的字段定义**：哪些数据跨账号共享（车辆配置 / 车家联动），哪些独立（导航习惯 / 音乐）。schema 设计？
8. **卖车 / 注销的数据清理**：彻底清理的校验流程。合规要求（个保法）？
9. **访客账号的权限白名单**：访客能做什么 / 不能做什么。权限配置和检查机制？
10. **账号变更事件的广播**：账号切换后所有服务（音乐 / 导航 / 空调 / 主动服务）立刻切换。事件总线设计？

**5.5 E-F8 品类聚合 Agent 层（咖啡/外卖/出行等）**

**5.5.1 用户价值**

**做什么**：用户说"我要咖啡"时，**不需要自己选瑞幸还是星巴克**——座舱 AI 通过**品类聚合 Agent**（咖啡 Agent / 外卖 Agent / 出行 Agent 等）**自动择优推荐**（根据用户偏好 + 距离 + 价格 + 评分），用户只需确认下单。**品类 Agent 内部接入多个 MCP**（例如咖啡 Agent 同时连瑞幸 / 星巴克 / Manner / 库迪），对用户呈现**统一的品类入口**（"我要咖啡"），而不是零散的渠道列表（"叫瑞幸 API""叫星巴克 API"）。这是把"MCP 技术层"做成"用户可感知的产品层"的关键——**用户想的是品类（咖啡 / 外卖 / 出行），不是渠道（瑞幸 / 美团 / 滴滴）**。

**不做什么**：

* 不做"把所有 MCP 平铺给用户"（用户得记"咖啡该叫哪个""外卖该叫哪个"→ 认知负担）
* 不做"每次都问用户选哪家"（"星巴克还是瑞幸？"→ 把选择痛苦甩给用户）
* 不做"V1 上所有品类"（品类太多维护成本爆炸；V1 只做 P0 三个：咖啡 / 外卖 / 出行）
* 不做"品类 Agent 替用户做主"（择优推荐 ≠ 强制下单；用户可"指定渠道"兜底："我要瑞幸的咖啡"）
* 不做"品类边界模糊的细拆"（"奶茶"算咖啡还是新建饮品 Agent？V1 用粗粒度，看数据再拆）

**核心价值主张**：

* **按品类建 Agent，不按渠道**——用户说意图（"咖啡"），AI 内部决策渠道
* **多渠道聚合 + 择优推荐**——每个品类 Agent 内接多个 MCP，自动择优
* **统一下单交互**——用户不需要学每个 App 的下单流程
* **用户可指定渠道兜底**——不信任择优时可"我要瑞幸"绕过

**5.5.2 三层架构**

|  |
| --- |
| Plain Text ┌─────────────────────────────────┐ │ 用户层：统一说"我要 xx" │ ← 用户只说意图 ├─────────────────────────────────┤ │ 聚合 Agent 层（核心） │ ← 把意图路由到对应子 Agent │ ┌──────┬──────┬──────┬──────┐ │ │ │咖啡 │外卖 │出行 │购物 │ │ 子 Agent（按品类，不按渠道） │ │Agent │Agent │Agent │Agent │ │ │ └──────┴──────┴──────┴──────┘ │ ├─────────────────────────────────┤ │ MCP 接入层（原始第三方） │ ← 技术层，用户不感知 │ 瑞幸/星巴克/美团/饿了么/滴滴... │ └─────────────────────────────────┘ |

**关键设计点**：

* **品类 Agent 内部做"渠道聚合 + 择优推荐"**
* **择优算法**：距离 × 偏好 × 价格 × 评分（权重可调）
* **用户偏好画像**：喝过什么、常去哪家、口味偏好（与 04 记忆联动）
* **上下文判断**：在哪里、什么时间、预算档位

|  |  |  |  |
| --- | --- | --- | --- |
| 品类 Agent | 接的 MCP（举例） | V1 优先级 | 择优维度 |

**5.5.4 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景 | 品类 Agent 行为 | 关键约束 |
| S1 | **早高峰要咖啡**："我要咖啡" | 咖啡 Agent：查附近瑞幸/星巴克/Manner → 根据用户偏好（常喝瑞幸）+ 距离（瑞幸 500m vs 星巴克 1.2km）→ 推荐"瑞幸 xx 店，美式，500m，预计 5 分钟" | 推荐 1 个；用户可"换一个" |
| S2 | **指定渠道**："我要星巴克的咖啡" | 咖啡 Agent：跳过择优 → 直接调星巴克 MCP | 用户可绕过择优 |
| S3 | **外卖午餐**："点个外卖" | 外卖 Agent：NLU 补全"什么菜系？" → 用户："川菜" → 查美团/饿了么川菜 → 择优推荐 | 多轮对话补全参数 |
| S4 | **打车回家**："叫个车回家" | 出行 Agent：查滴滴/高德打车 → 择优（等待时间 + 价格）→ 推荐"滴滴快车，3 分钟到，预计 25 元" | 与 E-F6 跨端联动（手机已登录滴滴） |
| S5 | **充电**："找个充电桩" | 出行 Agent：查特来电/星星/小桔 → 择优（距离 + 空闲桩数 + 价格）→ 推荐"特来电 xx 站，2km，5 个空闲桩" | 与导航联动 |
| S6 | **择优失败**：所有渠道都无货 | 咖啡 Agent：返回"附近咖啡店都满了，要不要看 3km 外的？" | 失败路径清晰 |
| S7 | **用户偏好学习**：连续 3 次选瑞幸 | 咖啡 Agent：更新用户偏好权重（瑞幸 +0.2）→ 下次优先推瑞幸 | 与 04 记忆联动 |
| S8 | **多品类组合**："点咖啡 + 叫车" | 咖啡 Agent + 出行 Agent 并行 → 咖啡先下单 → 车 5 分钟后到 | 与 06 Agent 编排联动 |
| S9 | **离线降级**：无网络 | 品类 Agent 不可用 → 降级到"直接打开 App"（E-F1 深度接入的 App 仍可用） | 与 12 弱网降级联动 |
| S10 | **新品类请求**："我要买书" | 当前无"购物 Agent" → 降级到"打开淘宝 App" + 提示"V1.5 会支持语音购物" | 优雅降级 |

**5.5.5 成功指标**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方式 | 备注 |
| 品类 Agent 使用率 | ≥ 40% | 用品类意图（"咖啡"）÷ 总第三方服务请求 | 核心指标 |
| 择优推荐接受率 | ≥ 70% | 用户直接确认 ÷ 推荐次数 | 择优算法有效性 |
| 指定渠道使用率 | ≤ 20% | 用户说"我要 xx 的咖啡" ÷ 总咖啡请求 | 择优信任度（越低越好） |

| 择优失败率 | ≤ 10% | 所有渠道都无货 ÷ 总请求 | 渠道覆盖度 |
| 用户偏好学习准确率 | ≥ 80% | 第 5 次推荐与用户历史偏好一致 | 记忆有效性 |

**5.5.7 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 择优推荐不准 | 偏好画像不准 / 算法权重不对 | 用户每次都"换一个" | A/B 测试权重 + 偏好学习优化 |
| 所有渠道都无货 | 渠道覆盖不足 / 时段问题 | 用户失望 | 扩大搜索半径 / 推荐替代品类 |
| 多轮补全失败 | NLU 不理解 / 用户不耐烦 | 用户放弃 | 补全问题简化 / 提供默认选项 |
| 品类边界模糊 | "奶茶"算咖啡还是饮品？ | 用户困惑 | V1 粗粒度（奶茶归咖啡），V1.5 细拆 |
| 指定渠道失败 | 用户说"我要瑞幸"但瑞幸 MCP 挂了 | 用户失望 | 降级到其他渠道 + 说明原因 |
| 品类 Agent 响应慢 | 多个 MCP 串行查询 | 用户等待 | 并行查询 + 超时降级 |
| 用户偏好冲突 | 今天想换口味但 AI 还推老偏好 | 用户烦躁 | 偏好有"遗忘机制" + 情境感知（今天心情不同） |
| 品类 Agent 与品牌方利益冲突 | 瑞幸投诉"凭什么星巴克排前面" | 商业风险 | 商业决策（老师拍）；技术上支持"品牌方付费优先" |

**6. 自我反驳与边界**

**6.1 核心 App 5-8 个够吗?用户真的不需要更多?**

**自反**:小鹏接了 20+ App,我们只接 8 个会不会被比?**反驳自反**:深度 > 数量。8 个做深(每个 5-10 个复杂指令)比 20 个做浅(每个 1-2 个简单指令)更有价值。营销上我们用"真实使用场景故事"取代"接入数量"。

**6.2 开放协议接长尾 App 会不会引入风险?**

**自反**:开放 = 安全/性能/体验都可能失控。**对冲方案**:(1)严审核(功能/安全/隐私三关)(2)沙箱执行(3)动态下架(出问题立即关)(4)用户可见接入来源(不是官方的,明确标出)

**6.3 支付多层看起来很安全,但用户嫌麻烦怎么办?**

**自反**:免密 < 200 元能覆盖 80%,用户其实无感;只有 20% 非常规场景走扫码/人脸。**结论**:不是打扰用户,是**在用户无感层面分级管控**。

**6.4 车家互联会不会分散精力?**

**自反**:车家生态是小米/蔚来战场,我们做有多大胜算?**对冲方案**:V1 只做"下班到家预热"一个场景,小而美。其他场景 V1.5 根据生态布局决定。

**6.5 跨端延续会不会被平台(iOS/Android)限制?**

**自反**:iOS 后台受限严格,跨端同步延迟大。**对冲方案**:(1)用户主动触发(上车后"是否继续" 提示)(2)不依赖实时推送,用状态同步 + 上车时主动拉取
