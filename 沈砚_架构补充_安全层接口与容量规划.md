## 补充 A：安全层 ↔ AI 层接口定义

> 背景：主文档 ⑦-21 提到"AI 层 → 安全层 → CAN 总线"的三层结构，但两层之间的接口协议未定义，会导致 AI 团队和安全层团队在联调时才发现接口不匹配。本节补全。

### A.1 整体消息流

```
AI Sub-bot 生成车控意图
        ↓
[接口A] VehicleCommandRequest（AI→安全层）
        ↓
安全层执行校验（规则引擎，<10ms）
        ↓ 通过                    ↓ 拒绝
CAN 帧发送             [接口B] VehicleCommandResponse（拒绝原因）
        ↓                               ↓
CAN 执行结果                    TTS 告知用户
        ↓
[接口B] VehicleCommandResponse（成功/失败）
        ↓
AI Sub-bot 更新任务状态
```

### A.2 接口 A：VehicleCommandRequest（AI → 安全层）

**协议**：同进程 gRPC（端侧）或 HTTP/2（云端下发到端侧安全层）

**消息格式（Protobuf 定义）：**
```protobuf
message VehicleCommandRequest {
  string trace_id = 1;          // 全链路追踪ID
  string session_id = 2;        // 当前对话会话ID
  CommandType command_type = 3; // 枚举：WINDOW/AC/SEAT/LOCK/LIGHT/...
  map<string, string> params = 4; // 参数键值对，如 {position: "driver", action: "open", degree: "50"}
  float confidence = 5;         // AI 置信度 0.0-1.0
  string source_text = 6;       // 原始 NLU 识别文本（供审计）
  int64 timestamp_ms = 7;       // 请求时间戳
  bool user_confirmed = 8;      // 是否已经过用户二次确认
}
```

**CommandType 枚举（V1 白名单，未列出的类型安全层直接拒绝）：**

| 枚举值 | 允许操作 | 最高风险动作禁止 |
|--------|----------|-----------------|
| WINDOW | 开/关/调节角度 | 行驶中全开（车速>80km/h 限制开合角度≤30%） |
| AC | 温度/风速/模式 | 无限制 |
| SEAT | 角度/位置调节 | 驾驶中禁止主驾座椅前后移动 |
| LOCK | 解锁/上锁 | 行驶中禁止解锁 |
| LIGHT | 车内氛围灯 | 无限制（车外灯不在V1范围） |
| SUNROOF | 开合 | 雨天传感器触发时禁止开启 |

**绝对禁止（硬编码，不受 confidence 影响）：**
- 转向系统、制动系统、油门系统
- 发动机启停（V1不做远程启动）
- ADAS 参数修改

### A.3 接口 B：VehicleCommandResponse（安全层 → AI）

**消息格式：**
```protobuf
message VehicleCommandResponse {
  string trace_id = 1;
  ResponseStatus status = 2;  // SUCCESS / REJECTED / TIMEOUT / PARTIAL
  RejectionReason reason = 3; // 拒绝原因枚举
  string user_message = 4;    // 建议 TTS 播报的文本（安全层生成，保证措辞准确）
  int64 executed_at_ms = 5;   // 实际执行时间戳（CAN 帧发送时刻）
}

enum RejectionReason {
  UNKNOWN = 0;
  SPEED_LIMIT = 1;        // 车速限制
  CONFIDENCE_LOW = 2;     // AI 置信度不足
  COMMAND_FORBIDDEN = 3;  // 命令类型禁止
  USER_NOT_CONFIRMED = 4; // 高风险操作未经用户确认
  SENSOR_BLOCK = 5;       // 传感器条件不满足（如雨天不开天窗）
  SYSTEM_BUSY = 6;        // CAN 总线繁忙
}
```

### A.4 安全层校验规则（完整清单 V1）

| 规则ID | 条件 | 动作 |
|--------|------|------|
| S-001 | confidence < 0.7 | REJECTED（CONFIDENCE_LOW），触发二次确认 |
| S-002 | confidence 0.7-0.89 AND 高风险操作 | REJECTED（USER_NOT_CONFIRMED），等用户确认 |
| S-003 | confidence ≥ 0.9 | 直接执行 |
| S-004 | 车速 > 120km/h | 仅允许 AC/LIGHT，其余 REJECTED（SPEED_LIMIT） |
| S-005 | 车速 > 30km/h | 禁止 LOCK 解锁，禁止 SEAT 主驾前后移动 |
| S-006 | 雨量传感器 > 阈值 | SUNROOF 开启 REJECTED（SENSOR_BLOCK） |
| S-007 | CommandType 不在白名单 | REJECTED（COMMAND_FORBIDDEN），记录告警 |
| S-008 | CAN 帧超时（>500ms） | TIMEOUT，重试1次，仍失败则 REJECTED |

---

### A.5 超时与并发

- 安全层校验超时：**10ms**（规则引擎，同步阻塞，必须在这个时间内返回）
- 并发请求：同一时刻只处理1个 VehicleCommand（队列等待，FIFO）
- 队列超时：排队超过 3s → 整体 TIMEOUT，AI Sub-bot 告知用户"指令繁忙，请稍后重试"
- 审计：所有 Request + Response 写入端侧审计日志（不可删除，车控合规要求）

## 补充 B：云端容量规划

> 背景：主文档 ⑦-28 给了监控指标，但没有给容量规划数字。基础设施团队需要知道"买多少机器"。

### B.1 用量模型（灰度→正式）

| 阶段 | DAU | 高峰系数 | 峰值 QPS（L3 LLM） |
|------|-----|---------|-------------------|
| 内测（M+1） | 100 | 3x | 0.5 QPS |
| 灰度5%（M+3） | 5,000 | 3x | 25 QPS |
| 灰度20%（M+5） | 20,000 | 3x | 100 QPS |
| 正式V1（M+7） | 100,000 | 3x | 500 QPS |

**QPS 计算逻辑：**
`峰值 QPS = DAU × 5次/天 × 20%（L3比例） × 3（早晚高峰集中系数） / 86400 × 3600`

高峰期（8:00-9:00 / 17:30-18:30）：流量集中在2h内，实际峰值约为均值的 **3x**。

### B.3 自动扩容策略

```yaml
# 扩容规则（云端 K8s HPA 配置示意）
qwen7b-router:
  min_replicas: 2        # 最少保持2个实例（防止冷启动延迟）
  max_replicas: 50
  scale_up:
    cpu_threshold: 70%   # CPU > 70% 触发扩容
    latency_threshold: 500ms  # P95 延迟 > 500ms 触发扩容
    cooldown: 60s        # 扩容冷却60s
  scale_down:
    cpu_threshold: 30%
    cooldown: 300s       # 缩容冷却5min（防止流量波动频繁缩容）

qwen72b-planner:
  min_replicas: 1
  max_replicas: 20
  scale_up:
    latency_threshold: 2000ms  # P95 延迟 > 2s 触发
    cooldown: 120s       # 72B 实例启动慢，冷却更长
```

**冷启动时间：**
- Qwen2.5-7B（A10G）：模型加载约 **30s**
- Qwen2.5-72B（4×A100）：模型加载约 **180s**

应对冷启动：维持 min_replicas 常驻，避免从0扩容；峰前（早7:50/下午5:20）提前触发预扩容（基于历史流量的定时扩容）。

### B.4 成本控制策略

**主备模型降级（控制 72B 用量）：**
- 正常：L3 路径先用 Qwen2.5-7B（路由）+ Qwen2.5-72B（规划）
- 成本预警（当日费用 >80% 月预算）：Qwen2.5-72B 降级为 DeepSeek-V2.5（成本≈60%）
- 成本熔断（当日费用 >100% 月预算）：72B 关闭，全部用 7B 处理（功能降级但可用）

**API 计费 vs 自托管对比（V1 建议 API 计费）：**

| 方案 | 适用阶段 | 月成本（正式V1，100K DAU） | 控制权 |
|------|----------|--------------------------|--------|
| 阿里云 API 计费 | 灰度期（M+1~M+5） | ≈ ¥19,200（7B+72B合计） | 低，依赖供应商 |
| 自托管 GPU | 正式后（M+7+） | ≈ ¥345,000/月（GPU+运维） | 高，数据不出域 |
| 混合：7B自托管+72B API | M+7 过渡 | ≈ ¥135,000/月 | 中 |

> 建议：灰度期用 API，正式上量后评估自托管（数据合规 + 规模效益两个维度综合决策）。自托管节点必须在国内（汽车数据安全管理规定要求）。

### B.5 多地域部署

| 地域 | 覆盖城市 | V1 实例比例 |
|------|----------|------------|
| 华北（北京） | 华北、东北 | 30% |
| 华东（上海） | 华东、华南 | 40% |
| 华南（广州） | 华南、西南 | 20% |
| 西南（成都） | 西南 | 10% |

跨地域路由：用户请求就近接入（DNS 负载均衡），单地域故障时流量切到其他地域（<30s 感知到故障，<60s 完成切换）。
