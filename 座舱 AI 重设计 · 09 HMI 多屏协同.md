**§1 背景与目标**

**1.1 为什么做 HMI 多屏协同**

**核心问题**：座舱屏幕越来越多(仪表+中控+HUD+后排+投影),但多屏 ≠ 好体验——信息在多个屏幕间割裂、重复、冲突,用户反被屏幕绑架。

三大矛盾：

1. **信息割裂**：导航在 HUD,音乐在中控,空调在实体键——同一时刻注意力被三个方向撕扯
2. **场景不匹配**：驾驶时中控弹娱乐推荐,驻车时仪表还显示车速——信息和场景错配
3. **焦点失控**：多个屏幕同时抢注意力,用户不知道该看哪里

**AI 时代的机会**：

* **AI 焦点管理**：系统决定"此刻最重要信息在哪块屏显示",用户不用选
* **场景化布局**：驾驶/泊车/驻车/休憩四种模式,每屏自动切换信息架构
* **卡片化**：信息以卡片为单位,跨屏流转(音乐卡片从主驾屏滑到后排屏)

**目标**：

* **主目标**：驾驶场景下,关键信息(导航/车速/警示)100% 通过 HUD+仪表呈现,中控不抢视线
* **效率目标**：常用操作从"找 App→点开→操作"变为"一句话→卡片弹出→确认"
* **安全目标**：驾驶模式下中控非安全信息自动降亮度/隐藏

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **视线不乱飞**：该看的(HUD/仪表)和想看的(中控)自动分离,不会误看
* **场景自动适配**：开车时中控只留导航+音乐,驻车时展开全部功能
* **一句话弹卡片**：不需要翻菜单,"订位"→美团卡片自动弹出

**对乘员**：

* **独立屏幕**：后排独立娱乐屏,不影响驾驶员
* **内容流转**：前排分享视频到后排,一句话搞定

**2.2 反价值(不做什么)**

* **不做全屏都塞满**：留白比塞满好——驾驶时中控可以是空的,HUD 够用就行
* **不做花哨动画**：场景切换要快(0.5s 内),不做 3 秒过渡动画
* **不做强迫学习**：基本操作不依赖 AI,物理键+触屏永远可用

**3. 核心能力要求**

**3.1 多屏分工**

1. **仪表** — 驾驶关键信息(速度/档位/导航箭头/电量)
2. **中控** — 主交互屏(AI 卡片/任务列表/娱乐)
3. **HUD** — 最关键提示(转向/风险预警)
4. **后排**(可选)— 独立娱乐/儿童内容
5. **投影**(可选)— 驻车影院

**3.2 卡片化信息架构**

1. **AI 输出卡片** — 任务进度/结果/建议都用卡片
2. **卡片优先级** — 安全 > 任务 > 推荐
3. **卡片生命周期** — 自动消失/用户关闭/固定
4. **卡片跨屏** — 中控卡片可推到仪表/HUD

**3.3 驾驶/泊车/驻车交互分层**

1. **驾驶模式** — 仪表+HUD 主导,中控辅助,禁用复杂交互
2. **泊车模式** — 中控全屏影像,仪表档位,HUD 隐藏
3. **驻车模式** — 中控娱乐,后排独立,仪表息屏

**3.4 AI 输出与屏幕配合**

1. **语音+卡片** — AI 说话同时中控显卡片
2. **屏幕选择** — AI 决策输出到哪个屏(驾驶时 HUD,驻车时中控)
3. **焦点管理** — 同一时刻只有一个屏是"主焦点"

**3.5 焦点管理**

1. **单焦点原则** — 避免多屏同时抢注意力
2. **焦点切换** — 用户视线/语音指令切换焦点
3. **驾驶优先** — 驾驶模式下仪表/HUD 永远优先

**5.1 H-F1 多屏分工与协同**

**5.1.1 用户价值**

**做什么**：车上 4-5 块屏（仪表 / 中控 / HUD / 后排屏 / 可选投影）**各司其职不互相抢戏**——仪表管驾驶硬信息、HUD 管抬头关键提示、中控管主交互、后排屏管独立娱乐。用户无论坐哪个位置都清楚"这块屏给我用什么"，不会被多屏同步的信息洪流淹没。

**不做什么**：

* 不做"每块屏都显示一样内容"（浪费屏幕资源）
* 不做"仪表显示推荐卡片"（干扰驾驶关键信息）
* 不做"HUD 全屏动画"（HUD 只做最关键提示）
* 不做"后排屏和前排屏强绑定"（后排乘客该有自由，不该被主驾打扰）

**核心价值主张**：

* **角色明确**：每块屏有一个首要职责
* **仪表 / HUD = 驾驶硬信息**：速度 / 档位 / 导航箭头 / 风险预警
* **中控 = 主交互**：AI 卡片 / 任务列表 / 娱乐控制
* **后排屏 = 独立娱乐**：独立内容源 + 独立音频
* **协同不等于同步**：各屏做各屏该做的事，协同是"必要时把卡片从 A 推到 B"

|  |  |  |  |
| --- | --- | --- | --- |

| S1 | **A1 早高峰通勤**：司机 8:00 通勤路上 | 仪表：导航箭头 + 距目的地时间 + 车速 / 中控：当前播客卡片 + 通勤模式快捷 / HUD：下一步转向（仅距离 500m 时显示）/ 后排屏：关闭 | 驾驶中中控文字信息限 < 20 字；HUD 只在关键决策点触发 |

**5.1.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 多屏同步延迟 | 渲染引擎瓶颈 | 中控更新了但仪表没跟上 | 关键信息（导航转向）在仪表优先；中控异步 |
| 跨屏推送失败 | 底层通信失败 | 用户说推但没推过去 | 推送前确认目标屏可用；失败明确反馈 |
| 后排屏独立干扰不够 | 音频泄露到前排 | 后排看动画影响主驾 | 独立音频通道 + 定向扬声器（见 10）；无法实现时物理音量受控 |
| HUD 信息过密 | 多条信息同时显示 | 反而干扰视线 | HUD 单时刻最多显示 1-2 个元素；超出则排队 |
| 仪表被挤压 | 信息太多挤进仪表 | 驾驶关键信息被遮挡 | 仪表内容白名单硬编码；非驾驶类信息不得进入 |
| 副驾误被判定为主驾 | 识别失败 | 副驾操作触发了"主驾"的规则 | 默认当作"驾驶员操作"（保守态度）；副驾主动标识 |
| 后排屏儿童模式错判 | OMS 识别失败 | 成人看到儿童内容 | 儿童模式必须 OMS 确认 + 用户手动确认 |

**5.1.6 技术钩子（给沈砚）**

1. **多屏渲染引擎**：统一引擎（如 Qt / React Native）还是各屏独立？性能 / 开发成本 / 一致性？
2. **跨屏通信机制**：总线 / 事件 / RPC。延迟 / 可靠性 / 重试机制？
3. **副驾识别**：触控热区 + 视线 + 话筒方向。多信号融合置信度？
4. **HUD 内容优先级**：多条信息排队机制。FIFO 还是优先级队列？显示时长？
5. **后排屏独立性**：独立 OS 还是主系统分区？安全隔离（不让后排屏影响主系统）？
6. **各屏资源配额**：CPU / GPU / 内存在多屏间的分配？驾驶中中控降级优先级？
7. **卡片跨屏简化渲染**：同一卡片在中控 / 仪表 / HUD 的不同 UI 变体。自动适配还是手动定义每种变体？
8. **仪表内容白名单**：白名单定义在哪？越界时如何拦截？
9. **失败降级链**：HUD 关闭 → 仪表补位、后排屏无法独立 → 主屏兼任。降级策略的配置？
10. **多屏协同的 A/B 测试**：默认 HUD on/off、中控卡片数量上限、仪表布局等都需要 A/B。测试框架？

**5.2 H-F2 卡片化信息架构**

**5.2.1 用户价值**

**做什么**：座舱 AI 的所有**主动产出**（任务进度 / 结果呈现 / 建议 / 通知 / 关怀动作）**统一用"卡片"承载**——不弹对话框、不占全屏、不随机浮动。每张卡片有**明确的生命周期**（出现 / 停留 / 消失 / 固定），有**优先级**（安全 > 任务 > 推荐），用户能**一键清空所有卡片**回到干净的主界面。

**不做什么**：

* 不做"对话框式弹窗"（用户必须点"确定"才能继续 → 驾驶中极度危险）
* 不做"无主次的信息流"（所有卡片平级 → 用户不知道该看哪个）
* 不做"不会消失的卡片"（驻留太久 → 屏幕被垃圾信息占满）
* 不做"卡片间随意跳转"（用户点 A 跳到 B 再跳到 C → 迷路）

**核心价值主张**：

* **统一视觉语言**——所有 AI 产出视觉风格一致，用户眼熟不用重新学习
* **明确生命周期**——自动消失 / 用户关闭 / 固定三档，不堆积
* **优先级可见**——颜色 + 位置 + 大小共同表达重要程度

**5.2.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景 | 卡片类型 + 生命周期 | 关键约束 |
| S1 | **任务进度卡片**：用户说"订会议室 + 导航过去"，任务开始 | 任务卡片，位置：中控右上，生命周期：任务进行中保留，完成后 5s 自动消失 | 任务卡片始终在同一位置（用户眼熟） |
| S2 | **结果呈现卡片**：导航路线规划完成 | 结果卡片，位置：中控中央，生命周期：10s 自动消失或用户固定 | 用户可"钉住"卡片（固定在中控，直到手动取消） |
| S3 | **建议卡片**：AI 推荐"要不要顺路吃个饭" | 建议卡片，位置：中控底部，生命周期：5s 自动消失或用户响应 | 不打断当前任务；建议类优先级最低 |
| S4 | **通知卡片**：微信消息 / 电话来电 | 通知卡片，位置：中控顶部横条，生命周期：10s 或用户响应 | 电话来电强制显示；其他通知驾驶中可批量推迟 |
| S5 | **关怀动作卡片**：娃哭关怀触发 | 关怀卡片，位置：中控小浮标，生命周期：3s 自动消失 | 关怀卡片极简（一句话 + icon），不打扰 |
| S6 | **安全告警卡片**：胎压异常 / 碰撞预警 | 告警卡片，位置：仪表全屏 + HUD 同步，生命周期：用户主动确认消失或告警解除 | 最高优先级；覆盖其他所有卡片 |
| S7 | **卡片优先级冲突**：导航进行中来了微信消息 | 任务卡（导航）保留；通知卡在顶部横条临时显示 10s | 任务卡优先级 > 通知卡；两者可共存但不能互相覆盖 |
| S8 | **用户一键清空**：用户说"清一下" / 点"全部关闭"按钮 | 所有非安全类卡片消失；回到主界面 | 安全告警类不会被一键清空（避免误操作） |
| S9 | **卡片跨屏推送**：用户说"把这个放到仪表上" | 卡片在中控保留，同时推到仪表（简化版） | 跨屏推送后中控可选关闭；仪表版简化 |
| S10 | **驾驶中卡片限流**：驾驶中同时应显示 5 张卡片 | 最多同时显示 2 张（优先级最高的 2 张）；其他排队 | 驾驶中卡片上限硬约束 2 张 |

**5.2.3 成功指标**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方式 | 备注 |
| 卡片视觉风格一致性 | 100% | 所有 AI 卡片通过设计规范审查 | 硬约束 |

| 告警卡片首次显示延迟 | ≤ 500ms | 从触发到卡片显示 | 硬约束 |

**5.2.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 卡片渲染卡顿 | 动画 / 资源加载慢 | 卡片卡住不消失 | 性能预算 + 降级动画（低端车型用静态） |
| 同时多卡片冲突 | 优先级判断逻辑错误 | 卡片互相覆盖 | 优先级硬编码；单元测试覆盖所有组合 |
| 卡片消失后信息丢失 | 用户想回看 | 用户错过关键信息 | "卡片历史"入口（可查最近 10 条被动消失的卡片） |
| 告警卡片被其他屏幕干扰 | Z-index 问题 | 告警被其他 UI 遮挡 | 告警强制最高 Z-index；遮挡主动上升 |
| 视觉风格不一致 | 多团队参与 | 品牌感被稀释 | 设计系统 + 组件库 + CI 检查 |
| 卡片文案过长 | 内容超限 | 信息被截断 | 卡片文案字数硬限制；超出则显示"更多"按钮 |
| 固定卡片被误关闭 | 用户手滑 | 重要信息丢失 | 固定卡片关闭前二次确认 |

**5.2.6 技术钩子（给沈砚）**

1. **卡片系统架构**：自研还是开源（React Native / Flutter / 原生）？性能 / 跨屏兼容 / 开发成本？
2. **卡片优先级队列**：数据结构设计。优先级动态变化（如告警升级）如何处理？
3. **卡片生命周期管理**：每张卡片独立计时器 vs 中心化调度。性能差异？
4. **渲染性能预算**：单张卡片 CPU / 内存占用。多卡片并发上限？
5. **设计规范 CI 检查**：自动化检查卡片视觉风格。工具链？
6. **跨屏简化渲染**：同一卡片在不同屏的变体定义。手动还是自动？
7. **卡片历史存储**：最近 10 条被动消失的卡片。存储位置 / 查询接口？
8. **Z-index 管理**：告警强制最高、任务中等、通知较低。规则在哪维护？
9. **卡片 A/B 测试**：默认停留时长、驾驶中上限数、视觉样式都需要测试。框架？
10. **卡片与语音的协同**：TTS 说话时卡片如何配合？同步还是异步？语音结束后卡片保留多久？

**5.3 H-F3 驾驶 / 泊车 / 驻车分层**

**5.3.1 用户价值**

**做什么**：车在不同状态下用户的注意力分配完全不同——**驾驶中专注路面，泊车中专注影像，驻车后才有空玩屏**。座舱 HMI 必须在三种状态间**自动切换交互层级**，让屏幕在每种状态下都"恰到好处"——不在驾驶中干扰、不在驻车时显得冷清、不在泊车时显示无关信息。

**不做什么**：

* 不做"一刀切"（不管什么状态都显示一样内容）
* 不做"切换迟钝"（车从前进切倒车 5 秒后才切影像 → 危险）
* 不做"用户手动切换"（用户开车没空切层级，必须自动）
* 不做"复杂分层"（用户记不住 5 种状态各用什么 UI）

**核心价值主张**：

* **三层而非五层**——驾驶 / 泊车 / 驻车三档够用，不要过度细分
* **自动切换 + 手动可调**——默认自动，但用户可强制（如驻车不想娱乐就手动切回简洁模式）
* **切换不打扰**——切换动画 < 500ms，不阻塞用户当前任务

**5.3.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 状态 | 触发条件 | 各屏 UI 表现 | 关键约束 |
| S1 | **驾驶模式**（车速 ≥ 5 km/h） | 车速持续 ≥ 5 km/h | 仪表：完整驾驶信息 / HUD：转向 + 风险预警 / 中控：简化（卡片限 2 张，文字 < 20 字）/ 后排屏：保持当前 | 中控复杂操作禁用（如长视频 / 大量文字阅读） |
| S2 | **泊车模式**（档位 R 或主动启动 APA） | 档位切到 R 或用户启动自动泊车 | 仪表：360 影像 + 档位 / HUD：隐藏 / 中控：360 影像全屏 + 障碍物提示 / 后排屏：暂停（避免干扰） | 影像延迟 ≤ 100ms；360 影像优先级最高 |
| S3 | **驻车模式**（档位 P + 车速 0） | 档位 P 持续 ≥ 3s | 仪表：可息屏（用户开关）/ HUD：隐藏 / 中控：解锁影院 / 娱乐 / 设置全功能 / 后排屏：独立娱乐 | 任何"准备出发"信号（解开安全带 / 踩刹车）→ 中控退出全屏 |
| S4 | **泊车 → 驾驶切换**：完成泊车，切回 D 档 | 档位 R → D | 中控：360 影像渐变到导航主屏（500ms 过渡）/ HUD：恢复显示 / 仪表：恢复完整 | 切换动画不阻塞用户操作 |
| S5 | **驾驶 → 驻车**：到达目的地停车 | 档位切 P 且车速 0 持续 3s | 中控：从简化版渐变到完整功能 / 仪表：可息屏 / HUD：隐藏 / 后排：独立内容 | 用户已配置的"到达模式"动作（音量降 / 灯光柔）一并触发（与 D-F1 关怀联动） |
| S6 | **用户手动覆盖**：驻车后用户不想看影院模式 | 用户手动切"简洁模式" | 中控简洁版（仅显示导航 / 时间 / 播放控制）；忽略默认娱乐 | 用户偏好记忆；下次驻车默认偏好版 |
| S7 | **泊车异常**：自动泊车失败需用户接管 | APA 失败信号 | 中控：影像保留 + 接管提示 / HUD：警示 / 仪表：警示图标 | 警示明确不混淆"失败"与"完成" |
| S8 | **驾驶中误判驻车**：等红灯（车速 0 但很快走） | 车速 0 持续 < 30s 且档位仍是 D | 不切驻车模式；保持驾驶 UI | 驻车切换需要档位 P + 持续时间双重判断 |
| S9 | **驻车中突发驾驶意图**：用户系安全带 + 启动 | 解开安全带 → 系上 → 启动 | 中控从娱乐渐变到驾驶简化版（500ms） | 切换不阻塞动画必须可被用户操作打断 |

|  |  |  |  |

| 指标 | 目标值 | 测量方式 | 备注 |
| 状态识别准确率 | ≥ 99% | 状态识别正确 ÷ 总状态切换 | 硬约束 |
| 切换延迟 | ≤ 500ms | 状态变化到 UI 更新完成 | 用户感知"流畅" |

**5.3.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 误判驻车（等红灯） | 仅看车速忽略档位 | 驾驶中切到驻车 UI | 状态判断加档位 + 持续时间双重条件 |
| 切换动画卡顿 | 渲染慢 / 资源未加载 | UI 闪烁不流畅 | 资源预加载 + 动画降级（低端车用渐变 → 切换） |
| 泊车影像延迟 | 摄像头 → 渲染链路慢 | 影像跟不上车移动 | 链路优化；超时降级到"低分辨率 + 高帧率" |
| 用户偏好不记忆 | 偏好存储 bug | 用户每次驻车都要手动切"简洁版" | 偏好持久化 + 跨会话同步 |
| 驾驶中泊车信号误触发 | 倒车信号误读 | 中控误切影像模式 | 倒车信号需要档位 + 车速 + OBD 多重确认 |
| 驻车自动息屏过快 | 息屏阈值低 | 用户还在看时屏幕黑了 | 用户操作（触摸 / 视线 / 语音）刷新息屏倒计时 |
| 状态切换打断用户操作 | 切换动画阻塞 | 用户点的按钮没生效 | 切换动画期间用户操作排队执行 |
| 多状态快速切换 | 边界场景反复切（停车场缓慢出入） | UI 反复闪烁 | 切换需要"持续 N 秒"防抖 |

**5.3.6 技术钩子（给沈砚）**

1. **状态识别信号源**：车速 / 档位 / 安全带 / 启动信号 / OBD。多信号融合优先级？
2. **状态切换防抖**：边界场景（停车场缓慢出入）的切换防抖。机制？
3. **UI 渲染过渡动画**：500ms 过渡如何不卡顿？资源预加载？
4. **影像低延迟链路**：摄像头 → 渲染。延迟预算如何拆？硬件需求？
5. **用户偏好存储**：用户的"驻车默认简洁版"偏好。端侧 / 云端？跨车型迁移？
6. **状态切换的事件广播**：所有应用 / agent 都要知道当前状态。事件总线？
7. **驾驶中 UI 限制的执行机制**：白名单 / 黑名单？哪一层强制？
8. **车型差异**：低端车型可能没 HUD / 没后排屏。状态切换是否需要根据车型变化？
9. **状态切换的回退**：切换后系统异常如何回滚到上一稳定状态？
10. **多用户共享车的偏好冲突**：丈夫喜欢简洁版、妻子喜欢娱乐版。按用户画像加载？

**5.4 H-F4 AI 输出屏幕选择**

**5.4.1 用户价值**

**做什么**：AI 要说话 / 显示卡片时，**自动选择最合适的那块屏**——驾驶中的导航提示去 HUD 而不是中控、关怀动作卡片去中控而不是仪表、告警去仪表 + HUD 同步、驻车娱乐提示去后排屏。用户不需要思考"这个信息应该出现在哪"，AI 替他决定。决策错了用户烦，决策对了用户无感——而无感就是成功。

**不做什么**：

* 不做"所有输出都去中控"（偷懒做法 → 中控信息过载）
* 不做"所有屏同步显示"（信息重复 → 分心）
* 不做"用户手动指定屏"（驾驶中没空操作）
* 不做"AI 决策黑盒"（用户想推到别的屏必须能手动覆盖）

**核心价值主张**：

* **决策规则明确**——每类 AI 输出有默认屏 + 备选屏
* **用户可覆盖**——说"把这个放到仪表"就跨屏推送（见 H-F1 S7）
* **学习用户偏好**——用户多次把某类卡片推到某屏 → 记录偏好，下次默认改

**5.4.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | AI 输出类型 | 默认屏 | 备选屏 | 决策因素 |
| S1 | 导航下一步转向 | HUD | 仪表（HUD 关闭时） | 驾驶中不能看中控 |
| S2 | 安全告警（胎压 / 碰撞） | 仪表 + HUD 同步 | 中控（补充信息） | 最高优先级 + 多屏同步强化 |
| S3 | 任务进度（订会议室 / 导航规划） | 中控 | 无 | 任务主屏 = 中控 |
| S4 | 关怀动作反馈 | 中控小浮标 | 无 | 极简 + 不干扰 |
| S5 | 推荐 / 建议 | 中控底部 | 无 | 非紧急 + 可忽略 |
| S6 | 通知（微信 / 电话） | 中控顶部横条 | 仪表（驾驶中电话） | 电话驾驶中需显眼 |

**5.5 H-F5 焦点管理**

**5.5.1 用户价值**

**做什么**：车上多块屏可能同时有信息显示，**但用户的注意力只有一份**。焦点管理就是**明确告诉用户"此刻该看哪里"**——驾驶时焦点永远在仪表 + HUD（路面优先），驻车影院时焦点在中控大屏，泊车时焦点在中控影像。任意时刻**只有一块屏是"主焦点"**，其他屏是辅助。不是屏幕越多越花哨越好，是用户始终知道"当前我在哪"。

**不做什么**：

* 不做"多屏同时抢注意力"（全闪烁 / 全高亮 → 用户不知道看哪）
* 不做"焦点随意跳"（A 屏闪一下 B 屏又闪一下 → 晕）
* 不做"用户手动管理焦点"（驾驶中没空切）
* 不做"隐式焦点"（用户不知道哪块屏是主焦点 → 无效）

**核心价值主张**：

* **单焦点原则**——同一时刻只有一块屏高亮 / 提示音 / 边框闪烁
* **焦点切换明确**——有视觉暗示（渐变 + 微震动 + 微音）告诉用户"焦点刚切了"
* **驾驶优先硬约束**——驾驶模式下仪表 / HUD 永远有焦点权；中控只能"辅助"

**5.6 H-F6 虚拟形象（V1 动态 + 6 状态 + 模板换肤）**

**5.6.1 用户价值**

**做什么**：座舱 AI 不再是冰冷的"语音 + 卡片"，而是有一个**会动的虚拟形象**——它有眼睛 / 嘴 / 表情，会**根据当前状态做出对应表现**：休眠时安静呼吸 / 被唤醒时睁眼面向用户 / 聆听时嘴部音波环绕 / 思考时眼神微转 / 任务失败时歪头皱眉 / 任务成功时点头微笑。**用户能"看见 AI 在想什么"**——这是把"无形语音助手"做成"有温度伙伴"的关键一步。**V1 同时开放模板换肤**（5-10 套官方形象：机器人 / 宠物 / 拟人 / 抽象），用户可以选自己喜欢的形象，但**不能上传任意图**（合规 / 内容审核风险大）。

**不做什么**：

* 不做"NOMI 屏的话痨吉祥物"（不需要持续吸引视觉注意力）
* 不做"V1 静态 icon"（推翻 5/2 的判断 → 动态形象是 V1 体验差异化的核心）
* 不做"V1 上 3D 写实形象"（成本高、效果不一定好；建议 2D Live2D 起步，由沈砚拍）
* 不做"用户上传任意图"（V1 只做模板换肤，AIGC / 自上传放 V2）
* 不做"驾驶中夸张动作"（驾驶安全优先，所有动画驾驶时简化）
* 不做"自动换形象"（用户主动选；不替用户做主）

**核心价值主张**：

* **6 状态闭环**：休眠 / 唤醒 / 聆听 / 思考 / 失败 / 成功——每个状态都有明确的视觉 + 音效
* **状态可连**：每个状态有进入条件 / 退出条件 / 兜底超时，不会卡死
* **模板换肤**：5-10 套官方形象 + 用户偏好记忆
* **驾驶简化**：驾驶模式下动画退化为表情 + 光效（不大幅动作）

**5.6.2 6 个核心状态定义**

|  |  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- | --- |
| # | 状态 | 进入条件 | 退出条件 | 视觉表现 | 音效 | 关键约束 |
| 1 | **休眠中** | 无唤醒 + 无主动服务 ≥ 60s | 唤醒触发 / 主动服务触发 | 闭眼 / 慢呼吸动画 / 低亮度 30% | 无 | 每次呼吸约 4s |
| 2 | **被唤醒时** | 唤醒词命中 / 视觉唤醒 | 切到聆听 / 超时 3s 切回休眠 | 睁眼 + 面向用户 + 高光 | 短促"叮" < 200ms | 切换动画 ≤ 300ms |
| 3 | **聆听中** | ASR 开始采集 | ASR 结束 | 嘴部 / 耳朵微动 + 周围音波环 | 无（避免干扰 ASR） | 音波环跟随实时音量 |
| 4 | **思考中** | NLU / LLM 调用中 | 调用返回 / 超时 5s | 眼睛微转 / 呼吸加快 / 光点旋转 | 无 | 超 5s 自动切失败状态 |
| 5 | **任务失败** | 工具调用失败 / NLU 不理解 | 致歉话术结束 + 用户响应 | 轻微歪头 / 皱眉 + 抱歉表情 | 柔和降调（< 300ms） | 不夸张；和 SOUL 的"承认失败 + 给备选"风格一致 |
| 6 | **执行成功** | 任务完成 | 1.5s 后切回休眠 | 点头 / 微笑 + 短暂亮光 | 清脆确认音 < 300ms | 不庆祝过度（避免 NOMI 那种话痨欢呼） |

**状态切换硬约束**：

* 切换动画 ≤ 300ms
* 任意状态都有**兜底超时**（最长 30s 强制回休眠）
* **驾驶模式下所有大幅动画退化为表情 + 光效**（避免分散视觉注意）
* 音效全局可关闭（部分用户不喜欢 AI 发声）
* 6 状态以外的"过场"用渐变补全（避免突跳）

| S12 | **形象首次设置**：新车主第一次开车 | 引导用户从 5 套模板里选 1 套 | 新手引导 |

**5.6.7 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 状态切换闪烁 | 状态机抖动 | 形象表情乱跳 | 防抖（300ms 内同方向不切）+ 渐变过渡 |
| 状态卡死 | 退出条件未触发 | 形象一直在某状态 | 兜底超时强制回休眠（30s） |
| 思考状态过长 | LLM 卡 | 用户以为 AI 死了 | 5s 超时切失败 + 致歉文案 |
| 切换动画卡顿 | 渲染瓶颈 | 形象 lag | 性能预算 + 低端车型简化 |
| 驾驶中动画过抢眼 | 简化逻辑 bug | 干扰司机 | 驾驶状态硬编码"简化模式" |
| 失败状态过夸张 | 美术 / 动画过度设计 | 用户感觉 AI 在卖惨 | 美术规范约束；不卖惨 |
| 成功状态过庆祝 | 美术 / 动画过度 | 用户觉得话痨 | 1.5s 自动收回；不持续庆祝 |
| 模板换肤后状态错位 | 各形象骨骼不一致 | 某些状态在某形象下没动画 | 6 状态动画模板统一；CI 测试每个形象 |
| 形象与人格不匹配 | 与 P-F1/P-F2/P-F3 没联动 | 形象长得软萌但说话很硬 | 形象选择影响 TTS 音色 + 话术风格（需打通） |
| 用户换形象时丢上下文 | 切换打断任务 | 任务被中断 | 切换不打断当前任务；只换皮肤 |
| 离线时形象失效 | 资源依赖云端 | 形象消失 | 形象资源端侧持久化 |
| 海外形象文化敏感 | 本地化不到位 | 用户反感 | 海外版本地化设计 + 区域审核 |

**5.6.8 技术钩子（给沈砚）**

1. **形象渲染技术选型**：3D（VRM）vs 2D（Live2D）vs Spine 骨骼动画。**V1 我倾向 2D Live2D**（成本 1/3、效果够），最终由沈砚拍。
2. **状态机架构**：6 状态 + 切换 + 防抖 + 兜底超时。状态机库（如 XState）vs 自研？
3. **动画模板抽象**：6 套状态动画统一模板；形象只提供骨骼 + 贴图 + 颜色。资产格式定义？
4. **状态切换防抖**：300ms 内同方向不重复切。算法？
5. **驾驶状态简化模式**：动画 LOD 分级。如何在驾驶状态下自动降级？
6. **音效系统**：6 状态对应音效；与 TTS（08）/ 关怀模式（D-F1）联动。优先级？
7. **形象资源端侧化**：换肤资源端侧持久化；离线可用。包大小预算？
8. **多形象切换不打断任务**：切换只换皮肤不动状态机。架构隔离？
9. **形象与人格 / TTS 联动**（与 P-F1/P-F2/P-F3 + 08）：形象切换后 TTS 音色 / 话术风格自动适配。打通？
10. **CI 测试**：每个形象 × 6 状态 × 切换组合的自动化测试。框架？
11. **OTA 模板下发**：新模板（如节日皮肤）通过 OTA 推。资源管理？
12. **A/B 测试**：动画时长 / 状态切换阈值 / 默认形象等。框架？
13. **性能预算**：单形象 CPU / GPU / 内存上限。低端车型降级方案？
14. **海外本地化**：部分形象在不同文化下含义不同。区域开关？
15. **可观测**：状态切换链路 / 切换耗时 / 卡死事件埋点。看板？

**5.6.9 与其他文档的交叉点（沈砚需关注）**

* **08 TTS**：形象的"嘴部动画"需要与 TTS 音频精确同步（口型 lip-sync）；状态音效不能与 TTS 同时响
* **D-F1~D-F7（05 主动服务）**：关怀触发时形象状态切换（无叮声 + 柔和动画）
* **D-F7（05 / 11 P-F1）**：人格切换是否同步换形象？还是同形象不同语气？建议**人格 ≠ 形象**——同一个形象可以承载 3 种人格语气
* **04 状态管理**：用户的形象选择跨会话持久化
* **13 S-F4**：自定义形象的内容合规审查
* **12 弱网降级**：离线时形象资源仍可用（端侧持久化）
* **I-F4（01 打断与恢复）**：用户打断 AI 说话时，形象立即切到"聆听"状态

**6. 自我反驳与边界**

**6.1 要不要做 NOMI 屏那种独立 AI 屏?**

**自反**:NOMI 屏是品牌记忆点,但增加 BOM 成本。**对冲方案**:V1 不做独立屏,用中控顶部"AI 状态条"替代(成本低,功能够)。V1.5 视用户反馈决定。

**6.2 卡片太多会不会让用户分心?**

**自反**:会。**对冲方案**:(1)驾驶模式限制卡片数量(最多 2 个)(2)卡片自动消失(3)用户可一键清空

**6.3 后排屏是刚需吗?**

**自反**:理想做了,但我们 V1 要不要?**对冲方案**:V1 不做后排屏(BOM 成本),V1.5 视家庭用户占比决定。
