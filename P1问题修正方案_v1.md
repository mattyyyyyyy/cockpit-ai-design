## 产品设计 P1（4条）

### P1-1：冷启动期（前14天）体验设计补全

**问题**：节律服务需14天，用户前两周感知AI和传统车机无异，缺乏留存钩子。

**修正：渐进式成长感知设计**

冷启动分三阶段，每阶段有明确的"里程碑时刻"：

| 阶段 | 时间 | 用户感知 | 实现方式 |
|------|------|---------|---------|
| **探索期** | 第1-3天 | AI在学习我 | 首次使用后：TTS"我会记住你的习惯，越用越懂你" |
| **积累期** | 第4-13天 | AI记住了具体的事 | 每记住一个偏好→轻提示："我记住了你喜欢听爵士乐" |
| **解锁期** | 第14天 | 惊喜感 | TTS+卡片："你已用车14天，智能节律已开启，试试说'今天怎么走？'" |

**积累期轻提示规则**：
- 每天最多1次，不在行驶中触发
- 仅在首次记住某偏好时触发（不重复提示同类偏好）
- 话术克制："记住了""学到了"，不说"我发现你""我注意到"

### P1-2：主动服务的用户认知教育

**问题**：AI第一次主动推荐时，用户不知道是"学到的"还是"广告"，产生不信任。

**修正：首次主动服务强制附带解释**

每类主动服务**首次触发时**，TTS末尾追加触发原因（后续同类触发不重复）：

| 服务类型 | 话术示例 |
|----------|---------|
| 节律驱动推荐 | "…（因为你每周一到周五这个时间都会路过这里，我帮你记住了）" |
| 情境驱动推荐 | "…（因为你的电量低于20%，附近有充电桩）" |
| 关怀服务 | "…（我注意到你开车超过3小时了）" |

**"学到"vs"广告"的视觉区分**（HMI卡片）：
- AI学习触发：卡片左上角显示"根据你的习惯" + 脑波小图标
- 系统推送（天气/路况）：显示"当前状况" + 信号图标
- 不显示"推荐"或"广告"字样

**用户可查触发记录**：设置→AI助手→主动服务记录，每条显示触发原因，可逐条关闭。

### P1-3：儿童模式细分与家长控制出口

**问题**：OMS检测到儿童直接触发过滤，无年龄细分，无临时关闭出口。

**修正：三层儿童模式架构**

**层1 - 年龄段配置（用户手动设置，影响内容过滤强度）**：

| 年龄段 | 内容过滤 | 音量上限 | 主动服务 |
|--------|---------|---------|---------|
| 婴幼儿（0-3岁） | 严格（仅儿歌/白噪音） | 50% | 关闭，只保留安全告警 |
| 儿童（4-12岁） | 中等（过滤成人/暴力） | 70% | 仅天气/目的地信息 |
| 青少年（13-17岁） | 轻度（过滤18+） | 100% | 全开 |
| 未配置 | 中等（默认） | 70% | 仅天气/目的地信息 |

**层2 - OMS自动检测触发**（检测到儿童 → 激活对应档位）：
- 账号已配置儿童年龄段 → 使用配置档位
- 账号未配置 → 使用"未配置"默认档位
- 置信度 <0.85 → 不触发，保持当前模式

**层3 - 家长临时控制出口**：
- 方案A：方向盘按键长按3s → 语音验证家长身份（声纹）→ 临时解除儿童模式（本次行程有效）
- 方案B：中控设置页→儿童模式→"临时关闭（本次行程）"

两套方案V1同时实现，方案A适合行驶中，方案B适合驻车时。

### P1-4：虚拟形象驾驶中简化的跳变问题

**问题**：驾驶/驻车模式切换时，虚拟形象硬切简化版会产生视觉跳变。

**修正：平滑过渡动画规范**

**触发阈值**（以最后一个满足条件的状态为准，防抖5s）：
- 进入驾驶简化：车速 >20km/h 持续 5s
- 退出驾驶简化：车速 <5km/h 持续 5s（或档位切换到P）

**过渡动画规范**：

| 方向 | 动画内容 | 时长 |
|------|---------|------|
| 全形象 → 驾驶简化 | 形象缩小+淡出至角落光点（ease-out） | 800ms |
| 驾驶简化 → 全形象 | 光点扩散+形象淡入（ease-in） | 600ms |

**驾驶简化模式定义**（比全形象轻量）：
- 只保留：环形光效（7种颜色对应6个状态）+ 音频可视化波形
- 关闭：骨骼动画、表情变化、嘴型同步
- 位置：从中控中央缩小至右上角状态条（占用面积 <5%）
- NPU节省：从 4 TOPS → 0.5 TOPS（释放算力给DMS）

## Harness Engineering P1（4条）

### P1-5：置信度阈值的模型版本漂移

**问题**：NLU模型OTA后置信度分布变化，硬编码阈值（0.7/0.9）会导致二次确认频率异常。

**修正：置信度校准（Calibration）机制**

**什么是置信度漂移**：新模型可能系统性地输出更高或更低的原始概率，同样的输入在旧模型输出0.85，新模型可能输出0.72——即使"确定程度"实际一样。

**解决方案：模型附带校准参数**

每次模型OTA同步下发校准配置：
```json
{
  "model_version": "nlu-v2.3.1",
  "calibration": {
    "type": "temperature_scaling",
    "temperature": 1.15,
    "comment": "该模型置信度系统性偏高，除以1.15校准"
  },
  "thresholds": {
    "low_confidence": 0.65,
    "high_confidence": 0.88
  }
}
```

校准在置信度输出后、阈值比较前执行：
```python
calibrated_score = raw_score / calibration.temperature
is_high_confidence = calibrated_score >= thresholds.high_confidence
```

**校准参数来源**：每次模型训练后，在内测集上运行ECE（Expected Calibration Error）评估，自动生成temperature参数，随模型OTA包下发。不用人工调。

**监控**：OTA后7天内监控"二次确认触发率"，若与OTA前相差 >20%，自动告警（校准参数可能有误）。

### P1-6：七大线束的优先级冲突仲裁

**问题**：安全红线/降级路径/人类监督等线束同时触发时，执行顺序未定义。

**修正：线束冲突仲裁表**

**优先级顺序（从高到低）**：

```
① 安全红线（Safety Guardrails）     ← 最高，不可被任何其他线束覆盖
② 失败隔离（Failure Isolation）     ← 防止故障扩散，比降级更紧迫
③ 降级路径（Degradation Path）      ← 保证可用性
④ 置信度阈值（Confidence Threshold）← 决策质量控制
⑤ 人类监督（Human-in-the-Loop）    ← 用户确认，可被①②③覆盖
⑥ 状态机（State Machine）           ← 流程管理，兜底
⑦ 可观测性（Observability）         ← 诊断，不影响执行逻辑
```

**典型冲突场景处理**：

| 冲突场景 | 执行逻辑 |
|---------|---------|
| 安全红线触发 + 人类监督待确认 | 安全红线优先：立即中断，取消待确认项，TTS播报安全告警 |
| 失败隔离（熔断）+ 降级路径 | 同时生效：熔断该Sub-bot的同时触发降级（两者不互斥） |
| 降级路径触发 + 置信度低需确认 | 降级路径优先：先用降级方案执行，置信度确认在降级方案内完成 |
| 多个Sub-bot同时熔断 | 按Sub-bot优先级：车控>导航>娱乐>通信>信息，高优先级先恢复 |

**实现**：Harness协调器（单独模块）在每次线束激活时查仲裁表，决策延迟 <1ms（内存查表）。

### P1-7：降级切换的用户感知时机

**问题**："云端超时3s→切L2"，用户等3s什么都看不到，然后突然切换。

**修正：分级感知时间线**

```
请求发出
  ↓ 800ms  → 无响应 → 显示"正在思考…"动效（HMI + TTS静默）
  ↓ 2000ms → 无响应 → 后台预启动L2路径（端侧模型预热）
  ↓ 2500ms → 无响应 → TTS轻提示："稍等一下"（用户感知到延迟，不焦虑）
  ↓ 3000ms → 超时   → L2结果无缝接替（用户感知"慢了点"，不是"卡住了"）
  ↓ 如果L2也超时（+3s）→ TTS："网络有点慢，用本地方案为你处理"
```

**HMI动效规范**：
- 0-800ms：无变化（快速请求不需要提示）
- 800ms-3s："思考中"波形动画（低调，不遮挡内容）
- 3s切换：L2结果直接替换，无切换动画（避免"二次等待感"）

**端侧模型预热时机**（2000ms时）：
- 在后台异步加载端侧NLU模型（若未常驻）
- 不影响当前云端请求（并行）
- 预热完成后等待超时切换信号

### P1-8：trace_id 跨端云传播规范

**问题**：端侧trace_id和云端LLM request_id是两个孤立系统，故障时无法串联。

**修正：统一trace传播协议**

**trace_id格式**（端侧生成）：
```
格式：{device_id_hash_8}-{session_id_8}-{sequence_4}-{timestamp_8}
示例：a3f8c1d2-9e7b4f3a-0042-6839fa21
```

**传播规则**：所有跨系统调用必须透传trace_id：

```
端侧 → 云端LLM：HTTP Header: X-Trace-Id: {trace_id}
云端LLM → 工具调用：透传 X-Trace-Id
工具返回 → 云端LLM → 端侧：Response Header: X-Trace-Id: {original}
```

**云端LLM接入要求**（Qwen/DeepSeek API）：
- 请求时携带：`{"extra_headers": {"X-Trace-Id": trace_id}}`
- 响应中要求返回：厂商request_id存入span，与trace_id关联
- 关联存储：`trace_span: {trace_id, llm_request_id, latency_ms, model, tokens}`

**端侧日志格式**（所有关键节点）：
```json
{"ts": 1715529600123, "trace_id": "a3f8c1d2-...", "span": "asr_end", "latency_ms": 148, "level": "INFO"}
```

**云端日志对齐**：Datadog/自研日志平台按trace_id聚合，一次请求全链路可在单一视图查看（端侧KWS→ASR→云端NLU→工具调用→TTS→播报完成）。

## Agent Engineering P1（3条）

### P1-9：Sub-bot 执行结果反馈 Mother-bot 协议

**问题**：Sub-bot执行失败后（目的地关闭/歌曲下架），Mother-bot如何感知并重新规划，协议未定义。

**修正：标准化Sub-bot响应格式**

所有Sub-bot响应必须遵循统一Schema：
```json
{
  "node_id": "nav_to_destination",
  "status": "FAILED",
  "error_code": "DESTINATION_CLOSED",
  "error_message": "目的地已关闭（来自高德API）",
  "is_retryable": false,
  "suggested_action": "ASK_USER_FOR_ALTERNATIVE",
  "partial_result": null,
  "metadata": {"closed_reason": "business_hours", "reopen_at": "2026-05-14T09:00:00"}
}
```

**Mother-bot处理逻辑（按error_code）**：

| error_code | Mother-bot行为 |
|-----------|--------------|
| TIMEOUT | 重试1次（不同备选工具），失败则降级 |
| DESTINATION_CLOSED | TTS询问用户："目的地已关门，要换一个吗？" |
| CONTENT_UNAVAILABLE | 自动换备选（歌曲下架→播相似歌曲，告知用户） |
| PERMISSION_DENIED | TTS："需要先授权XX功能，要现在设置吗？" |
| QUOTA_EXCEEDED | 切备选MCP，不告知用户 |
| NON_RETRYABLE | TTS播报error_message，不重试 |

**动态重规划触发**：
```python
if sub_bot_result.status == "FAILED" and not sub_bot_result.is_retryable:
    mother_bot.replan(
        failed_node=sub_bot_result.node_id,
        reason=sub_bot_result.error_code,
        user_context=current_context
    )
    # replan内部：更新DAG（跳过/替换失败节点）→ 继续执行
```

### P1-10：并行Agent调用上限与乱序处理

**问题**：多意图并行调用Sub-bot没有最大并发数限制，结果乱序返回没有处理规范。

**修正：并发控制 + 有序聚合**

**并发上限**：同时最多 **3个** Sub-bot并发（超过3个时，优先级低的等待）

Sub-bot优先级（并发排队用）：
```
车控 > 导航 > 通信 > 娱乐 > 信息/生活服务
```

**实现（asyncio Semaphore）**：
```python
MAX_CONCURRENT_SUBBOT = 3
semaphore = asyncio.Semaphore(MAX_CONCURRENT_SUBBOT)

async def call_subbot_with_limit(subbot, task):
    async with semaphore:
        return await subbot.execute(task)

# 并行调用，保留调用顺序
tasks_with_index = [(i, task) for i, task in enumerate(dag_tasks)]
results = await asyncio.gather(
    *[call_subbot_with_limit(subbot, task) for _, task in tasks_with_index],
    return_exceptions=True
)
# results顺序与tasks_with_index一致（asyncio.gather保证顺序）
```

**乱序问题**：`asyncio.gather` 本身保证返回顺序与输入顺序一致（即使并发执行顺序不同），不需要额外排序。

**超时聚合**：单个Sub-bot超时不阻塞其他：
```python
results = await asyncio.gather(*tasks, return_exceptions=True)
# TimeoutError视为FAILED，不影响其他Sub-bot的结果
```

**TTS播报顺序**：按DAG拓扑排序播报（依赖关系优先），不按完成时间播报（避免"先完成先说"导致逻辑混乱）。

### P1-11：LLM Context Window 截断策略

**问题**：复杂任务规划时工具返回结果可能超出2000 token限制，没有定义截断策略。

**修正：分层Context管理**

**Context预算分配**（总预算 4096 token，Qwen2.5-7B路由；72B规划 8192 token）：

| 内容 | 7B路由预算 | 72B规划预算 | 优先级 |
|------|-----------|-----------|--------|
| System Prompt（人格/规则） | 固定500 | 固定800 | 最高，不截断 |
| 当前用户输入 | 固定200 | 固定300 | 最高，不截断 |
| 最近3轮对话 | 最多600 | 最多900 | 高 |
| 工具调用结果 | 最多1200 | 最多4000 | 中 |
| 历史摘要 | 剩余空间 | 剩余空间 | 低，优先截断 |

**工具结果截断规则**（超出预算时）：
1. 单个工具结果 >500 token → 提取关键字段（status/key_data），丢弃verbose字段
2. 多个工具结果总量超限 → 保留最近调用的结果，截断最早的
3. 截断后在context中注明：`[工具结果已压缩，完整结果见trace_id:xxx]`

**工具结果关键字段提取**（示例）：
```python
def compress_tool_result(result: dict, budget: int) -> str:
    full_json = json.dumps(result)
    if len(full_json) / 4 <= budget:  # 粗略token估算
        return full_json
    # 提取关键字段
    key_fields = {k: result[k] for k in ["status", "data", "error"] if k in result}
    return json.dumps(key_fields) + f" [已压缩，原始{len(full_json)}字符]"
```

**历史摘要生成**（防止对话历史膨胀）：
- 每5轮对话 → 后台异步生成摘要（Qwen2.5-7B，约100 token）
- 摘要替换原始5轮，节省 ~80% context占用
- 生成延迟不阻塞当前请求（异步任务，下次请求时生效）
