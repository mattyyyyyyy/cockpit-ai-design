**02 - 技术可行性边界框架**

**文档版本**: v0.1**创建时间**: 2026-05-10**作者**: 沈砚(后端架构师 + Agent 编排)**目标**: 给出座舱 AI 重设计的技术可行性边界,回答"能做到什么程度"

**1. 延迟边界**

**1.1 物理极限(不可突破)**

|  |  |  |
| --- | --- | --- |
| 环节 | 物理极限 | 原因 |
| 声音传播(1m) | ~3ms | 声速 340m/s |
| 4G 网络 RTT | 50-100ms | 基站 + 核心网 |
| 5G 网络 RTT | 10-30ms | 低延迟但覆盖有限 |
| CAN 总线延迟 | 10-50ms | 车内总线协议 |
| 屏幕刷新延迟 | 16-33ms | 60Hz 屏幕 |

**1.2 工程可达目标**

|  |  |  |  |
| --- | --- | --- | --- |
| 场景 | 目标延迟 | 工程手段 | 达成难度 |
| 唤醒词检测 | <200ms | 端侧轻量模型 + 硬件加速 | ★☆☆ |
| 简单车控(开空调) | <1s | 端侧直连 CAN,无云端绕行 | ★☆☆ |
| 复杂意图(导航+音乐) | <3s | 并行 Agent + 流式返回 | ★★☆ |
| 多轮对话补全参数 | <5s | 预测式执行 + 端侧缓存 | ★★★ |
| 断网场景车控 | <500ms | 端侧状态机 + 本地模型 | ★★☆ |

**关键约束**:

* 网络延迟不可控: 4G 环境下云端往返至少 100ms,弱网可能 >1s
* 大模型推理慢: 云端 70B 模型推理 1-3s,端侧 3B 模型 <500ms
* 串行累加: 每多一个 Agent,延迟至少 +200ms

**优化方向**: 并行化 / 流式返回 / 预测式执行 / 端侧缓存

**1.3 不同场景的延迟预算**

**场景 1: 简单车控(开空调、开窗)**

|  |
| --- |
| Plain Text 唤醒(200ms) + ASR(300ms) + 意图识别(100ms) + CAN 指令(50ms) = 650ms |

可达性: ★★★ 高 / 关键: 端侧直连,不走云端

**场景 2: 导航(搜索 POI + 规划路线)**

|  |
| --- |
| Plain Text 唤醒(200ms) + ASR(500ms) + NLU(300ms) + POI 搜索(800ms) + 路线规划(500ms) = 2.3s |

可达性: ★★☆ 中 / 关键: POI 搜索和路线规划可并行

**场景 3: 复杂意图(边开车边导航边播音乐)**

|  |
| --- |
| Plain Text 唤醒(200ms) + ASR(500ms) + 意图分解(300ms) + 并行执行(max(车控 500ms, 导航 2s, 音乐 800ms)) = 3s |

可达性: ★★☆ 中 / 关键: 并行 + 流式反馈

**场景 4: 多轮对话(订餐、订票)**

|  |
| --- |
| Plain Text 第一轮(3s) + 用户补充(1s) + 第二轮(2s) + 确认(1s) = 7s |

可达性: ★☆☆ 低 / 关键: 预测式执行(高置信度时跳过确认)

**2. 准确率边界**

**2.1 ASR 准确率**

|  |  |  |  |
| --- | --- | --- | --- |
| 场景 | 目标准确率 | 工程可达 | 约束条件 |
| 安静环境、标准普通话 | >98% | ✓ | 云端大模型 |
| 车内噪音(60dB)、标准普通话 | >95% | ✓ | 降噪 + 云端 |
| 车内噪音 + 方言 | >90% | △ | 需方言模型 |
| 高速行驶(80dB+)、方言 | >85% | △ | 硬件降噪 + 多模型 |
| 多人同时说话 | >70% | ✗ | 声源定位难 |

**2.2 NLU 准确率**

|  |  |  |  |
| --- | --- | --- | --- |
| 场景 | 目标准确率 | 工程可达 | 约束条件 |
| 高频场景(开空调、播音乐) | >98% | ✓ | 端侧规则 + 模型 |
| 中频场景(导航、查天气) | >95% | ✓ | 云端大模型 |
| 长尾场景(复杂组合意图) | >90% | △ | 需大模型 + 多轮确认 |
| 模糊意图("有点热") | >80% | △ | 需上下文推理 |
| 歧义意图("打开") | >70% | ✗ | 需多轮澄清 |

**2.3 Agent 编排准确率**

|  |  |  |  |
| --- | --- | --- | --- |
| 场景 | 目标准确率 | 工程可达 | 约束条件 |
| 单一意图(开空调) | >99% | ✓ | 规则引擎 |
| 简单组合(导航+音乐) | >95% | ✓ | 并行 Agent |
| 复杂组合 | >90% | △ | 需状态机 + 优先级 |
| 动态调整(用户中途改) | >85% | △ | 需打断机制 |
| 多轮协商(订餐) | >80% | ✗ | 需复杂状态管理 |

**5. 安全边界**

**5.1 车控安全红线**

**不可妥协**:

* 紧急制动 / 转向 / 安全带 / 行驶中开门 → 绝不允许 AI 控制

**需要确认**:

* 车窗(防止雨天误开) / 后备箱

**可直接控制**:

* 空调 / 座椅 / 音乐 / 灯光

**5.2 隐私分级**

* **L1 极敏感**(不上云): 语音原始数据、精确位置
* **L2 敏感**(脱敏后上云): 导航历史、POI 搜索、用户画像
* **L3 一般**(可上云): 车辆状态、环境数据

**5.3 AI 安全**

* **幻觉**: 工具调用前校验,低置信度拒绝
* **对抗样本**: 输入过滤 + 异常检测
* **偏见**: 按人群统计识别率,多模型兜底

**6. 可扩展性边界**

**6.1 用户规模**

|  |  |  |
| --- | --- | --- |
| 用户规模 | 架构形态 | 关键挑战 |
| <10 万 | 单地域、单集群 | 成本优化 |
| 10-100 万 | 多地域、多集群 | 延迟优化 |
| 100-1000 万 | 全球部署、边缘计算 | 一致性、合规 |
| >1000 万 | 混合云、端侧为主 | 成本、可靠性 |

**6.2 功能扩展**

**MVP 能力**: 车控 / 导航 / 娱乐**未来扩展**: 多模态 / 主动服务 / 社交 / 生态

**7. 不可行的事(明确边界)**

**7.1 技术上不可行**

* 完全自然对话(理解所有隐喻、反讽、双关)
* 100% 准确率
* 零延迟
* 零成本

**7.2 安全上不可行**

* AI 控制刹车/转向
* 无确认的高风险操作(如行驶中开门)
* 语音原始数据上云

**7.3 成本上不可行**

* 所有场景都用 70B 大模型
* 所有请求都记全量日志
* 所有功能都端侧部署

**8. 给陆行的问题清单**

1. **核心场景**: 用户最高频的 10 个场景是什么?
2. **延迟容忍度**: 用户能接受多少延迟?(1s/3s/5s)
3. **准确率要求**: 用户能接受多少错误率?(5%/10%/20%)
4. **成本预算**: 单用户年成本预算是多少?(¥1/¥5/¥10)
5. **安全红线**: 哪些功能绝不允许 AI 控制?
6. **隐私策略**: 用户是否接受语音/位置上云?
7. **降级策略**: 断网时哪些功能必须可用?
8. **扩展优先级**: 未来 1 年内要扩展哪些功能?
