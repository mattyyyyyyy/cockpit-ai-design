# 座舱 AI 迭代计划

> 2026-05-17 | 逻辑：用户痛点 → 具体做法 → 达成效果

---

## 1. AI 记忆系统

**痛点**
用户说过的、做过的、AI 全不记得。每次上车都是陌生人。"老样子""老地方"完全不懂。

**四层记忆架构**

**① 工作记忆（单次会话内）**
Redis，30轮对话，1小时 TTL。存当前对话历史、已填充的槽位、进行中的任务上下文。跨 channel 自动桥接——用户从点餐切到导航时，目的地自动带过去。

**② 用户偏好（永久）**
MySQL `user_preferences` 表。记录用户的习惯和喜好：

| category | key | value | 来源 |
|---------|-----|-------|------|
| food | kfc_regular | 香辣鸡腿堡套餐中可 | 下单3次自动学习 |
| food | starbucks_regular | 美式中杯少冰 | 用户说"记住" |
| shopping | taobao_address | 张江高科园区18号 | 下单地址提取 |
| movie | seat_preference | 中间靠后 | 选座2次自动学习 |
| navigation | home | 浦东新区xx路xx号 | 用户说"记住" |
| navigation | company | 张江高科技园区 | 导航3次自动学习 |

来源规则：用户明确说"记住" → confidence=1.0；同一选择重复3次 → confidence=0.85；工具执行成功后自动抽取关键字段写入。

**③ 任务状态（跨会话）**
MySQL `task_states` 表。记录每个任务的完整生命周期：

| 字段 | 含义 |
|------|------|
| task_id | 唯一标识 |
| user_id | 用户 |
| tool_code | 305.22（星巴克）/ 305.30（肯德基）等 |
| status | pending / running / done / failed / cancelled |
| context_json | 任务上下文快照（填了什么槽位、选了什么商品） |
| created_at / updated_at | 时间戳 |

用途：用户说"上次那个订单怎么样了"→ 查 task_states；系统重启 → 扫描 status=running 恢复；用户中途打断 → 下次上车 TTS："您上次点了一半的肯德基订单，要继续吗？"

**④ 节律记忆（V2）**
按 `(user_id, weekday, hour_bucket)` 存储行为规律，驱动主动推荐。

**指代词解析**
识别到"老地方/上次那个/老样子"时，查偏好表和最近任务记录，把模糊词替换成具体实体再走 NLU。

**歧义消除（和指代词不同）**
匹配到多个候选时（两个张伟、两家肯德基），TTS 播报让用户选一个，不自动猜。

## 2. Agent 架构：闭环动作框架

参考 CLAW（Closed-Loop Action Worker）设计理念——Agent 不是一次性执行指令，而是一个闭环控制器：感知→规划→执行→观察→调整。

**Mother-bot + 分域 Sub-bot**

```
用户输入
  ↓
Mother-bot（调度层）
  ├─ 意图识别 + 路由
  ├─ 多意图拆分
  ├─ 上下文管理
  └─ 任务生命周期管理
  ↓
Sub-bot（按域分工）
  ├─ 车控 Sub-bot    → 车窗/空调/座椅（不走LLM，直通）
  ├─ 导航 Sub-bot    → 路线/搜索/到达
  ├─ 生活 Sub-bot    → 星巴克/KFC/外卖
  ├─ 购物 Sub-bot    → 淘宝/京东
  ├─ 娱乐 Sub-bot    → 电影/音乐/视频
  └─ 通用对话 Sub-bot → 闲聊/搜索/深度思考
```

每个 Sub-bot 内部跑闭环：Perceive→Plan→Act→Observe→Adjust，循环直到任务完成或用户取消。

**模糊指令**

| 用户说的 | 处理方式 |
|---------|---------|
| "帮我搞点吃的" | 根据时间+位置+偏好推断：午餐→推荐常点的肯德基或星巴克 |
| "来个老样子" | 查偏好表，匹配最近高频消费记录，确认后执行 |
| "随便来杯咖啡" | 从偏好表推荐或给选项 |

**多指令（一句话多个意图）**

| 用户说的 | 处理方式 |
|---------|---------|
| "开空调，然后帮我导航去公司" | 拆分两个意图：车控即时执行 + 导航可能需要确认路线 |
| "点个肯德基，然后导航过去" | 串行：先点餐 → 完成后自动把门店地址传给导航 |
| "买两张电影票，顺便点个星巴克" | 并行：购票和点咖啡同时进行 → 都完成后问要不要导航 |

Mother-bot 判断意图间的依赖关系——无依赖并行，有依赖串行。

**全双工（用户随时插话）**

| 场景 | 处理方式 |
|------|---------|
| AI 在念搜索结果，用户说"第二个" | 停止 TTS → 选择第二个 → 继续流程 |
| AI 在确认订单，用户说"等一下加个蛋挞" | 暂停确认 → 回到槽位填充 → 追加 → 重新确认 |
| AI 在播导航，用户说"帮我点杯咖啡" | 导航挂起 → 切到星巴克 Sub-bot → 点完后恢复导航 |

ASR 始终在线，检测到用户开口立即打断 TTS。Mother-bot 判断插话是"补充当前任务"还是"新任务"。

**指令纠错**

| 用户说的 | 处理方式 |
|---------|---------|
| "导航到虹桥…不对，是浦东机场" | 检测"不对" → 替换目的地槽位 |
| "两杯美式…不，一杯美式一杯拿铁" | 回滚已填充槽位 → 填入新内容 |

识别纠错关键词（"不对/不是/等等/我说错了/应该是"），触发槽位回滚或替换。

**指令补充**

| 用户说的 | 处理方式 |
|---------|---------|
| "帮我点杯美式"（停顿）"少冰的" | 合并为："美式少冰" |
| "导航到公司"（停顿）"走高速" | 追加路线偏好 |

用户停顿后继续说，如果新内容能匹配当前任务的空槽位，自动合并，不当成新指令。

**拒识**

| 用户说的 | 处理方式 |
|---------|---------|
| 噪音/非语言声 | ASR 置信度 <0.3 → 静默忽略 |
| "帮我买100瓶茅台" | 数量异常 → "您确定要买100瓶吗？" |
| 违规指令 | 安全分类器过滤 → 拒绝+引导 |

三层过滤：① ASR 置信度过低直接忽略 ② 安全分类器过滤危险/违规 ③ 业务规则过滤异常参数。

## 3. 新增业务场景

### 通用订购流程

肯德基、星巴克、电影票、淘宝这四个场景底层流程一样：**多轮槽位填充 + 确认 + 支付**，差异只在细节。

| 步骤 | 做什么 | 各场景差异 |
|------|-------|-----------|
| ① 意图识别 | NLU 识别"点肯德基/买东西/买电影票" | 触发词不同 |
| ② 槽位填充 | 逐步收集完成任务所需信息 | 槽位表不同 |
| ③ 偏好注入 | 从偏好表读取常用选项自动填充 | 减少问答轮次 |
| ④ 确认 | TTS 播报摘要，等用户说"确认" | 行驶中只播语音 |
| ⑤ 执行 | 调平台 API 下单/支付 | 不同第三方 API |

### 肯德基点餐（305.30）

**槽位表**

| 槽位 | 必填 | 偏好可填 |
|------|------|---------|
| 商品列表 | ✅ | ✅ "老样子"→香辣鸡腿堡套餐 |
| 门店 | ✅ | ✅ 最近门店自动选 |
| 取餐方式 | ✅ | ✅ 默认上次选择 |
| 收货地址（外卖时） | 外卖必填 | ✅ 偏好表读取 |
| 支付方式 | ✅ | ✅ 默认银联 |

**交互示例**
```
用户："帮我点个肯德基"
AI：（查偏好）"您上次点的香辣鸡腿堡套餐，这次还是一样吗？"
用户："对，再加个蛋挞"
AI："好的，香辣鸡腿堡加蛋挞，张江店自取，42元，确认下单吗？"
用户："确认"
AI："已下单，预计15分钟可取。10秒内说取消可撤销。"
```

### 淘宝购物（305.31）

**槽位表**

| 槽位 | 必填 | 偏好可填 |
|------|------|---------|
| 商品描述 | ✅ | ❌ 需要搜索 |
| 具体商品 | ✅ | ❌ 搜索后用户选 |
| 收货地址 | ✅ | ✅ 偏好表 |
| 支付方式 | ✅ | ✅ 默认 |

**交互示例**
```
用户："帮我在淘宝买个车载手机支架"
AI：（搜索）"找到3个：第一个XX磁吸款39.9元月销1万+，第二个XX夹式29.9元，第三个XX出风口款49.9元。要哪个？"
用户："第一个"
AI："XX磁吸支架39.9元，寄到张江高科园区18号，确认购买吗？"
用户："确认"
AI："已下单，预计后天到。10秒内说取消可撤销。"
```

行驶中最多播报3个选项（TTS 列太多记不住），驻车时屏幕显示商品卡片。

### 电影票（305.32）

**槽位表**

| 槽位 | 必填 | 偏好可填 |
|------|------|---------|
| 电影名 | ✅ | ❌ 需要用户说 |
| 影院 | ✅ | ✅ 最近影院 |
| 场次 | ✅ | ❌ 需要选 |
| 座位 | ✅ | ✅ 偏好"中间靠后" |
| 人数 | ✅ | ❌ 需要问 |

**交互示例**
```
用户："帮我买两张今晚的电影票"
AI："想看什么电影？"
用户："黑客帝国5"
AI："万达影城张江店，今晚19:30和21:00，选哪个？"
用户："早点那个"
AI："19:30场，8排12、13座中间靠后，两张96元，确认吗？"
用户："好的"
AI："已购票。要导航到万达影城吗？"
```

购票后自动衔接导航（跨场景联动）；座位偏好可学习；人数必须明确问。

## 4. 危险操作可以撤销

**痛点**
AI 执行了操作用户想取消，但来不及了。

**做法**
所有有副作用的操作执行完之后，TTS 播"已完成，10秒内说取消可以撤销"，5秒时再提醒一次。用户说"取消"就调 cancel 接口：
- 导航：停止导航
- 下单：取消订单
- 支付：发起退款

撤销窗口只靠语音，不弹屏幕。10秒超时后锁定。行驶中（车速 > 10km/h）500元以上支付直接拒绝。

## 5. 响应有反馈

**痛点**
用户说了话等3秒没动静，不知道 AI 在处理还是没听到，重说一遍导致两条一起执行。

**做法**
渐进式感知反馈：
- 0–800ms：不做提示
- 800ms 后：Live2D 切"思考"状态
- 2500ms 后：TTS 播"稍等一下"
- 3000ms 后：云端还没回来，切端侧简化版

幂等 Key：每次工具调用带唯一 ID，重复触发时相同 Key 不重复执行。

## 6. 工具出错时 AI 主动说出来

**痛点**
工具超时了 AI 打个日志，用户只看到转圈。

**做法**
- 工具超时：TTS 播"星巴克点单现在有点问题，要我帮您记下来稍后提醒吗？"
- 云端慢：TTS 播"网络慢了一点，我先帮您处理车内操作"
- 完全离线：TTS 播"现在网络不好，我只能控制车内设备"

按工具独立熔断：连续3次失败暂停30秒，不影响其他工具。

## 7. AI 主动推送

**痛点**
AI 完全被动，用户不说话它就没反应。

**规则推送（先上，确定性高）**

车机每10秒上报状态包 `{speed, soc, location, timestamp}`，后端跑规则检查：

| 触发条件 | 播报内容 | 用户可说 |
|---------|---------|---------|
| SOC < 20% + 附近有充电桩 | "电量偏低，附近有充电桩，要导航过去吗？" | 好的 / 不用 |
| 连续行驶 > 2h + 附近服务区 | "已连续行驶2小时，前方有服务区" | 需要 / 继续走 |
| 进入常用停车场附近 | "快到了，要帮您找停车位吗？" | 要 / 不用 |

**跨场景联动推荐**

| 触发事件 | 联动推荐 |
|---------|---------|
| 刚买了电影票 | "要导航到影院吗？路上顺便点杯咖啡？" |
| 导航到商圈附近 | "附近有您常去的肯德基，要提前点餐吗？" |
| 周五晚上出发 | "今晚有几部新电影上映，要看看吗？" |

**防打扰**：每次出行最多主动推2条；行驶中只播 TTS 不弹卡片；同类被拒2次后静默30天。

**偏好驱动推荐（P1记忆做完后）**
- 常在工作日早上点咖啡 → 出发时问"要提前下单美式吗？"
- 每周三接孩子 → 周三下午问"要导航去学校吗？"

**节律学习（V2）**
AI 从历史行为自动发现规律，不需要规则和用户设置。
