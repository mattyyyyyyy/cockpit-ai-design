**§1 背景与目标**

**1.1 为什么做 NLU 与意图理解**

**核心问题**：ASR 把声音变成文字,但"打开空调 22 度"和"空调调冷一点"对系统来说是同一件事——这层"理解"是座舱 AI 从"语音遥控器"升级为"智能助手"的关键鸿沟。

三大矛盾：

1. **表达多样性**：同一意图有无数种说法("太热了"/"降温"/"空调开大点"/"我快烤熟了"),传统规则匹配覆盖率 <60%
2. **上下文依赖**："调大一点"必须结合上文(音量?温度?风量?)和视线(看哪个面板?),脱离上下文无法理解
3. **指代歧义**："把那个关掉"——"那个"是什么?需要视觉+对话历史+屏幕状态联合消解

**AI 时代的机会**：

* **大模型理解**：LLM 的泛化能力让"太热了"→ 降温意图的识别不再依赖人工规则
* **上下文建模**：多轮对话状态追踪(DST)让"调大一点"有据可依
* **多模态消解**：视线/屏幕焦点/手势信息辅助指代消解

**目标**：

* **主目标**：意图识别覆盖率 >95%(含非标准表达),Top 1 准确率 >90%
* **多轮目标**：3 轮内指代消解成功率 >85%
* **延迟目标**：端侧 NLU 推理 <300ms,云端 <1.5s

**§2 用户价值**

**2.1 核心价值主张**

**对驾驶员**：

* **不用背指令**：说人话就行——"我冷了"比"空调温度上调两度"自然 10 倍
* **不用重复说**：连续指令省略上下文("导航去公司"→"不走高速"→"避开学校"),系统记住上文
* **不用精确表达**：模糊意图也能理解("有点吵"→ 降低音量/关窗,系统主动确认)

**对乘员**：

* **跨域理解**：一句话跨多个服务("导航去最近的商场顺便订个位")不再被拆成两个独立指令

**2.2 反价值(不做什么)**

* **不做过度推理**："我饿了"→ 推荐餐厅是合理的,但"我饿了"→ 自动下单外卖是越界
* **不做无确认执行**：高代价动作(支付/发消息/改导航终点)必须确认,不能"理解了就直接做"
* **不做方言歧视**：方言/口音用户不应获得更低质量的理解——端侧必须覆盖主流方言

**3.2 边界场景**

|  |  |  |
| --- | --- | --- |
| 场景 | 挑战 | 处理策略 |
| **ASR 识别错误传入 NLU** | "开空调"被识别为"开空掉" | 意图层面的容错:基于上下文纠偏,或回溯 ASR 候选 |
| **多意图冲突** | "开窗"和"开空调"同时出现在一句话 | 分别识别 + 分别确认执行条件(如开窗时自动关空调?) |
| **儿童/噪音伪指令** | 非目标语音被 ASR 传入 | NLU 置信度门控:<0.5 直接拒识 |
| **极端长句** | "帮我导航去机场不走高速顺便找个加油站还要避开学校" | 拆解为子意图序列,逐个确认 |

**3.3 非场景(明确不做)**

* **对话闲聊**：不做"你觉得人生的意义是什么"——NLU 只处理可执行的意图
* **情感理解**：不做"你听起来很生气"——情绪感知归 OMS,不归 NLU
* **跨用户意图串联**：不做把副驾和后排的话拼成一个指令

**§4 非功能需求、SLO 与取舍**

**4.1 性能指标(SLO)**

|  |  |  |  |
| --- | --- | --- | --- |
| 指标 | 目标值 | 测量方法 | 降级阈值 |
| **意图 Top 1 准确率** | >90% | 标注集评估 | <80% 触发规则兜底 |
| **意图覆盖率** | >95% | 非标准表达识别率 | <85% 提示用户"换个说法" |
| **槽位填充准确率** | >92% | 实体抽取 F1 | <80% 主动追问缺失槽位 |
| **指代消解成功率** | >85%(3 轮内) | 多轮对话评测集 | <70% 放弃消解,直接追问 |
| **NLU 推理延迟** | <300ms(端侧)/<1.5s(云端) | E2E 延迟 | 端侧 >500ms/云端 >3s 降级 |

**4.2 可靠性与降级**

1. **云端不可达**：端侧小模型兜底(覆盖 Top 200 高频意图),复杂/长尾意图提示"暂不支持,稍后重试"
2. **模型置信度低**(<0.5)：拒识 + 提示用户重说或换说法
3. **多轮状态丢失**：降级为单轮模式,每轮独立理解,不做上下文关联

**4.3 安全与合规**

* **高代价意图确认**：支付/发消息/删除数据类意图,必须二次确认,不可自动执行
* **隐私边界**：对话内容仅用于当前意图理解,不存储原文(仅存结构化意图+槽位)
* **拒识日志**：所有拒识事件记录意图+置信度,用于模型迭代,但不记录原始语音

**4.4 关键取舍**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 取舍点 | 选项 A | 选项 B | 决策 | 理由 |
| **模型架构** | 纯规则(确定性高) | 纯大模型(泛化好) | **规则兜底 + 大模型主力** | 规则保底线,大模型扩上限 |
| **意图粒度** | 粗粒度(10 域) | 细粒度(100+ 意图) | **粗粒度域 + 细粒度意图** | 域级别规则可管,意图级别模型泛化 |
| **多轮深度** | 无限轮 | 限制 5 轮 | **限制 10 轮,超限重新开始** | 防止状态累积错误 |
| **置信度策略** | 低置信直接拒识 | 低置信也执行+确认 | **分档:高直接执行/中确认/低拒识** | 体验与安全平衡 |

**3. 核心能力要求**

**3.1 意图理解**

1. **垂域意图分类** — 车控/导航/音乐/空调/充电/通讯 等 15-20 个核心域
2. **跨域意图识别** — "带杯咖啡去公司"要识别为"导航公司 + 顺路咖啡"跨域意图
3. **长句多意图拆解** — C1 嘉兴行程这种长句,NLU 要拆成 5 个子意图并排序
4. **泛化能力** — 同一意图的不同表达(如"我冷""空调太低""调高点温度")都能识别

**3.2 槽位填充**

1. **标准槽位** — 地点/时间/联系人/金额/品牌 等
2. **模糊槽位** — "找个地方吃饭"的"地方"用上下文+偏好推断
3. **多值槽位** — "给妈妈和爸爸发消息"的联系人多值
4. **缺失槽位回填** — "导航"未给地点 → 反问"去哪里"

**3.3 指代消解**

1. **语言指代** — "那个""这个""上次那家"
2. **视觉指代** — 结合视线追踪("把那个调小"视线指向音乐卡片)
3. **时间指代** — "昨天""刚才""上个礼拜"
4. **对话指代** — "也给我一份"(承接上文推荐)

**3.4 多轮理解**

1. **上下文保持** — 10+ 轮对话内连续上下文
2. **任务切换** — 用户从 A 任务切到 B,旧上下文不污染新任务
3. **主题跟踪** — 识别"还在聊同一件事"还是"换话题了"

**5. 功能点展开**

**5.1 N-F1 垂域意图分类**

*待填*

**5.1.1 一句话定义**

**做什么**：把用户一句话归到 **15-20 个核心垂域** 中的一个（车控/导航/音乐/空调/充电/通讯/天气/日程/娱乐/购物/外卖/知识问答/闲聊/设置/多媒体/出行服务/车辆服务/个性化），输出 domain + intent + 置信度，给后续槽位填充和 Agent 编排用。

**不做什么**：

* 不做"跨域意图识别"（那是 N-F2 的事：一句话含两个域）
* 不做"意图执行"（执行是 Agent 编排 / 文档 06 的事）
* 不做"闲聊回复生成"（闲聊识别出来扔给 LLM，NLU 只负责判归类）

**5.1.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期意图识别 |
| S1 | **导航基础** | "导航去浦东机场" → domain=导航, intent=start\_navigation, slot.destination=浦东机场, 置信度 98% |
| S2 | **车控基础** | "打开空调" → domain=空调, intent=turn\_on, 置信度 97% |
| S3 | **模糊表达**（同义泛化） | "我有点冷" → domain=空调, intent=raise\_temperature, 置信度 85%（不是字面的"冷"，是要调温） |
| S4 | **同词多域**（歧义） | "周杰伦" → 可能是 音乐.play\_artist 或 知识问答.query\_person，上下文无暗示时**默认音乐**（频次更高）+ 次选知识问答作为备选 |
| S5 | **闲聊识别** | "你好烦" → domain=闲聊, intent=emotion\_complaint, 置信度 92% → 扔给 LLM 生成共情回复，不走 Agent 执行 |
| S6 | **低置信度兜底** | "那个那个你懂我意思吗" → 任何域置信度 < 60% → 反问"您是想……"或建议用户重说 |

**5.1.3 输入 / 触发条件**

* **输入**：I-F2 ASR 输出的文本 + I-F5 音区信息（主驾/副驾/后排）+ 上下文（最近 5 轮意图）
* **触发**：ASR 输出稳定文本（VAD 结束或 3s 静音）
* **前置条件**：NLU 模型已加载；意图域表已更新到最新版本

**5.1.4 执行流程**

|  |
| --- |
| Plain Text ASR 文本输入  ↓ 文本预处理（分词 + 归一化 + 标点修复）  ↓ 意图分类器（两路并行）  ├─ 端侧小模型（高频垂域 TOP10，毫秒级）：车控/空调/音乐/导航/通讯/天气/日程/设置/娱乐/多媒体  └─ 云端大模型（兜底复杂/低频域 + 泛化）：购物/外卖/知识问答/闲聊/出行服务/车辆服务/个性化 + 端侧兜不住的  ↓ 端侧结果置信度 ≥ 85% → 直接用端侧（延迟 ≤ 50ms） 端侧置信度 60-85% → 等云端结果融合（延迟 ≤ 400ms），取高置信度 端侧 < 60% 或无 → 完全依赖云端（延迟 ≤ 500ms）  ↓ 同词多域歧义消解  ├─ 有明确上下文（刚聊过音乐/导航）→ 继承上下文域  ├─ 无上下文 → 频次优先 + 次选备选  └─ 仍不确定 → 反问  ↓ 输出 domain + intent + 置信度 → 送槽位填充（N-F3） |

**关键时间预算**：

* 端侧高频域意图识别延迟：**≤ 50ms**（P95）
* 云端低频/泛化意图识别延迟：**≤ 500ms**（P95）
* 端云融合总延迟（端侧低置信度）：**≤ 400ms**（P95）

**5.1.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **高频域误分类**（"打开空调"识别成"打开音乐"） | 用户信任崩溃，再也不敢用 | 高频域训练样本 10 万+，端侧模型准确率 ≥ 97% 是硬线；误分类样本实时回流训练 |
| **同词多域歧义判错**（"周杰伦"场景错归类） | 用户意图被扭曲 | 上下文优先 + 频次兜底 + 次选备选；低置信度反问不硬判 |
| **云端弱网/超时，低频域无响应** | 用户说"帮我点份外卖"没反应 | 云端超时 ≤ 1.5s → 降级到端侧兜底（返回"抱歉现在网络不稳，请您手动操作"）；不让用户干等 |
| **新域增加慢**（市场推出新功能但 NLU 不认） | 新功能用户不会用 | 意图域表热更新机制（不发版就能加新域）；新域上线先云端后端侧 |
| **方言/非标准表达识别率低** | 南方用户普通话不标准被误判 | 训练集覆盖主要方言口音；端侧小模型对高频方言句式做增强 |

**Paper Tiger（看起来像风险但不是）**

* "15-20 个域不够用"—— 实证：华为/理想 20 个域覆盖 95% 场景；剩下 5% 走闲聊+LLM 兜底
* "端侧小模型准确率打不过云端大模型"—— 高频域（车控/空调/导航）模式固定，端侧小模型完全够用（实测 ≥ 97%）；云端只需兜复杂的

**Elephant（团队可能没讨论够）**

* **意图域表的"治理权"**：谁决定新增/删除域？产品 or 工程？怎么避免域表膨胀到 50+ 个变成维护噩梦？**建议**：产品定义域的边界，工程决定实现路径；新域必须过"月活 >10%"或"场景关键"两个门槛才加
* **闲聊域的"诱捕"边界**：用户说"今天天气不错" → 算 天气.query\_weather 还是 闲聊.small\_talk？**边界决策：含明确查询动作才走垂域，否则走闲聊**
* **多语言/方言的长期投入**：V1 普通话 + 四川/广东口音；出海时的多语言（英文/日文）成本极高，产品侧要提前规划
* **意图分类的"可观测"**：线上用户每句话的意图分类日志要不要存？**存下来训练迭代必需，但隐私边界需法务**

**5.1.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |

**5.2 N-F2 跨域意图识别**

*待填*

**5.2.1 一句话定义**

**做什么**：识别用户一句话里隐含的**多个垂域意图**或**跨 App 组合意图**，拆成结构化的"主意图 + 副意图"或"意图序列"，让 Agent 编排知道该同时/依次调用哪些服务。

**不做什么**：

* 不做"单域意图分类"（N-F1 的事）
* 不做"意图执行"（Agent 编排 / 文档 06）
* 不做"长句时序解析"（那是 A-F1 / §5.8 的事，跨域意图是单句）

**5.2.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期意图识别 |
| S1 | **带杯咖啡去公司**（导航+外卖组合） | "带杯咖啡去公司" → 主意图=导航.公司 + 副意图=外卖/顺路咖啡店，标记 时序=并行，顺路优先 |
| S2 | **订座餐厅+导航** | "帮我订今晚 7 点那家湘菜馆的位置再导航过去" → 意图1=餐厅预订 + 意图2=导航(依赖意图1 结果)，标记 时序=串行，前置依赖 |
| S3 | **通话+音量** | "打电话给老婆，声音调小点" → 意图1=通讯.call(对象=老婆) + 意图2=音量.调低，标记 时序=并行，同时执行 |
| S4 | **查天气+设闹钟** | "明天会下雨吗，帮我设个 7 点的闹钟" → 意图1=天气.查询(明天) + 意图2=闹钟.设置(7:00)，时序=并行 |
| S5 | **歧义边界**（看起来跨域其实单域） | "导航去那家刚才评价高的咖啡馆" → 这是**单域（导航）+指代（咖啡馆）**，不是跨域；NLU 需要判别 |
| S6 | **失败场景** | "帮我搞一下那个事情" → 跨域置信度低 → 反问"您是想订外卖、导航还是别的？" |

**5.2.3 输入 / 触发条件**

* **输入**：I-F2 ASR 文本 + N-F1 单域意图分类结果（如果单域置信度已 > 90% 则跳过跨域识别）+ 上下文
* **触发**：单域 NLU 检测到句子含两个动词/两个名词域，或含典型连接词（"再"、"然后"、"并且"、"顺便"、"同时"）
* **前置条件**：单域 NLU 已输出；跨域模型可用

**5.2.4 执行流程**

|  |
| --- |
| Plain Text 单域 NLU 输出  ├─ 单域置信度 ≥ 90%，句子无连接词 → 跳过跨域识别  └─ 含连接词 OR 单域置信度模糊 → 启动跨域识别  ↓ 跨域识别（云端 LLM 为主）  ├─ 拆句：分词 + 句法依存分析（识别多个谓语）  ├─ 域匹配：每个子谓语映射到垂域  └─ 关系识别：并行 / 串行 / 依赖 / 顺路  ↓ 时序结构输出  ├─ 主意图（核心诉求）  ├─ 副意图（附加诉求）  └─ 执行关系（并行 / 串行 / 前置依赖）  ↓ 送给 Agent 编排执行（文档 06）  └─ Agent 编排决定谁先谁后、谁并发 |

**关键时间预算**：

* 跨域识别延迟：**≤ 800ms**（P95）（云端 LLM 调用）
* 端云融合兜底：端侧已有单域结果，跨域识别失败也不会让用户白等

|  |  |  |

**5.3 N-F3 槽位填充与回填**

*待填*

**5.3.1 一句话定义**

**做什么**：从用户语句中提取**结构化参数**（槽位 slot），填入意图模板，形成可执行的指令；当槽位缺失时，**主动反问**补齐（回填）。

**不做什么**：

* 不做"意图分类"（N-F1/N-F2 的事）
* 不做"指令执行"（Agent 编排 / 文档 06）
* 不做"槽位验证"（比如"温度 100°C"不合理 → 那是 Agent 编排层的业务逻辑校验）

**5.3.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期槽位填充 |
| S1 | **完整槽位** | "导航去浦东机场 T2 航站楼" → intent=导航.start, slot.destination=浦东机场T2, 完整度 100% → 直接执行 |
| S2 | **缺槽位回填** | "导航去机场" → slot.destination=机场（模糊） → TTS 反问"您是去浦东机场还是虹桥机场？" → 用户"浦东" → 回填 slot.destination=浦东机场 |
| S3 | **多槽位** | "把空调调到 22 度自动模式" → intent=空调.set, slot.temperature=22, slot.mode=auto |
| S4 | **隐式槽位**（需推理） | "我有点冷" → intent=空调.raise\_temp, slot.delta=+2°C（隐式推理） |
| S5 | **槽位纠正** | "导航去浦东机场" → 系统开始导航 → 用户"不对，是虹桥" → slot.destination 被覆盖为虹桥机场 |
| S6 | **槽位冲突** | "把温度调到 22 度，不对 24 度" → 后者覆盖前者，slot.temperature=24 |

**5.3.3 输入 / 触发条件**

* **输入**：N-F1/N-F2 意图分类结果 + ASR 文本 + 上下文（前 3 轮槽位）
* **触发**：意图分类完成，进入槽位填充阶段
* **前置条件**：意图模板已定义（每个 intent 需要哪些 slot + 必填/可选）

**5.3.4 执行流程**

|  |
| --- |
| Plain Text 意图分类结果 + ASR 文本  ↓ 槽位提取（NER + 规则 + LLM）  ├─ 端侧 NER（高频槽位：地点/时间/温度/音量/联系人）  ├─ 规则引擎（数字/单位/枚举值）  └─ 云端 LLM（复杂槽位：模糊地点/隐式意图）  ↓ 槽位完整度检查  ├─ 必填槽位全满 → 直接执行  ├─ 必填槽位缺失 → 回填流程  └─ 可选槽位缺失 → 用默认值  ↓ 回填流程（缺槽位时）  ├─ 生成反问（TTS）："您是去浦东机场还是虹桥机场？"  ├─ 等待用户回答（ASR）  ├─ 提取回答中的槽位值  └─ 回填到原意图 → 重新检查完整度  ↓ 槽位验证（基础）  ├─ 类型检查（温度是数字、地点是字符串）  ├─ 范围检查（温度 16-32°C）  └─ 冲突检查（同一槽位多次赋值取最后）  ↓ 输出完整意图+槽位 → Agent 编排执行 |

**关键时间预算**：

* 槽位提取延迟：**≤ 100ms**（端侧 NER + 规则）
* 云端 LLM 槽位提取：**≤ 500ms**（复杂槽位）
* 回填一轮延迟：**≤ 3s**（TTS 反问 + 用户回答 + 槽位提取）

**5.3.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **槽位提取错**（"浦东机场"识别成"浦东"） | 导航到错误地点 | 高频槽位（地点/联系人）训练样本 10 万+；槽位提取准确率 ≥ 95% 是硬线 |
| **回填死循环**（反问 3 次用户还是没说清楚） | 用户烦躁放弃 | 回填最多 2 轮；第 3 轮直接"抱歉我没理解，请您手动操作" |
| **隐式槽位推理错**（"我有点冷" → 调到 30°C） | 用户不舒服 | 隐式槽位保守推理（+2°C 而非 +10°C）；推理结果 TTS 确认"已调高 2 度" |
| **槽位冲突处理错**（"22 度不对 24 度" → 还是 22） | 用户意图被忽略 | 同一槽位多次赋值，**后者覆盖前者**（时间戳优先） |
| **回填反问不清晰**（"您要去哪？" 太宽泛） | 用户不知道怎么回答 | 反问带选项："您是去浦东机场还是虹桥机场？"（≤ 3 个选项） |

**Paper Tiger（看起来像风险但不是）**

* "槽位模板维护成本高"—— 实证：20 个垂域 × 平均 5 个 intent × 平均 3 个 slot = 300 个槽位定义，工程可控

**5.4 N-F4 语言指代消解**

**5.4.1 一句话定义**

**做什么**：在**纯语言层**（无视觉信号辅助）解析用户语句中的**代词和时间词**指代的具体对象。包括：代词（那个/这个/它/他）、时间词（刚才/昨天/上次）、对话回指（"也给我一份"）。

**不做什么**：

* 不做"视觉指代消解"（那是 N-F5 + 01 文档 I-F6 的事）
* 不做"槽位填充"（N-F3 的事）
* 不做"跨轮意图跟踪"（那是 N-F6 多轮上下文的事）

**N-F4 和 01 文档 I-F6 的边界**：

* **I-F6 (01 层)**：多模态**输入融合** —— 视线 + 手势 + 上下文三路加权得到候选实体 ID
* **N-F4 (03 层)**：**纯语言层**指代消解 —— 只靠对话历史和时间词解析（视线不可用时的兜底，也是大多数场景的主力）

**5.4.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期消解 |
| S1 | **对话回指** | 上一轮"附近有什么咖啡店" → 系统返回 3 家 → 用户"去第一家" → 第一家 = 星巴克陆家嘴店（对话记忆第 1 个候选） |
| S2 | **时间指代** | "回刚才那条路" → 刚才 = 5 分钟前的导航路径（时间窗推理） → 调用历史路径 |
| S3 | **代词延续** | "导航去陆家嘴星巴克" → 成功 → 用户"它关门了吗" → 它 = 陆家嘴星巴克（槽位回指） → 查店铺营业状态 |
| S4 | **多候选歧义** | "附近有星巴克和肯德基" → 用户"带我去那家" → 那家 有歧义 → TTS"您是去星巴克还是肯德基？" |

**5.4.3 输入 / 触发条件**

* **输入**：ASR 文本 + 对话历史（最近 5 轮的意图 + 槽位 + 返回结果）+ 当前时间戳
* **触发**：ASR 文本含代词（那个/这个/它/他/她/那家/这家）或时间词（刚才/昨天/上次/前面）
* **前置条件**：对话上下文可用；指代消解模型已加载

**5.4.4 执行流程**

|  |
| --- |
| Plain Text ASR 文本 → 指代检测（代词/时间词）  ├─ 无指代词 → 跳过  └─ 含指代词 → 启动消解  ↓ 候选实体收集（对话记忆）  ├─ 最近 5 轮的槽位值（高权重）  ├─ 最近 5 轮的返回结果（如搜索结果列表）  ├─ 长期画像实体（家人/常用地点）  └─ 时间窗事件（"刚才"= 5 分钟内）  ↓ 候选打分（多因素加权）  ├─ 实体类型匹配：人/地点/物（"它"→ 物体，"他/她"→ 人）权重 0.3  ├─ 时间距离：越近权重越高（指数衰减）权重 0.3  ├─ 槽位位置：上一轮槽位最优先 权重 0.2  └─ 上下文连贯性：主题一致性 权重 0.2  ↓ Top-1 置信度判定  ├─ ≥ 85% → 直接消解  ├─ 60-85% → TTS 确认"您说的是 X 吗？"  └─ < 60% → TTS 澄清"您是指 X 还是 Y？"  ↓ 输出消解后的实体 → 送 N-F3 槽位填充 |

**关键时间预算**：

* 指代消解延迟：**≤ 150ms**（端侧+对话记忆查询）

**5.5 N-F5 视觉指代消解**

*待填*

**5.5.1 一句话定义**

**做什么**：接收 01 层 I-F6（多模态输入融合）送来的**视觉候选实体**（视线命中的控件/车外 POI + 手势指向的对象），在 NLU 层将其与用户语句中的代词**绑定成结构化指代结果**，送给槽位填充或 Agent 编排执行。

**不做什么**：

* 不做"视觉识别"（那是 01 层 DMS/OMS 和视觉感知的事）
* 不做"视线命中计算"（01 层 I-F6 已完成）
* 不做"纯语言指代"（N-F4 的事）

**N-F5 的本质**：把视觉感知结果**翻译成 NLU 能用的槽位值**，并决定"是否采纳视觉候选 vs 语言上下文候选"。

**5.5.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期消解 |
| S1 | **屏幕控件**（主场景） | I-F6 输出「视线命中=音乐卡片, 手势指向=音乐卡片」+ 用户"这首别放了" → N-F5 绑定 那个/这首 = 当前音乐 → 送 intent=音乐.stop |
| S2 | **车外 POI** | I-F6 输出「视线朝车右前方, 候选=金茂大厦」+ 用户"那是什么" → N-F5 绑定 那 = 金茂大厦 → 送 intent=知识问答.query\_place(金茂大厦) |
| S3 | **视觉+语言冲突**（难点） | I-F6 输出「视线命中=导航卡片」 + N-F4 输出「对话回指=刚才的咖啡店」 + 用户"它还开着吗" → **语言候选优先**（对话连贯性）+ TTS 确认"您说的是刚才那家咖啡店吗？" |
| S4 | **视觉信号低置信度** | I-F6 输出「视线命中置信度=55%」 + 用户"把那个调小" → N-F5 降级，等 N-F4 补位；若都失败则反问 |
| S5 | **多屏指代**（HUD+中控+仪表） | 视线命中 HUD 导航卡片 + 用户"关掉" → 关 HUD 导航提示（不是关导航本身）→ N-F5 需要知道"屏幕区域" → "功能" 的映射 |
| S6 | **失败场景** | I-F6 无输出（视觉不可用）+ N-F4 也无上下文 → 降级纯反问 |

**5.5.3 输入 / 触发条件**

* **输入**：I-F6（01 层）的视觉候选实体列表 [{entity\_id, type, confidence, source=视线/手势}] + ASR 文本（含代词）+ N-F4 的语言候选列表
* **触发**：ASR 文本含代词 + I-F6 有视觉候选输出
* **前置条件**：I-F6 接口已打通；视觉候选的 entity\_id 能映射到业务实体

**5.5.4 执行流程**

|  |
| --- |
| Plain Text ASR 代词检测 + I-F6 视觉候选 + N-F4 语言候选  ↓ 候选合并（视觉候选 + 语言候选）  ├─ 视觉候选：entity\_id, type, 置信度, 来源=视线/手势  └─ 语言候选：entity\_id, type, 置信度, 来源=对话历史  ↓ 仲裁策略（场景化优先级）  ├─ 屏幕控件指代（代词+动词是屏幕控件操作）→ 视觉优先  ├─ 车外 POI 指代（"那是什么"+视线朝车外）→ 视觉优先  ├─ 对话延续指代（槽位已确定，用户用"它"延续）→ 语言优先  ├─ 冲突（视觉和语言都高置信度）→ TTS 确认  └─ 都不够高 → 反问或放弃  ↓ Top-1 置信度判定  ├─ ≥ 85% → 绑定 + 送槽位填充  ├─ 60-85% → TTS 确认  └─ < 60% → 反问  ↓ 输出结构化 entity → 送 N-F3 槽位填充 |

**关键时间预算**：

* N-F5 消解延迟：**≤ 100ms**（候选合并 + 仲裁，端侧）
* 总链路（I-F6 + N-F4 + N-F5）：**≤ 400ms**（P95）

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |

**5.6.1 一句话定义**

**做什么**：维护**用户对话状态**（当前话题、当前意图、待填槽位、历史槽位、未完成任务），在连续多轮对话中**保持理解连贯**，并在用户切换话题时**干净切换**不污染新任务。

**不做什么**：

* 不做"长期记忆"（那是文档 04 状态管理·记忆 的事，比如"老婆喜欢喝拿铁"这种长期画像）
* 不做"意图分类"（N-F1 的事，但多轮理解依赖意图分类结果）
* 不做"任务状态跟踪"（Agent 编排 / 文档 06 的事）

**N-F6 的本质**：**会话级状态机**，管对话的"当前焦点"、"历史焦点"、"何时切换焦点"。

**5.7.1 一句话定义**

**做什么**：当 NLU 输出**置信度不足**或**多个候选意图相当**时，主动用 TTS **反问澄清**，引导用户补全或选择，避免硬猜导致错误执行。

**不做什么**：

* 不做"槽位回填"（那是 N-F3 的事，槽位回填是已知意图缺槽位）
* 不做"指代反问"（那是 N-F4/N-F5 的事）
* 不做"任务执行"（澄清完成后才走 Agent 编排）

**A-F6 的本质**：**是 NLU 的"我不确定"机制** —— 不强行执行不确定的任务，是产品克制感的体现。

| --- | --- | --- |
| # | 场景 | 预期澄清 |

| S3 | **跨域歧义** | "去陆家嘉" → ASR 不确定（陆家嘴 or 陆家嘉？）→ 跨意图反问"您是想去陆家嘴还是别的地方？" |

**5.7.3 输入 / 触发条件**

* **输入**：N-F1/N-F2 意图分类结果（含置信度 + 候选）+ N-F3 槽位完整度 + 当前会话状态
* **触发**：满足以下任一条件
* Top-1 意图置信度 < 60%
* Top-1 和 Top-2 差距 < 10%（强歧义）
* 高风险动作（删除/支付/分享）+ 任何置信度不足
* 关键槽位多个候选都合理
* **前置条件**：澄清话术模板已配置；TTS 可用

**5.7.4 执行流程**

|  |
| --- |
| Plain Text NLU 输出  ↓ 触发条件检查  ├─ 单意图置信度 ≥ 85% + 槽位完整 → 直接执行（不走澄清）  ├─ 单意图 60-85% → 静默确认（TTS 简短确认"好，导航去机场"）  ├─ 单意图 < 60% 或 多意图竞争 → 启动澄清  └─ 高风险 + 不完美匹配 → 强制澄清（不放过）  ↓ 澄清话术生成  ├─ 模板 1（多候选）："您是想 A 还是 B？"  ├─ 模板 2（无候选）："您是想做什么？我可以帮您 X、Y、Z"  ├─ 模板 3（高风险）："您确认要 X 吗？这会..."  └─ 模板 4（关键槽位）："是 A 还是 B？"  ↓ TTS 反问 + 等待用户回答  ↓ 用户回答 → 重新 NLU  ├─ 澄清成功 → 执行  ├─ 仍模糊 → 第 2 轮反问（最多 2 轮）  └─ 第 3 轮失败 → 兜底"抱歉，请您手动操作" |

**关键时间预算**：

* 澄清话术生成：**≤ 100ms**（端侧模板）
* 一轮澄清总延迟：**≤ 4s**（TTS 反问 1.5s + 用户回答 + 重 NLU）

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |
| **澄清过度**（高频域低置信度也澄清，用户被频繁反问） | 用户体验差，关闭语音 | 阈值精调，高频域用更高 confidence 才澄清；用户偏好可调"少反问" |
| **澄清不足**（该澄清不澄清，硬猜执行错误） | 执行错任务，用户崩溃 | 高风险动作必须澄清（不依赖置信度阈值）；多候选差距 < 10% 必须澄清 |
| **澄清话术不清晰** | 用户不知道怎么回答 | 模板审核 + 用户测试；候选数 ≤ 3 个 |
| **澄清死循环** | 反问 3 次还没解决 | 上限 2 轮；第 3 轮兜底"请手动操作" |
| **跨意图澄清错** | 把单域歧义当成跨意图反问 | 澄清触发条件分层：先单域内 → 跨域间 |

**Paper Tiger（看起来像风险但不是）**

* "用户讨厌被反问"—— 实证：用户讨厌的是**反问错地方**（高频明确指令也反问），合理的反问用户接受度高
* "澄清增加交互成本"—— 反问 1 轮 4s，比执行错任务后纠错（30s+）成本低得多

**Elephant（团队可能没讨论够）**

* **澄清的"个性化"**：有些用户喜欢"快+宁错不问"，有些喜欢"稳+宁问不错"。**用户偏好可配置**（设置中"宁问不错 / 平衡 / 宁错不问"三档）
* **澄清的话术风格**：直接型（"是 A 还是 B？"）vs 温柔型（"我不太确定您是想 A 还是 B 呢"）。**默认直接，AI 人设跟随用户偏好**
* **高风险动作的硬性规则**：删除/支付/分享 + 不完美匹配必澄清，不论置信度。**这是产品红线，不能因优化体验而放松**

**5.7.6 SLO / 验收指标**

|  |  |  |
| --- | --- | --- |
| 指标 | 目标值 | 测试条件 |
| 澄清触发准确率（该澄清的都澄清） | ≥ 95% | 500 条歧义测试集 |
| 澄清避免率（不该澄清的不澄清） | ≥ 90% | 500 条明确指令测试集 |
| 澄清成功率（一轮澄清后正确执行） | ≥ 85% | 1000 条澄清场景测试 |
| 澄清话术清晰度（用户理解率） | ≥ 90% | 用户测试 |
| 高风险动作澄清覆盖率 | 100% | 所有高风险动作（删除/支付/分享） |
| 澄清死循环避免率（≤ 2 轮兜底） | 100% | 第 3 轮必兜底 |

**分级阈值**：

* 置信度 ≥ 85%：直接执行
* 60-85%：静默简短确认
* < 60% 或多候选 < 10% 差距：启动澄清
* 高风险 + 任何不完美匹配：强制澄清

| 关 | 自问 | 答 |

| 失败路径 | 最糟糕失败？ | 高风险动作硬执行错误（"删除照片"误执行 → 数据丢失）。**高风险必澄清是红线，不能妥协** |

3. **澄清状态机**：澄清进行时是否屏蔽其他指令？回答后状态怎么转

**5.8 A-F1 多步复合任务解析（NLU 部分）**

**5.8.1 一句话定义**

**做什么**：把用户一句话里的**多步时序任务链**（"去嘉兴先充电再接朋友买咖啡"）拆解成**有序子任务列表**，标注依赖关系和时序约束，送给 Agent 编排执行。

**不做什么**：

* 不做"任务执行"（Agent 编排 / 文档 06 的事）
* 不做"单句跨域识别"（N-F2 的事，跨域是两个域并发，不是长任务链）
* 不做"任务进度跟踪"（06 的事）

**A-F1 和 N-F2 的边界**：

* **N-F2 跨域**：典型句式是"A 且 B"或"A 顺便 B"，2 个意图**并行/轻依赖**（"订餐厅再导航过去"）
* **A-F1 复合**：典型句式是"先 A 再 B 再 C"或"一路上 A 然后 B"，3+ 步**强时序/强依赖**（"去嘉兴先充电再接朋友再买咖啡"）

**5.8.2 核心场景脚本**

|  |  |  |
| --- | --- | --- |
| # | 场景 | 预期解析 |
| S1 | **经典复合任务**（C1 嘉兴） | "今天去嘉兴，先充电再接朋友买咖啡" → 子任务： 1. 导航.到嘉兴（目标） 2. 顺路.找充电站（前置到达） 3. 接人.朋友（接完再走） 4. 顺路.买咖啡（最后） **时序：1依赖2→接3→顺4** |
| S2 | **日程安排** | "明天 8 点闹钟，9 点前到公司，路上顺便取快递" → 子任务： 1. 闹钟.设定(8:00) 2. 导航.公司(到达时间 9:00 前) 3. 顺路.取快递点 **时序：1独立，2和3强关联** |
| S3 | **车辆服务链** | "明天早上检查胎压，上车前启动空调" → 子任务： 1. 车辆.胎压检测(明晨) 2. 空调.预启动(明晨上车前) **时序：都是明天执行，有时间约束** |
| S4 | **任务修改**（长任务中途） | 导航到一半 → R2"改到虹桥，咖啡不要了" → 系统修改子任务： 1. 导航.destination: 虹桥机场（修改） 2. 顺路.咖啡（取消） **时序：保留，子任务替换** |
| S5 | **时序解析歧义** | "去嘉兴，先接朋友再充电"（先接再充 vs 先充再接）→ 按句法顺序默认"先接再充"，置信度中等 → TTS 确认"先接朋友再充电，对吗？" |
| S6 | **任务超长** | "先 A 再 B 再 C 再 D 再 E 再 F" 6+ 步 → 要么拆成多个任务，要么提示"太长了，您能分几次说吗？" |

**5.8.3 输入 / 触发条件**

* **输入**：ASR 文本（长句） + N-F2 跨域识别结果（已知含多意图） + N-F3 槽位 + 上下文
* **触发**：
* 句中含 3+ 个动作动词
* 句中含典型时序词（先...再...然后...顺便...最后...）
* 单句长度 > 25 字
* **前置条件**：跨域识别已完成；复合任务模型可用

**5.8.4 执行流程**

|  |
| --- |
| Plain Text 长句 ASR 输入  ↓ 触发条件判断  ├─ 动词数 < 3 且无时序词 → 走 N-F2 跨域即可  └─ 动词数 ≥ 3 OR 含时序词 → 启动复合任务解析  ↓ 任务拆解（云端 LLM）  ├─ Step 1: 拆分子句（识别每个独立任务）  ├─ Step 2: 每个子句意图分类 + 槽位提取  ├─ Step 3: 时序关系识别（先/后/顺路/同时）  └─ Step 4: 依赖关系识别（B 依赖 A 完成后执行）  ↓ 时序结构化输出  ├─ 任务列表（有序 subtask[]）  ├─ 每个 subtask 的 { intent, slots, 前置依赖, 时间约束 }  └─ 整体任务 ID（用于后续跟踪）  ↓ 置信度检查  ├─ 整体 ≥ 80% → 确认总览："要帮您导航嘉兴，顺路充电、接朋友、买咖啡，对吗？" + 用户同意后执行  ├─ 60-80% → 详细确认每步  └─ < 60% → 建议用户"分开说几次会更准"  ↓ 送 Agent 编排（文档 06）执行  └─ 任务 ID 传给 N-F6 会话状态跟踪 |

**关键时间预算**：

* 复合任务解析延迟：**≤ 1.5s**（P95）（云端 LLM 调用）
* 总览确认 UX：**≤ 3s**（解析 + TTS 播报）

**5.8.5 失败路径与降级（pre-mortem）**

**Tiger 级风险（真问题）**

|  |  |  |
| --- | --- | --- |
| 风险 | 后果 | 缓解 |

| **任务修改失败**（中途改变主意时系统跟不上） | 旧任务继续执行导致错乱 | 任务 ID 绑定 + 修改识别机制；R2"改xxx"时强制对比旧任务做差异修改 |

**Paper Tiger（看起来像风险但不是）**

* "复合任务用户不会用"—— 实测：家庭场景（接送娃 + 顺路）、通勤场景（导航 + 充电 + 取快递）天然含复合；用户会自然说
* "复合任务识别率永远不高"—— LLM 当前在简单时序解析上已达 80%+；工程+模型迭代可上 85%+

**Elephant（团队可能没讨论够）**

* **和 Agent 编排（06）的合同**：
* A-F1 输出：{ task\_id, subtasks[], dependencies[], time\_constraints[] }
* 06 接管执行 + 进度跟踪 + 失败处理
* 合同必须和沈砚对齐
* **任务进度的 UX 反馈**：执行中是否实时告知用户"第 1 步完成，开始第 2 步"？**建议：关键步骤告知，小步骤静默**
* **任务失败的回退**：第 3 步充电站没位置了，怎么办？**建议：每个子任务有失败兜底方案**，产品层定义（如充电站没位 → 推荐下一个）
* **时间约束的现实冲突**："8 点闹钟，9 点前到公司" → 如果导航预测 9:30 到，怎么办？**产品决策：提前告警，让用户决定**（早起 or 延迟）

## 补充：NLU 层复杂语境处理规则

> 以下内容补充 NLU 在处理模糊指令、多指令拆分、纠错/补充合并时的具体规则，与 06 Agent 编排中的 CLAW 闭环框架配合。

### 模糊指令的 NLU 处理

NLU 层遇到模糊指令时的处理策略：

| 模糊类型 | 示例 | NLU 处理 |
|---------|------|---------|
| 指代词 | "去老地方""来个老样子" | 查 04 章偏好表/最近 task_states，替换为具体实体后重新识别 |
| 泛化表达 | "帮我搞点吃的""买个东西" | 输出泛化意图 + confidence，由 Agent 层结合时间/位置/偏好推断具体子意图 |
| 省略主语 | "热一点""大杯" | 结合对话状态（当前 channel）判断修改的是空调温度还是咖啡杯型 |
| 反义/情感 | "太冷了" | NLU 标注为反义表达 + 当前 channel，Agent 层推断：车内→升温 / 点餐→换热饮 |

**置信度输出**：所有模糊指令识别结果必须输出 confidence 分数。confidence < 0.75 时触发 TTS 确认（"您是说去张江吗？"），< 0.5 时直接问用户（"我没太听清，方便再说一遍吗？"）。

### 多指令拆分

用户一句话包含多个意图时，NLU 层负责拆分：

| 输入 | NLU 输出 |
|------|---------|
| "开空调然后导航去公司" | [{ intent: "vehicle_control", action: "ac_on" }, { intent: "navigation", dest: "公司" }] |
| "点个肯德基然后导航过去" | [{ intent: "305.30", keyword: "肯德基" }, { intent: "navigation", dest: "依赖305.30结果" }] |
| "买电影票顺便点杯咖啡" | [{ intent: "305.32" }, { intent: "305.22" }] |

NLU 层只负责拆分和标注依赖关系（"然后"=串行，"顺便/同时"=并行），不负责执行编排。执行编排由 06 Agent 的 Master Agent 处理。

### 纠错识别

NLU 层需要识别用户的纠错意图：

| 纠错关键词 | NLU 标注 | 后续处理 |
|-----------|---------|---------|
| "不对""不是""错了" | correction_full（完全否定） | 丢弃前半段，只处理纠错词后的内容 |
| "应该是""改成" | correction_replace（替换） | 替换对应槽位 |
| "等等""等一下" | correction_pause（暂停） | 暂停当前流程，等用户补充 |

纠错识别的优先级高于意图识别——先判断是否是纠错，再做意图分析。

### 补充合并

用户停顿后继续说的内容，NLU 层需要判断是"补充当前任务"还是"新指令"：

| 判断依据 | 处理 |
|---------|------|
| 新内容能匹配当前任务的空槽位 | 合并为同一指令（"美式"+"少冰"="美式少冰"）|
| 新内容是当前任务已有槽位的修改 | 替换槽位值（"两杯"→"一杯"）|
| 新内容是完全不同的意图 | 当成新指令，交给 Mother-bot 路由 |

停顿窗口：ASR 完成后等待 1.5 秒，期间用户继续说的内容走补充合并逻辑；超过 1.5 秒视为当前指令结束。
