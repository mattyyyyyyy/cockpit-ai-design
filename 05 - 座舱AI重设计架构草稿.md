**05 - 座舱AI重设计架构草稿**

**座舱 AI 重设计架构草稿**

**文档版本**: v0.2**创建时间**: 2026-05-10**作者**: 沈砚(后端架构师 + Agent 编排)**目标**: 基于前四份调研文档,以 **Harness Engineering**(给 AI 引擎装线束)为顶层设计哲学,给出座舱 AI 重设计的端云架构草稿

**1. 设计原则**

**1.1 核心原则**

1. 安全优先: 车控安全红线不可妥协,端侧必须兜底
2. 端云协同: 端侧保证可用性,云侧提供智能
3. 失败优雅: 每一步都有降级路径,不追求完美
4. 可观测: 全链路 trace\_id,故障可定位
5. 简单: 能一个 Agent 解决的不用两个,能串行的不用并行

**1.2 反模式清单**

避免的设计:

* 纯云端架构(断网全挂)
* 纯端侧架构(能力受限)
* 过度分解 Agent(失败面扩大)
* 无状态机的编排(调试难)
* 同步阻塞的工具调用(延迟爆炸)
* 全量日志(成本爆炸)
* 无置信度阈值的执行(误操作风险)

**1.3 Harness Engineering 原则(顶层设计哲学)**

**核心理念**: AI 引擎需要配套"线束系统"才能安全可控。类比汽车——引擎(大模型)再强,没有线束(控制/保护/监督)就是危险品,不能上车。

**为什么这是顶层原则**: 1.1(核心原则)和 1.2(反模式)讲的是"做什么/不做什么",Harness Engineering 讲的是"为什么这么做"——所有架构决策都在给 AI 引擎装线束。

**七大线束组件**:

**1. 状态机(State Machine)——流程线束**

* 显式定义任务流转: IDLE → LISTENING → UNDERSTANDING → EXECUTING → DONE/FAILED/CANCELLED
* 每个状态有明确的进入条件、退出条件、超时策略
* 不允许隐式状态散落在各 Agent 的 if/else 里
* 作用: 防止 AI 流程"走飞",任何时刻都能回答"我现在在哪一步"

**2. 置信度阈值(Confidence Threshold)——决策线束**

* 每个 Agent 输出必须带置信度分数(0-1)
* 高风险操作(车控/支付)要求置信度 > 0.9
* 低置信度时触发二次确认或降级
* 示例: "我理解你要打开车窗,确认吗?"(置信度 0.7)
* 作用: 防止 AI 在不确定时瞎执行,造成误操作

**3. 降级路径(Degradation Path)——可用性线束**

* 每个 Agent 调用都有 Plan B
* 云端大模型超时 → 端侧小模型
* 复杂意图识别失败 → 规则引擎兜底
* 全链路失败 → 提示用户手动操作
* 作用: 防止单点故障导致整个 AI 不可用

**4. 安全红线(Safety Guardrails)——安全线束**

* 车控指令必须端侧二次校验(不信任云端结果)
* 高速行驶时禁用分心操作(视频播放/复杂多轮交互)
* 紧急场景(碰撞预警/急刹)可打断任何 AI 流程
* 隐私敏感数据(原始语音/精确位置/通话记录)不上云
* 作用: 防止 AI 能力强但触碰安全/隐私红线

**5. 可观测性(Observability)——诊断线束**

* 全链路 trace\_id: 从语音输入到执行结果
* 关键节点埋点: ASR 耗时 / NLU 置信度 / 工具调用结果 / 降级触发次数
* 分级日志: ERROR 必采样 / WARN 10% / INFO 1% / DEBUG 只在灰度环境
* 故障可回放: 保留失败案例的完整上下文(脱敏后)
* 作用: 故障 5 分钟定位,而不是"玄学 bug 复现不了"

**6. 人类监督(Human-in-the-Loop)——兜底线束**

* 高风险操作强制二次确认(车控/支付/发消息)
* 低置信度时主动询问而非猜测
* 用户可随时打断/撤销/重试
* 提供"退出 AI 模式"的物理按钮(硬件级兜底)
* 作用: 防止 AI 越俎代庖,人始终是最终决策者

**7. 失败隔离(Failure Isolation)——韧性线束**

* 一个 Sub-bot 挂了不影响其他(导航挂了,车控还能用)
* 工具调用失败不崩溃整个会话
* 云端故障时端侧独立运行
* 熔断机制: 连续失败 3 次 → 暂停该 Agent 5 分钟,走降级
* 作用: 防止局部故障扩散成全系统崩溃

**反例对照表(没有线束的 AI)**:

|  |  |
| --- | --- |
| 反例 | 后果 |
| 云端大模型直接控制车窗 | 断网时全挂 |
| 意图置信度 0.3 就执行 | 误操作频发,用户投诉 |
| 没有状态机 | 调试时不知道卡在哪一步 |
| 全量日志上传 | 成本爆炸,隐私风险 |
| 用户无法打断 AI 流程 | 体验差,失去控制感 |
| Agent 之间无隔离 | 一个挂全挂 |
| 工具调用无降级 | 弱网场景完全不可用 |

**正例对照表(有线束的 AI)**:

|  |  |
| --- | --- |
| 正例 | 收益 |
| 端侧状态机 + 云端大模型 | 断网降级到端侧小模型 |
| 置信度 < 0.8 二次确认 | 误操作率 < 1% |
| trace\_id 串全链路 | 故障 5 分钟定位 |

**2. 端云分工架构**

**2.1 三层架构**

* **云端(Cloud)**: 大模型(70B) / ASR-TTS(云端版) / Agent 编排器 / POI 搜索 / 路线规划 / 内容服务
* **边缘(Edge,可选)**: 区域缓存 / 负载均衡
* **端侧(Vehicle)**: 唤醒词 / 端侧 ASR / 端侧 NLU / 小模型(3B) / 状态机(编排) / 规则引擎(兜底) / CAN 总线 / 离线地图 / 本地缓存

通信协议: WebSocket(实时对话) + MQTT(车控指令) + HTTP/2(内容拉取)

**2.2 分工原则**

|  |  |  |  |
| --- | --- | --- | --- |
| 能力 | 端侧 | 云端 | 分工依据 |
| 唤醒词检测 | ✓ | - | 低延迟、隐私 |
| ASR(高频场景) | ✓ | - | 端侧小模型够用 |
| ASR(长尾场景) | - | ✓ | 云端精度高 |
| NLU(高频意图) | ✓ | - | 端侧规则+模型 |
| NLU(复杂意图) | - | ✓ | 需大模型推理 |
| Agent 编排(简单) | ✓ | - | 状态机驱动 |
| Agent 编排(复杂) | - | ✓ | 需动态规划 |
| 车控指令 | ✓ | - | 安全、低延迟 |
| POI 搜索 | 离线缓存 | ✓ | 实时数据在云端 |
| 路线规划 | 离线地图 | ✓ | 实时路况在云端 |
| 大模型推理 | 小模型(3B) | 大模型(70B) | 算力+成本权衡 |

**2.3 三级降级策略**

**L1 - 全功能**(网络正常): 云端大模型 + 完整 Agent 编排,所有功能可用

**L2 - 核心功能**(网络抖动): 端侧小模型 + 高频场景,车控/简单导航/音乐可用,复杂意图降级或提示"网络不好"

**L3 - 安全兜底**(完全断网): 端侧规则引擎 + 离线地图,只保留车控、离线导航、本地音乐

切换规则:

* 云端超时 3s → 切 L2
* 网络完全断开 → 切 L3
* 网络恢复 → 逐步恢复 L1

**3. Agent 编排架构**

**3.1 选型: 层级分派(Mother-bot + Sub-bots)**

为什么选这个:

* 车内场景分领域明确(车控/导航/娱乐/通信/信息)
* 各领域独立演进
* 易于 A/B 测试和灰度
* 失败隔离好(一个 Sub-bot 挂了,其他不受影响)

为什么不选其他:

* 单 Agent: 能力上限受限,难扩展
* 串行流水线: 延迟累加,错误传染严重
* 纯并行: 聚合逻辑复杂,冲突难处理
* 对等协商: 协商不收敛,生产慎用

**3.2 拓扑**

Mother Agent(意图路由,云端,可端侧降级) → 5 个 Sub-bot:

* 车控 Sub-bot(端侧) → CAN 总线
* 导航 Sub-bot(云端) → 地图 API
* 娱乐 Sub-bot(云端) → 音乐 API
* 通信 Sub-bot(云端) → 通信 API
* 信息 Sub-bot(云端) → 信息 API

**3.3 Sub-bot 清单**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | Sub-bot | 部署 | 职责 | SLO |
| 1 | 车控 | 端侧 | 空调、车窗、座椅、灯光 | P99 <500ms,成功率 >99% |
| 2 | 导航 | 云端(端侧兜底) | POI 搜索、路线规划、实时路况 | P99 <3s,成功率 >90% |
| 3 | 娱乐 | 云端 | 音乐、电台、有声书、视频 | P99 <2s,成功率 >95% |
| 4 | 通信 | 云端 | 电话、短信、微信接入 | P99 <2s,成功率 >95% |
| 5 | 信息 | 云端 | 天气、新闻、股票、百科 | P99 <3s,成功率 >90% |

**3.4 Mother Agent 职责**

**1. 意图路由**: NLU 识别意图领域 → 路由表分派 Sub-bot → 调用 Sub-bot

**2. 多 Sub-bot 协调**: 用户复杂意图(边开车边导航边播音乐) → NLU 分解 → 并行调用 → 超时保护(3s) → 部分结果聚合

**3. 冲突解决**: 按优先级排序: 安全 > 车控 > 导航 > 娱乐 > 信息

**3.5 状态机**

任务生命周期: CREATED → ROUTED → RUNNING → SUCCESS / FAILED / TIMEOUT / CANCELLED

状态转换规则:

* CREATED → ROUTED: Mother 路由成功
* ROUTED → RUNNING: Sub-bot 开始执行
* RUNNING → SUCCESS: 工具调用成功
* RUNNING → FAILED: 工具调用失败,无重试余地
* RUNNING → TIMEOUT: 超过超时阈值
* RUNNING → CANCELLED: 用户新意图到达

**4. 上下文管理**

**4.1 共享黑板模式**

Mother Agent 写入意图和参数 → Sub-bot 读取并执行 → 用户反馈写入触发调整。版本号 + 锁保证一致性。

**4.2 上下文生命周期**

|  |  |  |  |
| --- | --- | --- | --- |
| 层级 | 存储 | TTL | 内容 |
| 短期(单次交互) | 内存 | 5 分钟 | 当前意图、参数、中间结果 |
| 中期(多轮对话) | 端侧 Redis | 30 分钟 | 最近 3 轮对话、用户偏好 |
| 长期(用户画像) | 云端数据库 | 永久(用户可删) | 常用 POI、音乐偏好、习惯 |

**4.3 隐私边界**

端侧处理(不上云): 语音原始数据、精确位置坐标

脱敏后上云: ASR 文本(脱敏)、位置(城市级别)、用户画像(匿名化)

**5. 可观测性**

**5.1 链路追踪**

trace\_id 贯穿全链路。例如一次车控请求:

* 唤醒词检测: 50ms
* ASR(端侧): 300ms
* NLU(云端,Mother 路由): 200ms
* 车控 Sub-bot(端侧)+ CAN 指令: 150ms
* TTS(端侧): 300ms
* 总延迟: 1000ms

**5.2 关键指标**

* **延迟**: P50/P90/P99(按场景),各环节拆分
* **成功率**: ASR / NLU / Agent / 工具
* **可用性**: 云端可用率 / 端侧兜底触发率 / 网络可用率
* **用户体验**: 点赞/点踩率、重试率、放弃率

**5.3 日志策略**

* **全量日志**(高价值): 车控指令(审计)、异常事件、用户反馈
* **采样日志**(低价值): 成功请求 10% / 性能数据 1%
* **不记录**(隐私): 语音原始数据、精确位置坐标

**6. 部署架构**

**6.1 云端**

多地域部署(华北/华东/华南/西南),每地域主备双活,跨地域最终一致。服务拆分: 网关层 / Agent 层 / 工具层 / 数据层。

**6.2 端侧资源占用估算**

* 内存: 2GB(模型 + 运行时)
* 存储: 5GB(模型 + 缓存)
* CPU: 峰值 50%,平均 10%
* NPU: 峰值 80%,平均 20%

**6.3 OTA 策略**

模型更新: 增量更新 → 后台下载 → 验证后切换 → 失败回滚。灰度: 1% → 10% → 50% → 100%,每阶段观察 24 小时,异常立刻回滚。
