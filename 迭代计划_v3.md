## 1. AI 记忆系统（全面升级）

**痛点**
用户说过的、做过的、AI 全不记得。每次上车都是陌生人。

**四层记忆架构**

**① 工作记忆（单次会话内）**
Redis，30轮对话，1小时 TTL。存当前会话的对话历史、已填充的槽位、进行中的任务上下文。跨 channel 自动桥接——用户从星巴克点单切到导航时，目的地自动带过去。

**② 用户偏好（永久）**
MySQL `user_preferences` 表。记录用户的习惯和喜好：

| category | key | value | 来源 |
|---------|-----|-------|------|
| food | kfc_regular | 香辣鸡腿堡套餐中可 | 下单3次自动学习 |
| food | starbucks_regular | 美式中杯少冰 | 用户说"记住" |
| shopping | taobao_address | 张江高科园区18号 | 下单地址提取 |
| movie | seat_preference | 中间靠后 | 选座2次自动学习 |
| navigation | home | 浦东新区xx路xx号 | 用户说"记住" |
| navigation | company | 张江高科技园区 | 导航3次自动学习 |

来源规则：用户明确说"记住" → confidence=1.0；同一选择重复3次 → confidence=0.85；工具执行成功后自动抽取关键字段写入。

**③ 任务状态（跨会话）**
MySQL `task_states` 表。记录每个任务的完整生命周期：

| 字段 | 含义 |
|------|------|
| task_id | 唯一标识 |
| user_id | 用户 |
| tool_code | 305.22（星巴克）/ 305.30（肯德基）等 |
| status | pending / running / done / failed / cancelled |
| context_json | 任务上下文快照（填了什么槽位、选了什么商品） |
| created_at / updated_at | 时间戳 |

用途：
- 用户说"上次那个订单怎么样了"→ 查 task_states
- 系统重启 → 扫描 status=running 的任务恢复
- 用户中途打断 → 下次上车 TTS："您上次点了一半的肯德基订单，要继续吗？"

**④ 节律记忆（V2）**
按 `(user_id, weekday, hour_bucket)` 存储行为规律，驱动主动推荐。

**指代词解析**
识别到"老地方/上次那个/老样子"时，查偏好表和最近任务记录，把模糊词替换成具体实体再走 NLU。

**歧义消除**
匹配到多个候选时（两个张伟、两家肯德基），TTS 播报让用户选一个，不自动猜。

## 2. 新增业务场景

### 通用订购流程

肯德基、星巴克、电影票、淘宝购物这四个场景的底层流程是一样的，都是**多轮槽位填充 + 确认 + 支付**，差异只在细节交互。

**通用流程（5步）**

```
用户发起 → 意图识别 → 槽位填充（多轮）→ 确认 → 执行/支付
```

| 步骤 | 做什么 | 各场景差异 |
|------|-------|-----------|
| ① 意图识别 | NLU 识别到"点肯德基/买个东西/买电影票" | 触发词不同，后续 Agent 不同 |
| ② 槽位填充 | 逐步收集完成任务所需的全部信息 | 每个场景的槽位表不同（见下面） |
| ③ 偏好注入 | 从偏好表读取用户常用选项，自动填充已知槽位 | 减少问答轮次 |
| ④ 确认 | TTS 播报订单摘要，等用户说"确认" | 行驶中只播语音，驻车可显示卡片 |
| ⑤ 执行 | 调用对应平台 API 下单/支付 | 接入不同第三方 API |

### 场景一：肯德基点餐（305.30）

**槽位表**

| 槽位 | 必填 | 示例值 | 偏好可填 |
|------|------|-------|---------|
| 商品列表 | ✅ | 香辣鸡腿堡套餐×1 | ✅ "老样子" |
| 门店 | ✅ | 张江店 | ✅ 最近门店自动选 |
| 取餐方式 | ✅ | 到店自取 / 外卖 | ✅ 默认上次选择 |
| 收货地址 | 外卖时必填 | 张江高科园区18号 | ✅ 偏好表读取 |
| 支付方式 | ✅ | 银联预绑定 | ✅ 默认上次 |

**交互示例**

```
用户："帮我点个肯德基"
AI：（查偏好表，发现常点香辣鸡腿堡套餐）
    "您上次点的香辣鸡腿堡套餐，这次还是一样吗？"
用户："对，再加个蛋挞"
AI：（追加商品，查最近门店）
    "好的，香辣鸡腿堡套餐加一个蛋挞，张江店到店自取，总共42元，确认下单吗？"
用户："确认"
AI：（调用 KFC API 下单 + 银联支付）
    "已下单成功，预计15分钟可取。10秒内说取消可以撤销。"
```

### 场景二：淘宝购物（305.31）

**槽位表**

| 槽位 | 必填 | 示例值 | 偏好可填 |
|------|------|-------|---------|
| 商品描述 | ✅ | 车载手机支架 | ❌ 需要搜索 |
| 具体商品 | ✅ | 某品牌磁吸支架 ¥39.9 | ❌ 搜索后用户选 |
| 收货地址 | ✅ | 张江高科园区18号 | ✅ 偏好表 |
| 支付方式 | ✅ | 银联 | ✅ 默认 |

**交互示例**

```
用户："帮我在淘宝上买个车载手机支架"
AI：（搜索淘宝商品 API，返回3个结果）
    "找到3个：第一个是XX品牌磁吸款39.9元，月销1万+；
     第二个是XX品牌夹式29.9元；第三个是XX品牌出风口款49.9元。要哪个？"
用户："第一个"
AI：（查偏好表取默认地址）
    "好的，XX品牌磁吸支架39.9元，寄到张江高科园区18号，确认购买吗？"
用户："确认"
AI：（下单 + 支付）
    "已下单，预计后天到。10秒内说取消可以撤销。"
```

**淘宝特殊点**：搜索结果多，行驶中最多播报3个选项（TTS 列举太多用户记不住）；驻车时可在屏幕上显示商品卡片列表。

### 场景三：电影票购买（305.32）

**槽位表**

| 槽位 | 必填 | 示例值 | 偏好可填 |
|------|------|-------|---------|
| 电影名 | ✅ | 黑客帝国5 | ❌ 需要用户说 |
| 影院 | ✅ | 万达影城（张江店） | ✅ 最近影院 |
| 场次 | ✅ | 今晚19:30 | ❌ 需要选 |
| 座位 | ✅ | 8排12座、8排13座 | ✅ 偏好中间靠后 |
| 人数 | ✅ | 2张 | ❌ 需要问 |

**交互示例**

```
用户："帮我买两张今晚的电影票"
AI："想看什么电影？"
用户："黑客帝国5"
AI：（查附近影院场次）
    "最近的万达影城张江店，今晚有19:30和21:00两个场次，选哪个？"
用户："早点那个"
AI：（查座位，结合偏好"中间靠后"）
    "19:30场，帮您选了8排12、13座，中间靠后的位置，两张共96元，确认吗？"
用户："好的"
AI：（购票 + 支付）
    "已购票成功，要导航到万达影城吗？"
```

**电影特殊点**：购票成功后自动衔接导航（跨场景联动）；座位偏好可学习；人数需要明确问，不能猜。

## 3. Agent 架构设计：闭环动作框架

参考 Harness CLAW（Closed-Loop Action Worker）和 OpenCLAW 的设计理念——Agent 不是一次性执行指令，而是一个**闭环控制器**：感知→规划→执行→观察→调整。

### 核心架构：Mother-bot + 分域 Sub-bot

```
用户输入
  ↓
Mother-bot（调度层）
  ├─ 意图识别 + 路由
  ├─ 多意图拆分
  ├─ 上下文管理
  └─ 任务生命周期管理
  ↓
Sub-bot（按域分工）
  ├─ 车控 Sub-bot    → 车窗/空调/座椅（不走LLM，直通）
  ├─ 导航 Sub-bot    → 路线/搜索/到达
  ├─ 生活 Sub-bot    → 星巴克/KFC/外卖
  ├─ 购物 Sub-bot    → 淘宝/京东
  ├─ 娱乐 Sub-bot    → 电影/音乐/视频
  └─ 通用对话 Sub-bot → 闲聊/搜索/深度思考
```

每个 Sub-bot 内部跑一个**闭环**：

```
Perceive（感知）→ Plan（规划）→ Act（执行）→ Observe（观察）→ Adjust（调整）
    ↑                                                        ↓
    └────────────── 循环直到任务完成或用户取消 ──────────────────┘
```

### 解决全部复杂语境

**模糊指令**

| 用户说的 | AI 怎么处理 |
|---------|-----------|
| "帮我搞点吃的" | 模糊意图 → Mother-bot 根据时间+位置+偏好推断：午餐时间→推荐常点的肯德基或星巴克 |
| "来个老样子" | 查偏好表，找到最近频次最高的消费记录，确认后执行 |
| "去老地方" | 查导航偏好，匹配高频目的地 |
| "随便来杯咖啡" | 用星巴克 Sub-bot，但不填具体商品槽位 → AI 从偏好表推荐或给选项 |

处理机制：Mother-bot 先尝试从偏好和上下文中补全信息，能补全就补全后确认，不能就主动问。

**多指令（一句话多个意图）**

| 用户说的 | AI 怎么处理 |
|---------|-----------|
| "开空调，然后帮我导航去公司" | Mother-bot 拆分为两个意图 → 车控 Sub-bot 开空调（即时执行）+ 导航 Sub-bot 导航（可能需要确认路线） |
| "点个肯德基，然后导航过去" | 生活 Sub-bot 点餐 → 完成后自动把门店地址传给导航 Sub-bot |
| "买两张电影票，顺便点个星巴克带过去" | 娱乐 Sub-bot 购票（先）→ 生活 Sub-bot 点咖啡（并行/后）→ 两个都完成后问要不要导航 |

处理机制：Mother-bot 判断意图之间的依赖关系——无依赖的并行执行，有依赖的串行执行（先点餐再导航到门店）。

**全双工（用户随时插话）**

| 场景 | AI 怎么处理 |
|------|-----------|
| AI 在念搜索结果，用户突然说"第二个" | 立即停止 TTS → 选择第二个结果 → 继续流程 |
| AI 在确认订单，用户说"等一下，加个蛋挞" | 暂停确认 → 回到槽位填充阶段 → 追加商品 → 重新确认 |
| AI 在播报导航路线，用户说"帮我点杯咖啡" | 导航任务挂起（继续导航但停止播报）→ 切到星巴克 Sub-bot → 点完后恢复导航播报 |

处理机制：ASR 始终在线（不是说完一段才识别），检测到用户开口立即打断当前 TTS 输出。Mother-bot 判断插话是"补充当前任务"还是"新任务"——补充的回到当前 Sub-bot，新任务的开启新 Sub-bot 同时挂起当前任务。

**指令纠错**

| 用户说的 | AI 怎么处理 |
|---------|-----------|
| "帮我导航到…不对，帮我点个肯德基" | 检测到"不对/等等/我说错了" → 丢弃前半段 → 只处理"点个肯德基" |
| "两杯美式…不，一杯美式一杯拿铁" | 回滚已填充的"两杯美式"槽位 → 填入"一杯美式一杯拿铁" |
| "导航到虹桥…是浦东机场" | 替换目的地槽位"虹桥"→"浦东机场" |

处理机制：识别纠错关键词（"不对/不是/等等/我说错了/应该是"），触发后：丢弃纠错词前的内容（如果是完全否定），或替换对应槽位（如果是部分修正）。

**指令补充**

| 用户说的 | AI 怎么处理 |
|---------|-----------|
| "帮我点杯美式"（停顿）"少冰的" | 合并为一条："美式少冰" |
| "导航到公司"（停顿）"走高速" | 追加路线偏好到导航参数 |
| "买张电影票"（停顿）"两张" | 修改人数槽位 |

处理机制：用户停顿后又继续说，如果新内容能填入当前任务的空槽位或修改已有槽位，就自动合并，不当成新指令。判断依据是内容是否能匹配当前任务的 schema。

**拒识（AI 不该做的事）**

| 用户说的 | AI 怎么处理 |
|---------|-----------|
| "帮我往前面那辆车扔个东西" | 安全拒识 → "这个我没法帮您" |
| "@#$%^&*（噪音/非语言声）" | ASR 置信度极低 → 静默忽略，不回应 |
| "讲个黄色笑话" | 内容拒识 → "我不太擅长这个，要不我给您讲个有趣的谜语？" |
| "帮我买100瓶茅台" | 数量异常检测 → "您确定要买100瓶吗？" |

处理机制：三层过滤——① ASR 置信度 <0.3 直接忽略；② 安全分类器过滤危险/违规指令；③ 业务规则过滤异常参数（数量/金额/频率异常）。

## 4. 危险操作可以撤销

（保留 v2 内容不变）

所有有副作用的操作执行完之后，TTS 播"已完成，10秒内说取消可以撤销"。撤销窗口只靠语音，不弹屏幕卡片。行驶中（车速 > 10km/h）500元以上的支付直接拒绝。

---

## 5. 响应有反馈

（保留 v2 内容不变）

渐进式感知反馈：800ms 动效 → 2500ms TTS "稍等" → 3000ms 切备用方案。幂等 Key 防重复执行。

---

## 6. 工具出错时 AI 主动说出来

（保留 v2 内容不变）

工具超时 → 明确话术告知用户 + 提供替代方案。按工具独立熔断。
