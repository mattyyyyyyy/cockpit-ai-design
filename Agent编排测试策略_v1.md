## 二、T1 模糊指令处理

### T1-1 泛化食物指令 + 有偏好

**前置条件**：
- 偏好表：KFC confidence=0.85（3天内点过2次）
- 当前时间：14:00 工作日
- 当前位置：张江

**输入**：`"帮我搞点吃的"`

**期望行为**：
1. NLU 返回 intent=food_order, confidence=0.6, slots=空
2. TaskOrchestrator 查偏好表 → KFC 命中
3. 查位置 → 最近 KFC 1.2km
4. **不直接下单**，TTS 推荐确认："你上次点的肯德基香辣鸡腿堡套餐，这次还是一样吗？"

**分支验证**：
| 用户回复 | 期望 |
|---------|------|
| "对"/"好的" | 走 FoodHandler KFC 下单流程 |
| "不要，随便来杯咖啡" | 切咖啡意图，查偏好→星巴克 |
| "算了不吃了" | Task cancel，TTS "好的" |
| 10s 无响应 | TTS 再问一次，20s 仍无响应→关闭 |

---

### T1-2 泛化食物指令 + 无偏好

**前置条件**：新用户，偏好表为空

**输入**：`"帮我搞点吃的"`

**期望行为**：
1. 偏好表查询为空 → 无法推荐
2. TTS 直接问："想吃什么？附近有肯德基、星巴克、麦当劳"
3. 不做任何猜测

---

### T1-3 模糊指代 · 歧义可消解

**前置条件**：
- 偏好表："公司" confidence=0.90, "家" confidence=0.88
- 当前时间：08:30 工作日
- 当前位置：家附近

**输入**：`"去老地方"`

**期望行为**：
1. NLU → intent=navigation, dest="老地方"（指代词）
2. 候选："公司" vs "家"，差距<0.1 → 触发消歧
3. **上下文辅助**：工作日早晨 + 从家出发 → 倾向"公司"
4. TTS 确认："导航到公司，走吗？"（不自动导航）

---

### T1-4 模糊指代 · 歧义不可消解

**前置条件**：
- 偏好表："公司" confidence=0.55, "健身房" confidence=0.52
- 当前时间：周六 14:00
- 当前位置：商场

**输入**：`"去老地方"`

**期望行为**：
1. 两个候选 confidence 都不高，且上下文无法判断
2. 直接问用户："您想去公司还是健身房？"
3. 不做任何猜测性推荐

## 三、T2 多意图拆分与编排

### T2-1 串行依赖（下单→导航）

**输入**：`"帮我点个肯德基，然后导航过去"`

**期望行为**：
1. NLU 拆分：[kfc_order, navigation(dest=依赖kfc)]
2. DAGExecutor 构建：Task1(kfc) → Task2(nav)
3. Task1 完成 → 取 store_address → 注入 Task2.dest
4. Task2 开始导航
5. TTS："肯德基已下单，正在导航到张江店"

**失败分支**：Task1 失败（门店打烊）
- Task2 跳过（依赖未满足）
- TTS："张江KFC今天已打烊，要换一家吗？"

---

### T2-2 并行无依赖（电影票 + 咖啡）

**输入**：`"买两张电影票，顺便点杯星巴克"`

**期望行为**：
1. NLU 拆分：[movie_ticket, starbucks_order], relation=parallel
2. DAGExecutor 并行启动两个 Task
3. 星巴克有偏好（美式少冰）→ 自动填槽→等确认
4. 电影票需要用户选片→优先交互
5. TTS："想看什么电影？星巴克帮你点了常喝的美式少冰"

---

### T2-3 混合编排（并行→串行）

**输入**：`"买电影票和星巴克，然后导航去影院"`

**期望行为**：
1. DAG：[movie + starbucks] 并行 → nav 串行
2. 并行阶段两个都完成后 → nav 启动
3. nav.dest = movie.result.cinema_address

---

### T2-4 单意图误拆检测

**输入**：`"导航到肯德基"`

**期望行为**：
1. NLU 判断这是**单意图**（导航到KFC门店），不是"点KFC + 导航"
2. 只创建一个 nav Task，dest=最近KFC
3. **不触发** FoodHandler

---

## 四、T3 全双工与任务管理

### T3-1 打断 · 补充当前任务

**场景**：AI 正在读电影票选项

**AI**："今天有《黑客帝国5》和《流浪地球3》——"
**用户打断**："黑客帝国5"

**期望行为**：
1. ASR 检测到用户开口 → 立即停 TTS
2. NLU 判断"黑客帝国5"能填当前 Task 的 movie_name 槽
3. 不创建新 Task，补充到当前 Task
4. 继续选场次流程

---

### T3-2 打断 · 切换新任务

**场景**：AI 正在确认 KFC 订单

**AI**："确认下单香辣鸡腿堡套餐——"
**用户打断**："先帮我导航到公司"

**期望行为**：
1. 停 TTS
2. NLU 判断"导航到公司"是新意图，不属于当前 KFC Task
3. TaskRegistry.suspend(kfc_task) → 保留已填槽位
4. 新建 nav Task → 执行
5. 导航完成后 → **不自动恢复** KFC Task
6. TTS（导航完成后）："之前的肯德基订单还要继续吗？"

---

### T3-3 并行任务跨 Task 纠错

**场景**：电影票和星巴克并行进行中

**AI**："19:30和21:00两个场次——"
**用户**："等一下，咖啡改成拿铁"

**期望行为**：
1. NLU 识别"咖啡"→ 匹配 Task2(Starbucks)，不是 Task1(Movie)
2. "改成"→ correction，修改 Task2.product 槽位
3. 然后**自动切回** Task1 继续场次选择
4. TTS："好的改成拿铁了。电影19:30和21:00选哪个？"

---

### T3-4 所有任务都挂起后的恢复

**场景**：有2个挂起的任务（KFC + 电影票），用户说"继续"

**期望行为**：
1. "继续" → NLU 无法判断恢复哪个
2. TaskRegistry.get_suspended() → [kfc, movie]
3. TTS："之前有肯德基订单和电影票两个没完成，先处理哪个？"

## 五、T4 纠错与补充

### T4-1 单次纠错

**用户**："导航到虹桥机场"
**AI**："好的，导航到虹桥机场"
**用户**："不对，浦东机场"

**期望行为**：
1. NLU 检测"不对" → correction_replace
2. 替换 dest: "虹桥机场" → "浦东机场"
3. TTS："好的，改成浦东机场"
4. 直接导航（单次纠错仍有信心）

---

### T4-2 连续纠错 · 信心降级

**用户**："导航到虹桥机场"
**用户**："不对，浦东机场"
**用户**："等等，不是机场，是浦东火车站"

**期望行为**：
1. 连续纠错2次 → confidence 降到 < 0.7
2. **强制确认**：TTS "浦东火车站，确认出发吗？"
3. 用户说"确认" → 导航
4. 用户说"算了" → cancel

---

### T4-3 停顿补充

**用户**："点杯美式"
（停顿 1.2s）
**用户**："少冰的"

**期望行为**：
1. 第一句 → 创建 Task，product=美式，ice=空
2. VAD 检测停顿 1.2s（<1.5s 窗口）→ 等待补充
3. "少冰的" → 填入 ice=少冰
4. 合并为完整指令：美式少冰
5. TTS："美式少冰，确认下单吗？"

## 六、T5 跨域联动

### T5-1 电影→导航→咖啡链式推荐

**前置条件**：
- 用户有咖啡偏好
- 打扰预算剩余 2 次/周

**触发**：电影票购买成功

**期望行为**：
1. EventBus emit task_completed(movie)
2. NavHandler 收到 → 生成导航建议（优先级高）
3. FoodHandler 收到 → 生成咖啡建议（优先级低）
4. **只推导航**（打扰预算控制，一次推一个）
5. 导航途中经过星巴克 → 再推咖啡

---

### T5-2 打扰预算耗尽

**前置条件**：本周已推荐 3 次（打扰预算已满）

**触发**：电影票购买成功

**期望行为**：
1. EventBus 发出事件
2. NavHandler 生成建议 → 检查打扰预算 → 已满
3. **不推荐**，静默
4. 日志记录：suggestion_suppressed, reason=budget_exhausted

---

### T5-3 主动服务 · SOC 低电量

**前置条件**：
- SOC = 18%（<20% 阈值）
- 驾驶状态：等灯

**期望行为**：
1. 车辆状态事件 → EventBus
2. 主动服务规则匹配 → 创建 charging Task
3. 查最近充电站 → 2km
4. TTS："电量剩 18%，附近 2 公里有充电站，要导航过去吗？"
5. 用户同意 → 导航
6. 用户拒绝 → 本次不再提醒，下次降到 10% 再提醒一次

## 七、T6 任务持久化与恢复

### T6-1 车辆重启恢复

**前置条件**：
- task_states 表有 status=running 的 KFC 订单
- context: { items: "香辣鸡腿堡+蛋挞", stage: "confirming" }

**触发**：车辆重启完成

**期望行为**：
1. TaskOrchestrator 启动 → 扫描 task_states
2. 发现未完成任务
3. TTS："您上次点了一半的肯德基——香辣鸡腿堡加蛋挞，要继续吗？"

**分支验证**：
| 用户回复 | 期望 |
|---------|------|
| "好" | 从 confirming 恢复 |
| "不要了" | cancel + 清理 task_states |
| 无响应 10s | 不再打扰，保留 task_states |

---

### T6-2 超期任务清理

**前置条件**：task_states 有 24 小时前的 running 任务

**期望行为**：
1. 启动时检测 → 超过 TTL（24h）
2. 自动标记为 expired
3. 不向用户提示
4. 日志记录：task_expired

## 八、T7 安全边界

### T7-1 行驶中大额支付

**前置条件**：车速 85km/h，行驶中

**输入**：`"帮我转账5000块给张三"`

**期望行为**：
1. Harness 安全层拦截：行驶中 + 金额>500 + 不可逆
2. **直接拒绝**，不进入 Handler
3. TTS："行驶中不支持大额转账，请停车后再操作"

---

### T7-2 行驶中小额支付

**前置条件**：车速 35km/h，行驶中

**输入**：`"帮我充个话费50块"`

**期望行为**：
1. 安全检查：行驶中 + 金额<50 + 话费（低风险）
2. 通过 → 但需声纹确认
3. TTS："确认充值50元话费？请说确认"

---

### T7-3 等灯时中额支付

**前置条件**：车速 0km/h，等红灯（<60s）

**输入**：`"帮我付停车费280块"`

**期望行为**：
1. 安全检查：等灯 + 金额 50-500 + 预授权品类（停车）
2. 通过 → 展示卡片 + TTS 确认
3. TTS："停车费280元，确认支付吗？"

---

### T7-4 NLU 置信度过低

**输入**：`"嗯嗯那个就是"` （无意义语句）

**期望行为**：
1. NLU confidence < 0.3
2. TaskOrchestrator 直接忽略
3. 不做任何回应（避免误触发）
4. 如果 confidence 在 0.3-0.5：TTS "没听清，能再说一遍吗？"

## 九、T8 工具故障与熔断

### T8-1 单工具超时→熔断

**场景**：星巴克 MCP 连续超时 3 次

**期望行为**：
1. 第1次超时 3s → 重试
2. 第2次超时 3s → 重试
3. 第3次超时 3s → CircuitBreaker OPEN
4. TTS："星巴克服务暂时不可用，要换瑞幸吗？"
5. 30s 后 HALF_OPEN → 探测一次

---

### T8-2 熔断不扩散

**场景**：星巴克 MCP 熔断中，用户说"导航到公司"

**期望行为**：
1. NavHandler 正常工作（CircuitBreaker 独立）
2. 导航正常执行
3. 星巴克熔断状态不影响任何其他 Handler

---

### T8-3 云端 LLM 超时→端侧降级

**场景**：Qwen-72B 云端调用超时 3s

**期望行为**：
1. L1 模式不可用 → 自动切 L2
2. 端侧小模型（3B）接管简单意图
3. TTS 语气变简洁（模板话术，非 LLM 生成）
4. 复杂意图 → TTS "网络不太好，稍后再试"
5. 云端恢复后 → 自动切回 L1

## 十一、测试执行规范

### 输入格式

每条测试用例应包含：

```json
{
  "test_id": "T1-1",
  "category": "模糊指令",
  "priority": "P0",
  "preconditions": {
    "preference_table": {...},
    "driving_state": "parked",
    "time": "14:00",
    "location": "张江"
  },
  "input": {
    "type": "voice",
    "text": "帮我搞点吃的",
    "asr_confidence": 0.92
  },
  "expected": {
    "nlu_intent": "food_order",
    "nlu_confidence_range": [0.5, 0.7],
    "action": "recommend_confirm",
    "tts_contains": ["肯德基", "还是一样"],
    "side_effects": [],
    "created_tasks": 1,
    "task_state": "slot_filling"
  },
  "branches": [
    {"user_reply": "对", "expected_action": "execute_order"},
    {"user_reply": "算了", "expected_action": "cancel"}
  ]
}
```

### 自动化等级

| 等级 | 说明 | 场景占比 |
|------|------|---------|
| L1 全自动 | Mock MCP + 断言输出 | 60%（T1/T2/T4/T7/T8）|
| L2 半自动 | 需要模拟 ASR 输入流 | 25%（T3 全双工）|
| L3 手动 | 需要真实 TTS 播放验证 | 15%（T5/T9 主观体验）|

### 回归触发条件

以下变更必须跑全量测试：
- TaskOrchestrator 核心逻辑修改
- NLU 模型或 prompt 更换
- 新增/修改 Handler
- 安全规则表变更
- DAGExecutor 调度逻辑修改

以下变更只需跑相关子集：
- 单个 MCP Server 接口变更 → 跑该 Handler 的测试
- TTS 话术调整 → 跑 tts_contains 断言
- 偏好表字段变更 → 跑 T1 + T9
