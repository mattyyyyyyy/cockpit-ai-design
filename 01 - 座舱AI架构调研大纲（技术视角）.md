**01 - 座舱AI架构调研大纲（技术视角）**

**座舱 AI 架构调研大纲 - 技术视角**

**文档版本**: v0.1**创建时间**: 2026-05-10**作者**: 沈砚(后端架构师 + Agent 编排)**目标**: 为座舱 AI 重设计提供技术侧的调研框架和失败模式预清单

**1. 座舱 AI 的技术分层**

**1.1 经典四层模型**

|  |
| --- |
| Plain Text ┌─────────────────────────────────────┐ │ 交互层 (Interaction) │ 语音/触控/手势/视觉 ├─────────────────────────────────────┤ │ 理解层 (Understanding) │ ASR → NLU → 意图识别 → 槽位填充 ├─────────────────────────────────────┤ │ 执行层 (Execution) │ Agent 编排 → 工具调用 → 车控/导航/娱乐 ├─────────────────────────────────────┤ │ 反馈层 (Feedback) │ TTS → 视觉反馈 → 多模态输出 └─────────────────────────────────────┘ |

**1.2 端云分工的技术边界**

|  |  |  |  |
| --- | --- | --- | --- |
| 能力 | 端侧(车机) | 云侧 | 分工依据 |
| 唤醒词检测 | ✓ | - | 低延迟、隐私 |
| ASR(语音识别) | 轻量模型 | 完整模型 | 网络可用时云端精度更高 |
| NLU(语言理解) | 高频场景 | 长尾场景 | 端侧覆盖 80% 高频,云端兜底 |
| 车控指令 | ✓ | - | 安全关键、断网可用 |
| 导航/POI 搜索 | - | ✓ | 实时数据依赖 |
| 娱乐内容 | 缓存 | ✓ | 内容库在云端 |
| 大模型推理 | 小模型 | 大模型 | 算力 + 成本权衡 |
| Agent 编排 | 简单流程 | 复杂流程 | 端侧状态机,云侧动态编排 |

**关键权衡**:

* **延迟 vs 能力**: 端侧快但能力受限,云侧慢但能力强
* **隐私 vs 精度**: 端侧数据不出车,云侧精度更高
* **成本 vs 体验**: 端侧算力有限,云侧按调用计费
* **安全 vs 智能**: 车控必须端侧兜底,AI 增强可选

**1.3 断网降级策略**

**三级降级**:

1. **L1 - 全功能**(网络正常): 云端大模型 + 完整 Agent 编排
2. **L2 - 核心功能**(网络抖动): 端侧小模型 + 高频场景
3. **L3 - 安全兜底**(完全断网): 车控 + 导航基础功能

**技术实现**:

* 端侧缓存高频意图的执行路径
* 云端响应超时(如 3s)自动切 L2
* 断网检测 → 立刻切 L3,不等超时

**2. Agent 编排的典型拓扑**

**2.1 单 Agent(最简形态)**

|  |
| --- |
| Plain Text 用户 → [超级 Agent] → 工具调用 → 车控/导航/娱乐 |

**优点**: 简单、延迟低、状态单一**缺点**: 能力上限受限、难以扩展、黑盒调试难

**适用场景**: MVP、高频简单场景(开空调、播音乐)

**2.2 串行流水线**

|  |
| --- |
| Plain Text 用户 → [意图识别 Agent] → [参数补全 Agent] → [执行 Agent] → 反馈 |

**优点**: 职责清晰、每步可观测**缺点**: 延迟累加、错误传染

**失败模式**:

* 意图识别错 → 后续全错
* 参数补全超时 → 整条链路卡死

**适用场景**: 需要多轮对话补全参数的场景(订餐、订票)

**2.3 并行 Fan-out/Fan-in**

|  |
| --- |
| Plain Text  ┌→ [车控 Agent] ─┐ 用户 → [路由 Agent] ─┼→ [导航 Agent] ─┼→ [聚合 Agent] → 反馈  └→ [娱乐 Agent] ─┘ |

**优点**: 并行降低延迟、各领域独立演进**缺点**: 聚合逻辑复杂、部分失败处理难

**失败模式**:

* 某个 Agent 超时 → 聚合等还是放弃?
* 多个 Agent 返回冲突结果 → 谁优先?

**适用场景**: 复杂意图(边开车边导航边播音乐)

**2.4 层级分派(Mother-bot + Sub-bots)**

|  |
| --- |
| Plain Text 用户 → [Mother Agent] → 任务分解  ├→ [Sub-bot 1: 车控专家]  ├→ [Sub-bot 2: 导航专家]  └→ [Sub-bot 3: 娱乐专家]  ← 结果聚合 ← 各 Sub-bot |

**优点**: 可扩展、专家分工、易于 A/B**缺点**: 复杂度高、状态同步难、调试成本高

**失败模式**:

* Mother 分派错误 → 任务到错误的 Sub-bot
* Sub-bot 之间需要协作 → 谁来协调?
* 某个 Sub-bot 挂了 → Mother 怎么兜底?

**适用场景**: 大规模、多领域、需要独立迭代的场景

**2.5 对等协商(Peer-to-Peer)**

|  |
| --- |
| Plain Text 用户 → [Agent A] ⇄ [Agent B] ⇄ [Agent C] → 反馈 |

**优点**: 灵活、去中心化**缺点**: 协商成本高、可能死锁、难以预测

**失败模式**:

* 协商不收敛 → 无限循环
* 多个 Agent 抢占资源 → 死锁

**适用场景**: 研究场景,生产慎用

**3. 关键技术挑战清单**

**3.1 延迟挑战**

|  |  |  |  |
| --- | --- | --- | --- |
| 环节 | 目标延迟 | 典型瓶颈 | 优化方向 |
| 唤醒词检测 | <200ms | 端侧算力 | 轻量模型、硬件加速 |
| ASR | <500ms | 网络 + 云端推理 | 流式识别、端云混合 |
| NLU + 意图识别 | <300ms | 模型推理 | 端侧缓存高频意图 |
| Agent 编排 | <1s | 多轮调用、工具延迟 | 并行、预测式执行 |
| 工具调用(车控) | <200ms | CAN 总线延迟 | 端侧直连、无云端绕行 |
| TTS | <500ms | 合成质量 vs 速度 | 流式合成、端侧缓存 |

**P99 目标**: 从唤醒到反馈 <3s(网络正常)、<1s(断网场景的车控)

**3.2 可靠性挑战**

**车联网特有的不可预测性**:

* **弱网**: 隧道、地库、偏远地区 → 云端不可用
* **版本分裂**: 不同车型、不同 OTA 版本 → 能力差异
* **硬件差异**: 算力、传感器、屏幕 → 体验不一致
* **地域漫游**: 跨省、跨国 → 法规、语言、内容差异

**Agent 编排的可靠性**:

* **工具调用幂等**: 重试不能导致重复执行(如重复开窗)
* **超时语义**: 超时后工具是否已执行? 如何回滚?
* **错误传染**: 一个 Agent 失败,下游如何隔离?
* **上下文一致性**: 多 Agent 看到的上下文是否一致?

**3.3 安全挑战**

**车控安全红线**:

* 绝不允许 AI 控制: 刹车、转向、安全带、行驶中开门
* 需要确认的操作: 开窗(防雨天误开)、开后备箱
* 可以直接控制: 空调、座椅、音乐、灯光

**隐私边界**:

* 不上云: 语音原始数据、精确位置坐标
* 脱敏后上云: ASR 文本、城市级位置、用户画像

**AI 安全**:

* 幻觉检测: 大模型编造不存在的 POI、路线
* 对抗样本: 特定语音模式绕过检测
* 偏见: 对特定口音、方言的识别率低

**3.4 成本挑战**

**云端成本**(百万日活估算):

* 大模型推理: ¥30,000/月
* ASR/TTS: ¥200,000/月
* 存储: ¥10,000/月
* 带宽: ¥40,000/月
* **合计**: ¥280,000/月 = ¥336 万/年

**单用户年成本**: ¥3.36

**端侧成本**(单车):

* 车规芯片: ¥2,000-5,000(一次性)
* 存储: ¥200-500
* OTA 流量: ¥10-50/次
* **生命周期成本**(5 年): ¥2,500-6,000

**4. 失败模式预清单**

**4.1 单点失败**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败点 | 检测方法 | 兜底方案 | 监控指标 |
| 云端大模型挂了 | 超时/5xx | 切端侧小模型 | fallback\_rate |
| ASR 服务挂了 | 超时/5xx | 端侧 ASR | asr\_timeout\_rate |
| 某个 Agent 超时 | asyncio.TimeoutError | 降级到规则引擎 | agent\_timeout\_rate |
| CAN 总线故障 | CANError | 提示用户手动操作 | can\_error\_rate |
| 网络完全断开 | requests.RequestException | 切 L3 安全兜底 | offline\_mode\_rate |

**4.2 级联失败**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 检测方法 | 兜底方案 | 监控指标 |
| 意图识别错 → 执行错 | 置信度 <0.8 | 多轮确认 | clarification\_rate |
| 某个 Agent 慢 → 整体超时 | 并行超时 | 用部分结果 | partial\_result\_rate |
| 工具调用失败 → 雪崩 | 连续失败 >5 次 | 熔断器 | circuit\_breaker\_open |

**4.3 上下文一致性失败**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 检测方法 | 兜底方案 | 监控指标 |
| 多 Agent 看到不同版本 | 版本号不一致 | 重试或拒绝 | version\_mismatch\_rate |
| 用户中途改变意图 | 新意图到达 | 取消旧任务 | task\_cancellation\_rate |

**4.4 资源耗尽**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 检测方法 | 兜底方案 | 监控指标 |
| 并发请求过多 | QPS 超限 | 限流 + 降级 | rate\_limit\_exceeded |
| 端侧内存不足 | psutil.virtual\_memory() | 卸载低优先级模型 | memory\_usage |
| 存储满 | shutil.disk\_usage() | 清理旧数据 | disk\_usage |

**5. 可观测性要求**

**5.1 链路追踪**

**trace\_id 贯穿全链路**:

|  |
| --- |
| Plain Text [trace\_id=abc123] ├── 唤醒词检测 (50ms) ├── ASR (端侧, 300ms) ├── NLU (云端, 200ms) ├── Agent 编排 (串行, 500ms) │ ├── 车控 Agent (100ms) │ └── 导航 Agent (400ms) └── TTS (端侧, 300ms)  总延迟: 1350ms |

**5.2 关键指标**

**黄金指标**(Golden Signals):

* **延迟**: P50/P90/P99 延迟
* **流量**: QPS、日活用户
* **错误**: 错误率、超时率
* **饱和度**: CPU/内存/队列深度

**业务指标**:

* ASR 准确率
* NLU 准确率
* Agent 编排成功率
* 用户满意度(点赞/点踩)

**5.3 日志策略**

**全量日志**(高价值):

* 车控指令(安全审计)
* 异常事件(故障排查)
* 用户反馈(模型改进)

**采样日志**(低价值):

* 成功请求(采样 10%)
* 性能数据(采样 1%)

**不记录**(隐私):

* 语音原始数据
* 精确位置坐标

**6. 技术选型框架**

**6.1 大模型选型**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| 模型 | 参数量 | 推理延迟 | 成本 | 适用场景 |
| 端侧小模型 | 1.8B-3B | 100-500ms | 一次性 | 高频场景 |
| 云端中模型 | 7B-13B | 500-1s | 中 | 中频场景 |
| 云端大模型 | 70B+ | 1-3s | 高 | 长尾场景 |

**我的建议**: 端侧 3B + 云端 13B,70B 只用于极少数复杂场景

**6.2 Agent 框架选型**

|  |  |  |  |
| --- | --- | --- | --- |
| 框架 | 优点 | 缺点 | 适用场景 |
| LangChain | 生态丰富、工具多 | 抽象重、性能差 | 快速原型 |
| LangGraph | 状态机清晰、可视化好 | 学习曲线陡 | 复杂编排 |
| CrewAI | 多 Agent 协作成熟 | 重量级 | 大规模团队 |
| AutoGen | 灵活、可定制 | 文档少 | 研究场景 |
| 自研 | 完全可控 | 开发成本高 | 特定需求 |

**我的建议**: 先自研 200 行编排器跑通 MVP,再考虑框架

**6.3 端云通信协议**

|  |  |  |  |
| --- | --- | --- | --- |
| 协议 | 优点 | 缺点 | 适用场景 |
| HTTP/2 | 成熟、工具多 | 弱网性能差 | 非实时场景 |
| WebSocket | 双向、低延迟 | 连接保活成本高 | 实时对话 |
| MQTT | 轻量、弱网友好 | 生态不如 HTTP | IoT、车联网 |
| gRPC | 高性能、类型安全 | 调试难 | 内部服务 |

**我的建议**: WebSocket(实时对话) + MQTT(车控指令) + HTTP/2(内容拉取)
