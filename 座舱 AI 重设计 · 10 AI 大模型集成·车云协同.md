**§1 背景与目标**

**1.1 为什么做 AI 大模型集成与车云协同**

**核心问题**：大模型能力上车不是"装个模型就完了"——端侧算力有限、网络不稳定、隐私要求高,必须精细分工"什么在车上做,什么在云端做"。

三大矛盾：

1. **算力 vs 延迟**：端侧算力有限但延迟低,云端算力强但依赖网络——什么放哪?
2. **能力 vs 隐私**：云端大模型更强但要上传数据,端侧隐私好但能力弱
3. **固定 vs 进化**：端侧模型出厂固定,OTA 更新慢;云端模型持续进化,但版本管理复杂

**目标**：

* **主目标**：端侧覆盖 Top 200 高频意图(延迟 <500ms),云端覆盖长尾意图(延迟 <3s)
* **可用性目标**：断网场景下核心功能可用率 >90%
* **成本目标**：单车云端推理成本 <¥3/月(稳态)

**§2 用户价值**

**2.1 核心价值主张**

* **快**：高频操作端侧响应,体感即时
* **强**：复杂/长尾需求走云端,能力不妥协
* **稳**：断网也能用核心功能,不会突然变傻
* **隐私**：敏感数据不出车,云端只处理非隐私请求

**2.2 反价值**

* **不做全云端**：网络是变量,全云端意味着隧道/地库里系统失能
* **不做全端侧**：端侧能力有限,强行全端侧 = 能力缩水
* **不做静默上传**：任何数据上云必须用户知情(设置页可查看/关闭)

**2. 车云分工原则(V1)**

**2.1 端侧职责**

1. **KWS/ASR** — 流式 ASR 端侧
2. **意图分流** — 简单垂域意图端侧判断(车控/空调/座椅)
3. **DMS/OMS** — 端侧实时感知
4. **紧急响应** — 风险预警(疲劳/电量)端侧兜底
5. **弱网降级** — 无网时基本功能可用

**2.2 云端职责**

1. **复杂 NLU** — 跨域/长句/指代用云端大模型
2. **任务规划** — 多步任务拆解云端
3. **工具调用** — Agent 编排云端
4. **长期记忆** — 记忆检索云端
5. **TTS 合成** — 复杂情感化 TTS 云端

**2.3 路由决策**

1. **延迟优先** — 响应 < 500ms 的走端侧
2. **复杂度优先** — 长上下文/多步规划走云端
3. **隐私优先** — 敏感数据(车内画面/声纹)不出车端
4. **网络优先** — 弱网自动降级到端侧

**3. 模型选型(V1 候选)**

**3.1 端侧模型**

* **ASR**:端侧流式模型(约 50-100M 参数)
* **KWS**:专用小模型(< 10M)
* **DMS/OMS**:视觉小模型(约 100-300M)
* **意图分类**:垂域 NLU 小模型(< 100M)
* **V1 不上端侧 7B+ 大模型**(V1.5 再评估)

**3.2 云端模型**

* **主 LLM**:自研 or 供应商大模型(100B+ 参数级)
* **Embedding**:用于记忆检索(向量化)
* **TTS**:云端 TTS 模型(支持流式+情感)
* **VLM**:用于视觉+语音指代消解(视沈砚评估是否必要)

**3.3 成本控制**

1. **端侧模型免费**(一次性部署成本)
2. **云端按调用量计费**
3. **预算控制** — 单用户日均 API 调用上限
4. **缓存** — 相同/相似 query 缓存结果

**5. OTA 策略**

1. **模型版本管理** — 每个版本有明确 changelog
2. **灰度发布** — 5% → 20% → 100%
3. **回滚机制** — 单用户可一键回滚到上版本
4. **用户告知** — 大模型升级必须告知用户(不让用户"感觉车在背着自己变")
5. **放弃条件对应** — §5.0.2 ❌4"大模型升级不告知用户"在这里落地

**8. 功能点展开**

**8.1 M-F1 车云路由决策**

**8.1.1 用户价值**

**做什么**：车上的每一次 AI 请求（唤醒 / ASR / 意图理解 / 任务规划 / TTS）都要决定\*\*"端侧跑还是云端跑"\*\*——用户不关心这个决策本身，但关心结果：**延迟够快、隐私不泄露、弱网下还能用、成本可控**。路由决策引擎是"看不见的指挥家"，让整套 AI 系统在延迟 / 质量 / 成本 / 隐私四个目标之间做最优取舍。

**不做什么**：

* 不做"全云端"（延迟高 / 弱网失灵 / 隐私敏感数据上云 / 成本不可控）
* 不做"全端侧"（端侧算力有限 / 复杂任务质量打折）
* 不做"静态路由"（写死规则 → 遇到新场景扩展难）
* 不做"用户可见的路由细节"（用户不该知道"这次去了云端"）

**核心价值主张**：

* **四维决策**：延迟 / 隐私 / 复杂度 / 网络，任一维度触发都会影响路由
* **动态调整**：网络变差时自动降级、成本预算用完时降级、敏感数据强制端侧
* **降级可预期**：降级路径用户感知一致（质量略降但功能可用）

**8.1.2 场景脚本**

|  |  |  |  |  |
| --- | --- | --- | --- | --- |
| # | 场景 | 路由决策 | 路由理由 | 关键约束 |
| S1 | **车控指令**："把空调调到 22 度" | 端侧 | 延迟 < 500ms + 端侧意图分类足够 | 车控永不走云端（安全 + 延迟） |
| S2 | **复杂指令**："订个周六家附近有儿童餐的餐厅，8 人，导航过去" | 云端 LLM | 多步规划 + 跨 App 组合 + 长上下文 | 端侧降级路径：分句后依次处理 |
| S3 | **隐私敏感**：用户说"帮我查一下日历里的私人会议" | 端侧优先；无端侧 → 云端脱敏后处理 | 日历内容不宜明文上云 | 隐私数据清单由隐私团队维护 |
| S4 | **弱网场景**：地库 / 隧道 / 信号差 | 强制端侧 + 缓存 | 网络不可用 | 用户感知：功能降级但可用；恢复网络自动回切云端 |
| S5 | **成本预算触顶**：用户今日已调用云端 200 次 | 端侧兜底 + 提示 | 成本控制 | 预算耗尽后非关键指令走端侧；关键指令（安全 / 紧急）仍可用云端 |
| S6 | **边缘场景**：简单但端侧模型置信度低 | 先端侧快速返回初版 → 云端后续校正 | 速度与质量兼顾 | 适用于"先给个回答再完善"场景 |
| S7 | **缓存命中**：重复 / 相似 query（"导航去公司"每天都说） | 直接查缓存 | 缓存命中即返回 | 缓存命中率 ≥ 30%（核心指标） |
| S8 | **路由失败降级**：云端接口超时 500ms | 降级到端侧兜底 | 超时保底 | 降级对用户透明；日志记录失败 |
| S9 | **A/B 实验**：部分用户用新路由模型 | 灰度路由 | 实验验证新策略 | 灰度比例可控；异常自动回退 |
| S10 | **模型版本差异**：端侧刚 OTA 新版 vs 云端稳定版 | 灰度 A/B 路由 | 版本共存期 | 用户看不到"版本"细节，只看到效果 |

**8.1.5 失败路径**

|  |  |  |  |
| --- | --- | --- | --- |
| 失败场景 | 原因 | 用户感知 | 补救措施 |
| 云端接口超时 | 网络波动 / 云端压力 | 响应慢 / 无响应 | 端侧兜底 + 降级提示 |
| 端侧模型推理失败 | 模型崩溃 / OOM | 端侧无响应 | 自动重启 + 云端兜底 |
| 路由判错（该云端走端侧） | 决策模型 bug | 质量下降 | 用户反馈回流 → 模型重训 |
| 缓存过期 | 数据老化 | 返回旧信息 | 缓存 TTL + 主动刷新机制 |
| 成本预算耗尽但需求未完成 | 预算过低 | 非关键需求被拒 | 透明降级；用户可付费升级预算（V2） |
| 隐私数据被意外上云 | 规则漏洞 | 合规风险 | 隐私清单硬编码 + CI 自动检查 |
| 版本不一致导致结果不同 | 端云版本差 | 相同 query 结果不同 | 版本标注 + 一致性监控 |
| A/B 实验失败 | 新策略引入问题 | 部分用户体验差 | 自动回退机制 |

**8.1.6 技术钩子（给沈砚）**

1. **路由决策引擎架构**：规则引擎 + ML 模型混合？决策逻辑在哪一层（端侧 / 边缘 / 云端）？
2. **决策因子数据获取**：网络状态 / 算力占用 / 隐私标记 / 用户偏好。这些实时获取的延迟？
3. **缓存机制**：语义缓存（相似 query 命中）vs 精确缓存。实现技术？命中率优化？
4. **成本监控**：单用户 / 单请求级别的成本追踪。实时监控 + 预算控制？
5. **端侧模型推理框架**：TensorFlow Lite / ONNX / 自研？多模型调度？
6. **端云版本一致性**：端侧 vs 云端模型版本差异管理。灰度策略？
7. **隐私清单维护**：敏感数据自动识别。规则 / 正则 / NLU？
8. **降级策略的透明度**：降级时 UI 是否提示？还是完全无感？
9. **A/B 测试框架**：路由策略的灰度和分组。实验平台？
10. **监控告警**：路由错误率 / 成本 / 延迟的全链路观测。工具链？

**8.2 M-F2 端侧模型推理**

**8.2.1 用户价值**

**做什么**：车机上跑一组**轻量级 AI 模型**（KWS / ASR / 意图分类 / DMS / OMS），让**60% 以上的常见指令不用联网就能响应**——"打开空调"、"调音量"、"下一首"、"导航回家"这些高频场景延迟 < 500ms，弱网 / 无网时仍可用。端侧模型是"离线兜底"，不是"全能替代云端"。

**不做什么**：

* 不做"端侧 7B+ 大模型"（V1 不跟极氪激进路线，见 §9.1）
* 不做"端侧处理所有场景"（复杂任务质量打折 → 用户体验差）
* 不做"端侧模型无限扩张"（车机算力 / 存储有限）
* 不做"端侧模型不更新"（模型老化 → 准确率下降）

**核心价值主张**：

* **高频场景端侧化**——车控 / 空调 / 音乐 / 导航回家等垂域指令端侧全覆盖
* **延迟硬约束**——端侧推理 < 500ms（含 ASR + 意图 + 执行）
* **弱网兜底**——无网时基本功能可用

**8.2.2 场景脚本**

|  |  |  |  |

| # | 场景 | 端侧模型链路 | 关键约束 |

**8.3 M-F3 云端模型调用**

**8.3.1 用户价值**

**做什么**：复杂任务（多步规划 / 跨域指令 / 长上下文 / 情感化 TTS / VLM 视觉指代）走**云端大模型**，让座舱能处理"订个有儿童餐 8 人位的餐厅 + 导航过去 + 顺路买杯咖啡"这种端侧搞不定的复杂指令。云端调用是**端侧的"大脑"补充**——速度比端侧慢，但能解决端侧不能解决的问题。

**不做什么**：

* 不做"全云端"（延迟高 / 弱网失灵 / 隐私敏感）
* 不做"任意 LLM 切换"（不同 LLM 行为不一致 → 用户感觉 AI 性格变了）
* 不做"无成本约束的调用"（百万用户 = 月千万级成本）
* 不做"调用失败后死磕重试"（重试 3 次失败必须降级）

**核心价值主张**：

* **流式响应**——LLM 流式输出，TTS 边收边说，不让用户等"长答案完整生成"
* **成本可控**——单用户日均预算 + 缓存 + 端侧分流
* **质量可监控**——LLM 输出的准确率 / 幻觉率 / 用户满意度全链路埋点

**8.4 M-F4 模型 OTA 升级**

**8.4.1 用户价值**

**做什么**：座舱 AI 模型不能一锤子买卖——上线后还要持续迭代（修 bug / 加场景 / 提准确率）。**OTA 升级机制**让车机模型可以**远程更新**，但**不能让用户感觉"车在背着自己变"**——升级前用户可见 changelog、灰度发布、可一键回滚、不频繁打扰。模型更新是产品的"持续供血"，但供血方式必须有边界。

**不做什么**：

* 不做"静默升级不告知用户"（违反《05.0.2 ❌4》原则）
* 不做"高频升级"（每周升级 → 用户疲劳）
* 不做"全量推送"（一次升级所有车 → 出问题影响全网）
* 不做"不可回滚"（升级后出问题用户无法回退）

**核心价值主张**：

* **灰度发布**：5% → 20% → 100%，每阶段观察 24-72 小时
* **可回滚**：单用户可一键回滚到上版本
* **changelog 透明**：升级前用户可见"改了什么"
* **不频繁**：每月最多 1 次主动升级（紧急修复除外）

**8.4.2 场景脚本**

|  |  |  |  |
| --- | --- | --- | --- |
| # | 场景 | OTA 行为 | 关键约束 |
| S1 | **正常升级**：每月 1 次新版 ASR 模型 | 推送通知 + changelog + 用户选择"立即升级 / 稍后 / 跳过本次" | 不强制；用户可推迟到下次 |
| S2 | **后台下载**：用户选"立即升级" | 后台下载 + 验证完整性 + 等待车辆驻车后切换 | 不阻塞用户使用 |
| S3 | **静默升级**（用户已授权"自动升级"） | 后台升级 + 升级后显示一条简短通知 | 用户首次必须明确授权"自动升级" |
| S4 | **灰度阶段**：5% 用户先升级 | 监控 24h 后扩大到 20%，再 48h 后 100% | 每阶段异常 → 立即停止扩大 + 评估 |
| S5 | **紧急修复**：发现严重 bug | 加速灰度（5% → 50% → 100% 在 12h 内） | 紧急修复仍需用户告知，但流程加速 |
| S6 | **用户回滚**：升级后觉得不好用 | 设置 - AI 模型 - 回滚到上版本 | 一键回滚；保留至少 2 个旧版本 |
| S7 | **升级失败**：下载失败 / 验证失败 / 切换失败 | 自动重试 → 失败后保留旧版 + 提示用户 | 不让升级失败影响正常使用 |
| S8 | **升级中断电**：升级过程中车辆断电 | 启动后检测升级状态 → 回退到旧版 + 重新尝试 | 升级过程必须可中断恢复 |
| S9 | **多模型版本共存**：端侧 ASR v1 + 云端 v2（灰度期） | 路由层标注版本；监控两版本一致性 | 灰度期间允许版本差异；监控差异指标 |
| S10 | **跨车型差异**：8155 芯片车型 vs 8295 车型 | 不同车型推送不同版本（适配） | 推送规则按车型 / 配置匹配 |
